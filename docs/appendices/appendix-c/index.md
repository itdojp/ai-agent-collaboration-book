# 付録C: スキル評価ルーブリック

この付録は、第15章「スキルマップと評価」で使う詳細ルーブリックである。

評価の目的は、社員を一律に序列化することではない。目的は、AIエージェント協働に必要な能力を見える化し、不足領域を学習パス、OJT、権限付与、ガバナンスへ接続することである。

## C.1 レベル定義

| レベル | 名称 | 状態 | 評価の中心 |
|---:|---|---|---|
| Lv1 | AI利用者 | 基本ルールを守り、要約、下書き、整理に使える | 安全な基本利用 |
| Lv2 | 指示設計者 | 目的、前提、制約、出力形式を指定できる | Request ContractとContext基礎 |
| Lv3 | 業務適用者 | 自分の業務にAIを組み込み、出力をレビューできる | 業務適用とレビュー |
| Lv4 | エージェント運用者 | ツール、権限、ログ、評価、承認を含めて運用できる | Control層の設計 |
| Lv5 | チーム変革者 | チーム単位でユースケース、KPI、実験文化を設計できる | チーム運用と改善 |
| Lv6 | 全社設計者 | ガバナンス、投資判断、AI台帳、リスク管理、人材制度を設計できる | 全社展開と統制 |

## C.2 評価証拠レベル

| 証拠レベル | 名称 | 内容 | 用途 |
|---:|---|---|---|
| EV0 | 証拠なし | 本人ができると主張しているだけ | 判定には使わない |
| EV1 | 自己診断 | チェック項目への自己回答 | 学習パス設計の入口 |
| EV2 | 知識テスト | 用語、ルール、判断問題 | Lv1〜Lv2の補助証拠 |
| EV3 | 模擬成果物 | 演習ケースで作成した成果物 | Lv2〜Lv4の主要証拠 |
| EV4 | 実務成果物 | 実案件で作成しレビューされた成果物 | Lv3以上の強い証拠 |
| EV5 | 運用証跡 | ログ、承認、改善、再評価まで含む証跡 | Lv4以上の強い証拠 |

## C.3 領域別ルーブリック

### C.3.1 AIエージェント基礎理解

| レベル | 判定基準 |
|---:|---|
| Lv1 | AI出力が誤る可能性、社内ルール、入力禁止情報を理解している |
| Lv2 | チャットAI、RAG、ツール、AIエージェントの違いを業務例で説明できる |
| Lv3 | 自業務において、AIに任せる作業と人間が担う判断を区別できる |
| Lv4 | エージェント、ツール、ログ、HITL、評価を含む運用構成を説明できる |
| Lv5 | チーム内の誤解を是正し、共通言語を作れる |
| Lv6 | 全社のAIリテラシー基準、用語、教育体系を設計できる |

### C.3.2 課題・ユースケース設計

| レベル | 判定基準 |
|---:|---|
| Lv1 | AIに頼めそうな作業を挙げられる |
| Lv2 | AIに向く作業と向かない作業の基本分類ができる |
| Lv3 | 自業務をAI適用判断シートでリスク、ROI、適用段階に分類できる |
| Lv4 | 部門内のAI適用候補をポートフォリオ化できる |
| Lv5 | チームのAI投資優先順位、実験順序、撤退条件を設計できる |
| Lv6 | 全社のAI投資領域、リスク許容度、展開順序を設計できる |

### C.3.3 指示・依頼設計

| レベル | 判定基準 |
|---:|---|
| Lv1 | 簡単な依頼を出せる |
| Lv2 | 目的、背景、出力形式、制約、不明点の扱いを指定できる |
| Lv3 | Request ContractとAcceptance Criteriaを作り、修正指示を出せる |
| Lv4 | 複数ステップの依頼をTask Briefや評価条件へ接続できる |
| Lv5 | チーム標準の依頼テンプレート、NG例、添削基準を整備できる |
| Lv6 | 全社共通テンプレートと職種別テンプレートの運用を設計できる |

### C.3.4 コンテキスト設計

| レベル | 判定基準 |
|---:|---|
| Lv1 | 関連資料をAIに渡す必要性を理解している |
| Lv2 | 必要情報、不要情報、情報鮮度、要確認事項を簡単に整理できる |
| Lv3 | Context Packを作成し、確定、推定、仮定、要確認を分けられる |
| Lv4 | RAG、メモリ、ツール結果、ログを含むコンテキスト管理を設計できる |
| Lv5 | チームのContext Pack標準、情報鮮度ルール、更新ルールを運用できる |
| Lv6 | 全社ナレッジ、RAG、機密区分、アクセス権を含む文脈設計方針を定義できる |

### C.3.5 タスク分解・委任設計

| レベル | 判定基準 |
|---:|---|
| Lv1 | AIに作業を頼める |
| Lv2 | 大きな依頼を複数の小作業へ分けられる |
| Lv3 | Task Breakdown、Checkpoint Plan、Stop / Escalation Rulesを作れる |
| Lv4 | 複数AI、複数人、複数ツールを含む委任フローを設計できる |
| Lv5 | チーム標準の委任粒度、チェックポイント、エスカレーション運用を作れる |
| Lv6 | 全社的にAI委任できる業務、できない業務、承認条件を定義できる |

### C.3.6 検証・評価・レビュー

| レベル | 判定基準 |
|---:|---|
| Lv1 | 明らかな誤りや不自然な出力を見つけられる |
| Lv2 | 目的適合性、事実、根拠、リスクの基本確認ができる |
| Lv3 | AI出力レビュー表とReview Issue Logで採用可否を判断できる |
| Lv4 | Evaluation Plan、評価ケース、回帰評価を設計できる |
| Lv5 | チームのレビュー基準、レビュー強度、品質KPIを運用できる |
| Lv6 | 全社のAI品質保証、監査、評価標準を設計できる |

### C.3.7 ツール・権限・HITL設計

| レベル | 判定基準 |
|---:|---|
| Lv1 | AIにツールを使わせる危険性を理解している |
| Lv2 | 参照、下書き、実行、削除、外部送信の違いを説明できる |
| Lv3 | 簡単なツール権限マトリクスを作れる |
| Lv4 | Tool Spec Card、HITL承認フロー、Permission Testを設計できる |

| Lv6 | 全社のツール接続、外部MCP、権限付与、監査の統制を設計できる |

### C.3.8 ログ・観測・継続改善

## C.4 認定別の合格基準

| 認定 | 対象 | 必須成果物 | 最低証拠レベル |
|---|---|---|---|
| AI Safe User | 全社員 | 知識テスト、入力禁止情報分類、簡易レビュー課題 | EV2 |
| AI Work Designer | 業務担当者 | AI適用判断シート、Request Contract、Context Pack、AI出力レビュー表 | EV3 |
| AI Agent Operator | エンジニア、業務システム担当 | ツール権限マトリクス、HITL承認フロー、Agent Run Log、Security Checklist | EV3〜EV4 |
| AI Team Lead | 管理職、リーダー | チームAI活用方針、レビュー基準、AI実験設計、Failure Review運用案 | EV3〜EV4 |
| AI Governance Owner | AI推進担当、経営層、管理部門 | AI Use Policy、System Inventory、Risk Register、Training Register、90日展開案 | EV3〜EV5 |

## C.5 評価者向け採点観点

- 必須項目が揃っているか。
- 目的、前提、制約、成果物が明確か。
- 不確実性、仮定、要確認が分離されているか。
- 人間が判断すべき論点をAIに渡していないか。
- 顧客影響、法務、セキュリティ、個人情報、本番影響を考慮しているか。
- 高リスク操作にHITLが入っているか。
- ログ、レビュー、改善へ接続しているか。
- 失敗を隠さず、学習へ変換しているか。
- 評価結果が学習パスへ接続しているか。

## C.6 不合格にすべき例

- 機密情報や顧客情報を無条件にAIへ渡す設計になっている。
- AI出力を顧客へ自動送信するのに、人間承認がない。
- 契約、SLA、補償、法務、採用、人事、医療、金融などの判断をAIだけで完結させる。
- ツール権限がwriteまたはadminなのに、Permission Testやログがない。
- Prompt Injectionや外部入力のリスクを考慮していない。
- Review Issue LogでCriticalな誤りを見落としている。
- 失敗条件、停止条件、やめる条件がない実験を本番化しようとしている。
- 評価成果物に個人情報や顧客情報を未マスクで含めている。

## C.7 評価者校正

評価者は、少なくとも次を実施する。

1. 模範解答を読む。
2. NG例を読む。
3. 同じサンプル成果物を複数評価者で採点する。
4. 判定差分を記録する。
5. ルーブリックと採点基準を修正する。
6. 本番評価へ適用する。

## C.8 更新方針

このルーブリックは固定ではない。

次のタイミングで更新する。

- 新しいAIツールを導入したとき。
- 社内AI利用ポリシーが変わったとき。
- インシデントまたは重大なニアミスがあったとき。
- 新しい職種・業務でAI利用を開始したとき。
- 評価者間の判定差分が大きかったとき。
- 四半期または半期の定期レビュー時。

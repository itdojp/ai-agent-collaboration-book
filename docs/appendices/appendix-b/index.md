# 付録B: チェックリスト集

## B.0 本書活用開始チェック

- [ ] 自分の読者タイプを決めたか
- [ ] 対象業務を1つ選んだか
- [ ] AI活用の目的を1文で書いたか
- [ ] 扱う情報の機密度を確認したか
- [ ] 顧客影響、本番環境影響、外部送信の有無を確認したか
- [ ] 最初に読む章を3〜5章に絞ったか
- [ ] 最初に作る成果物を1〜3個に絞ったか
- [ ] AI出力をレビューする人を決めたか
- [ ] 30日以内の小実験を1つ書いたか
- [ ] 成功条件、失敗条件、停止条件を仮で書いたか
- [ ] 社内AI利用ポリシー、情報分類、禁止事項を確認する予定を入れたか


## B.1 AI利用前チェック

- [ ] 利用目的が明確か
- [ ] AIに任せる作業と人間が判断することを分けたか
- [ ] 入力してよい情報か確認したか
- [ ] 顧客情報、個人情報、認証情報が含まれていないか
- [ ] 出力形式を指定したか
- [ ] 不明点の扱いを指定したか
- [ ] 人間確認が必要な条件を指定したか

## B.2 AI出力レビュー

- [ ] 目的に合っているか
- [ ] 固有名詞、日付、数値、仕様が正しいか
- [ ] 根拠があるか
- [ ] 推測と確定情報が分かれているか
- [ ] 重要論点が抜けていないか
- [ ] 法務、セキュリティ、顧客影響のリスクがないか
- [ ] そのまま外部送信しないか

## B.3 顧客送信前チェック

- [ ] 事実関係を担当者が確認したか
- [ ] 原因未確定事項を断定していないか
- [ ] 社内情報や内部ログを含めていないか
- [ ] 契約、補償、SLA判断を勝手にしていないか
- [ ] 顧客名、宛先、添付ファイルが正しいか
- [ ] 承認者が確認したか

## B.4 ツール権限チェック

- [ ] read-only から始められるか
- [ ] 下書きと送信を分けたか
- [ ] 書き込み、削除、外部送信、本番変更で承認が入るか
- [ ] APIキーは最小権限か
- [ ] 一時トークンや環境分離を使っているか
- [ ] ツール実行ログが残るか

## B.5 セキュリティチェック

- [ ] 外部文書内の指示を命令として扱わないか
- [ ] AI出力コード、SQL、shellを未検証で実行しないか
- [ ] 機密情報をログに残しすぎていないか
- [ ] 外部AIのデータ保持条件を確認したか
- [ ] MCP / プラグイン / 外部SaaS の審査をしたか
- [ ] インシデント時の停止手順があるか

## B.6 判断前チェック

- [ ] AIの情報と人間の判断を分けたか
- [ ] 判断基準を明示したか
- [ ] やめる選択肢を検討したか
- [ ] 判断を延期するコストを見たか
- [ ] 感情的に避けている論点がないか
- [ ] 判断後に検証する指標があるか

## B.7 実験前チェック

- [ ] 仮説が明確か
- [ ] 最小実験になっているか
- [ ] 成功条件と失敗条件があるか
- [ ] やめる条件があるか
- [ ] 失敗しても安全な範囲か
- [ ] 学習ログを残すか

## B.8 タスク委任前チェック

- [ ] 親となる Request Contract があるか
- [ ] 使用する Context Pack が明確か
- [ ] タスクが1成果物に対応しているか
- [ ] AIに任せる作業と、AIに任せない判断を分けたか
- [ ] 入力と出力が明確か
- [ ] 受け入れ条件があるか
- [ ] 依存関係が整理されているか
- [ ] 中間チェックポイントを置いたか
- [ ] 停止条件が具体的か
- [ ] エスカレーション先が具体的か
- [ ] 顧客影響、法務、セキュリティ、本番環境、金銭影響の扱いを明示したか
- [ ] 長い作業では Restart Packet を残す設計になっているか
- [ ] 完了後にレビュー対象となる中間成果物が残るか


## B.9 AI出力レビュー前チェック

- [ ] AI出力を、未レビューのまま成果物として扱っていないか
- [ ] 利用先を確認したか
- [ ] レビュー強度を決めたか
- [ ] Request Contract と受け入れ条件を確認したか
- [ ] Context Pack の前提、情報鮮度、機密区分を確認したか
- [ ] Task Brief の範囲から逸脱していないか
- [ ] 事実と推測が分かれているか
- [ ] 重要な固有名詞、日付、数値、仕様を確認したか
- [ ] 根拠が存在し、対象範囲に合っているか
- [ ] 論理の飛躍がないか
- [ ] 重要論点、例外、リスクが抜けていないか
- [ ] 顧客、法務、セキュリティ、本番環境への影響を確認したか
- [ ] AIが人間の判断領域に踏み込んでいないか
- [ ] 採用、修正後採用、差し戻し、不採用、エスカレーションを判定したか
- [ ] Review Issue Log を残したか
- [ ] 次回改善先を決めたか

## B.10 ツール権限・HITL設計チェック

- [ ] ツール単位ではなく、操作単位で権限を定義したか
- [ ] no-access / read-only / draft-only / proposal-only / approval-required / limited-write / write / admin禁止を使い分けたか
- [ ] AIにadmin権限、権限変更権限、広い本番権限を与えていないか
- [ ] 外部送信、更新、削除、本番操作、権限変更に承認ゲートがあるか
- [ ] 下書き、提案、実行を分離したか
- [ ] ドライランと差分プレビューを設計したか
- [ ] 承認者が判断できる承認パケットを定義したか
- [ ] 承認者、代替承認者、承認期限、未承認時動作を決めたか
- [ ] 承認者に、変更前、変更後、差分、根拠、リスク、ロールバック方法が提示されるか
- [ ] 承認内容と実行内容が一致することをログで確認できるか
- [ ] APIキー、トークン、認証情報をAIに見せない設計になっているか
- [ ] AI用のサービスアカウントは最小権限か
- [ ] スコープを顧客、プロジェクト、環境、期間、件数で絞ったか
- [ ] レート制限、重複実行防止、冪等性を検討したか
- [ ] 失敗時にAIが勝手に再試行または権限昇格しないか
- [ ] ロールバックできない操作をAI単独にしていないか
- [ ] 禁止操作リストを作成したか
- [ ] Permission Testを作成したか
- [ ] スコープ外データ、承認なし実行、外部文書内命令、禁止操作のテストを含めたか
- [ ] 実行結果を第8章のAgent Run Logへ接続できるか

## B.11 ログ・トレース・継続改善チェック

- [ ] 1回の業務実行をRunとして定義したか
- [ ] Run IDとCorrelation IDを定義したか
- [ ] Request Contract、Context Pack、Task Brief、AI出力、レビュー、承認、ツール実行をRunに紐づけたか
- [ ] Tool Execution Logに、操作、スコープ、権限、承認ID、実行結果、エラーが残るか
- [ ] Approval Logに、承認者、判断理由、承認パケット、承認内容と実行内容の差分が残るか
- [ ] Review Issue LogとEvaluation PlanがAgent Run Logに接続されているか
- [ ] ログに残す情報、残さない情報、ID参照にする情報を分けたか
- [ ] 認証情報、APIキー、トークン、パスワードをログに残さない設計になっているか
- [ ] 個人情報、顧客機密、本番データのマスキング方針を決めたか
- [ ] ログの保存先、保持期間、アクセス権を決めたか
- [ ] 失敗分類を定義したか
- [ ] 失敗ログをImprovement Backlogへ接続したか
- [ ] Review Issueを評価ケースへ追加する流れがあるか
- [ ] Run Reviewの頻度、参加者、確認観点を決めたか
- [ ] 品質、リスク、効率、コスト、学習メトリクスを定義したか
- [ ] ログ自体が新たな情報漏洩源になっていないか


## B.12 セキュリティ・プライバシー設計チェック

- [ ] 対象ユースケースのセキュリティレベルを決めたか
- [ ] 情報分類を行ったか
- [ ] AIに入力してはいけない情報を明示したか
- [ ] 顧客情報、個人情報、認証情報、本番データを最小化したか
- [ ] 外部文書は命令ではなくデータとして扱う設計か
- [ ] プロンプトインジェクションテストを作ったか
- [ ] AI出力コード、SQL、shell、設定変更を未検証で実行しない設計か
- [ ] 顧客送信、契約、SLA、補償、原因断定にHITLがあるか
- [ ] ツール権限は操作単位で最小化されているか
- [ ] AIにadmin権限、権限変更権限、広い本番権限を与えていないか
- [ ] RAG検索で、顧客、部門、プロジェクトを横断しないか
- [ ] ベクトルDBに古い文書、削除済み文書、権限外文書が残らない設計か
- [ ] 長期メモリに顧客機密、個人情報、外部文書内命令を保存しない設計か
- [ ] 外部ツール、MCP、プラグイン、SaaS連携を審査したか
- [ ] ログ、トレース、評価データに機密情報を残しすぎていないか
- [ ] マスキング、保持期間、アクセス権、削除条件を決めたか
- [ ] セキュリティインシデントの即時停止条件を決めたか
- [ ] 例外承認が必要な残リスクを記録したか


## B.13 ガバナンス・統制設計チェック

- [ ] AI利用ポリシーがある
- [ ] 許可、条件付き許可、要承認、禁止が例付きで定義されている
- [ ] AI System Inventoryがある
- [ ] ツール名ではなくユースケース単位で管理している
- [ ] Business Owner、System Owner、Data Owner、Risk Ownerが定義されている
- [ ] 情報分類、外部送信、顧客影響、ツール権限が記録されている
- [ ] G0〜G5などのリスク分類がある
- [ ] 高リスク用途の承認フローがある
- [ ] Use Case Approval、Tool Action Approval、Risk Acceptanceを分けている
- [ ] 例外承認が期限付きで管理されている
- [ ] ベンダー、外部AI、MCP、SaaSを審査している
- [ ] 教育履歴を管理している
- [ ] ログ、承認、レビュー、評価の証跡がある
- [ ] インシデント時の停止・通知・復旧手順がある
- [ ] 定期棚卸しと廃止手順がある
- [ ] コスト、品質、リスク、教育、実験のKPIがある

## B.14 AI時代の判断力チェック

- [ ] 判断対象を1文で書ける
- [ ] 判断の種類を、採用、修正、棄却、保留、実験化、拡大、縮小、停止、やめる、方向転換、リスク受容のいずれかに分類した
- [ ] 判断レベルJ0〜J5を決めた
- [ ] 判断者、助言者、承認者を分けた
- [ ] 判断期限を決めた
- [ ] AIから得た情報、AIの推奨、人間の判断を分けた
- [ ] 判断基準をAI分析の前に定義した
- [ ] Decision Criteria Matrixを作成した
- [ ] 選択肢を同じ基準で比較した
- [ ] やめる選択肢、縮小する選択肢、実験化する選択肢を検討した
- [ ] 判断を延期した場合のコストを確認した
- [ ] 許容できるリスクと許容できないリスクを分けた
- [ ] 残リスクを受容する場合、Risk Acceptance Memoに期限、Owner、再評価条件を記録した
- [ ] 感情的に避けている論点を確認した
- [ ] 言えていない懸念を確認した
- [ ] 判断後に見る指標を設定した
- [ ] 見直し日を設定した
- [ ] Decision Review Logで判断後レビューを行う予定がある
- [ ] 判断結果をImprovement Backlog、Evaluation Plan、Governance Reviewへ接続できる

## B.15 困難な会話・対立処理チェック

- [ ] AIを論破ではなく論点整理に使っている
- [ ] Connection Debt診断を作成した
- [ ] 言っていないことを、人格評価ではなく業務論点に変換した
- [ ] 放置している業務コスト、関係コスト、判断コストを書いた
- [ ] 対立の種類を、事実、優先順位、範囲、責任、品質、リスク、価値観、行動、関係、権限のいずれかに分類した
- [ ] 相手側の合理性をPerspective Mapで整理した
- [ ] 事実、解釈、感情、要求を分けた
- [ ] 会話で決めること、決めないことを分けた
- [ ] Difficult Conversation Briefを作成した
- [ ] 伝える内容が、観察、業務影響、要望になっている
- [ ] 相手に聞く質問を準備した
- [ ] Conversation Planで会話の順序を決めた
- [ ] 必要なら人事、法務、セキュリティ、コンプライアンスへエスカレーションする条件を確認した
- [ ] バックチャネルを増やさないルールを決めた
- [ ] 会話後にAgreement Logを残す予定がある
- [ ] Follow-up ContractでOwner、期限、確認方法を決める予定がある
- [ ] 合意をRequest Contract、Context Pack、Task Brief、Decision Brief、Review Issue Log、AI Risk Register、Improvement Backlogのいずれかへ反映する予定がある
- [ ] 対立を小実験へ変換できる場合、第13章のAI実験設計シートへ接続できる

## B.16 失敗耐性・実験文化チェック

- [ ] 実験の目的が、効率化、改善、学習、展開のどれか明確である
- [ ] 実験レベルX0〜X5を決めた
- [ ] 対象業務、対象者、対象データ、対象期間を限定した
- [ ] 仮説が検証可能な文になっている
- [ ] 最小実験になっている
- [ ] 成功条件がある
- [ ] 失敗条件がある
- [ ] 即時停止条件がある
- [ ] やめる条件がある
- [ ] 使ってよいデータ、使ってはいけないデータを分けた
- [ ] 顧客影響、本番影響、個人情報、契約、法務、セキュリティがある場合、X3以上として扱った
- [ ] 必要なRequest Contract、Context Pack、Task Briefを確認した
- [ ] ツールを使う場合、HITLと権限を確認した
- [ ] Run Log、Learning Log、Improvement Backlogの記録先を決めた
- [ ] 実験後の判断日を決めた
- [ ] 続ける、変える、やめる、展開するの判断基準を決めた
- [ ] Failure Review Sheetで失敗を設計改善へ変換する予定がある
- [ ] Learning Logで再利用可能な学習を残す予定がある
- [ ] 成功事例だけでなく、失敗事例と廃棄判断を共有する予定がある
- [ ] 実験が常用化する場合、AI System Inventoryへ登録する条件を決めた
- [ ] 実験が低リスク探索に留まる場合、Side Quest Charterで入口と出口を決めた
- [ ] Pace / Spin Reviewで、実験速度と空回りを確認する予定がある


## B.17 セルフトーク・高レバレッジ環境チェック

- [ ] 止まっている行動を、観察可能な1文で書いた
- [ ] 「AI活用が進まない」ではなく、具体的な未実行行動に分解した
- [ ] 頭の中のセルフトークを書き出した
- [ ] セルフトークが守ろうとしているものを特定した
- [ ] 実リスクと想像上のリスクを分けた
- [ ] 顧客影響、個人情報、契約、本番環境、法務、セキュリティなどの許容不可リスクを確認した
- [ ] リスク制御策を決めた
- [ ] 成果物の扱いを Draft / Review Only / Experiment / Production Candidate に分けた
- [ ] DraftをProduction Candidateの基準で止めていない
- [ ] Production CandidateをDraftの基準で緩く扱っていない
- [ ] 完璧主義をAcceptance Criteria、レビュー観点、HITL、停止条件へ変換した
- [ ] レビューしてほしい観点を指定した
- [ ] レビューしてほしくない観点を指定した
- [ ] レビュー指摘を成果物、プロセス、判断、リスク、自分の人格に分類した
- [ ] 自己批判として混ざっている言葉を切り離した
- [ ] AIへの不安を、AIが担う作業、人間が担う判断、対話、責任へ分解した
- [ ] 24時間以内または次の業務日までにできる最小行動を定義した
- [ ] Minimum Action Contractに完了条件と時間上限を書いた
- [ ] レビュー依頼先を決めた
- [ ] レビュー結果をReview Issue Log、Learning Log、Improvement Backlogのいずれかへ接続した
- [ ] 同じ行動ブロッカーが複数人に出ている場合、組織側の改善に接続した
- [ ] 管理職は、行動停止を人格問題ではなく、成果物、レビュー観点、権限、リスク、期限の問題として扱っている
- [ ] 必要な場合、人事、労務、法務、セキュリティ、メンタルヘルス等の専門対応へエスカレーションする条件を確認した

## B.18 スキルマップ・評価設計チェック

- [ ] 12能力領域が定義されているか。
- [ ] Lv1〜Lv6の定義が、職位ではなく能力として定義されているか。
- [ ] ロール別に到達目標を定義しているか。
- [ ] 扱う情報区分と業務リスクを考慮しているか。
- [ ] AI利用範囲が、参照、下書き、提案、承認後実行、自動実行に分かれているか。
- [ ] 評価が自己診断だけに依存していないか。
- [ ] 知識テスト、実技課題、成果物レビュー、運用証跡を使い分けているか。
- [ ] Lv4以上の判定に、実技課題または実務成果物を含めているか。
- [ ] 高リスク業務の評価に、承認、ログ、セキュリティ、ガバナンスの観点が含まれているか。
- [ ] Role Skill Profileが作成されているか。
- [ ] Skill Diagnosis Formがあるか。
- [ ] Practical Assessment Packがあるか。
- [ ] Evidence Portfolioがあるか。
- [ ] Learning Gap Handoffがあるか。
- [ ] Certification Decision Recordがあるか。
- [ ] 評価者校正の仕組みがあるか。
- [ ] 模範解答、NG例、採点基準があるか。
- [ ] 評価に使うログや成果物のマスキング方針があるか。
- [ ] 評価結果の利用目的が明示されているか。
- [ ] 評価結果が研修、OJT、Office Hour、権限付与、再評価へ接続しているか。
- [ ] 評価制度そのものの更新周期が決まっているか。

## B.19 組織展開・教材化チェック

- [ ] Rollout Charterが作成されている。
- [ ] 展開目的が、AI利用拡大だけでなく、業務成果、品質、安全性、学習に接続している。
- [ ] Executive Sponsorが明確である。
- [ ] AI CoEまたは暫定推進チームがある。
- [ ] Department Owner、Manager、AI Championの役割が定義されている。
- [ ] Security、Legal、HR、Platform / ITの関与ポイントが定義されている。
- [ ] RACIが定義されている。
- [ ] 全社員、業務担当者、管理職、エンジニア、経営層で学習パスが分かれている。
- [ ] AI利用ポリシー1枚紙がある。
- [ ] 入力禁止情報チェックリストがある。
- [ ] Request Contract、Context Pack、AI出力レビュー表がMVP教材に含まれている。
- [ ] セキュリティ基礎と情報分類が全社員向け教材に含まれている。
- [ ] 管理職向けに、レビュー、承認、判断、対立処理、実験文化の教材がある。
- [ ] エンジニア / AI推進担当向けに、ツール権限、HITL、ログ、評価、セキュリティの教材がある。
- [ ] 2時間ワークショップの設計がある。
- [ ] ワークショップの提出物が定義されている。

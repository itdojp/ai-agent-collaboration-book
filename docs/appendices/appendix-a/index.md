
### 最小版

軽量な作業では、次の7項目から始める。

```text

成果物：
対象範囲：
入力情報：
制約：
出力形式：

```

### 受け入れ条件テンプレート

```text
受け入れ条件：
- [ ] 目的に合っている
- [ ] 対象範囲が守られている
- [ ] 除外範囲が混ざっていない
- [ ] 指定した出力形式になっている
- [ ] 確定情報、推定、仮定、要確認が分かれている
- [ ] 禁止された判断や操作をしていない
- [ ] 人間確認が必要な事項が分離されている
- [ ] 残リスクが明示されている
```

### 記入例: 競合調査

```text
目的：法人向けバックアップサービスの価格改定検討のため、主要競合5社の比較表を作る。
最終成果物：経営会議前の論点整理に使うMarkdown表。
対象範囲：国内法人向けクラウドバックアップサービス。
除外範囲：個人向け、海外専業、単純なオンラインストレージ。
制約条件：不明な情報は推測せず「要確認」と書く。価格改定の結論は出さない。
出力形式：会社名、サービス名、価格体系、主要機能、対象顧客、強み、弱み、要確認事項、出典の表。
人間承認が必要な条件：価格判断、競合評価、顧客向け説明に使う場合。
停止条件：公開情報または許可された社内情報だけでは比較できない場合。
```

## A.4 Context Pack

### 用途

AIに渡す前提情報を構造化し、古い情報、未確認情報、機密情報の混入を防ぐ。第4章で作成する成果物である。

```text
案件名：
業務背景：
関係者：
現在の状況：
過去の経緯：
決定済み事項：
未決事項：
関連資料：
重要な制約：
優先順位：
情報の鮮度：
機密区分：公開 / 社内限定 / 機密 / 個人情報 / 認証情報
AIに参照させてよい範囲：
AIに参照させてはいけない範囲：
確定情報：
推定情報：
仮定：
要確認事項：
更新履歴：
```

### 情報鮮度ラベル

| ラベル | 意味 | 使い方 |
|---|---|---|
| 最新 | 現時点で有効 | 現行規程、最新仕様、直近ログ |
| 参考 | 背景理解には使えるが判断根拠にはしない | 過去調査、旧議事録 |
| 旧版 | 現在は置き換え済み | 旧価格表、旧SLA |
| 未確認 | 事実確認が必要 | Slackの発言、顧客推測、伝聞 |
| 禁止 | AIに参照させない | 認証情報、他顧客情報、未公開財務 |

### 確度分類

| 区分 | 意味 | AIへの扱わせ方 |
|---|---|---|
| 確定情報 | 確認済みの事実 | 事実として使ってよい |
| 推定情報 | 可能性はあるが未確定 | 断定禁止。推定と明記 |
| 仮定 | 作業上の仮置き | 仮定欄に分離 |
| 要確認事項 | 次に確認すべきこと | 本文と分けて出力 |

### 更新履歴テンプレート

```text
更新日時：
更新者：
追加された確定情報：
変更された前提：
破棄された推定：
新しい未決事項：
参照禁止になった情報：
次回レビュー予定：
```

## A.5 Task Brief / Task Breakdown / Restart Packet

### 用途

複雑な業務をAIに委任可能な単位へ分解し、各タスクの入力、出力、受け入れ条件、チェックポイント、停止条件、エスカレーション条件を明確にする。第5章で作成する成果物である。

### A.5.1 Task Brief

AIに渡す個別タスクの仕様書。Request Contract より小さく、Context Pack より作業寄りの文書として使う。

```text
タスク名：
目的：
親Request Contract：
使用するContext Pack：
AIに任せる作業：
AIに任せない判断：
入力：
出力：
受け入れ条件：
依存関係：
チェックポイント：
停止条件：
エスカレーション条件：
再開メモ：
ログに残す情報：
```

### 最小版

軽量な作業では、次の項目から始める。

```text
タスク名：
目的：
AIに任せる作業：
AIに任せない判断：
入力：
出力：
停止条件：
```

顧客影響、法務、セキュリティ、本番環境、金銭影響がある場合は最小版では不足する。フル版を使う。

### A.5.2 Task Breakdown

複数タスクを扱う場合に、全体構造を整理する。AIに一括で丸投げするためではなく、人間が委任範囲を設計するために使う。

```text

親Request Contract：
使用するContext Pack：

| ID | タスク | 目的 | 入力 | 出力 | AIの役割 | 人間確認 | 依存 | 停止条件 |
|---|---|---|---|---|---|---|---|---|
```

### A.5.3 Checkpoint Plan

AIが次の段階へ進む前に、人間が確認する地点を定義する。

```text
業務名：
対象タスク：

| ID | チェックポイント | 何を確認するか | 確認者 | 次へ進む条件 | 差し戻し条件 |
|---|---|---|---|---|---|
```

### A.5.4 Stop / Escalation Rules

AIが止まる条件と、誰に渡すかを定義する。リスクの高い業務では、Task Brief内に埋め込まず独立表として管理する。

```text
業務名：
対象範囲：

| 条件 | AIの動作 | エスカレーション先 | 渡す情報 | 再開条件 |
|---|---|---|---|---|
```

長い作業、中断する作業、別セッションで再開する作業のための引き継ぎメモ。Context Pack が業務前提をまとめるのに対して、Restart Packet は進行状況をまとめる。


### A.5.6 記入例: 障害タイムライン候補作成

```text
タスク名：障害タイムライン候補作成
目的：顧客向け報告書の前提となる時系列を整理する。
親Request Contract：障害報告書作成 Request Contract
使用するContext Pack：障害対応 Context Pack v2026-06-27

AIに任せる作業：
- 監視通知、チケット、作業メモから時刻付きイベントを抽出する
- イベントを時系列に並べる
- 発生、検知、対応開始、暫定復旧、完全復旧の候補を分ける
- 矛盾する時刻、未確認の時刻を要確認として分離する

AIに任せない判断：
- 正式な障害発生時刻の確定
- 正式な復旧時刻の確定
- 顧客向け説明への採用判断

入力：
- 監視通知一覧
- インシデントチケット
- 作業メモ
- Slack要約

出力：
- Markdown表
- 列は「時刻」「イベント」「情報源」「確度」「要確認事項」

受け入れ条件：
- 情報源が明記されている
- 確定、推定、要確認が分かれている
- 正式時刻を断定していない

チェックポイント：
- タイムライン候補を作成した時点で停止し、技術担当が確認する

停止条件：
- 入力資料間で時刻が矛盾する場合
- 認証情報、顧客個別情報、未許可ログが必要になる場合

エスカレーション条件：
- 監視ログと作業メモが矛盾し、正式な発生時刻判断が必要な場合は技術責任者へ渡す

再開メモ：
- 技術担当が正式時刻を確認した後、影響範囲候補整理へ進む
```

## A.7 ツール権限マトリクス / HITL承認フロー

### 用途

AIエージェントが外部ツールを使う範囲を、操作単位、スコープ単位、承認単位で定義する。第7章で作成する成果物である。

ツール権限マトリクスは「AIにどのツールを接続するか」を決める表ではない。AIが、どのツールで、どの操作を、どの対象に、どの条件で、誰の承認を受けて実行できるかを定義する表である。

### A.7.1 ツール権限マトリクス

```text
対象業務：
対象AIエージェント：
作成日：
```

# 第8章 ログ・トレース・継続改善

## この章で扱う能力

この章では、AIエージェントの実行を観測し、失敗、成功、承認、コスト、品質を改善へ接続する方法を扱う。読者は、AIエージェントの出力だけでなく、どの依頼を受け、どの前提を使い、どのツールを呼び、どこで人間が承認し、どこで失敗し、次に何を改善すべきかを記録できるようになる。

第7章では、AIにツールを使わせる前に、権限、スコープ、HITL、承認パケット、禁止操作、Permission Testを設計した。第8章では、その実行結果を観測可能にする。権限設計は「危険な操作を止める」ための仕組みである。ログとトレースは「何が起きたかを説明し、次に改善する」ための仕組みである。

この章で扱う中心能力は、次の7つである。

```text
1. AIエージェント実行をRun単位で記録する
2. ツール呼び出し、参照情報、承認、レビュー結果を関連づける
3. ログに残す情報と残さない情報を分ける
4. 失敗を分類し、原因を前工程へ戻す
5. 評価結果、レビュー結果、運用結果を改善バックログへ接続する
6. 品質、コスト、速度、リスクを継続的に測定する
7. 監査・インシデント対応・教材改善に使える証跡を残す
```

この章の成果物は、主に次である。

```text
Agent Run Log
Trace Event Record
Tool Execution Log
Approval Log
Run Review Report
Improvement Backlog
Observability Design Sheet
Metrics Dashboard Definition
```

```text

ログは監視のためだけにあるのではない。
再現、説明、監査、評価、改善、教育のためにある。

出力だけを残すのではなく、
依頼、前提、参照情報、ツール実行、承認、レビュー、失敗、改善アクションを
ひとつのRunとしてつなげて残す。
```

## なぜログとトレースが必要か

AIエージェントは、単なる文章生成ツールではない。依頼を受け、前提を読み、情報を検索し、ツールを呼び、判断候補を作り、人間承認を経て業務システムに影響を与える。

このような実行系では、最終出力だけを見ても十分ではない。

例えば、顧客返信案が誤っていたとする。その誤りの原因は複数あり得る。

```text
依頼が曖昧だった
Context Packの情報が古かった
参照したFAQが対象外だった
AIが過去チケットを誤って要約した
ツールが古いデータを返した
承認者が未確定事項を見落とした
レビュー基準にSLA確認が含まれていなかった
外部文書内の命令をAIが指示として扱った
```

出力だけでは、どれが原因だったか分からない。

ログがなければ、次の問題が起きる。

| 問題 | 内容 |
|---|---|
| 再現できない | どの入力、前提、ツール結果でその出力になったか分からない |
| 改善できない | プロンプト、Context Pack、ツール、レビュー基準のどこを直すべきか分からない |
| 監査できない | 顧客送信、本番変更、CRM更新などの責任境界が追えない |
| 評価できない | AI活用が品質を上げたのか、事故を増やしたのか測れない |
| 教材化できない | 成功例、失敗例、レビュー観点を組織学習へ変換できない |
| コスト管理できない | どのユースケースでコストが増えているか分からない |
| セキュリティ対応できない | 情報漏洩、外部入力、権限誤用の経路が追えない |

AIエージェント協働におけるログは、単なるシステムログではない。業務成果物である。

## 用語の整理

この章では、次の用語を使い分ける。

| 用語 | 意味 |
|---|---|
| Run | 1つの依頼から、出力、レビュー、承認、実行、改善記録までの一連の実行単位 |
| Agent Run Log | Run単位で残す業務ログ。依頼、Context Pack、ツール、レビュー、承認、改善を含む |
| Trace | AIエージェント内部または実行基盤上の処理の流れ。モデル呼び出し、ツール呼び出し、分岐、エラーなど |
| Event | Trace内の個別イベント。検索、ツール実行、承認待ち、例外、出力生成など |
| Correlation ID | 複数ログ、ツール、承認、チケットを同じRunに紐づけるID |
| Artifact | AIが作った成果物、レビュー表、差分、承認パケット、チケット、PRなど |
| Approval Log | 人間承認の内容、承認者、時刻、理由、条件を残すログ |
| Review Log | AI出力レビューの判定、Issue、Severity、修正指示を残すログ |
| Evaluation Log | 評価ケース、評価結果、回帰テスト結果、品質指標を残すログ |
| Improvement Backlog | 失敗・レビュー・評価から導いた改善項目の一覧 |
| Redaction | ログから機密情報や個人情報を除去またはマスクすること |
| Retention | ログの保持期間 |

ここで重要なのは、**Run**を単位にすることである。

チャット履歴だけを眺めても、業務上の証跡にはならない。逆に、システムログだけを見ても、業務上の目的や判断理由は分からない。AIエージェント協働では、業務のRunとシステムのTraceを関連づける必要がある。

```text
Run ID
  ├ Request Contract
  ├ Context Pack
  ├ Task Brief
  ├ AI出力
  ├ Review Issue Log
  ├ Tool Execution Log
  ├ Approval Log
  ├ 実行結果
  └ Improvement Backlog
```

## 観測性の基本原則

ログ設計では、次の原則を守る。

### 原則1: 出力だけでなく、前提と経路を残す

AIの最終出力だけを保存しても、改善には使いにくい。

最低限、次を残す。

```text
どの依頼だったか
どのContext Packを使ったか
どのTask Briefに基づいたか
どの情報を参照したか
どのツールを呼んだか
どのレビューで何が指摘されたか
誰が何を承認したか
結果として何が実行されたか
次に何を改善するか
```

### 原則2: ログは多ければよいわけではない

すべてを保存すれば安全になるわけではない。ログには機密情報、個人情報、認証情報、顧客情報、内部設定、脆弱性情報が含まれ得る。

したがって、ログ設計では次を決める。

```text
何を保存するか
何を保存しないか
何をID参照にするか
何をマスクするか
誰が見られるか
どれだけ保持するか
どの条件で削除するか
```

### 原則3: ログは改善先へ接続する

ログを残しても、誰も見なければ意味がない。ログは改善アクションへ接続する。

改善先は主に次である。

| 改善先 | 例 |
|---|---|
| Request Contract | 依頼に禁止事項、出力形式、不明点の扱いを追加する |
| Context Pack | 古い資料、未確定情報、機密区分を修正する |
| Task Brief | サブタスク粒度、停止条件、エスカレーション条件を修正する |
| Review Criteria | レビュー観点、Severity、承認条件を追加する |
| Tool Spec | 引数、戻り値、エラー、スコープを修正する |
| Permission | 権限を下げる、承認を追加する、スコープを狭める |
| Evaluation Plan | 失敗例を評価ケースに追加する |
| Training Content | 失敗事例、演習、チェックリストへ反映する |
| Governance | AI台帳、ポリシー、承認ルールへ反映する |

### 原則4: 監査ログと改善ログを分ける

監査ログは、後から事実確認するための証跡である。改善ログは、次に何を直すかを管理するための作業管理である。

この2つを混同しない。

| 種類 | 目的 | 主な内容 | 更新方法 |
|---|---|---|---|
| 監査ログ | 何が起きたかを証明する | 入力、出力、ツール、承認、時刻、実行者 | 原則として改変しない |
| 改善ログ | 次に何を直すかを管理する | 原因分類、改善案、担当者、期限、完了条件 | 継続的に更新する |

監査ログを後から都合よく編集してはいけない。一方、改善ログはIssueやBacklogとして状態を更新する。

### 原則5: 人間の判断も記録する

AIの出力だけではなく、人間が何を採用し、何を捨て、どのリスクを許容したかも記録する。

特に次は残す。

```text
AI出力をそのまま採用したのか
修正して採用したのか
差し戻したのか
不採用にしたのか
エスカレーションしたのか
どのリスクを許容したのか
どの判断を保留したのか
誰が承認したのか
```

AIエージェント協働では、最終責任は人間と組織に残る。したがって、人間の判断ログが必要である。

## 観測性レベル

AIエージェントの観測性は、段階的に高める。

| レベル | 状態 | 内容 | 問題 |
|---:|---|---|---|
| L0 | ログなし | 出力だけが残る | 再現、改善、監査ができない |
| L1 | チャット履歴のみ | 会話の断片が残る | 業務証跡として不十分 |
| L2 | 構造化Run Log | 依頼、前提、出力、レビュー、承認がRun単位で残る | ツール詳細や評価には不足 |
| L3 | Trace / Tool / Approval連携 | ツール呼び出し、エラー、承認パケット、実行結果が紐づく | 分析・評価の運用が必要 |
| L4 | Evaluation / Metrics連携 | 評価ケース、回帰評価、品質指標、コスト指標が見える | 改善バックログがないと形骸化する |
| L5 | 継続改善運用 | 失敗がIssue化され、教材、ポリシー、権限、評価に戻る | 運用責任者とレビュー会が必要 |

初期導入では、L2を最低ラインにする。顧客影響、外部送信、CRM更新、GitHub PR、本番影響がある場合は、L3以上を求める。社内AI基盤や複数部門で使うエージェントは、L4以上を目指す。

悪い状態は次である。

```text
AIが何をしたかはチャットを見れば分かるはずです。
```

これは運用ではない。チャット履歴は、業務Run、承認、ツール実行、評価、改善に接続されていない限り、証跡として弱い。

良い状態は次である。

```text
Run IDで、依頼、Context Pack、Task Brief、参照情報、ツール実行、レビュー、承認、最終出力、改善Issueを追える。
```

## Run IDとCorrelation ID

AIエージェント協働では、すべての関連ログに同じ識別子を持たせる。

```text
Run ID：AIエージェントの一回の業務実行を表すID
Correlation ID：外部ツール、承認、チケット、PR、メール下書きなどを同一Runに紐づけるID
```

例えば、顧客返信案作成のRunでは、次が同じRunに紐づく。

```text
Run ID: CS-2026-0627-00012
Request Contract ID: RC-CS-2026-0627-00012
Context Pack ID: CP-CUST-ACME-2026-0627
Task Brief ID: TB-REPLY-DRAFT-00012
FAQ検索ログID: FAQ-QUERY-9821
過去チケット検索ログID: TICKET-SEARCH-7712
メール下書きID: DRAFT-44201
Review Issue Log ID: RIL-00012
Approval Log ID: APPROVAL-22091
Improvement Backlog ID: IMP-00318
```

IDがなければ、後から調べるときに人間が記憶と検索に頼ることになる。これは属人的である。

## Agent Run Logの設計

Agent Run Logは、AIエージェント実行の業務単位ログである。システム内部の詳細ログではなく、業務責任者、管理職、AI推進担当、監査担当が読める形式にする。

最低限の項目は次である。

| 項目 | 内容 |
|---|---|
| Run ID | 一意な実行ID |
| 日時 | 開始、終了、主要イベント時刻 |
| 依頼者 | 誰が依頼したか |
| 利用者 / 承認者 | 誰が利用し、誰が承認したか |
| 目的 | 何のための実行か |
| 対象業務 | 顧客返信、競合調査、PR作成など |
| 利用したRequest Contract | 依頼仕様のIDまたはリンク |
| 利用したContext Pack | 前提情報のIDまたはリンク |
| 利用したTask Brief | サブタスク定義のIDまたはリンク |
| 参照情報 | 参照した社内文書、FAQ、チケット、URL、検索クエリなど |
| 使用ツール | ツール名、操作、対象スコープ |
| ツール実行結果 | 成功、失敗、エラー、返却概要 |
| AI出力 | 出力の要約、成果物リンク |
| レビュー結果 | 採用、修正後採用、差し戻し、不採用、エスカレーション |
| 承認結果 | 承認者、承認時刻、判断理由、条件 |
| 実行結果 | 外部送信、更新、PR作成、チケット作成など |
| 残リスク | 未解決のリスク、要確認事項 |
| 失敗・例外 | エラー、誤出力、承認拒否、権限不足など |
| 改善アクション | 改善バックログへのリンク |
| 機密区分 | ログ自体の機密レベル |
| 保持期間 | いつまで保存するか |

Agent Run Logは、1回の実行ごとに必ず詳細長文を書く必要はない。低リスク業務では簡易版でよい。ただし、顧客影響、外部送信、更新、本番影響、法務・金銭影響がある場合は詳細版を使う。

## Trace Event Recordの設計

Agent Run Logが業務視点のログだとすると、Trace Event Recordは実行過程のイベントログである。

Traceは、次のようなイベント列として扱う。

```text
Run開始
Request Contract読込
Context Pack読込
FAQ検索
過去チケット検索
AI出力生成
レビュー待ち
レビュー差し戻し
再生成
承認待ち
承認
メール下書き作成
Run終了
```

Trace Event Recordの項目は次である。

| 項目 | 内容 |
|---|---|
| Run ID | 紐づくRun |
| Event ID | 個別イベントID |
| Parent Event ID | 親イベントID。ツール呼び出しや分岐を階層化する |
| Event Type | model_call / tool_call / retrieval / approval / review / error / output / stop など |
| Actor | AI、ユーザー、承認者、ツール、システム |
| Timestamp | 開始・終了時刻 |
| Input Summary | 入力の要約。機密値は保存しない |
| Output Summary | 出力の要約。機密値は保存しない |
| Tool Name | ツール名 |
| Tool Action | 検索、作成、更新、送信、削除など |
| Status | success / failure / blocked / skipped / approved / rejected |
| Error Code | エラーがある場合 |
| Policy Result | 権限、HITL、禁止操作チェックの結果 |
| Related Artifact | 生成された成果物、承認パケット、レビュー表へのリンク |

Traceは技術者だけのものではない。業務側が読むAgent Run Logと、技術側が読むTraceをRun IDでつなぐことで、事故時に「業務上何が起きたか」と「システム上何が起きたか」を接続できる。

## Tool Execution Logの設計

AIにツールを使わせる場合、ツール実行ログは必須である。

特に記録すべき項目は次である。

| 項目 | 内容 |
|---|---|
| Run ID | 紐づくRun |
| Tool Execution ID | ツール実行ID |
| Tool Name | ツール名 |
| Action | 実行した操作 |
| Scope | 対象顧客、対象プロジェクト、対象環境など |
| Permission Level | read-only、draft-only、approval-requiredなど |
| Approval Required | 承認要否 |
| Approval ID | 承認ログへのリンク |
| Input Summary | 入力要約。機密値は保存しない |
| Output Summary | 返却結果要約 |
| Diff | 更新、作成、削除、送信の差分 |
| Status | success / failure / blocked |
| Error | エラー内容 |
| Retry Count | 再試行回数 |
| Rate Limit | レート制限の適用有無 |
| Rollback Reference | ロールバック方法または変更前値への参照 |

ツール実行ログでは、特に次を確認できるようにする。

```text
承認された内容と実行された内容が一致しているか
AIがスコープ外データへアクセスしていないか
禁止操作がブロックされたか
承認前に更新・送信・削除していないか
エラー時に勝手に再試行し続けていないか
```

## Approval Logの設計

HITLを入れても、承認ログが残らなければ意味がない。

Approval Logでは、次を残す。

| 項目 | 内容 |
|---|---|
| Approval ID | 承認ID |
| Run ID | 紐づくRun |
| 承認対象 | 外部送信、CRM更新、PR作成、本番操作など |
| 承認パケット | 承認者に提示した情報へのリンク |
| 承認者 | 誰が承認したか |
| 承認権限 | なぜその人が承認できるのか |
| 判断 | 承認 / 修正して承認 / 差し戻し / 拒否 / エスカレーション |
| 判断理由 | 承認・拒否の理由 |
| 条件付き承認 | 条件がある場合 |
| 承認時刻 | いつ承認したか |
| 実行時刻 | 承認後いつ実行されたか |
| 実行内容との差分 | 承認内容と実行内容の差分 |
| 残リスク | 承認時点で残したリスク |

承認ログの目的は、承認者を責めることではない。承認者が何を見て、どのリスクを理解し、何を条件に承認したかを説明できるようにすることである。

## Review / Evaluation Logの設計

第6章で扱ったAI出力レビューと評価も、ログに接続する。

Review Logでは、次を残す。

```text
レビュー対象
レビュー強度
レビュー観点
発見したIssue
Severity
判定
修正指示
改善先
```

Evaluation Logでは、次を残す。

```text
評価対象のエージェントまたはワークフロー
評価ケース
期待結果
実結果
スコア
失敗理由
前回との差分
改善アクション
```

レビューは単発の品質確認である。評価は仕組みとしての品質確認である。

例えば、ある顧客返信案で「原因未確定なのに断定した」というIssueが見つかった場合、その場で差し戻すだけでは不十分である。次回以降に同じ失敗を検出する評価ケースへ追加する。

```text
評価ケース：原因未確定の障害問い合わせ
期待結果：原因を断定しない。未確定事項として表現する。必要に応じて技術担当へエスカレーションする。
禁止出力：「原因は〇〇です」「再発しません」などの断定表現。
```

これにより、レビューで見つかった失敗が、仕組みの改善へ変わる。

## 何をログに残さないか

ログ設計で最も危険なのは、「後で困るから全部保存する」という考え方である。

ログに残すべきでないもの、または慎重に扱うべきものは次である。

| 種類 | 扱い |
|---|---|
| 認証情報 | 保存しない。マスクではなく原則記録対象外にする |
| APIキー、トークン | 保存しない。必要ならキーIDや権限名だけを残す |
| パスワード、秘密鍵 | 保存しない |
| 個人情報 | 最小化、マスク、ID参照、保持期間制限 |
| 顧客機密 | 原文保存を避け、文書ID、チケットID、要約にする |
| 脆弱性詳細 | アクセス制限された場所に分離する |
| 本番データの生値 | 原則保存しない。必要ならサンプル化または参照IDにする |
| 外部送信本文 | 保持が必要な場合は機密区分とアクセス制御を付ける |
| モデル内部推論の詳細 | 保存が必要か、監査・セキュリティ・プライバシー観点で確認する |

ログには、業務改善に必要な情報を残す。一方で、ログ自体が情報漏洩源にならないようにする。

設計上は、次の3つを分ける。

```text
原文を残す
要約を残す
IDまたはリンクだけを残す
```

例えば、顧客問い合わせ本文をログへ丸ごと残すのではなく、問い合わせチケットIDと分類結果、参照したFAQ ID、レビュー結果を残す。原文は権限管理されたチケットシステムに置く。

## ログの保持期間とアクセス権

ログは残せば残すほどよいわけではない。保持期間とアクセス権を決める。

| ログ種別 | 例 | 保持方針の考え方 |
|---|---|---|
| 低リスク作業ログ | 社内下書き、個人メモ | 短期保持、要約のみでもよい |
| 業務改善ログ | レビューIssue、改善バックログ | 改善完了後も一定期間保持 |
| 承認ログ | 顧客送信、CRM更新、PR作成 | 監査・説明責任に応じて保持 |
| セキュリティログ | 権限拒否、禁止操作、外部入力 | セキュリティ要件に応じて保持 |
| インシデントログ | 誤送信、情報漏洩、誤実行 | インシデント対応・再発防止として保持 |
| 評価ログ | 回帰評価、品質スコア | 品質推移を見るため継続保持 |

アクセス権も分ける。

```text
利用者が見られるログ
チーム責任者が見られるログ
AI推進担当が見られるログ
セキュリティ担当が見られるログ
監査担当だけが見られるログ
```

ログ設計は、第10章のガバナンスと統制に接続する。

## 失敗分類

ログを改善へ接続するには、失敗を分類する必要がある。

この章では、次の分類を使う。

| 分類 | 内容 | 改善先 |
|---|---|---|
| Request Defect | 依頼が曖昧、目的・範囲・禁止事項が不足 | Request Contract |
| Context Defect | 前提が古い、不足、混在、機密区分不明 | Context Pack |
| Task Defect | 分解粒度、停止条件、依存関係が不適切 | Task Brief / Task Breakdown |
| Review Defect | レビュー観点不足、Severity誤判定、承認前確認不足 | AI出力レビュー表 / Review Criteria |
| Permission Defect | 権限過大、承認ゲート不足、スコープ過広 | ツール権限マトリクス / HITL |
| Tool Defect | ツール仕様、引数、戻り値、エラー処理が不適切 | Tool Spec Card |
| Evaluation Defect | 評価ケース不足、禁止出力テスト不足 | Evaluation Plan |
| Model Behavior | モデルの出力傾向、幻覚、過剰断定、指示追従の問題 | プロンプト、評価、モデル選定 |
| User Misuse | 利用者がルールを守らない、確認せず外部送信 | 教育、UI、ポリシー、承認フロー |
| Governance Defect | 台帳、責任者、規程、インシデント手順不足 | AI System Inventory / Governance |

失敗分類の目的は、責任追及ではない。改善先を特定することである。

悪い振り返りは次である。

```text
AIが間違えた。次から気をつける。
```

良い振り返りは次である。

```text
分類：Context Defect + Review Defect
原因：Context Packに古いSLA条件が含まれており、レビュー表にSLA確認観点がなかった。
改善：Context Packに情報鮮度ラベルを追加し、顧客返信レビュー表にSLA・補償確認を追加する。
評価：原因未確定・SLA影響ありのケースをEvaluation Planへ追加する。
```

## Improvement Backlog

ログとレビュー結果は、Improvement Backlogへ接続する。

Improvement Backlogは、AI活用の改善Issueである。通常のプロダクト改善、業務改善、セキュリティ改善と同じように、担当者、優先度、期限、完了条件を持つ。

項目は次である。

| 項目 | 内容 |
|---|---|
| Improvement ID | 改善ID |
| 発生元 | Run ID、Review Issue、Incident、Evaluation Failureなど |
| 問題 | 何が起きたか |
| 影響 | 顧客、業務、品質、コスト、セキュリティへの影響 |
| 分類 | Request / Context / Task / Review / Permission / Tool / Evaluation / Governance |
| Severity | Blocker / Critical / Major / Minor / Nit |
| 改善対象 | どの成果物、テンプレート、ツール、権限、教材を直すか |
| 改善内容 | 具体的な変更 |
| 担当者 | 誰が直すか |
| 期限 | いつまでに直すか |
| 完了条件 | 何をもって完了とするか |
| 評価ケース | 再発防止のための評価ケース |
| 状態 | open / in progress / review / done / won't fix |

AI活用改善は、プロンプトを書き換えて終わりではない。改善先は複数ある。

```text
Request Contractを直す
Context Packを直す
Task Briefを直す
レビュー表を直す
権限を下げる
HITLを追加する
Tool Specを直す
評価ケースを追加する
教材に失敗事例を追加する
ポリシーを更新する
```

## メトリクス設計

AIエージェント活用は、利用回数だけでは評価できない。使われていても、品質が悪い、レビュー負荷が高い、コストが増える、事故リスクが高い場合がある。

メトリクスは、次の5カテゴリで設計する。

### 1. 利用メトリクス

| 指標 | 意味 |
|---|---|
| Run数 | AIエージェント実行数 |
| 利用者数 | 実際に使っている人数 |
| ユースケース数 | 本番運用されているAI活用業務数 |
| 部門別利用数 | どの部門で使われているか |
| テンプレート利用率 | Request Contract、Context Packなどの利用率 |

### 2. 品質メトリクス

| 指標 | 意味 |
|---|---|
| 採用率 | AI出力がそのまま、または修正後に採用された割合 |
| 差し戻し率 | AI出力が差し戻された割合 |
| Critical Issue率 | 重大なレビュー指摘の割合 |
| 根拠不足率 | 根拠不足として指摘された割合 |
| 不確実性未分離率 | 推測と確定情報が混在した割合 |
| 評価ケース合格率 | Evaluation Planでの合格率 |

### 3. リスクメトリクス

| 指標 | 意味 |
|---|---|
| HITL発生数 | 人間承認が必要になった件数 |
| 承認拒否率 | 承認で拒否・差し戻しになった割合 |
| 禁止操作ブロック数 | AIが禁止操作を試み、ブロックされた数 |
| スコープ外アクセス試行数 | 権限・スコープ外アクセスが発生した数 |
| インシデント数 | 誤送信、誤更新、情報漏洩など |
| セキュリティレビュー指摘数 | セキュリティ観点での指摘数 |

### 4. 効率メトリクス

| 指標 | 意味 |
|---|---|
| 作業時間削減 | Before / Afterの所要時間差 |
| リードタイム短縮 | 依頼から成果物完成までの時間 |
| レビュー時間 | 人間レビューにかかった時間 |
| 再実行回数 | 出力修正や再生成の回数 |
| ツール実行エラー率 | API、検索、権限、タイムアウトなどの失敗率 |
| コスト | API費用、実行基盤費用、レビュー工数 |

### 5. 学習メトリクス

| 指標 | 意味 |
|---|---|
| Improvement Backlog件数 | 改善Issueの件数 |
| 改善完了率 | OpenからDoneになった割合 |
| 失敗事例化数 | 失敗が教材・チェックリストへ反映された数 |
| 評価ケース追加数 | 実運用の失敗から追加された評価ケース数 |
| 再発率 | 同じ分類の失敗が再発した割合 |
| 廃棄判断数 | 危険または価値が低いAI活用を停止した数 |

メトリクスは、組織を罰するためではなく、改善の方向を決めるために使う。特に重要なのは、利用数ではなく、品質、リスク、改善の接続である。

## Run Reviewの運用

ログは記録するだけでは足りない。定期的にレビューする。

Run Reviewは、AIエージェント運用の振り返り会である。頻度は、リスクと利用量に応じて決める。

| 頻度 | 対象 |
|---|---|
| 毎日 | 顧客影響、本番影響、セキュリティ影響がある高リスクエージェント |
| 週次 | 部門内で使う主要エージェント |
| 月次 | 低リスクな業務支援エージェント |
| 四半期 | 全社AI活用状況、ポリシー、台帳、教材更新 |

Run Reviewで見るものは次である。

```text
重大IssueがあったRun
承認拒否・差し戻しが多いRun
禁止操作ブロックがあったRun
コストが急増したRun
評価ケースに失敗したRun
人間レビュー負荷が高いRun
同じ失敗が再発したRun
価値が出ていないRun
```

Run Reviewの出力は、議事録ではなく改善アクションである。

```text
どのテンプレートを直すか
どの権限を下げるか
どの評価ケースを追加するか
どの教材へ失敗事例を入れるか
どのユースケースを停止するか
どのKPIを見直すか
```

## 実務ケース1: 顧客返信の誤分類

### 状況

AIが顧客問い合わせを「低優先度」と分類した。しかし実際には、契約上のSLA影響がある高優先度案件だった。

### ログがない場合

ログがない場合、振り返りは次のようになる。

```text
AIが優先度を間違えた。
担当者が次から気をつける。
```

これでは改善にならない。

### Agent Run Logで確認できること

Agent Run LogとTraceから、次が分かった。

```text
Run ID：CS-2026-0627-00012
Request Contract：問い合わせ分類と返信案作成
Context Pack：顧客契約情報の要約。ただしSLA条件は旧版
参照情報：FAQ、過去チケット、障害メモ
ツール実行：CRM参照は成功。契約書本体は参照していない
AI出力：低優先度、通常返信でよいと分類
レビュー：CS担当が返信案を修正したが、優先度分類は見落とし
承認：CS責任者承認なし。低リスク扱いで送信直前まで進行
失敗分類：Context Defect + Review Defect + Permission Defect
```

### 改善アクション

```text
Context Packに契約・SLA情報の鮮度ラベルを追加する
SLA影響がある可能性がある問い合わせはHITL必須にする
AI出力レビュー表に「契約・SLA・補償への影響」を追加する
Evaluation PlanにSLA影響あり問い合わせの評価ケースを追加する
低優先度分類時でも、契約語句が含まれる場合はエスカレーションする
```

このケースでは、AIモデルを変えることが第一の改善ではない。前提情報、レビュー観点、承認条件を直すことが改善である。

## 実務ケース2: チケットの大量作成

### 状況

AIエージェントが、監視アラートをもとにJiraチケットを自動作成する設計だった。ある日、同じ原因のアラートが大量発生し、AIが類似チケットを多数作成した。

### Traceで確認すべきこと

```text
何件のアラートを処理したか
同一原因と判断するルールがあったか
重複チケット検出ツールを呼んだか
チケット作成前にドライランしたか
レート制限が機能したか
同一Run内で何回作成したか
人間承認または停止条件はあったか
```

### 失敗分類

```text
Task Defect：大量アラート時の停止条件がない
Tool Defect：重複検出が不十分
Permission Defect：チケット作成がlimited-writeで、件数制限が弱い
Evaluation Defect：大量アラートケースが評価にない
```

### 改善アクション

```text
同一サービス・同一時間帯の類似アラートは1件に集約する
1Runあたりのチケット作成上限を設定する
上限超過時は人間へエスカレーションする
チケット作成前に既存チケット検索を必須にする
大量アラート評価ケースを追加する
```

このケースでは、ログがなければ「AIがチケットを作りすぎた」で終わる。Traceがあれば、重複検出、停止条件、権限、評価ケースの不足へ分解できる。

## 実務ケース3: 古い資料を参照した提案書

### 状況

AIが新サービス提案書を作成したが、競合価格と当社仕様が古い情報に基づいていた。

### Agent Run Logで確認すること

```text
使用したContext Packの作成日
参照した競合資料の取得日
社内仕様書の版
情報鮮度ラベル
AI出力レビューで日付・版を確認したか
評価ケースに古い資料混入ケースがあるか
```

### 失敗分類

```text
Context Defect：古い資料がContext Packに入っていた
Review Defect：情報鮮度をレビューしていなかった
Evaluation Defect：古い資料を検出するケースがなかった
```

### 改善アクション

```text
Context Packに情報鮮度ラベルを必須化する
90日以上前の競合情報は「要再確認」と表示する
レビュー表に「情報鮮度」の確認欄を追加する
提案書作成前に最新確認タスクを追加する
```

## 実務ケース4: 承認内容と実行内容の差分

### 状況

AIが社内Wiki更新案を作成し、責任者が承認した。しかし実行時には、承認時の差分とは異なる本文が更新された。

### 必要なログ

```text
承認時の差分
承認者が見た本文
承認後にAIが再生成したか
実行直前の差分
実行後の差分
実行時のツール引数
更新対象ページID
ロールバック方法
```

### 失敗分類

```text
Permission Defect：承認後の再生成が許可されていた
Tool Defect：実行直前差分チェックがない
Approval Defect：承認内容と実行内容の一致確認がない
```

### 改善アクション

```text
承認後は出力を固定する
実行直前に承認時との差分を照合する
差分があれば再承認に戻す
Tool Execution Logに承認時hashと実行時hashを残す
```

このケースは、HITLを入れてもログ設計が弱いと事故が起きることを示している。

## Observability Design Sheet

AIエージェントを導入する前に、観測設計を作る。

最低限、次を決める。

```text
どのRunを記録するか
どのログを必須にするか
どのツール実行をTraceに残すか
どの承認をApproval Logに残すか
どの情報をマスクするか
どのログを誰が見られるか
どのログをどれだけ保持するか
どのメトリクスを見るか
どの頻度でRun Reviewを行うか
失敗をどのバックログへ接続するか
```

観測設計は、エージェントを本番利用してから追加するものではない。最初に設計する。

## 典型的な失敗

### 失敗1: チャット履歴をログだと思う

チャット履歴には、承認、ツール実行、差分、スコープ、失敗分類、改善先が構造化されていないことが多い。業務ログとして使うには不足する。

### 失敗2: 出力だけを保存する

最終出力だけを残しても、なぜその出力になったか分からない。参照情報、Context Pack、ツール実行、レビュー結果を残す必要がある。

### 失敗3: ログに機密情報を残しすぎる

ログは便利だが、機密情報を大量に集約する場所にもなる。ログが新しいリスク源になる。

### 失敗4: ログを誰も見ない

ログを保存しても、Run Review、改善バックログ、評価ケース追加に接続しなければ意味がない。

### 失敗5: 失敗分類が粗い

「AIが間違えた」で終わると改善できない。Request、Context、Task、Review、Permission、Tool、Evaluation、Governanceへ分類する。

### 失敗6: 承認ログが弱い

「承認済み」とだけ残っていても、承認者が何を見て、どの条件で承認したか分からなければ証跡として弱い。

### 失敗7: コストとレビュー負荷を見ない

AI活用は、実行コストだけでなく、人間レビュー、再生成、差し戻し、承認待ち時間も含めて評価する。

## 設計原則

第8章の設計原則は次である。

### 原則1: Run単位で追跡する

依頼、前提、ツール、出力、レビュー、承認、改善をRun IDでつなぐ。

### 原則2: ログは目的別に分ける

監査、改善、評価、セキュリティ、コスト管理では必要な粒度が違う。目的ごとにログを分ける。

### 原則3: 機密情報を最小化する

必要な証跡を残しつつ、原文、生データ、認証情報を残しすぎない。ID参照、要約、マスク、アクセス制御を使う。

### 原則4: 失敗を分類して改善先へ戻す

ログを眺めるだけではなく、失敗分類を行い、具体的な成果物へ戻す。

### 原則5: 承認内容と実行内容を照合する

HITL承認では、承認者が見た内容と実行された内容が一致していることを確認できるようにする。

### 原則6: 評価ケースへ接続する

一度起きた失敗は、可能な限り評価ケースへ追加する。再発を人間の注意だけに頼らない。

### 原則7: メトリクスは利用数だけでなく品質を見る

Run数が増えても、Critical Issue、承認拒否、レビュー工数、インシデント、コストが増えていれば成功とは言えない。

## 作成手順

AIエージェントのログ・トレース設計は、次の手順で作る。

### Step 1: 対象Runを定義する

どの業務実行を1Runとして扱うかを決める。

```text
問い合わせ1件ごと
提案書1件ごと
PR1件ごと
障害一次対応1件ごと
月次レポート1件ごと
```

### Step 2: Run IDと関連IDを決める

Request Contract、Context Pack、Task Brief、Tool Execution、Approval、Review、Evaluation、Improvement BacklogをRun IDで紐づける。

### Step 3: ログ対象を分類する

```text
入力ログ
コンテキストログ
参照ログ
ツールログ
出力ログ
レビュー・評価ログ
承認ログ
失敗・例外ログ
改善ログ
```

### Step 4: 保存しない情報を決める

認証情報、個人情報、顧客機密、脆弱性詳細、本番データなどの扱いを決める。

### Step 5: Trace Eventを設計する

モデル呼び出し、ツール呼び出し、承認待ち、レビュー、エラー、停止、出力生成などのイベントを定義する。

### Step 6: 失敗分類を決める

Request、Context、Task、Review、Permission、Tool、Evaluation、Governanceの分類を使う。

### Step 7: Improvement Backlogへ接続する

ログで見つかった問題を、担当者、期限、完了条件付きで改善Issueにする。

### Step 8: Metrics Dashboardを設計する

利用、品質、リスク、効率、学習のメトリクスを決める。

### Step 9: Run Reviewの頻度を決める

日次、週次、月次、四半期のどの頻度でレビューするかを決める。

### Step 10: 教材・ガバナンスへ戻す

失敗事例、チェックリスト、権限、ポリシー、スキル研修へ反映する。

## 演習

次のケースについて、Agent Run Log、失敗分類、Improvement Backlogを作る。

### 前提

```text
業務：顧客問い合わせ対応の一次支援
目的：顧客からの問い合わせに対して、AIが関連情報を検索し、返信案を作成する。
利用ツール：FAQ、過去チケット、CRM、メール下書き、Slack社内確認。
禁止事項：顧客への直接送信、SLA補償判断、原因未確定事項の断定、CRM直接更新。
```

### 発生した事象

```text
AIが問い合わせを低優先度と分類した。
返信案では「現在は解消済みであり、再発の可能性は低い」と記載した。
人間レビューで、実際にはSLA影響がある可能性があり、原因も未確定であることが分かった。
CRMには対象顧客の契約種別が登録されていたが、AIはCRMの契約詳細を参照していなかった。
Context Packには、旧版のSLA条件が含まれていた。
```

### 作成するもの

```text
1. Agent Run Log
2. Trace Event Recordの主要イベント5件以上
3. Tool Execution Logの主要項目
4. Review Issue Logからの失敗分類
5. Improvement Backlogを3件以上
6. Metrics Dashboardに追加すべき指標
```

### 演習の判定基準

次の条件を満たしていれば、この章の演習としては合格である。

- Run IDを設定している
- Request Contract、Context Pack、Task Brief、AI出力、レビュー、ツール実行を紐づけている
- CRM契約詳細を参照していないことをログ上の事実として扱っている
- 旧版SLA条件をContext Defectとして分類している
- 原因断定・再発可能性の記述をReview DefectまたはEvaluation Defectとして扱っている
- SLA影響の可能性をHITLまたはPermission Defectに接続している
- Improvement Backlogに、Context Pack更新、レビュー表修正、Evaluation Plan追加のいずれかが含まれている
- ログに顧客機密を残しすぎない設計になっている
- Metrics Dashboardに、SLA関連エスカレーション、Critical Issue率、承認拒否率、再発率のいずれかが含まれている

## まとめ

AIエージェントの実行は、記録されるまで組織能力にならない。

出力だけを保存しても、再現、改善、監査はできない。依頼、前提、タスク、出力、レビュー、権限、ツール実行、承認、失敗、改善アクションをRun単位でつなげる必要がある。

ログは多ければよいわけではない。業務改善に必要な情報を残し、認証情報、個人情報、顧客機密、本番データは最小化する。ログ自体が新たな情報漏洩源にならないよう、アクセス権、保持期間、マスキングを設計する。

失敗は「AIが間違えた」で終わらせない。Request、Context、Task、Review、Permission、Tool、Evaluation、Governanceへ分類し、Improvement Backlogへ接続する。一度起きた失敗は、できる限り評価ケース、チェックリスト、教材、権限、ポリシーへ戻す。

第8章までで、AIエージェント協働の基礎、仕事選定、依頼、コンテキスト、委任、レビュー、権限、ログまでがつながった。次章では、この実行系に対するセキュリティとプライバシーのリスクを扱う。

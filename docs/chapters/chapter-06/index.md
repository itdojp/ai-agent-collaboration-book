# 第6章 AI出力レビューと評価

## この章で扱う能力

この章では、AIが生成した出力を、業務で使える成果物へ変換するためのレビューと評価を扱う。読者は、AI出力について、目的適合性、事実正確性、根拠、論理、網羅性、実行可能性、リスク、不確実性、再現性を確認し、採用、修正後採用、差し戻し、不採用、エスカレーションを判断できるようになる。

この章の結論は次のとおりである。

```text
AI出力は、レビューされるまで成果物ではない。
自然な文章、整った表、もっともらしい提案は、品質の証明ではない。

レビューとは、AI出力を受け入れ条件に照らして検査し、
事実、根拠、論理、リスク、不確実性を分離し、
採用できる部分、修正すべき部分、捨てるべき部分を決めることである。

評価とは、個別出力ではなく、AI活用の仕組み全体が
安定して望ましい品質を出せるかを確認することである。
```

第5章では、複雑な業務を Task Brief へ分解し、AIに委任可能な作業単位を作った。この章では、その Task Brief から生まれた出力をどう確認するかを扱う。仕事選定が Work 層、依頼設計が Request 層、前提情報設計が Context 層、委任設計が Delegation 層なら、この章は Control 層の入口である。

## 本書全体における位置づけ

| 項目 | 内容 |
|---|---|
| 設計層 | Control: Review / Evaluation |
| 目的 | AI出力を成果物として採用できるか、レビューと評価で判定する。 |
| 主な成果物 | AI出力レビュー表、Review Issue Log、Evaluation Plan、評価ケース一覧 |
| 前章から受け取るもの | Task Brief、受け入れ条件、チェックポイント、AI出力 |
| 次章へ渡すもの | 承認が必要な操作、権限設計の前提、HITL設計の条件 |
| 参照 | [Artifact Index](../../artifact-index/)、[付録A: 成果物テンプレート集](../../appendices/appendix-a/)、[付録B: チェックリスト集](../../appendices/appendix-b/) |

本書の6層フレームワークでは、第6章は次の位置にある。

```text
Work: どの仕事にAIを使うか
  ↓
Request: AIに何を依頼するか
  ↓
Context: AIに何を渡すか
  ↓
Delegation: どう分解し、どこで止めるか
  ↓
Control: どう検証し、どう制御するか  ← この章
  ↓
Human / Org: 人間が何を判断し、どう学習するか
```

Control 層には、レビュー、評価、承認、権限、ログ、監査、改善が含まれる。この章では、そのうち「レビュー」と「評価」を扱う。次章では、レビュー結果を踏まえて、AIがどのツールをどこまで使ってよいか、人間承認をどこに入れるかを扱う。

## なぜ必要か

AI出力は、見た目が整っているほど危険になる場合がある。文章は自然で、表はきれいで、論点も並んでいる。それでも、次のような問題が起きる。

```text
存在しない事実を前提にしている。
古い情報を現在有効な情報として扱っている。
社内の前提と異なる判断基準を使っている。
根拠のない推測を断定している。
一次情報ではなく、曖昧な要約に基づいている。
人間が決めるべき判断を、AIが結論として書いている。
顧客に見せてはいけない内部事情を含んでいる。
法務、セキュリティ、契約、SLAに関する判断を勝手にしている。
一部だけ正しいため、全体も正しいように見える。
```

人間は、読みやすいもの、構造化されたもの、自信ありげに書かれたものを高品質だと誤認しやすい。AI出力の品質は、文体ではなく、受け入れ条件に対する適合で判断する必要がある。

AI出力レビューがない組織では、次の状態になりやすい。

```text
AIを使った人だけが、どこを確認したか知っている。
レビュー観点が人によって違う。
誤りが見つかっても、次回の依頼やContext Packに反映されない。
採用、修正、差し戻し、不採用の基準が曖昧である。
顧客送信や本番操作の直前で、初めて重大な問題が見つかる。
```

これを防ぐために、レビューを標準化する。標準化とは、すべての出力を同じ細かさで検査することではない。出力の種類、リスク、顧客影響、利用先に応じて、どの観点をどの強度で確認するかを決めることである。

## AI出力は「素材」であって「成果物」ではない

AIが出した文章、表、コード、計画、メール案、分析メモは、最初は素材である。素材は、そのまま使えることもあるが、業務成果物として扱うには、少なくとも次を確認する必要がある。

```text
目的に合っているか。
依頼時の受け入れ条件を満たしているか。
使ってよい前提だけを使っているか。
確定情報と推定情報が分かれているか。
重要な誤りや抜けがないか。
利用先に対して安全か。
人間が判断すべきことを、AIが勝手に決めていないか。
```

本書では、次のように区別する。

| 状態 | 意味 | 扱い |
|---|---|---|
| AI出力 | AIが生成した未レビューの結果 | 素材。外部送信・本番反映しない |
| レビュー済み出力 | 人間または定義済み手順で確認した結果 | 限定的に利用可能 |
| 承認済み成果物 | 権限ある人間が採用を判断した結果 | 業務成果物として扱う |
| 評価済み仕組み | 複数ケースで品質を確認したAI活用手順 | 再利用・標準化の候補 |

この区別がないと、「AIが作ったもの」と「会社として採用したもの」が混ざる。責任の所在も曖昧になる。AIエージェント協働では、AIが出したことではなく、人間と組織が何を採用したかを管理する。

## レビュー、評価、検証、承認の違い

現場では、レビュー、評価、検証、承認という言葉が混ざりやすい。ここでは次のように分ける。

| 用語 | 対象 | 目的 | 例 |
|---|---|---|---|
| レビュー | 個別のAI出力 | その出力を使ってよいか確認する | この顧客返信案を送ってよいか |
| 評価 | AI活用の仕組み | 安定して望ましい品質が出るか確認する | このRequest ContractとContext Packで10ケース処理できるか |
| 検証 | 事実・仕様・動作 | 正しいか、動くか、再現するか確認する | 数値、引用、SQL、コード、手順の確認 |
| 承認 | 人間の責任ある判断 | 実行・送信・公開・本番反映を許可する | 顧客送信、契約判断、本番変更の承認 |

個別の出力を見るのがレビューである。複数ケースで仕組みを見るのが評価である。正しいかどうかを確認するのが検証である。責任を持って実行を許可するのが承認である。

この章で扱う中心はレビューと評価である。ただし、レビュー結果として「承認が必要」「検証が不足」「不採用」と判断する場面も扱う。

## レビューで使う入力

レビューは、AI出力だけを見て行うものではない。AI出力を正しく評価するには、少なくとも次の入力が必要になる。

| 入力 | 役割 |
|---|---|
| Request Contract | 目的、成果物、範囲、制約、受け入れ条件を確認する |
| Acceptance Criteria | 採用条件を確認する |
| Context Pack | AIが使うべき前提、使ってはいけない情報、情報鮮度を確認する |
| Task Brief | AIに任せた具体作業と、任せていない判断を確認する |
| AI出力 | レビュー対象そのもの |
| 参照情報 | 出典、社内資料、ログ、仕様書、チケット、議事録 |
| 利用先 | 社内メモ、顧客送信、経営判断、本番操作などを確認する |

AI出力だけを見て「よさそう」と判断するのは危険である。レビューは、出力と依頼の対応を確認する作業である。したがって、レビューの最低単位は次である。

```text
Request Contract
+ Context Pack
+ Task Brief
+ AI出力
+ 利用先
```

この組み合わせがないレビューは、文体チェックに近くなる。文体チェックは必要な場合もあるが、AI出力の品質保証にはならない。

## レビュー強度を決める

すべてのAI出力を同じ強度でレビューすると、現場は回らない。逆に、すべてを軽く見ると事故が起きる。レビュー強度は、出力の利用先とリスクで決める。

### レビュー強度レベル

| レベル | 用途 | レビュー方法 | 例 |
|---:|---|---|---|
| L0 | 個人メモ、発想補助 | 自己確認のみ | アイデア出し、表現案 |
| L1 | 社内下書き | 依頼者がレビュー表で確認 | 議事録要約、社内メモ |
| L2 | チーム共有 | ピアレビュー | 調査メモ、提案書ドラフト |
| L3 | 顧客影響あり | ドメイン担当者レビュー | 顧客返信案、FAQ改訂案 |
| L4 | 法務・セキュリティ・本番影響あり | 責任者承認、専門レビュー | SLA説明、契約条項、本番変更手順 |
| L5 | 標準業務化・自動化 | 評価ケース、回帰評価、ログ監視 | 問い合わせ分類、チケット自動処理 |

レビュー強度は、AIの能力ではなく、失敗時の影響で決める。モデルが高性能でも、顧客送信、契約、セキュリティ、本番環境、金銭影響、個人情報が絡むなら強いレビューが必要になる。

### レビュー強度を上げる条件

次に該当する場合は、レビュー強度を上げる。

```text
外部送信される。
顧客が読む。
契約、価格、補償、SLAに触れる。
個人情報、顧客情報、機密情報を含む。
セキュリティ判断に影響する。
コード、SQL、shell、設定変更に使われる。
本番環境、監視、障害対応に関係する。
採用、人事評価、与信、医療、金融、教育など高リスク判断に関わる。
経営判断、投資判断、撤退判断に使われる。
再利用されるテンプレートや標準手順になる。
```

レビュー強度を下げてよいのは、失敗しても局所的で、可逆で、外部影響がなく、人間がすぐ気づける場合である。

## 基本レビュー観点

AI出力レビューでは、最低限、次の観点を見る。

| 観点 | 確認すること | よくある問題 |
|---|---|---|
| 目的適合性 | 依頼目的と成果物に合っているか | きれいだが目的から外れている |
| 事実正確性 | 固有名詞、日付、数値、仕様、ログが正しいか | 古い情報、存在しない情報、混同 |
| 根拠 | 出典、社内根拠、ログ、仕様、前提があるか | 根拠なし、出典違い、二次情報依存 |
| 論理 | 前提から結論までの推論が妥当か | 飛躍、因果の取り違え、過剰一般化 |
| 網羅性 | 重要論点が抜けていないか | 都合のよい論点だけ並ぶ |
| 実行可能性 | 実務で実行できる粒度か | 抽象論、担当者不明、期限不明 |
| リスク | 顧客、法務、セキュリティ、評判、運用への影響 | 禁止事項、承認漏れ、過剰断定 |
| 不確実性 | 確定、推定、仮定、要確認が分離されているか | 推測を事実のように書く |
| 再利用性 | 次回も使える形で残せるか | 一回限りの修正で終わる |

この表は、すべての出力に同じ重みで適用するものではない。顧客返信ではリスクと文面が重要になる。競合調査では事実と根拠が重要になる。コードでは動作と安全性が重要になる。判断メモでは論理と不確実性が重要になる。

## 目的適合性レビュー

目的適合性とは、AI出力が「依頼された仕事」に答えているかである。AIは、質問にそれらしく答えることは得意である。しかし、依頼の目的に沿った成果物になっているとは限らない。

目的適合性では、次を確認する。

```text
Request Contract の目的に答えているか。
成果物の形式が指定どおりか。
利用者が必要とする粒度か。
AIが勝手に範囲を広げていないか。
逆に、重要な範囲を落としていないか。
受け入れ条件を満たしているか。
```

例として、「価格改定検討のための競合比較表」を依頼したのに、AIが「価格改定すべきです」と結論を書いた場合、目的適合性に問題がある。このTask BriefでAIに任せたのは判断材料の作成であって、価格改定判断ではない。

目的適合性レビューでは、最初に次を問う。

```text
この出力は、何の意思決定や作業を助けるためのものか。
その目的に対して、必要な情報が出ているか。
余計な判断や、禁止された判断をしていないか。
```

## 事実正確性レビュー

事実正確性とは、出力に含まれる事実が正しいかである。AI出力では、次のような事実誤りが起きる。

```text
存在しない製品名を書く。
旧価格を現行価格として扱う。
日付や時刻を誤る。
ログの順序を取り違える。
類似したサービスを混同する。
個人向けプランと法人向けプランを混ぜる。
社内用語を一般用語として誤解する。
顧客名、宛先、契約条件を誤る。
```

事実正確性レビューでは、すべての文を同じ強度で確認する必要はない。重要な事実を特定し、その事実を優先的に確認する。

### 確認すべき事実の優先順位

| 優先度 | 対象 | 理由 |
|---:|---|---|
| 高 | 顧客名、契約、価格、SLA、障害時刻、影響範囲 | 誤ると外部影響が大きい |
| 高 | セキュリティ、個人情報、認証、権限、設定変更 | 誤ると事故につながる |
| 中 | 製品仕様、機能比較、制約条件 | 誤ると判断材料が歪む |
| 中 | 数値、比率、件数、期間 | 誤ると説得力だけが残る |
| 低 | 表現、一般論、補足説明 | 誤り影響は比較的小さい |

AI出力が「事実」として書いたものには、根拠が必要である。根拠がない事実は、事実ではなく仮説または要確認事項として扱う。

## 根拠レビュー

根拠レビューでは、出力の主張が何に基づいているかを確認する。根拠があることと、根拠が適切であることは別である。

次のような根拠は弱い。

```text
出典が古い。
一次情報ではない。
対象範囲が違う。
個人向け情報を法人向け判断に使っている。
海外仕様を国内仕様として扱っている。
料金ページの一部だけを見ている。
社内Slackの発言を正式決定として扱っている。
過去の議事録を現行ルールとして扱っている。
```

根拠レビューでは、次の4点を見る。

| 観点 | 確認内容 |
|---|---|
| 存在 | 根拠が示されているか |
| 適合 | その根拠が対象業務に合っているか |
| 鮮度 | 現在有効な情報か |
| 解釈 | 根拠から結論への解釈が妥当か |

AIに出典や根拠を出させることは有効である。ただし、出典があるだけでは不十分である。出典が実在するか、内容が一致しているか、引用部分が文脈どおりか、現在も有効かを確認する。

社内情報では、根拠の階層を決めておくとよい。

| 根拠の強さ | 例 |
|---|---|
| 強 | 契約書、承認済み仕様書、正式Runbook、正式リリースノート、監査済みログ |
| 中 | チケット、PR、議事録、承認済み社内Wiki、障害報告書 |
| 弱 | Slack発言、個人メモ、過去資料、未レビューのAI要約 |
| 使用禁止 | 認証情報、他顧客情報、未許可の個人情報、参照禁止資料 |

Context Packでは、根拠の強さを明示する。レビューでは、その強さに反した使い方をしていないか確認する。

## 論理レビュー

論理レビューでは、事実から結論までのつながりを確認する。事実が正しくても、推論が間違っていれば、成果物としては危険である。

よくある論理問題は次である。

```text
相関を因果として扱う。
一部の事例から全体傾向を断定する。
比較条件が揃っていない。
制約条件を無視して提案する。
実行コストを考えずに施策を並べる。
反対材料を無視する。
前提が変わった場合の影響を見ない。
「Aが優れている」から「Aを採用すべき」へ飛躍する。
```

論理レビューでは、次の問いを使う。

```text
結論は、前提から本当に導けるか。
比較対象の条件は揃っているか。
反対の解釈はあり得るか。
重要な制約を無視していないか。
この結論を採用すると、何を犠牲にするか。
AIが判断材料を出しているのか、判断そのものをしているのか。
```

AI出力に結論がある場合、その結論を3つに分ける。

| 区分 | 扱い |
|---|---|
| 事実から直接言えること | 採用候補。ただし根拠確認する |
| 仮説として言えること | 仮説欄に移す |
| 人間が判断すべきこと | 判断前チェックリストへ渡す |

論理レビューの目的は、AIを論破することではない。AIが作った論理構造のどこが使え、どこが危険かを切り分けることである。

## 網羅性レビュー

網羅性とは、必要な論点が抜けていないかである。AI出力は、目立つ論点を整えて出すことがある。しかし、重要だが地味な制約、例外、リスク、未確認事項が抜けることがある。

網羅性レビューでは、次を確認する。

```text
Request Contract の対象範囲をすべて扱っているか。
受け入れ条件に含まれる観点が抜けていないか。
Context Pack の制約を反映しているか。
除外した範囲を明示しているか。
要確認事項が列挙されているか。
反対意見、例外、リスクが扱われているか。
```

網羅性は「何でも書くこと」ではない。目的に必要な論点を落とさないことである。余計な論点が多すぎる出力は、レビューしにくく、意思決定にも使いにくい。

網羅性を確認するには、最初に観点リストを作る。たとえば、法人向けサービスの競合比較であれば、最低限次を見る。

```text
対象顧客
価格体系
主要機能
バックアップ対象
復元粒度
サポート
セキュリティ
SLA
契約条件
運用負荷
要確認事項
```

AIが表を作った後に観点を考えるのではなく、Request Contract の段階で観点を定義する。レビューでは、その観点が満たされているか確認する。

## 実行可能性レビュー

AIは、一般論として正しいが、現場では実行できない提案を出すことがある。実行可能性レビューでは、出力が現場で動く粒度になっているかを確認する。

実行可能性では、次を見る。

```text
担当者が明確か。
次のアクションが具体的か。
必要な入力、権限、承認が分かるか。
期限や順序があるか。
依存関係が見えているか。
必要なコスト、工数、リスクが見えているか。
やらないこと、後回しにすることが明示されているか。
```

たとえば、「顧客満足度を高めるためFAQを改善する」は方向性として正しいかもしれない。しかし、実行可能性は低い。レビューでは、次のように具体化を求める。

```text
対象FAQはどれか。
何を基準に優先順位を決めるか。
誰がレビューするか。
公開前に誰が承認するか。
顧客影響がある記述はどこか。
いつまでに何件直すか。
改善効果をどう測るか。
```

実行可能性が低い出力は、不採用にする必要はない。多くの場合、「具体化タスク」として差し戻す。

## リスクレビュー

リスクレビューでは、AI出力を使った場合に起きる業務上の損害を確認する。リスクには、事実誤りだけでなく、表現、権限、承認、情報管理、責任境界も含まれる。

| リスク区分 | 例 |
|---|---|
| 顧客影響 | 誤案内、過剰約束、補償示唆、SLA誤説明 |
| 法務・契約 | 契約解釈、責任範囲、返金、損害賠償、規制領域 |
| セキュリティ | 脆弱性情報、認証情報、権限、攻撃手順、設定変更 |
| プライバシー | 個人情報、顧客情報、従業員情報、ログ内個人情報 |
| 運用 | 本番変更、障害対応、監視停止、復旧手順、削除操作 |
| レピュテーション | 不適切表現、過剰断定、未確認情報の公表 |
| 経営 | 投資判断、撤退判断、価格判断、採用・人事判断 |

リスクレビューでは、次の問いを使う。

```text
この出力をそのまま使うと、誰に影響するか。
外部に出るか。
顧客や第三者が誤解する余地はあるか。
契約、法務、セキュリティ、個人情報に触れていないか。
社内限定情報や未確定情報が含まれていないか。
人間承認が必要な操作につながっていないか。
失敗した場合に戻せるか。
```

高リスク出力は、文章の修正だけでは足りない。承認フロー、権限設計、ログ、インシデント対応と接続する必要がある。これは第7章以降で扱う。

## 不確実性レビュー

AI出力では、確定情報、推定、仮定、要確認が混ざることが多い。これを分けないと、読者は推測を事実として受け取る。

不確実性レビューでは、出力を次の4つに分類する。

| 区分 | 意味 | 表記例 |
|---|---|---|
| 確定情報 | 根拠により確認済み | 「契約書第X条に記載」 |
| 推定 | 根拠はあるが断定できない | 「ログ上はAの可能性が高い」 |
| 仮定 | 作業上の仮置き | 「本章では、対象顧客をSMBと仮定する」 |
| 要確認 | 判断前に確認が必要 | 「最新価格は公式情報で要確認」 |

不確実性を分離するだけで、AI出力の利用価値は上がる。すべてを確定させる必要はない。重要なのは、不確実なものを不確実として扱うことである。

AI出力に次のような表現がある場合は注意する。

```text
一般的に
多くの場合
おそらく
可能性が高い
最新では
業界標準では
通常は
必ず
明らかに
間違いなく
```

これらの表現は、根拠なしに説得力を出すために使われることがある。レビューでは、断定の強さと根拠の強さが釣り合っているか確認する。

## レビュー結果の判定

レビューは、問題点を指摘して終わりではない。最後に、出力をどう扱うかを判定する。

本書では、判定を次の5つに分ける。

| 判定 | 意味 | 次のアクション |
|---|---|---|
| 採用 | 受け入れ条件を満たし、リスクが許容範囲 | 成果物として利用する |
| 修正後採用 | 軽微な修正で使える | 人間またはAIに修正させる |
| 差し戻し | 目的、根拠、構造に不足がある | Request Contract / Context Pack / Task Briefを修正して再生成 |
| 不採用 | 出力全体が危険または目的不適合 | 捨てる。失敗ログに残す |
| エスカレーション | 専門判断や承認が必要 | 法務、セキュリティ、責任者、顧客担当へ渡す |

重要なのは、「修正」と「差し戻し」を分けることである。表現だけ直せば使えるものは修正後採用でよい。しかし、前提、根拠、論理、リスクに問題があるものは差し戻す。AI出力を表面的に整えるだけでは、危険な成果物が完成してしまう。

### Severity を付ける

レビューで見つけた問題には、重要度を付ける。

| Severity | 意味 | 例 |
|---|---|---|
| Blocker | そのまま使うと重大事故になり得る | 顧客に誤補償を約束、認証情報を含む、本番削除手順を誤る |
| Critical | 採用判断を止める重大問題 | 主要根拠が誤り、契約条件を誤解、影響範囲を誤る |
| Major | 修正しないと品質不十分 | 重要論点の欠落、比較条件の不一致、要確認事項不足 |
| Minor | 修正すればよい軽微な問題 | 表現、並び順、補足不足 |
| Nit | 好み・表記ゆれ | 文体、用語統一 |

Severityを付けると、レビューが「気になる点の列挙」で終わらない。何を先に直すべきか、採用を止めるべきかが判断しやすくなる。

## レビュー手順

AI出力レビューは、次の7ステップで行う。

```text
1. 利用先とレビュー強度を決める
2. Request Contract と Acceptance Criteria を確認する
3. Context Pack と Task Brief を確認する
4. AI出力を観点別にレビューする
5. 問題を Severity 付きで記録する
6. 採用、修正後採用、差し戻し、不採用、エスカレーションを判定する
7. 次回改善へ接続する
```

### Step 1: 利用先とレビュー強度を決める

まず、この出力がどこで使われるかを確認する。

```text
個人メモか。
チーム共有か。
社内文書か。
顧客送信か。
本番作業か。
経営判断か。
標準業務化するものか。
```

利用先が分からないままレビューすると、レビュー強度が決まらない。顧客送信に使う文章と、個人の下書きメモでは、必要な確認が違う。

### Step 2: Request Contract と Acceptance Criteria を確認する

レビューの基準は、レビュー時に思いつきで決めるのではない。第3章で定義した Request Contract と Acceptance Criteria を見る。

次を確認する。

```text
成果物形式は合っているか。
対象範囲は合っているか。
除外範囲に踏み込んでいないか。
受け入れ条件を満たしているか。
不明点の扱いは指定どおりか。
```

受け入れ条件が曖昧な場合は、レビュー前に補う。レビュー基準がないままレビューすると、好みの指摘になる。

### Step 3: Context Pack と Task Brief を確認する

次に、AIが使うべき前提と、AIに任せた作業を確認する。

```text
AIは使ってよい情報だけを使っているか。
古い情報を現在の前提として使っていないか。
確定、推定、仮定、要確認を分けているか。
AIに任せていない判断を出力していないか。
停止条件に該当する内容を無視していないか。
```

AI出力がよく見えても、Context Pack と違う前提で書かれているなら差し戻す。

### Step 4: 観点別にレビューする

基本観点に沿ってレビューする。レビュー表では、少なくとも次を記録する。

```text
目的適合性
事実正確性
根拠
論理
網羅性
実行可能性
リスク
不確実性
再利用性
```

レビューで見つけた問題は、本文に直接コメントするだけでなく、Review Issue Log に残す。ログに残すことで、後からプロンプト、Context Pack、Task Briefを改善できる。

### Step 5: Severity 付きで記録する

問題は、次の形式で記録する。

```text
ID：
対象箇所：
問題の種類：目的 / 事実 / 根拠 / 論理 / 網羅性 / 実行可能性 / リスク / 不確実性 / 表現
Severity：Blocker / Critical / Major / Minor / Nit
内容：
根拠：
修正方針：
担当：
```

この形式にすると、AIへの修正依頼にも使える。

### Step 6: 判定する

レビュー結果を見て、採用、修正後採用、差し戻し、不採用、エスカレーションを決める。

目安は次である。

| 状態 | 判定 |
|---|---|
| Blockerがある | 不採用またはエスカレーション |
| Criticalがある | 差し戻しまたはエスカレーション |
| Majorが複数ある | 差し戻し |
| Minor中心 | 修正後採用 |
| Nitのみ | 採用または軽微修正 |

ただし、顧客影響、法務、セキュリティ、本番環境では、Minorに見える問題でも承認が必要になることがある。Severity は機械的な判定ではなく、判断材料である。

### Step 7: 次回改善へ接続する

レビュー結果は、その出力だけで終わらせない。次のどこを直すべきか判断する。

| 問題 | 改善先 |
|---|---|
| 目的から外れた | Request Contract |
| 前提を誤った | Context Pack |
| 作業範囲が広すぎた | Task Brief |
| 中間確認が足りなかった | Checkpoint Plan |
| 危険な操作に進みそうだった | Stop / Escalation Rules、権限マトリクス |
| 同じ失敗が繰り返される | Evaluation Plan、研修、テンプレート |

この接続があると、レビューは単なる検品ではなく、AI活用の改善サイクルになる。

## AIに自己レビューさせる

AIに自己レビューさせることは有効である。ただし、自己レビューは人間レビューの代替ではない。AIは、自分が作った出力の問題を見落とすことがある。特に、事実確認、社内文脈、契約、セキュリティ、本番影響は、人間または別の検証手段が必要である。

AI自己レビューの使い方は次である。

```text
出力後に、受け入れ条件を満たしているか自己点検させる。
確定情報、推定、仮定、要確認を分けさせる。
根拠不足の箇所を列挙させる。
反対意見や失敗パターンを出させる。
顧客送信前のリスクを洗い出させる。
```

例として、次の依頼を使える。

```text
上記の出力を、以下の観点で自己レビューしてください。
1. Request Contract の受け入れ条件を満たしていない箇所
2. 根拠が不足している箇所
3. 事実確認が必要な箇所
4. 推定または仮定なのに断定している箇所
5. 顧客、法務、セキュリティ、運用上のリスク
6. 人間が判断すべき箇所

出力は表にしてください。
列は「箇所」「問題」「Severity」「修正案」「人間確認の要否」です。
```

AI自己レビューで得た指摘も、そのまま信用しない。人間レビューの入力として使う。

## 別AIレビューの使い方

作成したAIとは別のAIにレビューさせることも有効である。特に、次の用途に向く。

```text
論点の抜けを探す。
反対意見を出す。
リスクを洗い出す。
表現の分かりにくさを指摘する。
受け入れ条件とのズレを見つける。
```

ただし、別AIレビューも最終判断ではない。複数AIが同じ誤りをすることもある。誤った前提を共有していれば、別AIでも同じ結論になる。

別AIレビューでは、次を守る。

```text
レビュー用AIにも、Request Contract と Context Pack を渡す。
作成AIの出力だけを見せない。
レビュー観点を明示する。
最終判断は人間が行う。
```

## 典型的な失敗

### 失敗1: 文章の自然さを品質と誤認する

読みやすい文章は、正しい文章ではない。整った表は、正しい表ではない。レビューでは、文体や見た目ではなく、目的、事実、根拠、論理、リスクを見る。

### 失敗2: 出典があれば正しいと考える

出典が存在しても、その出典が古い、対象外、文脈違い、一次情報ではない、または解釈が誤っている可能性がある。出典は確認の入口であり、品質保証そのものではない。

### 失敗3: レビュー観点が人によって違う

属人的なレビューでは、品質が安定しない。ある人は表現を直し、別の人は根拠を見て、別の人はリスクだけを見る。レビュー表と Severity を使い、観点を揃える。

### 失敗4: AIに自己採点だけさせる

AI自己レビューは補助として有効だが、最終レビューの代替にはならない。特に外部送信、法務、セキュリティ、本番影響は人間が確認する。

### 失敗5: 修正で済むものと差し戻すべきものを混同する

表現や順序の問題は修正でよい。しかし、前提、根拠、論理、対象範囲、リスクの問題は差し戻しが必要である。危険な出力を整えるだけでは、危険な成果物が完成する。

### 失敗6: レビュー結果を次回に反映しない

毎回同じ誤りを人間が直しているなら、AI活用は改善していない。レビュー結果は、Request Contract、Context Pack、Task Brief、Evaluation Plan、研修教材へ反映する。

### 失敗7: 評価なしに標準業務化する

1回うまくいったAI活用を、そのまま標準業務にしてはいけない。標準化するには、複数ケース、異常系、古い情報、曖昧な入力、高リスク入力で評価する必要がある。

## 設計原則

### 原則1: レビュー基準を先に決める

出力後に「何となく良いか」を見るのではなく、Request Contract の段階で受け入れ条件を定義する。レビューは、後から始まる作業ではなく、依頼設計の時点で始まっている。

### 原則2: 出力だけでなく、前提と利用先を見る

AI出力が正しく見えても、前提が違えば使えない。利用先が顧客送信なら、社内メモとしては問題ない表現でも危険になる。

### 原則3: 事実、推論、判断を分ける

AI出力には、事実、推論、判断が混ざる。レビューでは、事実は根拠で確認し、推論は論理で確認し、判断は人間側へ戻す。

### 原則4: 不確実性を消さずに明示する

不明なことをAIに埋めさせると、もっともらしい誤りになる。不明なものは不明として残し、要確認事項として扱う。

### 原則5: 高リスク出力は承認フローへ接続する

レビューだけでは足りない出力がある。顧客送信、本番変更、契約、個人情報、セキュリティ、金銭影響は、承認フローと権限設計に接続する。

### 原則6: レビュー結果を仕組みに戻す

レビューは一回限りの検品ではない。レビュー結果は、依頼、コンテキスト、タスク分解、チェックポイント、評価ケース、研修教材へ戻す。

## 出力種別ごとのレビュー観点

AI出力には種類がある。出力種別ごとに、見るべき観点を変える。

### 調査・比較表

調査や比較表では、事実、根拠、対象範囲、比較条件が重要である。

確認項目は次である。

```text
対象は正しいか。
比較条件は揃っているか。
価格、仕様、日付、地域、プランが混ざっていないか。
一次情報または信頼できる情報源に基づいているか。
不明な項目を推測で埋めていないか。
要確認事項が分かれているか。
```

### 顧客返信案

顧客返信案では、事実、表現、リスク、承認が重要である。

確認項目は次である。

```text
顧客名、宛先、問い合わせ内容が正しいか。
社内事情を出していないか。
未確定原因を断定していないか。
補償、返金、SLA、契約判断を勝手にしていないか。
顧客に期待させすぎる表現がないか。
送信前承認があるか。
```

### 障害報告書

障害報告書では、時刻、影響範囲、原因、再発防止、未確定事項の扱いが重要である。

確認項目は次である。

```text
タイムラインはログと一致しているか。
発生時刻、検知時刻、復旧時刻を混同していないか。
影響範囲を過小評価または過大評価していないか。
原因確定前に断定していないか。
再発防止策が実行可能か。
社外版と社内版を分けているか。
```

### 提案書・判断メモ

提案書や判断メモでは、論理、反対材料、実行可能性、判断責任が重要である。

確認項目は次である。

```text
判断材料と判断そのものが分かれているか。
前提条件が明示されているか。
選択肢が比較されているか。
反対材料やリスクがあるか。
やめる選択肢があるか。
判断後に検証する指標があるか。
```

### コード、SQL、コマンド、設定

コード、SQL、shell、設定変更では、文章レビューだけでは足りない。実行前にテスト、差分確認、権限確認、環境分離が必要である。

確認項目は次である。

```text
対象環境は本番ではないか。
読み取りだけか、書き込みがあるか。
削除、更新、権限変更が含まれていないか。
入力値の検証があるか。
ロールバック可能か。
テスト環境で動作確認したか。
レビュー済みPRまたは変更申請になっているか。
```

AI出力のコードやコマンドを、未検証で実行してはいけない。これは第9章のセキュリティでも再度扱う。

## 実務ケース1: 競合比較表のレビュー

### 状況

事業企画担当者が、法人向けバックアップサービスの価格改定を検討している。第2章でAI適用対象として選び、第3章で Request Contract を作り、第4章で Context Pack を作り、第5章で次の Task Brief に分解した。

```text
Task Brief：主要競合5社の価格体系と主要機能を比較表にする。
AIに任せる作業：公開情報をもとに比較表を作る。
AIに任せない判断：価格改定の採否、改定幅、競合優劣の最終判断。
出力：Markdown表。
受け入れ条件：各項目に情報源または要確認ラベルを付ける。
停止条件：価格情報が不明、または法人向けプランと個人向けプランが分離できない場合。
```

AIは、競合5社の比較表を出した。レビューでは、次を見る。

### レビュー観点

| 観点 | 確認内容 | よくある指摘 |
|---|---|---|
| 目的適合性 | 価格改定検討の材料になっているか | 一般的な企業紹介に寄っている |
| 事実正確性 | 価格、プラン、機能が正しいか | 個人向け価格を法人向け価格としている |
| 根拠 | 情報源があるか | 公式価格ページではなく二次記事に依存 |
| 比較条件 | 月額、年額、税込、税抜が揃っているか | 価格単位が混在している |
| 不確実性 | 不明な価格を推測していないか | 「要問い合わせ」を推測で埋めている |
| 判断境界 | 価格改定の結論を出していないか | 「当社は20%値下げすべき」と書いている |

### 判定例

```text
最終判断：差し戻し
理由：価格情報の根拠が不足し、法人向けと個人向けが混在している。価格改定判断にも踏み込んでいる。
修正指示：
- 公式情報で確認できる項目と要確認項目を分ける。
- 価格単位を月額 / 年額 / 税込 / 税抜で明記する。
- 価格改定の結論は削除し、論点メモに留める。
- 情報源が確認できない項目は「要確認」とする。
```

このケースでは、表の見た目を直すだけでは不十分である。根拠と対象範囲に問題があるため、差し戻しが妥当である。

## 実務ケース2: 顧客返信案のレビュー

### 状況

サポート担当者が、障害に関する顧客問い合わせへの返信案をAIに作らせた。Context Packには、障害の発生時刻、復旧時刻、暫定原因、社外向けに公開してよい情報、未確定事項が含まれている。

AI出力には、次の問題が含まれていた。

```text
「弊社の設定ミスが原因でした」と断定している。
「再発は絶対にありません」と書いている。
内部の監視システム名を記載している。
SLA補償について「対象になる可能性があります」と書いている。
```

### レビュー結果

| 問題 | Severity | 理由 | 判定 |
|---|---|---|---|
| 原因未確定なのに断定 | Critical | 顧客説明として誤る可能性が高い | 差し戻し |
| 再発しないと断定 | Critical | 過剰約束 | 差し戻し |
| 内部監視システム名を記載 | Major | 社内情報露出 | 修正必須 |
| SLA補償に言及 | Blocker | 契約・法務判断に該当 | エスカレーション |

この出力は、文章としては丁寧でも、そのまま送信できない。判定は「エスカレーション」である。法務または顧客責任者へ確認し、返信案は作り直す。

### 修正方針

```text
- 原因は「現在調査中」とする。
- 再発防止は「暫定対策」「恒久対策を検討中」に分ける。
- 内部システム名は削除する。
- SLA補償には触れず、契約担当から別途連絡する扱いにする。
- 送信前にサポート責任者が承認する。
```

このケースは、第7章のHITL承認フローに接続する。

## 実務ケース3: 障害報告書のレビュー

### 状況

第5章で分解した障害報告書作成タスクのうち、AIがタイムライン候補と影響範囲候補を作った。

AI出力は次のような表である。

```text
09:01 障害発生
09:05 監視アラート検知
09:12 一次対応開始
09:30 復旧
```

レビューでは、まず「09:01 障害発生」が本当に確定情報かを確認する。実際には、09:01は最初のエラーログ時刻であり、ユーザー影響開始時刻ではない可能性がある。この場合、「障害発生」と断定してはいけない。

修正後は、次のようにする。

```text
09:01 最初の関連エラーログを確認（情報源：監視ログ、確度：高）
09:05 監視アラート検知（情報源：監視アラート、確度：高）
09:12 一次対応開始（情報源：インシデントチケット、確度：高）
09:30 復旧確認（情報源：監視ログ、確度：要確認）
要確認：ユーザー影響開始時刻、復旧完了時刻、影響対象テナント
```

AIが作るタイムラインは、正式報告書ではなく候補である。レビューでは、時刻、意味、情報源、確度を分ける。

## 実務ケース4: 提案書ドラフトのレビュー

### 状況

新しい法人向けバックアップオプションの社内提案書をAIに作らせた。AIは、機能案、価格案、販売計画、開発計画をまとめた。

このとき、レビューでは次の境界を確認する。

```text
AIが整理してよいもの：
- 顧客課題
- 既存問い合わせ傾向
- 機能候補
- 競合比較
- 論点
- リスク
- 選択肢

AIが決めてはいけないもの：
- 価格
- 開発優先順位
- 事業化可否
- 顧客への発表時期
- 契約条件
```

AI出力に「価格は月額X円が適切」「Q3にリリースすべき」といった断定があれば、判断境界の逸脱である。レビューでは、「判断材料」と「判断」を分ける。

判断材料として使える表現は次である。

```text
価格検討に必要な論点は、原価、競合価格、既存プランとの整合、営業上の説明容易性である。
リリース時期の判断には、開発工数、既存ロードマップ、顧客要望の緊急度、サポート体制が必要である。
```

判断そのものは、管理職または経営側で行う。この接続は第11章で扱う。

## Review Issue Log

レビューで見つけた問題は、Review Issue Log に残す。これは、AI出力にコメントを入れるだけでは不足するためである。

Review Issue Log は、次の用途に使う。

```text
AIへの修正依頼を明確にする。
人間レビューの証跡を残す。
同じ失敗が繰り返されていないか見る。
Request Contract、Context Pack、Task Briefの改善点を見つける。
研修教材や失敗事例集へ転用する。
評価ケースを作る。
```

基本形は次である。

```text
レビュー対象：
レビュー日：
レビュー担当：

| ID | 対象箇所 | 問題種別 | Severity | 内容 | 修正方針 | 改善先 |
|---|---|---|---|---|---|---|
```

「改善先」には、次のような値を入れる。

```text
Request Contract
Context Pack
Task Brief
Checkpoint Plan
Stop / Escalation Rules
権限マトリクス
Evaluation Plan
研修教材
```

この列が重要である。レビューで見つけた問題を、次回の仕組み改善へ接続するためである。

## Evaluation Plan

レビューは個別出力を見る。評価は仕組みを見る。AI活用を標準業務にするには、評価が必要である。

たとえば、顧客問い合わせ回答案をAIで作る場合、1件の問い合わせでうまくいっても十分ではない。次のようなケースで評価する必要がある。

```text
単純なFAQで答えられる問い合わせ。
契約条件に関わる問い合わせ。
障害中の問い合わせ。
顧客が怒っている問い合わせ。
古いFAQに基づくと誤る問い合わせ。
個人情報を含む問い合わせ。
プロンプトインジェクションを含む問い合わせ。
複数製品が混ざった問い合わせ。
```

Evaluation Plan では、次を定義する。

```text
評価対象：どのAI活用手順を評価するか。
評価目的：何を確認するか。
評価ケース：どの入力で試すか。
期待結果：望ましい出力は何か。
禁止結果：出てはいけない出力は何か。
評価観点：事実、根拠、リスク、承認、表現など。
合格基準：どの水準なら運用可能か。
評価頻度：いつ再評価するか。
責任者：誰が評価し、誰が承認するか。
```

### 評価ケースの種類

評価ケースは、正常系だけでは不足する。最低限、次を含める。

| 種類 | 内容 | 目的 |
|---|---|---|
| 正常系 | 想定どおりの入力 | 基本性能を見る |
| 境界条件 | 情報が少ない、曖昧、複数解釈 | 不明点処理を見る |
| 異常系 | 矛盾、欠落、古い情報 | 停止・要確認を見る |
| 高リスク系 | 契約、顧客、セキュリティ、本番影響 | エスカレーションを見る |
| 悪意ある入力 | 外部命令、誘導、情報抽出 | 防御を見る |
| 回帰ケース | 過去に失敗した入力 | 改善が維持されるか見る |

Evaluation Plan は、AIエージェントを業務に組み込む前に作る。業務投入後は、モデル、プロンプト、Context Pack、ツール、社内ルールが変わるたびに再評価する。

## 評価指標

評価では、単純な正解率だけでは足りない。AI活用の種類に応じて、複数の指標を見る。

| 指標 | 内容 | 使いどころ |
|---|---|---|
| 目的適合率 | 依頼目的に合う出力の割合 | ほぼすべての用途 |
| 事実誤り率 | 重要事実の誤り数 | 調査、FAQ、報告書 |
| 根拠不足率 | 根拠なし主張の割合 | 調査、判断メモ |
| 要確認分離率 | 不確実性を分けられた割合 | 障害、競合調査、法務周辺 |
| エスカレーション適合率 | 人間へ渡すべきケースで渡せた割合 | 顧客対応、セキュリティ |
| 禁止出力発生率 | 出してはいけない情報や判断の発生率 | 全用途、特に高リスク業務 |
| レビュー工数 | 人間が修正に使った時間 | ROI評価 |
| 再生成率 | 何度作り直したか | 依頼設計・Context Packの品質確認 |
| 採用率 | 修正後も含め採用された割合 | 業務適用可否の判断 |

評価指標は多ければよいわけではない。業務目的に合う少数の指標を選ぶ。顧客返信案なら、禁止出力発生率、エスカレーション適合率、レビュー工数が重要である。競合調査なら、事実誤り率、根拠不足率、要確認分離率が重要である。

## 回帰評価

AI活用は一度作って終わりではない。次の変更があるたびに、品質が変わる可能性がある。

```text
モデルが変わる。
プロンプトが変わる。
Context Packが変わる。
社内WikiやFAQが更新される。
ツールの仕様が変わる。
権限が変わる。
社内規程が変わる。
対象業務が変わる。
```

回帰評価では、過去に重要だった評価ケースを再実行する。特に、過去に失敗したケースは残しておく。失敗ケースは、品質を守るための資産である。

回帰評価で確認することは次である。

```text
以前できていたことが壊れていないか。
禁止出力が再発していないか。
エスカレーション条件が維持されているか。
古い情報を使うようになっていないか。
レビュー工数が増えていないか。
```

回帰評価は、エンジニアだけのものではない。業務部門でも、重要なAI活用手順については、代表ケースを残して再評価する必要がある。

## レビュー結果をAIへ戻す

AI出力の問題を修正する場合、単に「もっと正確に」「詳しく」と言ってはいけない。修正指示は、レビュー結果に基づいて具体化する。

悪い修正指示は次である。

```text
もっとちゃんと調べて。
全体的に直して。
分かりやすくして。
リスクも考えて。
```

良い修正指示は次である。

```text
以下のReview Issue Logに基づいて修正してください。
- ID R-01: 価格情報について、法人向けプランと個人向けプランが混在しています。法人向けに限定し、確認できない項目は「要確認」としてください。
- ID R-02: 価格改定の結論を削除してください。このタスクでは判断材料の整理に留めます。
- ID R-03: 価格単位を「月額 / 年額 / 税込 / 税抜」で明示してください。

出力形式は同じMarkdown表とし、最後に「要確認事項」を箇条書きで付けてください。
```

修正依頼にも、Request Contract の考え方を使う。レビューと再依頼はつながっている。

## レビューを軽量化する

レビューは重要だが、重すぎると使われない。現場で使うには、軽量版を用意する。

### 3分レビュー

個人メモや社内下書きでは、次の3点だけ見る。

```text
目的に合っているか。
明らかな事実誤りがないか。
そのまま外部に出ないか。
```

### 10分レビュー

チーム共有や小さな業務成果物では、次を見る。

```text
目的適合性
事実正確性
根拠
不確実性
重要リスク
次アクション
```

### 30分レビュー

顧客影響、意思決定、標準化候補では、レビュー表を使う。

```text
Request Contract
Context Pack
Task Brief
AI出力
Review Issue Log
最終判定
改善先
```

レビュー標準化の目的は、現場を遅くすることではない。リスクに応じて、適切な強度のレビューを選べるようにすることである。

## 演習

次のAI出力をレビューする。

### 前提

```text
業務：新サービスの社内提案書作成
目的：法人向けバックアップオプションの事業化可否を検討するため、判断材料を整理する。
AIに任せてよいこと：顧客課題、競合比較、機能候補、論点、リスクの整理。
AIに任せないこと：価格決定、開発優先順位、事業化判断、顧客向け発表。
利用先：社内の事業検討会議。
```

### AI出力サンプル

```text
新オプションは月額9,800円で提供すべきです。
競合A社は同等機能を月額12,000円で提供しているため、当社は価格優位性を出せます。
Q3にリリースすれば市場機会を逃しません。
営業部門からの要望も強く、開発優先度は最上位にすべきです。
顧客には来月のニュースレターで先行告知できます。
セキュリティ面では一般的な暗号化を使えば問題ありません。
```

### 作成するもの

```text
1. AI出力レビュー表
2. Review Issue Log
3. 最終判断
4. 修正指示
5. 改善すべき前工程（Request Contract / Context Pack / Task Brief など）
```

### 演習の判定基準

次の条件を満たしていれば、この章の演習としては合格である。

- 価格決定、リリース時期、開発優先順位、顧客告知が人間判断領域であると指摘できる
- 競合A社の価格情報に根拠確認が必要であると指摘できる
- 「一般的な暗号化を使えば問題ない」がセキュリティレビュー不足であると指摘できる
- 少なくとも1件以上の Critical または Blocker を記録できる
- 最終判断を「差し戻し」または「エスカレーション」とし、理由を説明できる
- 修正指示が具体的である
- 改善先として、Request Contract、Task Brief、Stop / Escalation Rules のいずれかを挙げられる

## 章の受け入れ基準

この章を読み終えた時点で、読者は次をできる必要がある。

- AI出力、レビュー済み出力、承認済み成果物を区別できる
- レビュー、評価、検証、承認を区別できる
- 利用先とリスクに応じてレビュー強度を決められる
- Request Contract、Context Pack、Task Brief を使ってレビュー基準を確認できる
- 目的、事実、根拠、論理、網羅性、実行可能性、リスク、不確実性の観点でレビューできる
- Review Issue Log に問題を Severity 付きで記録できる
- 採用、修正後採用、差し戻し、不採用、エスカレーションを判定できる
- 個別レビューと仕組み評価の違いを説明できる
- Evaluation Plan の基本項目を設計できる
- レビュー結果を Request Contract、Context Pack、Task Brief、権限設計、評価ケースへ戻せる

評価ケースや回帰評価を本格運用する場合は、対象業務のリスク分類、データ保持、ログ保存、アクセス制御、監査要件を確認する必要がある。第7章以降では、レビュー結果を権限、承認、ログ、ガバナンスへ接続する。

## まとめ

AI出力は、レビューされるまで成果物ではない。文章が自然でも、表が整っていても、事実、根拠、論理、リスク、不確実性が確認されていなければ、業務成果物として扱ってはいけない。

レビューでは、Request Contract、Context Pack、Task Brief、AI出力、利用先をセットで見る。出力を、目的適合性、事実正確性、根拠、論理、網羅性、実行可能性、リスク、不確実性の観点で確認し、採用、修正後採用、差し戻し、不採用、エスカレーションを判定する。

評価では、個別出力ではなく、AI活用の仕組み全体を見る。複数ケース、異常系、高リスク系、過去の失敗ケースを使い、安定して望ましい品質が出るかを確認する。

次章では、レビュー結果を踏まえて、AIがどのツールをどこまで使ってよいか、どこで人間承認を入れるかを扱う。レビューで「危険」と分かったものは、権限とHITLで制御する必要がある。

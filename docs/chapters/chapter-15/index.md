# 第15章 スキルマップと評価

## この章を5分で使う場合

- まず作るもの: 対象ロールを1つ選び、Role Skill Profileで必要能力、目標レベル、評価証拠を仮置きする。
- 使うテンプレート: Skill Map、Role Skill Profile、Skill Diagnosis Form、Evidence Portfolio、Learning Gap Handoff。評価基準の詳細は [付録C: スキル評価ルーブリック](../../appendices/appendix-c/) を参照する。
- 読まなくてよい節: 今日の目的が能力定義と評価判定だけなら、教材体系、Office Hour、KPI、90日展開計画の設計は [第16章](../chapter-16/) に渡す。
- 管理職が確認すべき判断: この評価を人事査定ではなく能力開発とリスク管理として扱えているか。高リスク業務に必要な到達レベル、評価証拠、再評価条件が明確か。
- 次章または付録へ渡すもの: Learning Gap HandoffとCertification Decision Recordを第16章へ渡す。詳細ルーブリックは付録C、診断・認定問題は [付録G](../../appendices/appendix-g/) を使う。

## この章で扱う能力

この章では、AIエージェント協働に必要な能力を、個人の感覚や自己申告ではなく、観察可能な行動、成果物、実技課題、運用証跡で評価できる形にする。

第1章から第14章までは、AIエージェント協働を成立させるための設計要素を扱った。

- どの仕事にAIを使うか。
- AIに何を依頼するか。
- どの前提情報を渡すか。
- どの粒度で委任するか。
- 出力をどうレビューするか。
- ツール権限と人間承認をどう設計するか。
- ログと評価でどう改善するか。
- セキュリティ、プライバシー、ガバナンスをどう組み込むか。
- 人間が担う判断、対話、実験、行動をどう止めないか。

本章では、これらを **スキルマップ、レベル定義、評価ルーブリック、実技課題、評価証拠、学習不足の引き渡し条件** に変換する。

読者は本章を通じて、次のことができるようになる。

- AIエージェント協働能力を12領域に分けて説明できる。
- 職種ごとに必要な到達レベルを定義できる。
- Lv1〜Lv6の評価基準を、行動と成果物で判定できる。
- 自己診断、実技課題、レビュー証跡を組み合わせて評価できる。
- 診断結果を、第16章で扱う研修、OJT、Office Hour、認定運用へ渡すための条件に変換できる。
- AI活用能力評価を、人事評価ではなく能力開発とリスク管理に接続できる。

この章の目的は、社員に点数を付けることではない。目的は、組織がどの能力を持っていて、どの能力が不足し、どの不足が業務リスクや導入停滞を生んでいるかを見えるようにすることである。

## 第16章との責務境界

第15章と第16章は、どちらも人材育成に関係する。しかし、責務は分ける。

| 項目 | 第15章で扱うこと | 第16章で扱うこと |
|---|---|---|
| 主目的 | 能力を定義し、評価する | 評価結果をもとに、組織へ展開する |
| 主語 | 個人、ロール、評価者 | 部門、AI CoE、管理職、経営層 |
| 成果物 | Skill Map、Role Skill Profile、Evidence Portfolio、Learning Gap Handoff、Certification Decision Record | Role-based Learning Path、Curriculum Blueprint、Office Hour Runbook、KPI Dashboard、90-Day Rollout Plan |
| 認定 | 合格基準、評価証拠、判定記録を定義する | 認定をいつ導入し、どの権限・研修・運用へ接続するかを決める |
| 学習 | 不足能力と再評価条件を明確にする | 教材、ワークショップ、OJT、Office Hourとして提供する |
| KPI | 評価証拠と認定判定に必要なデータを定義する | 経営・現場・CoEが見る運用ダッシュボードを設計する |

本章では、ロール別の詳細カリキュラムや90日展開計画は作らない。それらは第16章の責務である。本章で作るのは、展開側が教材や支援を設計するための入力である。

## なぜスキルマップが必要か

AI活用が個人任せになると、組織内に次のようなばらつきが出る。

| 状態 | 起きる問題 |
|---|---|
| 一部の人だけが高頻度に使う | 成果が属人化し、再現できない |
| 便利ツールとしてだけ使う | 要約、下書きで止まり、業務プロセスが変わらない |
| プロンプト集だけが共有される | 文脈が違う業務では品質が再現しない |
| リスク判断が個人任せになる | 機密入力、誤送信、ツール誤用が起きる |
| 管理職が評価できない | AI成果物のレビュー基準が曖昧になる |
| エンジニアだけが詳しい | 業務部門側の課題設計やレビューが弱くなる |
| 経営層が効果を測れない | 投資判断、教育投資、ガバナンス設計が感覚になる |

AIエージェント協働は、プロンプトの上手さだけでは成立しない。仕事選定、コンテキスト設計、委任、レビュー、権限、ログ、セキュリティ、ガバナンス、判断、対話、実験、行動変容がつながって初めて、組織能力になる。

そのためには、次の3つを分ける必要がある。

| 区分 | 説明 |
|---|---|
| 能力 | 何ができるべきか |
| 証拠 | できることを何で確認するか |
| 学習 | 不足している場合に何を学ぶか |

スキルマップは、この3つを接続するための設計図である。

## 評価してはいけないもの

最初に、評価してはいけないものを明確にする。

| 評価してはいけないもの | 理由 |
|---|---|
| AI利用回数だけ | 回数が多くても、業務成果や安全性につながるとは限らない |
| 長いプロンプトを書けること | 長い指示は、構造化された良い依頼とは限らない |
| 特定ツールへの慣れ | ツールは変わる。評価すべきは原理と成果物である |
| AIへの好意的態度 | 好きかどうかではなく、安全に成果を出せるかを見る |
| 自信の強さ | 自信が高い人ほどリスクを見落とす場合がある |
| 最新モデル名の知識 | 変化が速く、能力評価の中核にはならない |
| 生成物の見た目 | 整った文章でも、事実やリスクが誤っていることがある |
| 失敗ゼロ | 実験文化を殺す。評価すべきは失敗から学ぶ設計である |

AI活用能力評価で見るべきなのは、AIを使ったかどうかではない。

見るべきなのは、次である。

- 適切な仕事を選べたか。
- 必要な前提を渡せたか。
- 成果物を定義できたか。
- 出力を検証できたか。
- 危険な操作を止められたか。
- 判断責任を人間側に残せたか。
- 失敗を学習に変えられたか。
- チームの標準として再現できるか。

## 評価原則

AIエージェント協働能力の評価では、次の原則を使う。

### 原則1: 成果物で評価する

「使えます」と言えるかではなく、成果物を作れるかで見る。

| 能力 | 評価する成果物 |
|---|---|
| 指示設計 | Request Contract |
| コンテキスト設計 | Context Pack |
| 委任設計 | Task Breakdown、Checkpoint Plan |
| レビュー | AI出力レビュー表、Review Issue Log |
| 権限設計 | ツール権限マトリクス、HITL承認フロー |
| 観測 | Agent Run Log、Improvement Backlog |
| セキュリティ | AI Security Checklist、Prompt Injection Test Sheet |
| ガバナンス | AI System Inventory、AI Risk Register |
| 判断 | Decision Brief、Kill Decision Sheet |
| 対立処理 | Difficult Conversation Brief、Agreement Log |
| 実験 | AI実験設計シート、Failure Review Sheet |
| 行動変容 | Action Blocker診断、Minimum Action Contract |

成果物があると、評価者が同じ基準でレビューできる。本人も、どこが不足しているかを理解しやすい。

### 原則2: 自己診断だけで評価しない

自己診断は入口として有効である。しかし、自己診断だけで能力判定してはいけない。

理由は単純である。

- 初学者は、自分が何を知らないかを知らない。
- 過信している人は、リスクを低く見積もる。
- 慎重な人は、実力より低く自己評価する。
- 役職が高い人ほど、弱点を自己申告しにくい場合がある。

そのため、評価は次の組み合わせで行う。

| 評価手段 | 用途 |
|---|---|
| 自己診断 | 本人の認識を把握する |
| 知識テスト | 基本概念と社内ルールの理解を見る |
| 実技課題 | 実際に成果物を作れるかを見る |
| 成果物レビュー | 現場の成果物品質を見る |
| 運用証跡 | 実務で安全に使えているかを見る |
| 面談・振り返り | 判断理由、不確実性、学習内容を見る |

### 原則3: 職種別に評価する

全員に同じ到達レベルを求める必要はない。

たとえば、全社員にRAGの実装能力は不要である。一方、エンジニアやAI推進担当には、ログ、評価、ツール権限、セキュリティの設計能力が必要になる。

営業担当者に求めるべきことは、顧客情報を安全に扱い、提案書や調査を適切にレビューし、顧客に見せる前にリスクを判断できることである。GitHub AgentOpsの詳細までは不要である。

評価は、職種ごとの役割とリスクに合わせる。

### 原則4: 高リスク業務では評価基準を上げる

同じAI利用でも、リスクが違えば必要スキルは変わる。

| 利用例 | 必要レベル |
|---|---|
| 社内メモの要約 | 低い |
| 顧客返信案の作成 | 中程度 |
| 顧客へ自動送信 | 高い |
| 契約書レビュー補助 | 高い |
| 本番環境変更 | 非常に高い |
| 採用・人事評価・与信・医療判断 | 専門統制が必要 |

スキル評価は、業務リスクとセットで設計する。

### 原則5: 人事評価と能力開発を分ける

AI活用能力の評価は、人材育成、リスク管理、権限付与、研修設計に使うべきである。

そのまま人事評価に直結させると、次の問題が起きる。

- 失敗を隠す。
- 実験数が減る。
- リスクを正直に報告しなくなる。
- AI利用ログが監視として受け止められる。
- 自己診断が歪む。

特に初期導入では、評価は「能力開発」と「安全な権限付与」のために使う。報酬や査定に使う場合は、社内規程、労務、人事、法務、プライバシーの確認が必要である。

### 原則6: 評価者を校正する

同じ成果物でも、評価者によって判定がぶれる。

そのため、以下を用意する。

- 模範解答。
- NG例。
- 採点基準。
- 判定例。
- 複数評価者によるサンプル採点。
- 判定差分のすり合わせ。

評価者校正をしないルーブリックは、制度として使えない。

## 12能力領域

本書のスキルマップは、次の12能力領域で構成する。

| No | 能力領域 | 対応章 | 目的 | 主な証拠 |
|---:|---|---|---|---|
| 1 | AIエージェント基礎理解 | 第1章 | チャットAI、ワークフロー、エージェント、RAG、ツール、メモリの違いを理解する | 用語テスト、協働ブループリント |
| 2 | 課題・ユースケース設計 | 第2章 | AIに任せる仕事、任せない仕事を選ぶ | AI適用判断シート |
| 3 | 指示・依頼設計 | 第3章 | Request Contractを作る | Request Contract、Acceptance Criteria |
| 4 | コンテキスト設計 | 第4章 | 前提、鮮度、不確実性、機密区分を整理する | Context Pack |
| 5 | タスク分解・委任設計 | 第5章 | サブタスク、チェックポイント、停止条件を設計する | Task Breakdown、Checkpoint Plan |
| 6 | 検証・評価・レビュー | 第6章 | AI出力を受け入れ条件に照らして判定する | AI出力レビュー表、Review Issue Log |
| 7 | ツール・権限・HITL設計 | 第7章 | 操作単位で権限と承認を設計する | ツール権限マトリクス、HITL承認フロー |
| 8 | ログ・観測・継続改善 | 第8章 | 実行履歴、失敗、改善をRun単位で残す | Agent Run Log、Improvement Backlog |
| 9 | セキュリティ・プライバシー | 第9章 | 情報漏洩、外部入力、過剰権限、ログリスクを制御する | AI Security Checklist、Prompt Injection Test Sheet |
| 10 | ガバナンス・統制 | 第10章 | 台帳、リスク分類、承認、教育、監査、廃止を運用する | AI System Inventory、AI Risk Register |
| 11 | 判断・対話・実験・行動 | 第11〜14章 | 判断、対立処理、失敗耐性、行動停止を扱う | Decision Brief、Connection Debt診断、AI実験設計シート、Action Blocker診断 |
| 12 | 組織展開・人材育成 | 第16章 | 教材、認定、KPI、Office Hour、CoEを運用する | ロール別学習パス、90日展開計画 |

ここで重要なのは、能力領域を「AIに詳しいかどうか」で分けていないことである。

能力領域は、AIエージェント協働ループに対応している。

```text
仕事を選ぶ
→ 依頼を契約にする
→ Context Packを渡す
→ タスクを分解して委任する
→ 出力をレビューする
→ 権限と承認で制御する
→ ログから改善する
→ セキュリティと統制で守る
→ 人間が判断し、対話し、実験し、行動する
→ 組織展開へつなぐ
```

## 6層モデルと評価対象

第0章で導入した6層モデルに対応させると、評価対象は次のようになる。

| 層 | 評価する能力 | 主な評価課題 |
|---|---|---|
| Work | AIに使う仕事を選ぶ | 業務をAI適用判断シートで分類する |
| Request | AIへの依頼を設計する | 曖昧依頼をRequest Contractへ変換する |
| Context | 前提情報を設計する | 資料群からContext Packを作る |
| Delegation | タスクを委任する | 大きな仕事をTask Breakdownへ分解する |
| Control | レビュー、権限、ログ、セキュリティを設計する | 出力レビュー、権限設計、Run Log、Security Checklistを作る |
| Human / Org | 判断、対話、実験、行動、組織展開を設計する | Decision Brief、Connection Debt診断、AI実験設計、学習パスを作る |

この対応関係を使うと、組織は「どこで詰まっているか」を把握しやすい。

たとえば、AI出力の品質が低い場合、問題はAIモデルではなく、Request層やContext層にあるかもしれない。AI活用が広がらない場合、Human / Org層、つまり判断回避、失敗回避、評価不安、管理職レビュー不足に原因があるかもしれない。

## レベル定義

本書では、AIエージェント協働能力をLv1〜Lv6で定義する。

| レベル | 名称 | 状態 | 評価の中心 |
|---:|---|---|---|
| Lv1 | AI利用者 | 基本ルールを守り、要約、下書き、整理に使える | 安全な基本利用 |
| Lv2 | 指示設計者 | 目的、前提、制約、出力形式を指定できる | Request ContractとContextの基礎 |
| Lv3 | 業務適用者 | 自分の業務にAIを組み込み、出力をレビューできる | 業務適用とレビュー |
| Lv4 | エージェント運用者 | ツール、権限、ログ、評価、承認を含めて運用できる | Control層の設計 |
| Lv5 | チーム変革者 | チーム単位でユースケース、KPI、実験文化を設計できる | チーム運用と改善 |
| Lv6 | 全社設計者 | ガバナンス、投資判断、AI台帳、リスク管理、人材育成を設計できる | 全社展開と統制 |

レベルは、職位ではない。

若手でも特定領域でLv4相当の実務能力を持つことがある。経営層でも、Request Contract作成はLv2でも十分な場合がある。一方、経営層には全社設計、リスク許容度、投資判断に関するLv5〜Lv6相当の能力が必要になる。

したがって、レベルは一つの総合点ではなく、領域ごとに判定する。

```text
悪い例：
この人はAIレベル3である。

良い例：
この人は、Request設計はLv3、レビューはLv2、セキュリティはLv1、実験設計はLv3である。
```

## Lv1: AI利用者

Lv1は、安全な基本利用ができる状態である。

| 観点 | できること |
|---|---|
| 基礎理解 | AIの出力が常に正しいわけではないと理解している |
| ルール | 入力禁止情報、社外送信前レビュー、禁止用途を理解している |
| 利用 | 要約、下書き、表現修正、アイデア出しに使える |
| レビュー | 明らかな誤りや違和感を見つけられる |
| セキュリティ | 顧客情報、認証情報、機密情報を不用意に入力しない |
| 人間判断 | AIの回答をそのまま最終判断にしない |

### Lv1の合格証拠

- AI利用ポリシーの確認テストに合格している。
- 入力してよい情報といけない情報を分類できる。
- 簡単な依頼を安全に出せる。
- AI出力をそのまま顧客・外部へ出さない。

Lv1は全社員に求める最低ラインである。

## Lv2: 指示設計者

Lv2は、AIに作業を依頼するための最低限の仕様化ができる状態である。

| 観点 | できること |
|---|---|
| 目的 | 何のための作業かを書ける |
| 成果物 | 表、文章、箇条書き、JSON、メール案などの出力形式を指定できる |
| 前提 | 対象範囲、制約、利用してよい情報を指定できる |
| 不明点 | 不明な場合の扱いを指定できる |
| 品質 | 受け入れ条件を簡単に書ける |
| Context | 必要な背景を整理して渡せる |

### Lv2の合格証拠

- 曖昧な依頼文をRequest Contractに変換できる。
- 簡易Context Packを作れる。
- 受け入れ条件を3〜5個書ける。
- AI出力に対して、修正指示を具体的に出せる。

Lv2は、AIを業務で日常利用する人の標準ラインである。

## Lv3: 業務適用者

Lv3は、自分の業務にAIを組み込み、成果物をレビューできる状態である。

| 観点 | できること |
|---|---|
| 仕事選定 | 自分の業務をAI適用判断シートで分類できる |
| 委任 | 大きな仕事をサブタスクへ分解できる |
| レビュー | AI出力レビュー表で採用、差し戻し、不採用を判定できる |
| 実験 | 小さなAI活用実験を設計できる |
| 判断 | AI提案の採否理由を説明できる |
| 改善 | 失敗から次回のRequest、Context、レビュー条件を改善できる |

### Lv3の合格証拠

- 自分の実業務からAI適用候補を3件以上出せる。
- そのうち1件について、AI適用判断シート、Request Contract、Context Pack、AI出力レビュー表を作れる。
- AI出力の問題点をSeverity付きで記録できる。
- 小実験の成功条件、失敗条件、やめる条件を定義できる。

Lv3は、業務担当者にとっての実用ラインである。

## Lv4: エージェント運用者

Lv4は、AIエージェントをツール、権限、ログ、評価、承認を含む運用として設計できる状態である。

| 観点 | できること |
|---|---|
| ツール | ツール単位ではなく操作単位で権限を分類できる |
| HITL | 人間承認が必要な操作を設計できる |
| ログ | Agent Run Log、Tool Execution Log、Approval Logを設計できる |
| 評価 | Evaluation Plan、評価ケース、回帰評価を設計できる |
| セキュリティ | Prompt Injection Test、Data Classification、外部ツール審査ができる |
| 運用 | 失敗をImprovement Backlogへ接続できる |

### Lv4の合格証拠

- ツール権限マトリクスを作れる。
- Tool Spec Cardを作れる。
- HITL承認フローとPermission Testを設計できる。
- Agent Run LogとImprovement Backlogを設計できる。
- Security ChecklistとPrompt Injection Test Sheetを作れる。

Lv4は、エンジニア、AI推進担当、業務システム責任者、情報システム、セキュリティ担当に必要なラインである。

## Lv5: チーム変革者

Lv5は、個人のAI利用ではなく、チーム単位でAI活用を運用し、改善できる状態である。

| 観点 | できること |
|---|---|
| チーム方針 | チームのAI利用方針を作れる |
| レビュー運用 | チーム内のAI成果物レビュー基準を設計できる |
| 実験文化 | 実験数、学習ログ、廃棄判断を運用できる |
| 対立処理 | AI活用に関する未処理の対立を論点化できる |
| KPI | 時間削減、品質、リスク、実験、学習の指標を設計できる |
| 育成 | チームメンバーの学習パスを設計できる |

### Lv5の合格証拠

- チームのAI活用ポートフォリオを作れる。
- チーム標準のRequest Contract、Context Pack、レビュー表を整備できる。
- チームのAI活用KPIを設計できる。
- 失敗レビュー会を運用できる。
- Connection DebtやAction Blockerを、チームの運用改善へ接続できる。

Lv5は、マネージャー、リーダー、AI推進担当に必要なラインである。

## Lv6: 全社設計者

Lv6は、全社のAI活用を、ガバナンス、投資、人材育成、リスク管理、組織設計へ接続できる状態である。

| 観点 | できること |
|---|---|
| 全社方針 | AI利用ポリシー、リスク分類、承認方針を設計できる |
| 台帳 | AI System Inventoryを運用できる |
| 投資判断 | どの部門・業務へ投資するか判断できる |
| ガバナンス | AI Risk Register、Audit Evidence、Incident Runbookを運用できる |
| 人材育成 | 職種別スキルマップ、認定、研修体系を設計できる |
| 組織設計 | AI CoE、部門責任者、Office Hour、教材更新を運用できる |

### Lv6の合格証拠

- 全社AI活用ロードマップを作れる。
- AIリスク分類と承認フローを設計できる。
- AI System Inventoryを使った棚卸しを設計できる。
- AI活用能力の職種別到達基準を定義できる。
- 研修、認定、KPI、ガバナンスを接続できる。

Lv6は、経営層、AI CoE責任者、部門長、セキュリティ・法務・情シス責任者に必要なラインである。

| ログ改善 | 利用履歴を残す必要を理解している | Run単位で失敗と改善を記録できる | チームの改善バックログと評価ケースを運用できる |
| セキュリティ | 入力禁止情報を理解している | AI利用前にリスク分類できる | Agentic AI固有リスクを統制へ組み込める |
| ガバナンス | 社内ルールを守れる | 部門のAI利用を台帳化できる | 全社AIガバナンスを設計できる |
| 人間側能力 | AIの回答を参考にできる | 判断、対話、実験、行動停止を成果物化できる | チームの判断・対立・実験文化を改善できる |
| 組織展開 | 個人で使える | チームへ共有できる | 教材、認定、Office Hourを運用できる |

## 職種別の到達目標

すべての職種が、すべての領域で高レベルになる必要はない。職種ごとに重点を変える。

## 全社員

| 領域 | 目標レベル | 理由 |
|---|---:|---|
| 基礎理解 | Lv1〜Lv2 | AI出力の限界と社内ルールを理解するため |
| 依頼設計 | Lv2 | 日常業務で安全にAIを使うため |
| コンテキスト | Lv2 | 必要な前提を渡すため |
| レビュー | Lv1〜Lv2 | 明らかな誤りとリスクを見つけるため |
| セキュリティ | Lv1〜Lv2 | 入力禁止情報と外部送信リスクを避けるため |
| 人間側能力 | Lv1 | AIを最終判断にしないため |

全社員に求めるのは、AIを高度に使うことではなく、安全に使うことである。

## 業務担当者

| 領域 | 目標レベル | 理由 |
|---|---:|---|
| 課題設計 | Lv3 | 自業務でAIを使う場所を見つけるため |
| 依頼設計 | Lv3 | 業務成果物を作るため |
| コンテキスト | Lv3 | 業務前提を正しく渡すため |
| 委任 | Lv3 | 複雑な作業をサブタスク化するため |
| レビュー | Lv3 | 出力を業務成果物として採用できるか判断するため |
| 実験文化 | Lv3 | 小さなAI活用実験を回すため |

業務担当者は、AIを使って自分の業務を変える主体である。

## マネージャー

| 領域 | 目標レベル | 理由 |
|---|---:|---|
| 課題設計 | Lv4 | チーム内の優先順位を決めるため |
| レビュー | Lv4 | 部下のAI成果物を評価するため |
| ツール権限 | Lv3〜Lv4 | 承認条件を判断するため |
| ガバナンス | Lv3〜Lv4 | 部門ルールと台帳を運用するため |
| 判断 | Lv4〜Lv5 | 採用、停止、展開、リスク受容を判断するため |
| 対立処理 | Lv4〜Lv5 | AI活用で生じる未合意を処理するため |
| 実験文化 | Lv4〜Lv5 | 実験数、失敗共有、廃棄判断を運用するため |
| 人材育成 | Lv4〜Lv5 | チームメンバーの学習パスを作るため |

マネージャーに必要なのは、自分がAIを使えることだけではない。チームのAI利用をレビューし、止めるべきものを止め、伸ばすべきものを伸ばすことである。

## エンジニア

| 領域 | 目標レベル | 理由 |
|---|---:|---|
| 基礎理解 | Lv3 | エージェント、RAG、ツール、メモリを技術的に理解するため |
| 委任 | Lv3〜Lv4 | 開発タスクをAIに分解・補助させるため |
| レビュー | Lv4 | コード、設計、テスト、運用出力を検証するため |

エンジニアは、AIを使うだけでなく、AIが安全に行動できる環境を設計する。

## AI推進担当 / CoE

| 領域 | 目標レベル | 理由 |
|---|---:|---|
| 全領域理解 | Lv4以上 | 部門横断で支援するため |
| ガバナンス | Lv5〜Lv6 | ポリシー、台帳、承認、監査を運用するため |

| KPI | Lv5 | 効果測定と改善サイクルを運用するため |

AI推進担当は、個別相談に応じるだけでは不十分である。相談から標準テンプレート、失敗事例、評価基準、教材更新へ戻す必要がある。

## 経営層

| 領域 | 目標レベル | 理由 |
|---|---:|---|
| 課題設計 | Lv4〜Lv5 | どの業務へ投資するか判断するため |
| 判断 | Lv5〜Lv6 | リスク許容度、撤退、拡大を判断するため |
| ガバナンス | Lv5〜Lv6 | 全社方針、責任分界、監査を設計するため |
| 組織展開 | Lv5〜Lv6 | 人材育成、CoE、評価制度を設計するため |
| 対立処理 | Lv4〜Lv5 | 部門間の責任・予算・優先順位対立を処理するため |
| 実験文化 | Lv4〜Lv5 | 失敗を許容しつつ、無責任な失敗を防ぐため |

経営層は、AIの細かな操作を覚える必要はない。しかし、何に投資し、どこまで許容し、何を止めるかを判断する責任がある。

## 評価証拠レベル

評価では、証拠の強さも見る。

| 証拠レベル | 名称 | 内容 | 信頼度 |
|---:|---|---|---:|
| EV0 | 証拠なし | 本人が「できる」と言っているだけ | 低い |
| EV1 | 自己診断 | チェック項目に回答している | 低〜中 |
| EV2 | 知識テスト | 用語、ルール、判断問題に回答できる | 中 |
| EV3 | 模擬成果物 | 演習ケースで成果物を作れる | 中 |
| EV4 | 実務成果物 | 実案件で成果物を作り、レビューを受けている | 高い |
| EV5 | 運用証跡 | 実運用でログ、承認、改善、再評価まで残っている | 高い |

Lv判定では、できるだけEV3以上を使う。

特にLv4以上は、自己診断や知識テストだけでは判定しない。ツール権限、ログ、セキュリティ、ガバナンスに関わるため、実技課題または実務成果物が必要である。

## Practical Assessment Pack

実技課題は、AIエージェント協働能力を評価する中心である。

実技課題は、次の5種類に分ける。

| 課題タイプ | 評価する能力 | 例 |
|---|---|---|
| 変換課題 | 曖昧な依頼を仕様化する | 「競合調査して」をRequest Contractに変換する |
| 選別課題 | 必要情報と不要情報を分ける | 資料群からContext Packを作る |
| 検出課題 | 誤りやリスクを見つける | AI出力レビュー表を作る |
| 設計課題 | 業務フローや権限を設計する | ツール権限マトリクス、HITL承認フローを作る |
| 判断課題 | 採用、棄却、実験化、停止を判断する | Decision BriefとKill Decision Sheetを作る |

実技課題は、業務に近いケースを使う。

一般論の問題だけでは、実務能力は測れない。

```text
- Review Issue Log：有 / 無
- ツール権限マトリクス：有 / 無
- Agent Run Log：有 / 無
- Security Checklist：有 / 無
- Decision Brief：有 / 無
- AI実験設計シート：有 / 無
- Learning Log：有 / 無

評価結果：
- 領域別レベル：
- 強み：
- 不足：
- リスク：
- 推奨学習：
- 次回評価日：
```

Evidence Portfolioには、機密情報や顧客情報をそのまま入れない。評価用には、マスキング済み成果物、架空ケース、または社内規程に従って許可された情報だけを使う。

## Learning Gap Handoff

評価結果は、そのまま研修メニューに変換しない。まず、不足能力を展開側へ渡せる形にする。

Learning Gap Handoffは、評価章である第15章から、展開章である第16章へ渡す引き継ぎ成果物である。これは詳細な学習パスではない。詳細な教材、ワークショップ、Office Hour、OJT計画は第16章で設計する。

```text
Learning Gap Handoff

対象者 / ロール：
評価日：
評価者：
対象業務：
扱う情報区分：
現在の認定 / 権限：

不足領域：
| 領域 | 現在レベル | 目標レベル | 評価証拠 | 不足内容 | 業務リスク | 再評価成果物 |
|---|---:|---:|---|---|---|---|
|  |  |  |  |  |  |  |

優先度：高 / 中 / 低
推奨する支援形式：e-learning / 演習 / OJT / Office Hour / レビュー会 / 管理職支援 / セキュリティ相談
第16章への引き渡し事項：
再評価期限：
```

この成果物の目的は、学習内容を細かく指定することではない。目的は、どの不足が、どの業務リスクに接続しており、どの成果物で再評価すればよいかを明確にすることである。

| 不足の種類 | 第15章で決めること | 第16章へ渡すこと |
|---|---|---|
| 知識不足 | どの概念が不足しているか | 対応する教材モジュール |
| 成果物不足 | どのテンプレートを作れないか | 演習・添削・OJTの設計要求 |
| レビュー不足 | どの誤りを検出できないか | レビュー会・模範解答の設計要求 |
| リスク判断不足 | どのリスク分類が弱いか | セキュリティ・法務・管理職支援 |
| 運用証跡不足 | 実務で使った証拠がないか | パイロット、Office Hour、再評価設計 |

Learning Gap Handoffに、全社員向け、管理職向け、エンジニア向けといった詳細な学習パスを書き込まない。そこまで書くと、第16章のRole-based Learning Pathと重複する。

## Certification Criteria

認定制度を作る場合、第15章では **認定の合格基準** と **判定に必要な証拠** を定義する。

認定の導入順序、受講案内、研修日程、権限付与、更新運用は第16章で扱う。本章では、認定を「受講完了」ではなく、能力証拠で判定できるようにする。

| 認定 | 主な対象 | 到達基準 | 必須証拠 |
|---|---|---|---|
| AI Safe User | 全社員 | Lv1相当。社内ルール、入力禁止情報、基本レビューを理解している | 確認テスト、低リスク出力レビュー |
| AI Work Designer | 業務担当者 | Lv2〜Lv3相当。Request Contract、Context Pack、レビュー表を作れる | Request Contract、Context Pack、AI出力レビュー表 |
| AI Agent Operator | エンジニア、業務システム担当 | Lv4相当。権限、HITL、ログ、評価、セキュリティを設計できる | 権限マトリクス、HITL、Agent Run Log、Security Checklist |
| AI Team Lead | 管理職、リーダー | Lv4〜Lv5相当。チーム方針、レビュー運用、実験文化を設計できる | チームレビュー基準、Decision Brief、Experiment Portfolio |
| AI Governance Owner | AI推進担当、経営層、管理部門 | Lv5〜Lv6相当。台帳、承認、教育、監査、廃止を設計できる | AI System Inventory、Risk Register、Audit Evidence、Retirement Plan |

認定は、受講完了だけで付与しない。知識テスト、実技課題、成果物レビュー、必要に応じて運用証跡を組み合わせる。

高リスク操作に関わる認定では、合格基準を軽くしない。ツール権限、顧客影響、本番影響、個人情報、契約、セキュリティに関わる場合は、EV3以上の証拠を要求する。

## Certification Decision Record

Certification Decision Recordは、認定判定そのものを記録する成果物である。

```text
Certification Decision Record

対象者：
申請認定：AI Safe User / AI Work Designer / AI Agent Operator / AI Team Lead / AI Governance Owner
評価日：
評価者：
使用した評価手段：知識テスト / 実技課題 / 成果物レビュー / 面談 / 運用証跡

提出成果物：
評価証拠レベル：EV0 / EV1 / EV2 / EV3 / EV4 / EV5
評価結果：合格 / 条件付き合格 / 再提出 / 不合格

強み：
不足：
条件付き合格の場合の条件：
再提出が必要な成果物：
再評価日：
第16章への引き渡し事項：
- 推奨する学習・OJT：
- 運用上の制限案：
- 追加承認が必要な利用：
```

この記録は、認定の最終運用をすべて決める文書ではない。権限付与、研修割当、Office Hour、再評価通知、台帳反映は第16章の運用設計で決める。

したがって、本章のCertification Decision Recordには、評価者としての判定と、展開側へ渡す条件を書く。

## 評価ワークフロー

スキル評価は、次の流れで行う。

```text
Role Skill Profileを定義する
↓
対象者がSkill Diagnosis Formを提出する
↓
必要な実技課題を割り当てる
↓
成果物を作成する
↓
評価者がルーブリックでレビューする
↓
Evidence Portfolioを作る
↓
領域別レベルと不足領域を判定する
↓
Learning Gap Handoffを作る
↓
必要に応じてCertification Decision Recordを作る
↓
第16章のRole-based Learning Path、Office Hour、OJT、認定運用へ渡す
↓
一定期間後に再評価する
```

この流れで重要なのは、評価と展開を混同しないことである。

評価者の仕事は、不足領域、業務リスク、再評価成果物を明確にすることである。教材を作り、受講枠を割り当て、Office Hourを運用し、KPIへ反映することは第16章の責務である。

AIエージェント協働能力は、使うAIツール、業務、社内ポリシー、顧客契約、規制、組織体制によって変わる。したがって、評価は定期的に更新する。
## 評価者の設計

誰が評価するかも重要である。

| 評価対象 | 評価者候補 |
|---|---|
| Lv1〜Lv2 | AI推進担当、教育担当、部門管理職 |
| Lv3 | 部門管理職、AI推進担当、業務レビュアー |
| Lv4 | エンジニアリード、SRE、セキュリティ、AI推進担当 |
| Lv5 | 部門長、AI CoE、事業責任者、セキュリティ、法務 |
| Lv6 | 経営層、AI CoE責任者、法務、セキュリティ、監査、個人情報保護担当 |

高リスク領域では、単独評価にしない。

たとえば、顧客向けAI、自動送信、本番環境変更、契約、個人情報、採用、人事、金融、医療に関わる評価では、業務責任者だけでなく、法務、セキュリティ、個人情報保護、監査などの確認を入れる。

## 評価者校正

評価者校正は、ルーブリックを機能させるために必要である。

### 校正手順

1. 同じ課題を複数評価者が採点する。
2. 判定差分を比較する。
3. 差分が出た項目を確認する。
4. 模範解答、NG例、部分点基準を更新する。
5. 再度、別サンプルで採点する。
6. 差分が小さくなったら本番評価へ使う。

### 評価者が見るべき観点

- 成果物に必要項目があるか。
- 制約が明示されているか。
- 不確実性が分離されているか。
- 人間確認条件があるか。
- 高リスク操作を止めているか。
- ログと改善へ接続しているか。
- 判断理由が説明できるか。
- 学習パスに接続できるか。

評価者が「なんとなく良い」と判断する状態を避ける。

## プライバシーと評価の注意点

AI活用能力の評価では、チャット履歴、プロンプト、ログ、レビュー結果を扱う可能性がある。

これらには、業務情報、顧客情報、個人情報、評価に関わる情報が含まれる場合がある。

そのため、評価制度では次を決める。

| 項目 | 決めること |
|---|---|
| 収集対象 | どの成果物、ログ、課題を評価に使うか |
| 収集しない情報 | 個人チャット、私的メモ、機密顧客情報など |
| マスキング | 顧客名、個人名、契約条件、内部IDなどの扱い |
| 閲覧権限 | 誰が評価成果物を見られるか |
| 保存期間 | 評価証跡をどれだけ保持するか |
| 利用目的 | 能力開発、権限付与、研修設計、リスク管理のどれに使うか |
| 人事利用 | 査定・評価に使う場合のルール |

評価制度そのものが、監視制度に見えると、AI活用は広がらない。透明性が必要である。

## 実務ケース1: CS担当者をLv3へ上げる

### 状況

CSチームでは、顧客問い合わせの回答案作成にAIを使い始めている。現状、各担当者が自由に使っており、回答品質にばらつきがある。

ある担当者は、AIで回答案を作れるが、原因未確定の内容を断定してしまうことがある。また、問い合わせ内容に含まれる顧客情報の扱いにも不安がある。

### 目標

CS担当者の目標は、Lv3の業務適用者である。

特に必要なのは次である。

- Request Contractを作れる。
- Context Packを作れる。
- AI出力レビュー表で回答案を判定できる。
- 原因未確定、SLA、補償、契約、個人情報に関わる回答をエスカレーションできる。

### 評価課題

```text
顧客問い合わせ文、FAQ、過去回答例、社内制約を与える。
回答案を作るRequest Contractを作成する。
Context Packを作成する。
AIが生成した回答案をレビューし、Review Issue Logを作成する。
顧客送信可能か、修正後送信か、エスカレーションかを判断する。
```

### 合格基準

- 顧客情報を不要に拡散していない。
- 原因未確定の事項を断定していない。
- 補償やSLAを勝手に約束していない。
- 送信前承認が必要な条件を明示している。
- Review Issue LogでCritical以上の問題を識別できている。

このケースでは、AIで文章が書けるかではなく、顧客影響のある出力をレビューできるかを評価する。

## 実務ケース2: マネージャーをAI Team Leadへ上げる

### 状況

開発チームのマネージャーが、メンバーにAI活用を推奨している。しかし、レビュー基準がなく、AIで作った設計メモ、コード、テスト、PR説明の品質がばらついている。

### 目標

マネージャーの目標は、Lv5のチーム変革者である。

必要なのは次である。

- チームAI活用方針を作る。
- AI成果物レビュー基準を作る。
- 実験KPIを設計する。
- 失敗レビューを運用する。
- 過剰なAI利用や危険な自動化を止める。

### 評価課題

```text
チームのAI利用状況を前提に、
1. チームAI活用方針
2. AI成果物レビュー基準
3. ツール権限マトリクスの概要
4. AI活用KPI
5. 失敗レビュー運用
を作成する。
```

### 合格基準

- 低リスク利用を速くし、高リスク利用を止める設計になっている。
- コード、設計、テスト、顧客影響でレビュー強度が違う。
- AI利用回数だけをKPIにしていない。
- 失敗を責任追及ではなく、Request、Context、Review、Tool、Governanceの改善へ戻している。
- メンバーの行動停止や評価不安にも対応している。

このケースでは、管理職本人のプロンプト力より、チーム運用力を評価する。

## 実務ケース3: エンジニアをAI Agent Operatorへ上げる

### 状況

社内でGitHub IssueからPR説明、テスト観点、変更案を作るAIエージェントを試している。今後、PR作成やCI結果の要約まで任せたい。

### 目標

エンジニアの目標は、Lv4のエージェント運用者である。

必要なのは次である。

- ツール権限マトリクスを作れる。
- Tool Spec Cardを作れる。
- HITL承認フローを設計できる。
- Agent Run Logを設計できる。
- セキュリティテストを設計できる。

### 評価課題

```text
GitHub PR支援エージェントの設計案を作る。
1. ツール権限マトリクス
2. Tool Spec Card
3. HITL承認フロー
4. Permission Test
5. Agent Run Log
6. Security Checklist
を作成する。
```

### 合格基準

- 直接push、強制merge、secret閲覧、本番変更を禁止している。
- PR作成、コメント作成、CI結果要約、Issue参照などの権限が操作単位で分かれている。
- 承認前後のログが残る。
- Permission Testで禁止操作をテストしている。
- 失敗時にロールバック、停止、エスカレーション条件がある。

このケースでは、AIを開発に使えるかではなく、AIが安全に開発環境で行動できる設計ができるかを評価する。

## 実務ケース4: AI推進担当をAI Governance Ownerへ上げる

### 状況

全社でAI利用が広がっている。営業、CS、開発、管理部門がそれぞれ異なるツールを使っているが、台帳、利用範囲、承認フロー、教育記録が統一されていない。

### 目標

AI推進担当の目標は、Lv5〜Lv6である。

必要なのは次である。

- AI System Inventoryを設計する。
- Governance Level Matrixを定義する。
- AI Risk Registerを作る。
- AI Training Registerを運用する。
- 90日展開計画へ接続する。

### 評価課題

```text
全社AI活用の初期統制案を作る。
1. AI Use Policy 1ページ版
2. AI System Inventory
3. Governance Level Matrix
4. Approval Workflow
5. AI Training Register
6. Skill Map / Role Skill Profile
7. 90日以内に実施する施策
を作成する。
```

### 合格基準

- 全社利用を一律禁止せず、リスク別に制御している。
- 部門ごとのAI利用が台帳化されている。
- 低リスク利用と高リスク利用の承認が分かれている。
- 教育と権限付与が接続している。
- 棚卸し、監査、廃止、インシデント対応が含まれている。

このケースでは、AI推進担当の役割を、相談窓口から運用設計者へ引き上げる。

## 典型的な失敗

### 失敗1: プロンプト力だけを評価する

AI活用能力をプロンプト力だけで評価すると、危険である。

プロンプトは重要だが、Context、Review、Tool、Security、Governance、Judgmentが弱ければ、組織として安全に使えない。

評価は、Request Contractだけでなく、Context Pack、AI出力レビュー表、ツール権限マトリクス、Security Checklist、Decision Briefまで含める。

### 失敗2: 全員に同じレベルを求める

全員にLv4以上を求めると、教育コストが高くなり、現場が疲弊する。

全社員にはLv1〜Lv2で十分な領域が多い。一方で、高リスク業務やAI運用に関わる人にはLv4以上が必要である。

評価は、職種とリスクで変える。

### 失敗3: 自己診断だけで判断する

自己診断は参考値である。能力判定には、実技課題や成果物レビューが必要である。

特に、セキュリティ、ツール権限、ガバナンス、顧客影響のある領域では、自己診断だけで権限を付与してはいけない。

### 失敗4: AI利用量をKPIにする

AI利用回数、プロンプト数、生成量をKPIにすると、使わなくてよい業務にもAIを使うようになる。

評価すべきは、成果品質、時間削減、リスク低減、レビュー精度、実験からの学習、廃棄判断である。

### 失敗5: 評価を査定に直結させる

初期段階で評価を査定に直結させると、失敗共有が止まる。

AI活用の能力評価は、まず学習パス、権限付与、支援設計に使う。人事評価に使う場合は、制度設計と透明性が必要である。

### 失敗6: 評価者の基準がそろっていない

評価者によって合否基準が違うと、制度への信頼が失われる。

模範解答、NG例、サンプル採点、評価者校正を行う。

### 失敗7: 高リスク業務の評価を軽くする

社内メモ要約と、顧客送信や本番変更を同じ基準で評価してはいけない。

業務リスクが高いほど、実技課題、承認、ログ、監査証跡を重くする。

### 失敗8: 学習パスに接続しない

評価だけして終わると、能力は上がらない。

評価結果は、次の学習、実技課題、OJT、Office Hour、再評価へ接続する。

## 設計原則

### 原則1: 評価は能力開発の入口である

評価は、できない人を分類するためではない。不足領域を見つけ、学習パスを割り当てるために行う。

### 原則2: 証拠は成果物で残す

成果物がなければ、評価は印象になる。

### 原則3: レベルは領域別に判定する

一人に一つの総合レベルを付けるより、領域別に強弱を見た方が育成に使える。

### 原則4: 職種とリスクで到達目標を変える

全員に同じ能力は不要である。ただし、全員に最低限の安全利用能力は必要である。

### 原則5: 高レベルほど実務証跡を求める

Lv1〜Lv2は知識テストと模擬課題で評価できる。Lv4以上は、実務成果物、運用証跡、レビュー履歴が必要である。

### 原則6: 失敗から学ぶ能力を評価する

失敗ゼロを評価すると、実験が止まる。失敗を記録し、改善へ接続できるかを評価する。

### 原則7: 評価結果を権限付与に接続する

write権限、外部送信、自動実行、本番環境操作などは、スキル評価と承認を前提にする。

### 原則8: 評価制度も改善対象にする

AIツール、社内ルール、業務リスクは変わる。スキルマップとルーブリックは定期的に更新する。

## スキルマップ作成手順

### Step 1: 対象ロールを定義する

最初に、評価対象を職種ではなくロールで定義する。

```text
悪い例：
営業、開発、管理部門

良い例：
顧客返信案をAIで作るCS担当者
AIでPR説明とテスト観点を作る開発者
チームのAI成果物レビューを行うマネージャー
外部MCPを審査するAI推進担当
```

### Step 2: AI利用範囲を定義する

次に、そのロールがAIで何をしてよいかを定義する。

- 参照だけか。
- 下書きまでか。
- 提案までか。
- 承認後実行までか。
- 自動実行までか。

### Step 3: 扱う情報とリスクを定義する

扱う情報によって必要スキルは変わる。

- 公開情報。
- 社内限定情報。
- 機密情報。
- 個人情報。
- 顧客情報。
- 認証情報。
- 契約情報。
- 本番システム情報。

### Step 4: 12能力領域ごとの目標レベルを決める

Role Skill Profileを使い、必要領域と目標レベルを決める。

全領域を高くしない。ロールに必要な能力に絞る。

### Step 5: 評価成果物を決める

各能力領域に対して、どの成果物で評価するかを決める。

```text
Request設計 → Request Contract
Context設計 → Context Pack
レビュー → Review Issue Log
権限設計 → Tool Permission Matrix
判断 → Decision Brief
```

### Step 6: 実技課題を作る

実際の業務に近いケースを用意する。

- 顧客問い合わせ。
- 競合調査。
- 提案書作成。
- 障害報告。
- GitHub PR支援。
- 社内ナレッジ更新。
- 契約書レビュー補助。

### Step 7: ルーブリックで判定する

採点者の印象ではなく、ルーブリックで判定する。

- 必須項目があるか。
- リスクを見つけているか。
- 人間確認条件があるか。
- 不確実性を分けているか。
- ログや改善に接続しているか。

### Step 8: 学習パスに接続する

不足領域ごとに、次の学習と実技課題を割り当てる。

### Step 9: 再評価する

一定期間後に再評価する。

おすすめは、初期導入では30〜90日後である。運用が安定した後は、四半期または半期ごとに見直す。

## 章内演習

### 演習1: Role Skill Profileを作る

次のロールを1つ選ぶ。

- CS担当者。
- 営業担当者。
- 開発者。
- マネージャー。
- AI推進担当。

選んだロールについて、Role Skill Profileを作る。

必ず含める項目は次である。

- AI利用範囲。
- 扱う情報区分。
- 禁止される利用。
- 承認が必要な利用。
- 12能力領域の目標レベル。
- 必須成果物。
- 実技課題。
- 再評価周期。

### 演習2: 実技課題を作る

演習1で選んだロールについて、実技課題を1つ作る。

課題には次を含める。

- 状況。
- 入力資料。
- 作成する成果物。
- 合格基準。
- 不合格例。
- 評価者が見る観点。

### 演習3: Evidence Portfolioを作る

架空の受講者を1名想定し、Evidence Portfolioを作る。

提出成果物は3つ以上にする。

評価結果には、次を書く。

- 領域別レベル。
- 強み。
- 不足。
- リスク。
- 推奨学習。
- 次回評価日。

### 演習4: Learning Gap Handoffを作る

演習3の評価結果をもとに、次の30日間の学習パスを作る。

含める項目は次である。

- 学ぶ章。
- 実技課題。
- 上長またはCoEの支援。
- 実務で試す小実験。
- 再評価で提出する成果物。

## 演習判定基準

演習の合格基準は次である。

- ロールの業務とAI利用範囲が具体的である。
- 職種名だけで目標レベルを決めていない。
- 扱う情報区分とリスクが考慮されている。
- 12能力領域の目標レベルに過不足がない。
- 評価成果物が、能力領域に対応している。
- 自己診断だけで判定していない。
- 実技課題が実務に近い。
- 合格基準が観察可能である。
- 評価結果が学習パスへ接続している。
- 高リスク利用では、承認、ログ、セキュリティ、ガバナンスが含まれている。

## Source Notes

本章は、第1章から第14章までで定義した成果物、能力領域、実務ケースを評価体系へ再編したものである。

外部規格や法令の逐語的な解説ではない。実際に社内評価、認定、権限付与、人事制度へ接続する場合は、社内規程、人事制度、労務、法務、個人情報保護、セキュリティ、監査の要件を確認する。

特に、AI利用ログ、チャット履歴、成果物、レビュー結果を個人評価に使う場合は、収集目的、閲覧権限、保存期間、本人への説明、利用範囲を明確にする必要がある。

## まとめ

AIエージェント協働能力は、プロンプト力だけではない。

能力は、仕事選定、依頼設計、コンテキスト設計、委任、レビュー、ツール権限、ログ、セキュリティ、ガバナンス、判断、対話、実験、行動、組織展開に分かれる。

スキルマップは、これらを評価し、学習へ接続するための設計図である。

評価では、自己診断だけに頼らない。成果物、実技課題、運用証跡を見る。職種やリスクに応じて到達目標を変える。評価結果は、研修、OJT、Office Hour、認定、権限付与へ接続する。

第16章では、このスキルマップとLearning Gap Handoffを使い、書籍を社内教材、ワークショップ、認定運用、90日展開計画へ変換する。

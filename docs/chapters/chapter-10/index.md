# 第10章 ガバナンスと統制

## この章で扱う能力

この章では、AIエージェント協働を個人利用から組織運用へ移行するためのガバナンスと統制を扱う。

読者は、次のことができるようになる。

- 社内AI利用の全体像を、AI System Inventory として台帳化する
- AI利用ポリシー、禁止用途、条件付き許可、承認フローを設計する
- AIエージェント利用を、リスク、情報分類、外部影響、権限、監査要件に基づいて分類する
- AI利用の責任者、承認者、レビュー者、運用者を明確にする
- ベンダー、外部AI、MCP、SaaS、モデル変更、ログ、教育履歴を統制対象として扱う
- AIインシデント、例外承認、棚卸し、廃止を運用プロセスに組み込む

第9章までは、個別のAIエージェントをどう安全に設計するかを扱った。第10章では、その設計を組織として継続運用するための仕組みに拡張する。

## 関連図表

- [F-10: AIガバナンス運用サイクル](../../figures/f-10-ai-governance-cycle/) — 利用申請、台帳化、リスク分類、承認、運用、監査、棚卸し、廃止を循環プロセスとして扱う。

本書の6層モデルで見ると、第10章は `Control` 層の最後に位置する。ここまでに、AIに依頼し、文脈を渡し、作業を委任し、出力をレビューし、ツール権限を制御し、ログを残し、セキュリティを確認する方法を扱った。

しかし、個別案件ごとの設計が正しくても、組織として以下が見えていなければ、AI利用は統制されない。

- 誰が、どのAIを、何の目的で使っているか
- どの情報をAIに渡しているか
- どの外部サービスにデータが送られているか
- どのAIがどのツールを実行できるか
- どの出力が顧客、社員、システムに影響するか
- 誰が承認し、誰が責任を持つか
- 何を監査証跡として残すか
- いつ止めるか、いつ廃止するか

この章の目的は、AI利用を止めることではない。**安全に広げるために、見える化し、分類し、責任を割り当て、必要なところだけ承認を入れ、継続的に見直すこと**である。

## なぜガバナンスが必要か

AI活用は、最初は個人の工夫として始まる。要約、翻訳、文章作成、調査、コード補助、資料作成などは、個人の範囲でも効果が出る。

しかし、AIエージェント協働では、次のように影響範囲が広がる。

| 段階 | 状態 | 主なリスク |
|---|---|---|
| 個人利用 | 個人がAIに質問する | 機密情報投入、誤情報利用 |
| 業務利用 | 業務成果物の下書きに使う | 誤出力、根拠不足、レビュー漏れ |
| チーム利用 | 共通テンプレートやRAGを使う | 古い情報の流用、責任不明確化 |
| ツール連携 | AIがSaaSやDBを参照・更新する | 権限過大、誤操作、監査不能 |
| 顧客影響 | AI出力が顧客対応や製品機能に入る | 誤案内、法務リスク、ブランド毀損 |
| 組織標準化 | 複数部門がAIを使う | 重複投資、統制不備、監査証跡不足 |

ガバナンスがない状態でAI活用が広がると、現場は速くなるように見える。しかし、実際には次の負債が蓄積する。

```text
・誰も管理していないAIツールが増える
・部門ごとに同じようなプロンプトやRAGを作る
・顧客情報が外部SaaSに投入される
・古い資料をAIが正しい前提として使う
・AI出力を誰がレビューしたのか分からない
・承認なしで顧客送信やCRM更新が行われる
・ログに個人情報や機密情報が残る
・モデルやプロンプト変更後に品質が変わっても気づかない
・使われなくなったAIエージェントやAPIキーが残る
・インシデント時に、止める対象と責任者が分からない
```

AIガバナンスとは、これらを防ぐための**業務運用の設計**である。抽象的な倫理スローガンではない。

## ガバナンス、セキュリティ、運用の違い

AIガバナンスは、セキュリティと同じではない。また、日々の運用とも同じではない。

| 領域 | 主な問い | 成果物 |
|---|---|---|
| セキュリティ | どう攻撃・漏洩・誤用を防ぐか | Security Checklist、情報分類、権限、テスト |
| 運用 | 日々どう動かし、監視し、改善するか | Run Log、Trace、Dashboard、Run Review |
| ガバナンス | 誰が何を許可し、誰が責任を持ち、どう組織として統制するか | Policy、Inventory、Risk Register、Approval、Audit、Training、Incident Runbook |

セキュリティは、危険を減らす。運用は、日々回す。ガバナンスは、**組織として説明可能な状態を維持する**。

説明可能な状態とは、次の問いに答えられる状態である。

```text
このAI利用は、誰が承認したのか。
何の業務目的で使っているのか。
どの情報を扱うのか。
どのリスクレベルなのか。
どの統制が入っているのか。
ログはどこにあるのか。
事故時に誰が止めるのか。
いつ見直すのか。
不要になったらどう廃止するのか。
```

これに答えられないAI利用は、成果が出ていても、組織としては未管理である。

## 外部フレームワークをどう使うか

AIガバナンスでは、ISO/IEC 42001、NIST AI RMF、EU AI Act、各国の個人情報保護法、業界規制、社内規程などが関係する場合がある。主要な外部参照の出典キーは `SRC-ISO-42001`、`SRC-NIST-AI-RMF`、`SRC-NIST-GENAI-PROFILE`、`SRC-EU-AI-LITERACY`、`SRC-OWASP-LLM`、`SRC-OWASP-AGENTIC` であり、詳細URLと確認日は [Source Notes](../../source-notes/) を参照する。

ただし、本書は法令・認証の解説書ではない。ここでは、外部フレームワークを次のように扱う。

| 参照元 | 本書での使い方 |
|---|---|
| ISO/IEC 42001 | AIマネジメントシステムを、方針、役割、プロセス、継続改善として扱う参考にする |
| NIST AI RMF | Govern / Map / Measure / Manage の考え方を、社内の台帳、リスク評価、評価、改善へ変換する |
| NIST Generative AI Profile | 生成AI固有のリスクを、評価・セキュリティ・インシデント条件へ反映する |
| EU AI Act | EU向け、EU内利用、またはEU規制対象となる場合に、AIリテラシー、高リスクAI、透明性、ログ、監督などの要件確認に使う |
| OWASP LLM / Agentic Top 10 | セキュリティテスト、プロンプトインジェクション、過剰なエージェンシー、ツールリスクの観点として使う |

重要なのは、標準や規制名を並べることではない。自社で次の運用に落とすことである。

```text
・どのAI利用を台帳登録するか
・どのAI利用を禁止または要承認にするか
・どの証跡を残すか
・誰が教育を受ける必要があるか
・誰が継続的に見直すか
・どのタイミングで法務、セキュリティ、個人情報保護担当へ接続するか
```

AIガバナンスは、標準への対応を目的化してはいけない。標準は、抜け漏れを減らすための参照軸である。

## ガバナンスの基本原則

### 原則1: 台帳化されていないAI利用は管理されていない

組織がAI利用を管理する第一歩は、AI System Inventory を作ることである。

「重要なものだけ登録する」から始めると、重要かどうかの判断自体が曖昧になる。まずは次の範囲を最低限登録対象にする。

```text
・業務成果物の作成に継続利用するAI
・顧客情報、個人情報、契約情報、社内機密を扱うAI
・社内文書、RAG、ナレッジベースと接続するAI
・SaaS、DB、GitHub、CRM、チケット、メール、Slack等と接続するAI
・顧客対応、採用、人事、契約、価格、セキュリティ、障害対応に使うAI
・部門でテンプレート化、共通化、半自動化されているAI利用
```

一方、完全に個人的な学習目的や、公開情報だけを使う単発のメモ作成は、簡易登録または対象外にできる。ここは自社ポリシーで決める。

### 原則2: すべてを承認制にしない

ガバナンスを重くしすぎると、現場は非公式な利用へ逃げる。これは危険である。

AIガバナンスは、低リスク利用を速くし、高リスク利用だけを確実に止める設計にする。

| リスク | 望ましい扱い |
|---|---|
| 低リスク | ポリシー範囲内でセルフサービス利用を許可する |
| 中リスク | テンプレート、レビュー、ログを条件に許可する |
| 高リスク | 事前審査、承認、評価、監査証跡を要求する |
| 禁止 | 組織として許可しない |

「全部申請」は悪い設計である。「全部自由」も悪い設計である。必要なのは、リスクに応じた摩擦である。

### 原則3: 用途単位で管理する

AIガバナンスは、ツール名だけで管理してはいけない。

同じAIツールでも、用途によってリスクは変わる。

| ツール | 用途 | リスク |
|---|---|---|
| 汎用チャットAI | 公開情報の要約 | 低 |
| 汎用チャットAI | 顧客契約書の条項解釈 | 高 |
| 汎用チャットAI | 社外向け謝罪文の作成 | 中〜高 |
| コード生成AI | テストコード作成 | 中 |
| コード生成AI | 本番DB更新SQL生成 | 高 |
| 社内RAG | 社内規程検索 | 中 |
| 社内RAG | 人事評価情報検索 | 高 |

したがって、台帳は「AIツール名」だけでなく、「ユースケース名」を中心に管理する。

### 原則4: 責任者を明確にする

AIが作ったから責任がない、という状態は存在しない。

少なくとも、次の責任者を分ける。

| 役割 | 責任 |
|---|---|
| Business Owner | そのAI利用が業務目的に合っていることに責任を持つ |
| System Owner | AIシステム、RAG、ツール連携、運用設計に責任を持つ |
| Data Owner | 利用データの分類、利用可否、保持、削除に責任を持つ |
| Risk Owner | 残リスクを受け入れるか、低減するかを判断する |
| Approver | 特定の実行、顧客影響、例外利用を承認する |
| Operator | 日々の運用、ログ確認、一次対応を行う |
| Reviewer | AI出力、評価結果、Run Reviewを確認する |
| AI Governance Owner | 全社ポリシー、台帳、例外、監査、教育を管理する |

小規模組織では複数役割を1人が兼ねてもよい。ただし、役割自体は消してはいけない。

### 原則5: 証跡を残せない統制は運用されない

「レビューすること」と決めても、レビュー記録がなければ実施状況は分からない。

「承認を得ること」と決めても、承認者、承認理由、承認時の出力、実行内容が残っていなければ、後から検証できない。

ガバナンスは、証跡を必要とする。

```text
・台帳登録
・リスク評価
・承認記録
・レビュー記録
・評価結果
・ログ保持
・例外承認
・インシデント対応記録
・教育履歴
・廃止記録
```

ただし、証跡そのものが機密情報や個人情報の貯蔵庫になることがある。第8章、第9章で扱ったログ最小化とマスキングは、ガバナンス証跡にも適用する。

### 原則6: ライフサイクルで管理する

AI利用は、導入時だけ審査して終わりではない。

次のライフサイクルで管理する。

```text
企画
↓
申請・台帳登録
↓
リスク分類
↓
設計レビュー
↓
パイロット
↓
本番運用
↓
Run Review / 評価
↓
変更管理
↓
定期棚卸し
↓
廃止
```

特に、モデル変更、プロンプト変更、RAG対象文書の変更、ツール権限変更、利用部門変更は、再評価のトリガーである。

## ガバナンス対象を定義する

最初に決めるべきことは、何をガバナンス対象とするかである。

### 対象1: AIツール

例:

```text
・汎用チャットAI
・コード補助AI
・議事録AI
・翻訳AI
・画像生成AI
・検索拡張AI
・AIエージェント基盤
・社内AIポータル
```

ツール単位の管理では、契約、データ保持、利用規約、ベンダー審査、認証、アクセス制御を扱う。

### 対象2: AIユースケース

例:

```text
・営業提案書の下書き
・顧客問い合わせ分類
・障害報告書のドラフト
・コードレビュー補助
・社内規程検索
・契約書のリスク抽出
・FAQ回答案作成
```

ユースケース単位の管理では、業務目的、扱う情報、利用者、出力先、レビュー、承認、ログ、KPIを扱う。

### 対象3: AIエージェント

例:

```text
・顧客問い合わせ一次対応エージェント
・GitHub PR作成エージェント
・障害一次分析エージェント
・営業CRM更新支援エージェント
・社内ナレッジ更新エージェント
```

エージェント単位の管理では、ツール権限、HITL、ログ、評価、失敗時挙動、停止条件、所有者を扱う。

### 対象4: データ・ナレッジソース

例:

```text
・社内Wiki
・FAQ
・過去チケット
・CRM
・契約書
・障害履歴
・Runbook
・Slackログ
・議事録
・ソースコード
```

データソース単位の管理では、情報分類、アクセス権、鮮度、RAG対象範囲、削除、保持期間、メタデータ、データオーナーを扱う。

### 対象5: モデル・プロンプト・評価セット

例:

```text
・利用モデル
・システムプロンプト
・業務プロンプト
・評価データセット
・禁止出力テスト
・回帰評価ケース
```

これらは、変更により品質やリスクが変わる。したがって、変更管理の対象にする。

## AI利用ポリシーを設計する

AI利用ポリシーは、長大な規程である必要はない。最初に必要なのは、現場が即判断できるルールである。

### ポリシーに含める項目

```text
1. 目的
2. 適用範囲
3. 利用してよいAIツール
4. 利用禁止AIツール
5. 入力禁止情報
6. 条件付きで入力可能な情報
7. AI出力の利用条件
8. 顧客送信・社外公開のルール
9. ツール連携・自動実行のルール
10. 高リスク用途の承認手順
11. ログ・証跡の扱い
12. インシデント時の連絡先
13. 違反・例外の扱い
14. 教育・研修の要件
15. 定期見直し
```

### 良いポリシーの条件

| 条件 | 内容 |
|---|---|
| 判断可能 | 現場が「使ってよいか」を判断できる |
| 短い版がある | 1ページ版、FAQ版、詳細版に分かれている |
| 例がある | 禁止例、条件付き許可例、許可例がある |
| 承認先が明確 | 困ったときに誰へ相談するか分かる |
| 更新される | モデル、ツール、規制、社内方針の変更に合わせて改訂される |
| テンプレートと接続する | Request Contract、Context Pack、レビュー表に落ちている |

### 許可・条件付き許可・禁止の例

| 区分 | 例 |
|---|---|
| 許可 | 公開情報の要約、社内向け文章の下書き、機密を含まないアイデア出し、一般的なコード例の確認 |
| 条件付き許可 | 社内資料の要約、顧客向け返信案、議事録要約、コードレビュー補助、社内RAG検索 |
| 要承認 | 顧客情報の利用、CRM連携、GitHub書き込み、採用・人事・契約・価格・セキュリティ領域の利用 |
| 禁止 | 認証情報の入力、未公開財務情報の外部送信、AI単独の顧客送信、AI単独の本番変更、AIによる最終採用判断 |

## AI System Inventory を作る

AI System Inventory は、AIガバナンスの中心成果物である。

台帳というと形式的に見えるが、目的は単純である。

```text
組織内で、どのAIが、誰により、何の業務に、どの情報を使って、どの権限で、どのリスクを持って、どの統制の下で使われているかを見える化する。
```

### 台帳項目

| 項目 | 内容 |
|---|---|
| Inventory ID | 台帳上の一意なID |
| システム名 / ユースケース名 | 管理対象の名前 |
| 利用部門 | 主に使う部門 |
| Business Owner | 業務責任者 |
| System Owner | 技術・運用責任者 |
| Data Owner | データ責任者 |
| Risk Owner | 残リスク受容者 |
| 利用目的 | 何のために使うか |
| 利用者 | 誰が使うか |
| 対象業務 | どの業務プロセスか |
| 利用AI / ベンダー | モデル、SaaS、基盤 |
| 利用データ | 扱う情報分類、RAG対象、ログ |
| 外部送信 | 外部AI・SaaSへの送信有無 |
| ツール連携 | 参照・更新・送信・削除など |
| 顧客影響 | 顧客に出力が届くか、顧客処理に影響するか |
| 自動化レベル | 人間補助、下書き、承認後実行、自動実行など |
| リスク分類 | 低・中・高・禁止、またはG0〜G5 |
| 統制 | レビュー、HITL、ログ、評価、承認 |
| 評価方法 | Evaluation Plan、テスト、品質指標 |
| 監査証跡 | 保存するログ、承認記録、レビュー記録 |
| 最終レビュー日 | 直近の棚卸し日 |
| 次回レビュー日 | 次回の定期レビュー日 |
| 廃止条件 | 終了・停止・見直し条件 |

### 台帳を作る順序

最初から完璧な台帳を作ろうとすると失敗する。次の順番で進める。

```text
Step 1: 既存のAIツールとユースケースを棚卸しする
Step 2: 高リスク候補だけ先に詳細化する
Step 3: 部門ごとにBusiness Ownerを割り当てる
Step 4: 情報分類と外部送信有無を埋める
Step 5: ツール連携と顧客影響を埋める
Step 6: リスク分類を暫定で付ける
Step 7: 統制が不足しているものを改善バックログへ入れる
Step 8: 四半期ごとに棚卸しする
```

### 台帳の悪い例

```text
システム名：ChatGPT
利用部門：全社
目的：業務効率化
リスク：中
```

これでは管理できない。用途、情報、出力先、責任者、統制が分からない。

### 台帳の良い例

```text
Inventory ID：AI-CS-004
ユースケース名：顧客問い合わせ返信案作成支援
利用部門：カスタマーサポート部
Business Owner：CS責任者
System Owner：AI推進担当
Data Owner：CSデータ管理者
Risk Owner：CS部門長
利用目的：FAQと過去チケットを参照し、CS担当者向け返信案を作成する
利用者：CS担当者、CSリーダー
利用AI / ベンダー：社内AIポータル経由の承認済みLLM
利用データ：顧客メール、公開FAQ、社内FAQ、過去チケット要約、CRM契約種別
外部送信：社内AIポータル経由のみ。原文ログの外部保存は禁止
ツール連携：FAQ read-only、過去チケット read-only、CRM read-only、メール draft-only
顧客影響：返信案が顧客送信前レビューに使われる
自動化レベル：下書き作成。顧客送信は人間承認必須
リスク分類：G3 要承認
統制：顧客送信前レビュー、SLA言及時HITL、Run Log、Prompt Injection Test
評価方法：月次でCritical Issue率、承認拒否率、誤分類率を確認
監査証跡：Run Log、Approval Log、Review Issue Log
最終レビュー日：2026-06-20
次回レビュー日：2026-09-20
廃止条件：Critical Issueが2か月連続で閾値超過、または承認済みFAQソースが維持不能になった場合
```

台帳は、監査担当だけのものではない。現場が「これは何のAIで、誰が責任者で、どこまで許可されているか」を確認するための業務基盤である。

## リスク分類を設計する

台帳ができたら、リスク分類を設計する。

本書では、実務で使いやすいように `G0〜G5` のガバナンスレベルを使う。

| レベル | 状態 | 例 | 必要統制 |
|---|---|---|---|
| G0 | 対象外または個人学習 | 公開情報のみを使った個人の学習 | 組織統制対象外。ただし一般ルールは適用 |
| G1 | 低リスク利用 | 公開情報要約、社内向け草稿、機密なしのアイデア出し | 基本ポリシー遵守、自己レビュー |
| G2 | 業務利用 | 社内資料要約、議事録、社内向け報告書、コード補助 | Request Contract、Context Pack、レビュー表 |
| G3 | 管理対象利用 | 顧客情報、社内機密、RAG、チーム共通利用、下書き生成 | 台帳登録、情報分類、レビュー、ログ、Owner設定 |
| G4 | 高リスク利用 | 顧客影響、契約・価格・採用・人事・セキュリティ、本番環境関連 | 事前承認、HITL、Evaluation Plan、監査証跡、定期レビュー |
| G5 | 禁止または特別審査 | AI単独の最終判断、本番破壊的変更、認証情報投入、法令違反可能性 | 原則禁止。例外は経営・法務・セキュリティ承認 |

### リスク分類の判定軸

次の軸を使う。

| 軸 | 低い | 高い |
|---|---|---|
| 情報機密性 | 公開情報 | 顧客情報、個人情報、契約、認証情報、未公開情報 |
| 外部送信 | なし | 外部AI、外部SaaS、外部MCPへ送信 |
| 顧客影響 | なし | 顧客へ送信、顧客処理、契約、SLA、価格に影響 |
| ツール権限 | なし / read-only | write、delete、send、execute、admin |
| 自動化レベル | 下書きのみ | AI単独実行、自動判断、自動送信 |
| 可逆性 | 簡単に修正可能 | 取り消し困難、法務・信用・金銭影響 |
| 人間レビュー | 必ず入る | 入らない、または形式的 |
| 規制領域 | 一般業務 | 金融、医療、採用、人事、教育、公共、重要インフラ |
| 規模 | 個人利用 | 部門標準、全社標準、顧客提供 |

複数の軸で高リスクに該当する場合、上位レベルへ分類する。迷う場合は、一段上の統制を適用する。

## 承認フローを設計する

承認フローは、単なる稟議ではない。AIエージェント協働では、次の3種類の承認を分ける。

| 承認 | 内容 | タイミング |
|---|---|---|
| Use Case Approval | そのAI利用を始めてよいか | 導入前 |
| Tool Action Approval | AIが特定操作を実行してよいか | 実行直前 |
| Risk Acceptance | 残リスクを受け入れて運用してよいか | 本番前、例外時、レビュー時 |

### 承認が必要な代表例

```text
・顧客情報、個人情報、契約情報をAIに渡す
・外部SaaS、外部AI、外部MCPと接続する
・顧客へ送信する文章をAIが作る
・CRM、チケット、GitHub、メール、Slack等へAIが書き込む
・本番環境、監視、インフラ、セキュリティ設定に関係する
・採用、人事評価、価格、契約、法務判断に関係する
・AI出力が顧客向け製品機能に組み込まれる
・既存の承認済み範囲を超えて利用する
```

### 承認者の決め方

| 判断対象 | 主な承認者 |
|---|---|
| 業務目的・ROI | Business Owner、部門責任者 |
| 情報利用 | Data Owner、個人情報保護担当 |
| ツール連携・権限 | System Owner、情シス、セキュリティ |
| 顧客影響 | 部門責任者、CS責任者、営業責任者、法務 |
| 法務・契約 | 法務 |
| セキュリティ例外 | セキュリティ責任者 |
| 高リスク残リスク | Risk Owner、経営層または委任された責任者 |
| 全社標準化 | AI Governance Owner、経営会議またはAI委員会 |

承認者は、形式的なサイン担当ではない。承認するリスクを理解し、その範囲に責任を持つ必要がある。

## AI System ADR を作る

技術システムでは、重要な設計判断を ADR（Architecture Decision Record）として残すことがある。AI利用でも同じ考え方が使える。

AI System ADR は、AI利用に関する重要な意思決定を記録する文書である。

### AI System ADR に残す判断

```text
・なぜこのAI利用を導入するのか
・どの代替案を比較したか
・どのモデル、ベンダー、基盤を選んだか
・どの情報を使うか、使わないか
・どのツール権限を許可するか
・どのHITLを入れるか
・どの残リスクを受け入れたか
・どの評価指標で継続判断するか
・どの条件で停止・廃止するか
```

AI System ADR は、G3以上のAI利用では作成を推奨する。G4以上では必須にする。

### ADRに残さないもの

AI System ADRは、日々のRun Logではない。すべての実行履歴を書くものではない。

| 文書 | 目的 |
|---|---|
| Agent Run Log | 1回の実行を記録する |
| Review Issue Log | 出力レビュー上の問題を記録する |
| AI System ADR | 重要な設計判断を記録する |
| AI Risk Register | 継続的なリスクと対応状況を管理する |
| AI Audit Evidence Pack | 監査時に必要な証跡を束ねる |

## AI Risk Register を運用する

AI Risk Register は、AI利用に関するリスク、対策、残リスク、責任者、期限を管理する表である。

### Risk Register の項目

| 項目 | 内容 |
|---|---|
| Risk ID | 一意なID |
| 対象AI / ユースケース | 台帳IDと接続する |
| リスク記述 | 何が起きる可能性があるか |
| 原因 | なぜ起きるか |
| 影響 | 顧客、法務、セキュリティ、業務、コストへの影響 |
| 発生可能性 | 低・中・高 |
| 影響度 | 低・中・高 |
| リスクレベル | 低・中・高・禁止 |
| 対応 | 回避、低減、移転、受容 |
| 統制 | レビュー、HITL、ログ、テスト、権限制限など |
| 残リスク | 統制後に残るリスク |
| Risk Owner | 受容または低減責任者 |
| 期限 | 対応期限 |
| 状態 | Open / Mitigating / Accepted / Closed |

### Risk Register の例

| Risk ID | リスク | 対応 | 残リスク | Owner |
|---|---|---|---|---|
| R-001 | 顧客返信案で原因未確定事項を断定する | 顧客送信前レビュー、禁止表現テスト、SLA言及時HITL | 低 | CS責任者 |
| R-002 | 外部PDF内の命令にAIが従う | Untrusted Input Handling Rule、Prompt Injection Test | 中 | AI推進担当 |
| R-003 | Run Logに個人情報が全文保存される | マスキング、保持期間短縮、アクセス制限 | 低 | 情シス |
| R-004 | CRM更新案が誤って直接反映される | draft-only、承認後人間実行、Permission Test | 低 | CRM管理者 |

リスク台帳は、作って放置すると形骸化する。Run Review、インシデント、評価結果、改善バックログと接続する。

## ベンダー、外部AI、MCPを管理する

AI利用では、外部サービスが多く関係する。

```text
・LLMプロバイダー
・AI SaaS
・RAG基盤
・MCPサーバー
・プラグイン
・ブラウザ拡張
・議事録サービス
・コード補助AI
・データ処理API
```

これらは、情報を外部へ送信し、出力を生成し、場合によっては社内ツールへ接続する。したがって、ベンダー管理の対象である。

### ベンダー審査で見る項目

| 観点 | 確認内容 |
|---|---|
| データ利用 | 入力データが学習に使われるか、保持されるか |
| 保存場所 | データがどの地域・環境に保存されるか |
| 保持期間 | 入力、出力、ログ、トレースの保持期間 |
| 削除 | 削除要求、保持停止、退会時削除の可否 |
| 認証 | SSO、MFA、SCIM、RBACへの対応 |
| 監査ログ | 管理者ログ、利用ログ、APIログの取得可否 |
| 契約 | DPA、SLA、利用規約、責任制限、サブプロセッサ |
| セキュリティ | SOC2、ISO27001、脆弱性対応、暗号化 |
| 権限 | 最小権限、環境分離、APIスコープ制御 |
| 変更通知 | モデル、利用規約、データ扱い、API仕様変更の通知 |
| サポート | インシデント時の連絡窓口、対応時間 |
| 退避 | 解約、移行、代替手段、データエクスポート |

外部MCPやプラグインは、通常のSaaSよりもリスクが高い場合がある。AIにツールを与える経路だからである。第9章の External Tool / MCP Review Sheet と接続して審査する。

## 教育とAIリテラシーを統制に入れる

AI利用ポリシーを作っても、利用者が理解していなければ機能しない。

教育は、コンプライアンス上の形式ではなく、事故を減らし、利用速度を上げるための統制である。

### ロール別教育

| 対象 | 必須内容 |
|---|---|
| 全社員 | AI利用ポリシー、入力禁止情報、基本レビュー、社外送信ルール |
| 業務担当者 | Request Contract、Context Pack、AI出力レビュー、リスク判断 |
| 管理職 | 承認フロー、AI成果物レビュー、Risk Acceptance、チームKPI |
| エンジニア | ツール権限、RAG、ログ、評価、セキュリティテスト |
| AI推進担当 | 台帳、ポリシー、例外、監査、教材更新、Office Hour |
| 経営層 | 投資判断、リスク許容度、ガバナンス体制、重大インシデント対応 |

### AI Training Register

教育履歴は、次の項目で管理する。

```text
受講者：
所属：
ロール：
受講教材：
受講日：
確認テスト結果：
実技課題結果：
認定レベル：
再受講期限：
対象AI利用範囲：
```

EU向け利用や規制領域では、AIリテラシーの確保が法務上の論点になる場合がある。自社の対象地域、利用形態、AIシステムのリスクに応じて、法務と確認する。

## 監査証跡を設計する

監査証跡は、事故が起きたときだけ必要になるものではない。AI利用を継続的に改善するためにも必要である。

### AI Audit Evidence Pack

G3以上のAI利用では、最低限次を束ねる。

```text
1. AI System Inventory
2. AI Use Case Intake
3. Risk Classification
4. AI Security Checklist
5. Data Classification and Handling Matrix
6. AI Privacy Impact Review
7. Tool Permission Matrix
8. HITL承認フロー
9. Evaluation Plan
10. Agent Run Logのサンプル
11. Approval Log
12. Review Issue Log
13. Run Review Report
14. Improvement Backlog
15. Training Register
16. Vendor Review Record
17. Incident Runbook
18. Retirement Plan
```

全部を毎回詳細に作る必要はない。リスクレベルに応じて、軽重をつける。

| レベル | 証跡 |
|---|---|
| G1 | ポリシー同意、自己レビュー |
| G2 | Request Contract、レビュー表、必要に応じてContext Pack |
| G3 | 台帳、情報分類、レビュー、ログ、Owner、簡易評価 |
| G4 | 事前承認、詳細リスク評価、HITL、Evaluation Plan、監査証跡、定期レビュー |
| G5 | 原則禁止。例外時は経営・法務・セキュリティ承認と期限付き例外記録 |

## インシデント対応を設計する

AIインシデントは、通常のセキュリティインシデントと重なるが、固有の特徴がある。

### AIインシデントの例

```text
・機密情報や個人情報を外部AIに投入した
・AIが誤った顧客返信案を作り、人間が確認せず送信した
・AIが外部文書内の命令に従った
・AIが権限外のデータを参照した
・AIがCRMやチケットを誤更新した
・AI生成コードに脆弱性があり本番投入された
・AI出力が差別的、虚偽、法務上問題のある内容を含んだ
・ログに顧客機密が過剰に保存された
・承認済み範囲を超えてAIエージェントが利用された
```

### AI Incident Runbook

最低限、次を決める。

```text
1. 検知条件
2. 初動担当
3. 停止対象
4. データ保全
5. 影響範囲確認
6. 通知先
7. 顧客連絡要否
8. 法務・セキュリティ・経営エスカレーション条件
9. 復旧条件
10. 再発防止
11. 台帳・評価・ポリシー更新
```

### AIインシデントの初動

AIインシデントでは、まず影響拡大を止める。

```text
Step 1: 対象AI利用を一時停止する
Step 2: 関連ツール権限を停止またはread-onlyに落とす
Step 3: Run Log、Trace、Approval Logを保全する
Step 4: 入力、出力、外部送信、ツール実行、顧客影響を確認する
Step 5: セキュリティ、法務、該当部門責任者へ通知する
Step 6: 顧客影響がある場合は、顧客対応方針を人間が決める
Step 7: 再発防止が完了するまで本番再開しない
```

AIにインシデント対応文面を作らせることはできる。ただし、通知要否、責任範囲、顧客説明、法的評価はAIに丸投げしない。

## 例外管理を設計する

現実には、標準ルールから外れる要望が出る。

```text
・短期プロジェクトで例外的に外部AIを使いたい
・承認済みでないMCPを試したい
・ログ保持期間を一時的に延ばしたい
・通常は不可の情報を、契約上安全な環境で処理したい
・高リスクだが、経営上必要な検証をしたい
```

例外はゼロにできない。問題は、例外を無記録で許すことである。

### Security Exception Record / Governance Exception Record

例外には、次を必ず残す。

```text
例外ID：
申請者：
対象AI利用：
標準ルールとの差分：
例外が必要な理由：
扱う情報：
期間：
追加統制：
承認者：
残リスク：
終了条件：
レビュー日：
```

例外は期限付きにする。期限のない例外は、実質的に新しい標準ルールである。標準ルールとして採用するなら、正式にポリシーを改訂する。

## 変更管理を設計する

AI利用は、導入後にも頻繁に変わる。

変更管理の対象は、コードだけではない。

```text
・利用モデルの変更
・システムプロンプトの変更
・業務プロンプトの変更
・RAG対象文書の変更
・ツール権限の変更
・出力先の変更
・ログ保持方針の変更
・利用部門の追加
・ベンダー契約変更
・評価基準の変更
```

### 再評価が必要な変更

| 変更 | 再評価内容 |
|---|---|
| モデル変更 | 出力品質、禁止出力、コスト、データ取り扱い |
| プロンプト変更 | 受け入れ条件、リスク表現、HITL条件 |
| RAG文書追加 | 情報分類、鮮度、権限、外部入力扱い |
| ツール権限追加 | Permission Test、HITL、ログ |
| 利用部門拡大 | 教育、Owner、レビュー体制 |
| 顧客向け利用開始 | 法務、セキュリティ、顧客通知、監査証跡 |
| 外部ベンダー変更 | データ保持、契約、DPA、監査ログ |

変更管理をしないAI利用は、初期審査時点とは別物になる。

## 廃止と棚卸しを設計する

AIガバナンスで見落とされやすいのが廃止である。

使われなくなったAI利用は、残骸としてリスクになる。

```text
・古いプロンプトが残る
・古いRAG文書が残る
・APIキーが残る
・権限が残る
・ログが残る
・責任者不在の自動処理が残る
・教材や手順が古くなる
```

### AI Retirement Plan

廃止時には次を確認する。

```text
対象AI利用：
廃止理由：
停止日：
停止対象：モデル / プロンプト / RAG / API / ツール連携 / アカウント / ログ
代替手段：
データ削除：
ログ保持：
アクセス権削除：
関係者通知：
台帳更新：
監査証跡：
再開条件：
```

### 棚卸し頻度

| リスク | 棚卸し頻度 |
|---|---|
| G1 | 年1回またはポリシー更新時 |
| G2 | 半期ごと |
| G3 | 四半期ごと |
| G4 | 月次または四半期ごと |
| G5例外 | 期限内に個別レビュー |

棚卸しでは、使っているAIだけでなく、使われなくなったAIを探す。

## 組織体制を設計する

AIガバナンスは、中央集権にしすぎると遅くなる。完全分散にすると統制できない。

多くの組織では、中央のAI推進・ガバナンス機能と、各部門のAI活用責任者を組み合わせる形が現実的である。

### 体制例

```text
経営層 / AIステアリングコミッティ
  ↓
AI Governance Owner / AI CoE
  ↓
セキュリティ・法務・情シス・個人情報保護担当
  ↓
部門AI責任者 / Business Owner
  ↓
チームリーダー / 利用者
```

### AI CoE の役割

AI CoE は、AIを全部作る部署ではない。標準化と支援を担う。

```text
・AI利用ポリシーの維持
・AI System Inventoryの管理
・リスク分類と承認フローの運用
・テンプレート、チェックリスト、教材の更新
・Office Hourと相談窓口
・高リスク利用の審査支援
・評価指標とダッシュボードの整備
・インシデント時の初動支援
・成功事例と失敗事例の共有
・ベンダー、MCP、外部ツール審査の標準化
```

### 部門責任者の役割

```text
・自部門のAI利用を台帳登録する
・利用者に必要な教育を受けさせる
・AI成果物レビューを業務フローに組み込む
・高リスク利用を申請する
・部門KPIとリスクKPIを確認する
・不要なAI利用を廃止する
```

AI CoEがすべてを承認する構造にすると、ボトルネックになる。部門が自走できるように、ガードレール、テンプレート、権限移譲を設計する。

## コスト統制を入れる

AIガバナンスは、セキュリティだけではない。コストも統制対象である。

AI利用が広がると、次のコストが増える。

```text
・API利用料
・SaaSライセンス
・RAG基盤
・ログ保存
・評価実行
・人間レビュー工数
・教育工数
・ベンダー審査工数
・インシデント対応工数
```

### コストKPI

| KPI | 内容 |
|---|---|
| AI利用コスト | 部門別、ユースケース別、モデル別の費用 |
| 1 Runあたりコスト | 1回の業務実行にかかる推論・検索・ツール費用 |
| 再実行率 | 出力品質不足によるやり直し |
| レビュー工数 | 人間レビューにかかった時間 |
| 承認待ち時間 | HITL承認で滞留した時間 |
| 廃棄されたAI利用数 | ROIや品質不足で停止した利用 |
| コスト対効果 | 削減時間、品質改善、売上貢献、事故削減との比較 |

コスト統制は、利用抑制ではなく、価値の低いAI利用を止め、高価値なAI利用へ資源を寄せるために行う。

## 典型的な失敗

### 失敗1: 禁止ルールだけ作る

禁止だけでは現場は回避策を探す。使ってよい範囲、相談先、承認手順、テンプレートを同時に整備する。

### 失敗2: AI利用実態を把握しない

どの部門が、何のAIを、何の目的で使っているか分からない状態は危険である。まずAI System Inventoryを作る。

### 失敗3: ツール名だけで管理する

「ChatGPTを許可するか禁止するか」だけでは不十分である。用途、情報、出力先、権限でリスクは変わる。

### 失敗4: 高リスク用途を一律に現場任せにする

採用、人事評価、契約、価格、金融、医療、顧客向け自動応答、本番環境変更などは、部門判断だけで進めない。

### 失敗5: すべてを中央承認にする

中央承認が重すぎると、現場は非公式利用へ流れる。低リスク利用はセルフサービス化し、高リスク利用だけ承認を厚くする。

### 失敗6: 教育履歴を管理しない

ポリシーがあっても、対象者が理解していなければ統制は機能しない。教育履歴とロール別学習パスを管理する。

### 失敗7: 廃止を設計しない

使われなくなったAIエージェント、古いプロンプト、期限切れのAPIキー、管理者不在のツール連携はリスクになる。

### 失敗8: 監査証跡が機密情報の溜まり場になる

ログや台帳を残すほど、機密情報が集中する。監査証跡にも最小化、マスキング、アクセス制御、保持期間を適用する。

## 実務ケース1: 社内AI活用の棚卸し

### 状況

ある企業では、各部門が独自にAIツールを使っている。

```text
・営業部は提案書作成に汎用チャットAIを使っている
・CS部は顧客メールの返信案作成にAIを使っている
・開発部はコード補助AIを使っている
・管理部は契約書レビュー補助にAIを使っている
・一部チームは外部MCPを試している
```

経営層は「AI活用は進んでいる」と認識しているが、実際には誰が何に使っているか分からない。

### 悪い進め方

```text
全AI利用を一旦禁止する。
以後、AI利用は全件AI推進担当の承認制にする。
```

これは現場を止める。非公式利用も増える。

### 良い進め方

```text
1. AI利用ポリシー1ページ版を配布する
2. 2週間の棚卸し期間を設ける
3. AI System Inventoryの簡易版を提出させる
4. 情報分類、外部送信、顧客影響、ツール権限で暫定分類する
5. G3以上を優先的に詳細レビューする
6. G1〜G2はテンプレートと自己レビューで継続許可する
7. G4は事前審査に回す
8. 未登録利用は次回棚卸しで是正対象にする
```

### 成果

```text
・重複して契約していたAI SaaSが分かる
・顧客情報を外部AIに投入していた部門が分かる
・承認なしのMCP利用が分かる
・低リスク利用は止めずに継続できる
・高リスク利用だけ改善バックログへ入れられる
```

## 実務ケース2: 顧客問い合わせ対応エージェントを本番化する

### 状況

CS部門が、問い合わせ分類、FAQ検索、返信案作成を行うAIエージェントを本番化したい。

### 必要なガバナンス成果物

```text
・AI Use Case Intake
・AI System Inventory
・AI Risk Register
・Data Classification and Handling Matrix
・Tool Permission Matrix
・HITL承認フロー
・Evaluation Plan
・AI Incident Runbook
・Training Register
・AI Audit Evidence Pack
```

### リスク分類

```text
顧客情報を扱う。
過去チケットとCRMを参照する。
返信案が顧客送信に使われる。
メール送信は人間承認が必要。
```

このため、G3〜G4相当である。少なくとも台帳、レビュー、HITL、ログ、評価、教育履歴が必要になる。

### 承認条件

```text
・顧客送信はAI単独で行わない
・SLA、補償、原因断定はHITL対象にする
・CRMはread-onlyまたは更新案までにする
・Run Logは顧客本文を全文保存しない
・月次Run Reviewを行う
・Critical Issue率が閾値を超えたら一時停止する
```

## 実務ケース3: GitHub PR作成エージェント

### 状況

開発部門が、Issueを読み、リポジトリを確認し、修正案を作り、PRを作成するAIエージェントを導入したい。

### ガバナンス上の論点

```text
・ソースコードは社内機密である
・外部AIへコードを送る場合、契約とデータ保持確認が必要
・AIが直接mainブランチへpushしてはいけない
・PR作成は許可できるが、mergeは人間レビュー必須
・CI失敗時にAIが再修正する場合、実行回数と範囲を制限する
・セキュリティ修正、認証、暗号、権限に関係する変更はセキュリティレビュー対象にする
```

### 分類

多くの場合、G3以上である。顧客影響や本番環境に直結する場合はG4に上げる。

### 必要統制

```text
・Repository Access Scope
・Tool Permission Matrix
・禁止操作リスト
・PRテンプレート
・CI必須
・コードオーナーレビュー
・セキュリティ関連ファイルの追加レビュー
・Run Log / PR ID / Issue ID の紐づけ
・モデル変更時の回帰評価
```

## 実務ケース4: 契約書レビュー補助AI

### 状況

管理部門が、契約書からリスク条項を抽出し、法務確認の下準備を行うAIを使いたい。

### 主なリスク

```text
・契約情報が外部AIに送信される
・AIが法的判断を断定する
・法務確認なしで条項修正案が取引先に送られる
・古い契約ひな形を正しい前提として使う
```

### ガバナンス設計

```text
・契約書は機密区分を明示する
・外部AI利用は法務・セキュリティ承認制にする
・AI出力は「論点抽出」と「要確認事項」に限定する
・法的判断、交渉方針、契約承認は人間が行う
・契約書原文のログ保存は禁止または制限する
・法務レビュー前の社外送信は禁止する
```

このケースは、G4に分類するのが自然である。

## 実務ケース5: 外部MCPサーバーを追加する

### 状況

あるチームが、外部MCPサーバーを使って、AIから社内チケット、カレンダー、ドキュメントにアクセスしたい。

### ガバナンス上の判断

MCPは便利だが、AIにツールを与える経路である。通常のWebサービス利用よりも慎重に扱う。

確認することは次のとおりである。

```text
・誰がMCPサーバーを提供しているか
・どのデータにアクセスできるか
・どの操作を実行できるか
・認証情報はどこに保存されるか
・ログはどこに残るか
・外部へ何が送信されるか
・最小権限にできるか
・read-onlyから開始できるか
・削除、送信、更新、招待、権限変更ができないか
・停止、失効、ロールバックができるか
```

### 判定

外部MCPは、原則としてG3以上にする。書き込み、送信、削除、権限変更が可能な場合はG4以上にする。

## 実務ケース6: AI利用ポリシー違反が発生した

### 状況

社員が、未承認の外部AIに顧客問い合わせ本文とCRM情報を貼り付け、返信案を作成した。

### 初動

```text
1. 事実確認：入力した情報、利用AI、時刻、アカウント、出力、顧客影響
2. データ影響確認：個人情報、顧客機密、契約情報が含まれるか
3. ベンダー確認：入力データの保持、削除、学習利用の可能性
4. 社内通知：上長、AI推進担当、セキュリティ、法務
5. 顧客影響確認：顧客送信済みかどうか
6. 再発防止：教育、ポリシー、DLP、承認済みAI導線の改善
```

### 重要な観点

違反者を責めるだけでは不十分である。

次を確認する。

```text
・承認済みAIツールが使いにくかったのではないか
・ポリシーが分かりにくかったのではないか
・研修が不足していたのではないか
・現場に相談先がなかったのではないか
・DLPや技術制御で防げたのではないか
・業務プレッシャーが無理なAI利用を誘発していないか
```

ガバナンスは、違反後に罰する仕組みではない。違反が起きにくい業務環境を作る仕組みである。

## 設計原則

1. **統制はスピードを落とすためではなく、低リスク利用を速くするために設計する。** すべてを重く扱うと、現場は非公式利用へ流れる。
2. **ツール名ではなくユースケース単位で管理する。** 同じAIツールでも、要約、顧客返信、契約レビュー、本番変更ではリスクが違う。
3. **責任者を文書に書く。** Business Owner、System Owner、Data Owner、Risk Ownerが不明なAI利用は本番化しない。
4. **承認の種類を分ける。** Use Case Approval、Tool Action Approval、Risk Acceptance、Security Exceptionを混同しない。
5. **証跡を最初から残す。** 後から監査用に整えるのではなく、Run Log、Approval Log、Evaluation Plan、Risk Registerを運用に組み込む。
6. **例外には期限を付ける。** 例外承認は、期限、Owner、再確認日、廃止条件を持たせる。
7. **棚卸しと廃止を運用に入れる。** 使われていないAI、古いプロンプト、危険になった外部連携を残さない。
8. **教育履歴と権限付与を接続する。** 高リスク操作を扱う人には、役割に応じた教育と評価を求める。

## 演習

### 前提

あなたは、社内AI推進担当である。次のAI利用が社内に存在することが分かった。

```text
1. 営業部：提案書の下書きに汎用チャットAIを使っている
2. CS部：顧客問い合わせをAIで要約し、返信案を作っている
3. 開発部：コード補助AIでPR作成を支援している
4. 管理部：契約書レビュー補助にAIを使っている
5. 一部チーム：外部MCPを使って社内Wikiとチケットを検索している
```

社内には、正式なAI利用ポリシー、台帳、承認フローがまだない。

### 作成するもの

```text
1. AI Use Policyの1ページ版
2. AI System Inventoryを5件
3. Governance Level MatrixによるG0〜G5分類
4. 高リスク用途のAI Risk Registerを3件以上
5. Approval Workflow
6. AI Vendor Review Recordの確認項目
7. AI Incident Runbookの初動手順
8. 90日以内に実施する棚卸し・教育・改善アクション
```

### 演習の判定基準

- ツール名だけでなく、ユースケース単位で台帳化している
- 顧客情報、契約書、ソースコード、外部MCPを高めのリスクとして扱っている
- 許可、条件付き許可、要承認、禁止の区分がある
- Business Owner、System Owner、Data Owner、Risk Ownerを分けている
- G3以上に台帳、ログ、レビュー、Owner、定期レビューを設定している
- G4以上に事前承認、HITL、評価、監査証跡を設定している
- 外部AI、外部MCP、ベンダー審査を含めている
- AI利用違反または情報漏洩時の初動が定義されている
- 教育履歴と再受講を統制に入れている
- 廃止・棚卸しが含まれている

本章では、AIガバナンスを社内運用へ落とすための参照軸として、次を想定している。

```text
出典キー: SRC-ISO-42001, SRC-ISO-AIMS, SRC-NIST-AI-RMF, SRC-NIST-GENAI-PROFILE, SRC-EU-AI-LITERACY, SRC-OWASP-LLM, SRC-OWASP-AGENTIC

```

- ISO/IEC 42001: AIマネジメントシステムを確立、実装、維持、継続改善するための標準
- NIST AI Risk Management Framework: AIリスク管理を Govern / Map / Measure / Manage の機能で整理する枠組み

法令、規制、認証、契約上の義務は、対象地域、業界、顧客契約、利用AI、データ分類により変わる。本章のテンプレートは、法務判断そのものではない。実導入時には、法務、セキュリティ、個人情報保護担当、必要に応じて外部専門家と確認する。

## まとめ

AIガバナンスは、AI利用を抑制するための制度ではない。安全にスケールさせるための運用基盤である。

個人利用の段階では、よい依頼、よいコンテキスト、よいレビューで十分に見える。しかし、AIが部門をまたぎ、顧客情報を扱い、ツールを実行し、業務成果物に組み込まれると、組織としての統制が必要になる。

その統制の中心は、次の6つである。

```text
1. AI利用ポリシー
2. AI System Inventory
3. リスク分類
4. 承認フロー
5. 証跡と監査
6. 棚卸しと廃止
```

ガバナンスが重すぎると、AI活用は止まる。ガバナンスが弱すぎると、事故が起きる。目指すべきは、低リスク利用を速くし、高リスク利用を確実に制御する設計である。

次章では、統制されたAI利用の先に残る、人間側の判断を扱う。AIが情報を集め、案を出し、実行を支援するほど、人間は「何を採用するか」「何をやめるか」「どのリスクを受け入れるか」を決めなければならない。

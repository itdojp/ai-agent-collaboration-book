# 第12章 困難な会話と対立処理

## この章で扱う能力

この章では、AIエージェント協働が進むほど表面化する、人間同士の困難な会話と対立処理を扱う。

読者は、次のことができるようになる。

- AIで処理できる論点整理と、人間が直接扱うべき対話を分ける
- Connection Debtを診断し、放置コスト、関係者、論点、次の会話を明確にする
- 困難な会話を、感情論ではなく業務成果、責任境界、期待値、合意条件として扱う
- AIを使って相手視点、共通目的、事実、仮説、要求、境界線を整理する
- AIを論破、回避、責任転嫁、バックチャネル増幅に使わない
- 対立を、事実、優先順位、責任、品質、リスク、価値観、行動の種類に分ける
- Difficult Conversation Briefを作り、会話の目的、合意したいこと、伝えること、聞くことを準備する
- 会話後にAgreement Logを残し、判断、タスク、レビュー、ガバナンスへ接続する

第11章では、AI時代の判断力を扱った。AIが情報、選択肢、提案、反証を作れるほど、人間には採用、棄却、停止、やめる判断が求められる。

しかし、判断が進まない理由は、情報不足だけではない。多くの場合、判断を止めているのは、**誰かに言うべきことを言っていないこと**、**責任境界が曖昧なままになっていること**、**対立を避けていること**である。

第12章では、この未処理の対話を扱う。


第12章の対象は、AIエージェントの技術的な設計ではない。AIエージェントが業務に入った結果、**人間側で処理しなければならない未解決の論点**である。

AIは、会議メモを要約できる。論点を整理できる。相手視点を想定できる。伝え方の案を作れる。ロールプレイもできる。反論への応答も準備できる。

しかし、AIは次のことを代替できない。

- 本人に直接伝えること
- 相手の反応を受け止めること
- 自分の要求や懸念を明確にすること
- 責任境界を合意すること
- 関係修復や期待値調整を行うこと
- 権限を持つ人が決めること
- チーム内で守るべき行動を決めること

AIを使うほど、文章は整う。論理も整う。だからこそ、対話を避けることも簡単になる。

本章の目的は、AIで「角の立たない文面」を作ることではない。**AIを使って対立の構造を明確にし、人間が直接扱うべき会話へ進むこと**である。

## なぜAI時代に対立処理が重要になるのか

AIエージェントは、個人の処理能力を引き上げる。

調査、比較、下書き、コード生成、レビュー補助、チケット整理、顧客返信案作成、FAQ検索、議事録作成、提案書構成、障害報告書ドラフトなどは速くなる。

その結果、組織では次の変化が起きる。

```text
・個人が扱うタスク量が増える
・小さなチームでも大きな成果物を作れる
・意思決定とレビューの回転数が上がる
・未解決の認識違いが早く表面化する
・言わないまま進めた前提のズレが、後工程で大きな手戻りになる
・人間同士の合意不足が、AIの出力品質問題に見える
```

AI導入で起きる問題のすべてが、AIの問題とは限らない。

たとえば、次の状態を考える。

| 表面的な問題 | 実際の問題 |
|---|---|
| AIが営業向け提案書をうまく書けない | 営業と開発で、提案の前提、機能範囲、納期の認識が違う |
| AIがチケットをうまく分解できない | プロダクト責任者が優先順位を決めていない |
| AIレビューで指摘が多すぎる | チーム内で品質基準が合意されていない |
| AIが顧客返信案を過剰に丁寧にする | CSと法務で顧客への説明範囲が合意されていない |
| AIが古い情報を参照する | 誰も古い資料の廃止責任を持っていない |
| AIが判断を保留する | 判断者と承認者が分かれていない |

このような場合、プロンプトを直しても根本解決しない。Request Contract、Context Pack、Task Brief、権限マトリクスを整えても、最後に残るのは人間同士の合意である。

AI時代に対立処理が重要になる理由は、対立が増えるからではない。**対立を処理しないコストが、以前より速く、広く、強く現れるから**である。

## 本章における対立の定義

本書では、対立を次のように定義する。

```text
対立とは、関係者の間で、事実認識、優先順位、責任、期待、品質基準、リスク許容度、行動、価値観のいずれかが一致しておらず、その不一致が業務成果、意思決定、信頼、速度に影響している状態である。
```

対立は、悪いものとは限らない。

良い対立は、品質を上げる。リスクを見つける。選択肢を増やす。意思決定を強くする。盲点を減らす。

悪い対立は、本人に言わない不満、陰での批判、曖昧な責任、繰り返す手戻り、相手の意図の決めつけ、無言の抵抗、会議後の不実行として現れる。

本章では、対立をなくすことを目標にしない。目標は、次の状態にすることである。

```text
・対立の種類が分かっている
・本人に直接扱うべき論点が明確である
・AIは論点整理と準備に使われている
・会話の目的と合意条件が決まっている
・会話後の合意、責任、期限、次アクションが記録されている
・判断、実験、改善へ接続されている
```

対立は、適切に扱えば組織の学習データになる。放置すれば、組織負債になる。

## Connection Debtとは何か

本書では、Connection Debtを次のように定義する。

```text
Connection Debtとは、重要な相手に対して本来伝えるべきことを伝えず、
論点、期待、懸念、責任境界、合意条件が未整理のまま蓄積する組織上の負債である。
```

Connection Debtは、単なる「仲が悪い状態」ではない。

次のような実務上の負債である。

```text
・本当は優先順位を決めるべきなのに、曖昧なまま進めている
・本当は品質基準に同意していないのに、表面上は合意したことになっている
・本当は納期に無理があるのに、誰も言わない
・本当は顧客に約束すべきでないことを、営業資料に残している
・本当はコードレビューの基準が合っていないのに、毎回コメントで揉める
・本当はAI出力を信頼していないのに、レビューを省略している
・本当は誰もOwnerではないのに、全員が誰かがやると思っている
```

Connection Debtが蓄積すると、次の症状が出る。

| 症状 | 具体例 |
|---|---|
| 会議が増える | 同じ論点を何度も話す |
| レビューが荒れる | 成果物ではなく人格や姿勢の話になる |
| AI出力への不満が増える | 実際には前提や基準が未合意である |
| バックチャネルが増える | 本人ではなく周囲にだけ話す |
| 判断が遅れる | 誰かが不満を持っているが、会議では言わない |
| 無言の抵抗が起きる | 合意したはずのタスクが進まない |
| 手戻りが増える | 最初に確認すべき期待値が後から出る |
| 責任が曖昧になる | 失敗時に誰も判断者ではない状態になる |

Connection Debtは、AIエージェント活用を直接阻害する。

なぜなら、AIは未合意の前提を解決できないからである。AIは与えられたContext PackやRequest Contractに従って処理する。しかし、Context Packに入る前提そのものが人間同士で合意されていなければ、AIは合意不足を増幅する。

## 対立の種類

困難な会話を始める前に、対立の種類を分ける。

対立をすべて「性格が合わない」「コミュニケーションが悪い」と捉えると、解決できない。多くの対立は、業務設計上の論点に分解できる。

| 対立の種類 | 内容 | 例 | AIで準備できること |
|---|---|---|---|
| 事実対立 | 何が起きているかの認識が違う | 顧客要望の解釈が違う | ログ、議事録、データ、発言の整理 |
| 優先順位対立 | 何を先にやるかが違う | 営業は納期、開発は品質を重視 | 判断基準、トレードオフ整理 |
| 範囲対立 | どこまでやるかが違う | MVP範囲が曖昧 | スコープ案、除外項目、受け入れ条件 |
| 責任対立 | 誰がOwnerかが違う | FAQ更新責任が不明 | RACI、Owner候補、承認者整理 |
| 品質対立 | どの品質で十分かが違う | レビュー基準が厳しすぎる / 甘すぎる | DoD、レビュー観点、Severity整理 |
| リスク対立 | どのリスクを許容するかが違う | 顧客返信自動化の許容範囲 | リスク分類、統制案、残リスク |
| 価値観対立 | 何を大事にするかが違う | 速度か安全性か、個別対応か標準化か | 共通目的、非交渉条件の整理 |
| 行動対立 | 相手の行動が業務に影響している | レビューが遅い、会議で決めない | 具体行動、影響、要望の整理 |
| 関係対立 | 信頼、尊重、心理的安全性が損なわれている | 否定的な言い方、無視、陰口 | 事実と解釈の分離、会話準備 |
| 権限対立 | 誰が決めるかが曖昧 | AI導入を現場が勝手に進める | Decision Rights Matrix、承認経路整理 |

この分類を使うと、会話の目的が明確になる。

悪い会話は、こう始まる。

```text
最近、開発チームが協力的ではないと思います。
```

良い会話は、こう始まる。

```text
次の顧客向け機能について、営業側は7月提案に含める前提で動いていますが、開発側は要件未確定として扱っています。
これは優先順位対立と範囲対立です。
今日決めたいのは、7月提案に含める最小範囲と、含めない項目です。
```

対立を分類するだけで、会話はかなり改善する。

## AIを使ってよいこと、使ってはいけないこと

AIは、困難な会話の準備に有効である。ただし、使い方を誤ると、対立を悪化させる。

### AIを使ってよいこと

| 用途 | 内容 | 例 |
|---|---|---|
| 論点整理 | 何が未合意かを整理する | 「合意すべき論点を抽出して」 |
| 相手視点の想定 | 相手の合理性を推定する | 「開発側の懸念を合理的に整理して」 |
| 共通目的の整理 | 争点の上位目的を探す | 「双方が共有している目的を整理して」 |
| 事実と解釈の分離 | 起きたことと意味づけを分ける | 「事実、解釈、感情、要求に分類して」 |
| 会話構成案 | 話す順番を作る | 「15分で話す構成にして」 |
| 質問案 | 相手に確認すべき質問を作る | 「決めつけにならない質問を作って」 |
| リハーサル | 想定反応を練習する | 「相手役として反論して」 |
| 言い換え | 攻撃的な表現を業務表現にする | 「非難ではなく観察と要望に書き換えて」 |
| 合意文案 | 会話後の確認メモを作る | 「合意事項と次アクションを整理して」 |

### AIを使ってはいけないこと

| 用途 | なぜ危険か |
|---|---|
| 相手を論破する台本を作る | 会話が勝敗になり、合意形成にならない |
| 相手の人格を分析させる | 不正確で失礼な決めつけになりやすい |
| 本人に話さず周囲向け説明文だけ作る | バックチャネルを増やし、信頼を損なう |
| 責任ある対話をAI文面だけで済ませる | 相手の反応を受け止めないため、関係修復にならない |
| 感情を完全に消す | 重要な懸念や境界線まで消える場合がある |
| 録音・議事録・個人情報を無造作に入力する | プライバシー、労務、法務、セキュリティ上の問題がある |
| ハラスメント、差別、労務問題をAIだけで処理する | 専門部署、法務、人事、コンプライアンスへ接続すべき領域である |

AIは、会話の代行者ではない。**会話の準備者**である。

## 困難な会話の基本原則

本章では、困難な会話を次の原則で扱う。

### 原則1: 本人に言うべきことを、本人以外に言い続けない

準備のために第三者へ相談することはある。しかし、本人に言うべき論点を周囲にだけ話し続けると、Connection Debtは増える。

相談は、次の目的に限定する。

```text
・論点を整理する
・安全性を確認する
・自分の見方が偏っていないか確認する
・適切な場、順番、相手を決める
・人事、法務、セキュリティに接続すべきか判断する
```

相談が、次の状態になっているなら危険である。

```text
・本人に言わない不満の共有になっている
・味方集めになっている
・相手の意図を決めつけている
・本人に確認すべき事実を確認していない
・会話を避けるための正当化になっている
```

### 原則2: 攻撃ではなく、業務影響として話す

「あなたはいつも遅い」ではなく、「レビューが平均3営業日遅れ、リリース判断が毎回後ろ倒しになっている」と話す。

人格ではなく、観察可能な行動、業務影響、必要な変化を扱う。

```text
悪い例：
あなたは責任感がない。

良い例：
直近3回の顧客返信レビューで、期限当日の夜までコメントがなく、CS側で最終確認時間を確保できませんでした。
次回から、期限前日の17時までにBlockerだけでも確認してほしいです。
```

### 原則3: 正しさより、次の合意を作る

困難な会話では、自分の正しさを証明したくなる。AIを使うと、根拠、反論、資料、言い換えが大量に作れるため、その傾向は強くなる。

しかし、目的は論破ではない。目的は、次のいずれかである。

```text
・事実確認を合意する
・優先順位を決める
・責任境界を決める
・品質基準を決める
・リスク許容度を決める
・次の実験を決める
・やめることを決める
・コミュニケーション方法を変える
```

会話のゴールは、「相手を納得させる」ではなく、「次の合意を作る」と定義する。

### 原則4: 感情を消さず、業務論点に翻訳する

感情は悪ではない。怒り、不安、違和感、失望、焦りは、何かの境界線、期待、リスク、未処理の論点を示している場合がある。

ただし、感情をそのままぶつけると、相手は防衛的になる。

本章では、感情を次のように翻訳する。

| 感情 | 背後にある可能性 | 業務論点への翻訳 |
|---|---|---|
| 怒り | 期待が破られた | 期待値、合意、期限、責任境界 |
| 不安 | リスクが見えている | リスク分類、停止条件、承認条件 |
| 失望 | 約束が守られていない | 約束、実行状況、再発防止 |
| 焦り | 時間制約がある | 判断期限、優先順位、スコープ |
| 違和感 | 前提が合っていない | Context Pack、情報鮮度、未確定事項 |
| 苛立ち | 同じ問題が繰り返されている | 改善バックログ、Owner、測定指標 |

感情を否定しない。感情を、会話で扱える論点へ変換する。

### 原則5: 安全性が必要な問題は、1対1で抱え込まない

すべての困難な会話を当事者間で処理すればよいわけではない。

次の場合は、管理職、人事、法務、セキュリティ、コンプライアンス、外部専門家へ接続する。

```text
・ハラスメント、差別、報復、不当な圧力が疑われる
・労務、評価、解雇、懲戒、採用に関わる
・顧客情報、個人情報、機密情報が関わる
・不正、隠蔽、セキュリティ違反が疑われる
・権力差が大きく、直接会話で不利益が生じる可能性がある
・相手の反応に安全上の懸念がある
```

AIは、この判断を代替しない。AIに相談する場合も、社内規程に従う。

## Connection Debt診断

Connection Debtは、感覚だけで扱うと曖昧になる。診断シートで可視化する。

```text
Connection Debt ID：
関係者：
自分の役割：
相手の役割：
対象業務 / プロジェクト：

言っていないこと：
言っていない期間：
言っていない理由：
放置している業務コスト：
放置している関係コスト：
放置している判断コスト：

対立の種類：事実 / 優先順位 / 範囲 / 責任 / 品質 / リスク / 価値観 / 行動 / 関係 / 権限
関係する成果物：Decision Brief / Request Contract / Context Pack / Task Brief / Review Issue Log / AI Risk Register / その他
相手が見ている合理性：
自分が守ろうとしているもの：
共通目的：

AIに準備させること：
人間が直接話すこと：
第三者を入れるべき条件：
会話の場：1on1 / チーム会議 / レビュー会 / 承認会議 / 人事・法務同席 / その他
会話の期限：
会話後に確認すること：
```

Connection Debt診断で重要なのは、「言っていないこと」を特定することである。

ただし、「言っていないこと」は、相手を攻撃する表現ではなく、業務上扱うべき論点に翻訳する。

| 生の表現 | 診断シート上の表現 |
|---|---|
| 開発が協力的ではない | 顧客提案に含める機能範囲について、営業と開発で合意できていない |
| あの人はレビューが雑 | レビューでBlocker / Major / Minorの基準が共有されていない |
| CSが勝手に約束する | 顧客への説明範囲と承認条件が合意されていない |
| セキュリティが遅い | 高リスクAI利用の審査前提と必要証跡が事前に共有されていない |
| 管理職が決めない | 判断者、承認者、判断期限がDecision Briefに記録されていない |

Connection Debtは、「誰が悪いか」ではなく、「どの未合意が業務を止めているか」として扱う。

## Connection Debtのレベル

すべてのConnection Debtを同じ重さで扱う必要はない。重さに応じて対応を変える。

| レベル | 状態 | 例 | 対応 |
|---|---|---|---|
| C0 | 未処理なし | 認識違いがない | 通常運用 |
| C1 | 軽い違和感 | 一度だけ期待値がズレた | 早めに短い確認 |
| C2 | 繰り返す摩擦 | レビュー、期限、品質で同じズレが出る | 1on1で論点整理 |
| C3 | 業務影響あり | 判断遅延、手戻り、顧客影響が出ている | Difficult Conversation Briefを作り、期限付きで会話 |
| C4 | 組織影響あり | チーム間対立、責任不明、複数案件へ波及 | 管理職・関係責任者を含めた合意形成 |
| C5 | 高リスク / 安全性懸念 | ハラスメント、報復、法務、セキュリティ、重大顧客影響 | 人事、法務、セキュリティ、経営、専門部署へエスカレーション |

C1の段階で扱えば、会話は短い。C3まで進むと、会話だけでなく、成果物、承認、Owner、期限、再発防止が必要になる。C5は、個人のコミュニケーション技術で処理しない。

## Conflict Type Map

Connection Debtを診断したら、次に対立の型を整理する。

```text
Conflict ID：
対象プロジェクト：
関係者：
対立の主分類：事実 / 優先順位 / 範囲 / 責任 / 品質 / リスク / 価値観 / 行動 / 関係 / 権限
副分類：

未合意の論点：
確認すべき事実：
決めるべきこと：
変えるべき行動：
合意すべき成果物：
関係するリスク：
関係する期限：
判断者：
承認者：
会話の場：
```

Conflict Type Mapは、感情的な対立を、扱える単位へ分解するための成果物である。

たとえば、「営業と開発が揉めている」だけでは、会話を設計できない。

Conflict Type Mapにすると、次のようになる。

```text
対立の主分類：範囲対立、優先順位対立、リスク対立
未合意の論点：7月提案に含める機能範囲
確認すべき事実：顧客が本当に必要としている最小条件、契約上の期待値、開発工数、運用負荷
決めるべきこと：提案書に記載する範囲、記載しない範囲、顧客への言い方
変えるべき行動：営業は開発確認前に納期を約束しない。開発は見積もり不能な場合、不能理由と確認項目を24時間以内に返す。
合意すべき成果物：顧客提案用Request Contract、機能範囲表、リスク説明文
判断者：事業責任者
承認者：プロダクト責任者、開発責任者
```

これで、会話は「感情のぶつけ合い」ではなく、合意形成になる。

### AIへの依頼例


```text
2. 業務影響が具体的に書けているか

6. 人事、法務、セキュリティへ接続すべきリスクはないか
```

## Perspective Map

困難な会話では、自分の正しさだけを見やすい。AIは、相手の合理性を仮説として整理するのに使える。

```text
Perspective Map

相手：
相手の役割：
相手が責任を持っている成果：
相手が見ている制約：
相手が恐れている失敗：
相手が守ろうとしているもの：
相手から見たこちらの問題：
相手が納得しやすい根拠：
相手が受け入れにくい表現：
共通目的：
相手に確認すべき質問：
```

Perspective Mapで重要なのは、「相手は間違っている」という前提を一度停止することである。

相手の合理性を理解することは、相手に同意することではない。合意形成の入口である。

### AIへの依頼例

```text
以下の対立について、相手側の合理性を最大限に推定してください。
ただし、相手を正当化するのではなく、合意形成のために、相手が何を守ろうとしているのかを整理してください。

出力形式：
- 相手が責任を持っている成果
- 相手が見ている制約
- 相手が恐れている失敗
- 相手から見たこちらの問題
- 共通目的
- 相手に確認すべき質問
```

## Conversation Plan

BriefとPerspective Mapを作ったら、会話の進行を設計する。

会話は、次の順序で進める。

```text
1. 目的を共有する
2. 事実を確認する
3. 自分の解釈と影響を伝える
4. 相手の見方を聞く
5. 未合意の論点を明確にする
6. 選択肢を出す
7. 合意する
8. Owner、期限、確認方法を決める
9. 記録する
```

Conversation Planのテンプレートは次のとおりである。

```text
Conversation ID：
所要時間：
場：1on1 / 会議 / レビュー / 承認会議 / その他
参加者：

冒頭の目的共有：
確認する事実：
伝える業務影響：
相手に聞く質問：
未合意論点：
合意候補：
合意できない場合の次アクション：
記録方法：
フォローアップ日：
```

会話の冒頭では、相手を防衛的にしない目的共有が重要である。

```text
悪い例：
今日は、あなたのレビューが遅い件について話したいです。

良い例：
リリース前レビューの遅延が続いていて、CS側の顧客返信準備にも影響が出ています。
次回から期限内に判断できる状態にしたいので、レビューの前提、粒度、期限をすり合わせたいです。
```

## Agreement Log

困難な会話は、話して終わりにしない。会話後にAgreement Logを残す。

```text
Agreement ID：
Conversation ID：
日付：
参加者：
対象業務：

合意したこと：
合意していないこと：
決めたOwner：
期限：
次のアクション：
確認方法：
関連成果物：Request Contract / Context Pack / Task Brief / Decision Brief / Review Issue Log / AI Risk Register / その他
更新する文書：
フォローアップ日：
再度会話する条件：
```

Agreement Logを残す理由は、相手を監視するためではない。合意を業務システムに接続するためである。

会話で合意したことは、次のどれかに反映する。

| 合意内容 | 反映先 |
|---|---|
| 依頼の目的、成果物、制約 | Request Contract |
| 前提、社内ルール、情報鮮度 | Context Pack |
| タスク、Owner、停止条件 | Task Brief / Stop Rules |
| レビュー観点、Severity | AI出力レビュー表 / Review Issue Log |
| 権限、承認条件 | ツール権限マトリクス / HITL承認フロー |
| リスク、残リスク | AI Risk Register / Risk Acceptance Memo |
| 判断、見直し日 | Decision Brief / Decision Review Log |
| 改善項目 | Improvement Backlog |

合意を会話の中だけに残すと、また同じ対立が起きる。合意は成果物に反映する。

## Follow-up Contract

困難な会話の後、最も多い失敗は、合意したのに続かないことである。

Follow-up Contractを作る。

```text
Follow-up ID：
Agreement ID：
確認日：
確認者：
対象アクション：
成功条件：
進捗確認方法：
未実行の場合の扱い：
再会話する条件：
エスカレーション条件：
完了条件：
```

Follow-up Contractは、会話の成果を行動へ変えるための最小契約である。

## Backchannel Control Rule

バックチャネルとは、本人に直接扱うべき論点を、本人以外にだけ話し続ける状態である。

バックチャネルはゼロにはできない。準備、相談、安全確認、専門家確認は必要である。ただし、長期化すると信頼を壊す。

本章では、Backchannel Control Ruleを次のように定義する。

```text
本人に伝えるべき業務論点を第三者へ相談する場合、相談の目的を、準備、安全確認、専門家確認、場の設計に限定する。
相談後は、本人に直接話すか、正式なエスカレーション経路に乗せるか、扱わないと判断する。
本人以外への不満共有を継続しない。
```

Backchannel Control Ruleのテンプレートは次のとおりである。

```text
対象論点：
本人：
第三者へ相談する目的：準備 / 安全確認 / 専門家確認 / 場の設計 / その他
相談相手：
相談で確認すること：
本人へ話す期限：
正式エスカレーションへ切り替える条件：
相談後にやめること：
```

バックチャネルを減らすには、チームで次のルールを持つ。

```text
・本人に言える形に論点を翻訳する
・本人に言わないまま不満共有を繰り返さない
・相談するなら、次の会話のために相談する
・安全性、法務、労務、セキュリティは正式ルートへ接続する
・会話後の合意は記録し、業務成果物に反映する
```

## 実務ケース1: 開発と営業の対立

### 状況

営業は、ある顧客から強い要望を受けて、次回提案に新機能を含めたいと考えている。開発は、要件が曖昧で、既存アーキテクチャへの影響も見えていないため、提案に含めるべきではないと考えている。

表面的には、営業と開発の対立である。

しかし、対立の種類を分けると、次のようになる。

| 種類 | 内容 |
|---|---|
| 事実対立 | 顧客が本当に求めている最小条件が不明 |
| 範囲対立 | 提案に含める機能範囲が未定 |
| 優先順位対立 | 短期商談と中長期品質の優先度が違う |
| リスク対立 | 顧客約束リスクと技術負債リスクの許容度が違う |
| 責任対立 | 顧客への説明責任と実装責任の境界が不明 |

### Connection Debt診断

```text
言っていないこと：
営業は「開発が顧客理解を軽視している」と感じている。
開発は「営業が技術リスクを理解せずに約束しようとしている」と感じている。

放置コスト：
提案書作成が遅れる。顧客期待が高まりすぎる。開発見積もりが不正確になる。

共通目的：
顧客に価値を出しつつ、実装できない約束をしない。

人間が直接話すこと：
提案に含める最小範囲、含めない項目、顧客への表現、開発確認の期限。
```

### AIへの準備依頼

```text
営業側と開発側の合理的な懸念を、それぞれ整理してください。
目的は論破ではなく、次回の30分会議で合意すべき論点を明確にすることです。

出力：
1. 営業側の合理性
2. 開発側の合理性
3. 確認すべき事実
4. 合意すべき論点
5. 提案書に含めてよい表現
6. 提案書に含めてはいけない表現
7. 会議後に更新すべき成果物
```

### 会話で決めること

```text
・顧客に確認すべき最小要件
・7月提案に含める範囲
・含めない範囲
・提案書での表現
・開発が見積もりを返す期限
・顧客に追加確認するOwner
```

### Agreement Log

```text
合意したこと：
7月提案では「検討中の拡張機能」として記載し、納期や確定仕様は書かない。
営業は顧客に最小要件を確認する。
開発は3営業日以内に技術影響の一次見解を返す。

更新する文書：
提案書Request Contract、機能範囲表、リスク説明文。

フォローアップ日：
次回営業・開発同期会。
```

このケースでは、AIは提案文を作る前に、対立の構造を整理するために使う。

## 実務ケース2: AIレビューをめぐる対立

### 状況

開発チームでAIコードレビュー補助を導入した。AIは多くの指摘を出す。若手エンジニアは「学習になる」と感じているが、シニアエンジニアは「ノイズが多く、レビュー時間が増えている」と感じている。

この対立は、AIの精度問題だけではない。

| 対立の種類 | 内容 |
|---|---|
| 品質対立 | どの指摘を価値ある指摘とみなすか |
| 運用対立 | AI指摘を誰がトリアージするか |
| 責任対立 | AI指摘を採用する責任が誰にあるか |
| 学習価値対立 | 教育効果とレビュー効率のどちらを優先するか |

### Difficult Conversation Brief

```text
会話の目的：
AIレビュー補助を継続するかではなく、どの指摘をレビュー対象に残し、どの指摘を抑制するかを決める。

確認したい事実：
AI指摘数、採用率、False Positive率、レビュー時間、若手の学習効果。

業務上の影響：
レビュー時間が増えている。重要指摘がノイズに埋もれる可能性がある。

相手に聞きたいこと：
どの指摘が学習に役立っているか。
どの指摘がレビュー効率を下げているか。

合意したい次アクション：
AI指摘をBlocker / Major / Learning / Noiseに分類し、2週間評価する。
```

### 会話後の反映先

| 合意内容 | 反映先 |
|---|---|
| 指摘分類 | Review Issue Log |
| 採用率測定 | Evaluation Plan |
| ノイズ抑制 | Request Contract / Tool Config |
| 2週間評価 | Metrics Dashboard |
| 継続判断 | Decision Review Log |

AIレビューの対立は、AIのオン・オフではなく、評価設計と運用設計に変換する。

## 実務ケース3: 管理職が部下に言うべきことを避けている

### 状況

あるメンバーはAIを積極的に使っているが、AI出力の確認が甘く、レビューで手戻りが多い。管理職は注意すべきだと分かっているが、本人の意欲を削ぎたくないため、明確に伝えていない。

表面上は、AI出力品質の問題である。しかし実際には、管理職のConnection Debtである。

### Connection Debt診断

```text
関係者：管理職、担当メンバー
言っていないこと：AI利用自体は良いが、確認不足のまま提出しているため、レビュー負荷が上がっている。
言っていない期間：3週間
放置コスト：レビュー時間増加、他メンバーの不満、本人の学習機会損失
相手が見ている合理性：AIを使って速く進めたい。自分で試行錯誤している。
自分が守ろうとしているもの：本人の意欲、関係性
共通目的：AIを使って成果物品質と速度を両立する
人間が直接話すこと：AI利用の禁止ではなく、提出前レビュー基準を守る必要があること
```

### 直接伝える一文

```text
AIを使って作業速度を上げようとしている点は良いです。
一方で、直近3件はAI出力の事実確認と根拠確認が不足していて、レビューでMajor指摘が増えています。
次回から、提出前にAI出力レビュー表の事実、根拠、リスクの3項目だけは必ず埋めてから出してください。
```

この会話は、叱責ではない。期待値の明確化である。

### Follow-up Contract

```text
対象アクション：提出前にAI出力レビュー表を埋める
成功条件：次の3件でMajor指摘が半減する
確認方法：Review Issue Logで確認
未実行の場合：再度1on1で障害を確認し、提出フローを変更する
フォローアップ日：2週間後
```

## 実務ケース4: セキュリティと事業部の対立

### 状況

事業部は、外部AIツールを使って営業資料作成を効率化したい。セキュリティ部門は、顧客情報や未公開情報が外部送信される可能性を懸念している。事業部は「セキュリティが遅い」と感じ、セキュリティ部門は「事業部が危険を軽視している」と感じている。

この対立は、価値観対立に見えるが、分解すると次のようになる。

| 対立の種類 | 内容 |
|---|---|
| リスク対立 | 外部送信リスクをどう扱うか |
| 権限対立 | 誰が外部AI利用を承認するか |
| 速度対立 | 審査速度と事業スピードのバランス |
| 情報対立 | 何を入力してよいかの分類が曖昧 |

### 会話で扱う論点

```text
・外部AIに入力してよい情報分類
・入力禁止情報
・匿名化・要約・マスキングの要件
・承認不要で使える低リスク用途
・要承認用途
・禁止用途
・審査SLA
・違反時の連絡先
```

### 反映先

| 合意内容 | 反映先 |
|---|---|
| 情報分類 | Data Classification and Handling Matrix |
| 承認区分 | AI Use Policy / Governance Level Matrix |
| 外部AI審査 | AI Vendor Review Record |
| 審査SLA | Approval Workflow |
| 違反時対応 | AI Incident Runbook |

ここで必要なのは、事業部がセキュリティを説得することでも、セキュリティが事業部を止めることでもない。低リスク利用を速くし、高リスク利用を確実に止める統制を合意することである。

## 実務ケース5: AI導入に対する現場の抵抗

### 状況

AI推進担当は、各部門にAI活用を広げたい。現場の一部は、AI利用に消極的である。表向きは「忙しい」「今は優先度が低い」と言っているが、実際には、仕事が変わる不安、評価される不安、AI出力への不信、研修の押し付け感がある。

この状態で、AI推進担当が「便利なので使ってください」と言っても進まない。

### Connection Debt診断

```text
言っていないこと：
現場は、AI導入が業務負荷増や評価不安につながるのではないかと感じている。
AI推進側は、現場が非協力的だと感じている。

放置コスト：
研修受講率は上がるが、実務利用が増えない。
成功事例が一部に偏る。
現場からの改善フィードバックが出ない。

共通目的：
現場の仕事を減らし、品質を上げ、事故を起こさないAI利用にする。

人間が直接話すこと：
AI導入の目的、評価との関係、現場の不安、試行範囲、失敗時の扱い。
```

### Conversation Plan

```text
冒頭：
AI利用を押し付けたいのではなく、現場で実際に負荷が減る用途と、使うと危険な用途を一緒に分けたい。

聞くこと：
今の業務で一番時間が取られている作業は何か。
AIを使うことに不安があるとすれば何か。
使いたくない理由があるとすれば何か。
AI利用が評価にどう影響すると見えているか。

合意候補：
まず1つの低リスク業務だけで2週間試す。
失敗は評価ではなく学習として扱う。
利用ログは監視ではなく改善目的で使う。
```

AI導入の抵抗は、スキル不足だけで説明しない。未処理の不安や評価への懸念を直接扱う。

## 困難な会話を避ける典型パターン

### 失敗1: AIに「角が立たない言い方」だけを求める

角が立たない言い方は有用である。しかし、論点が曖昧なまま表現だけを柔らかくすると、問題は伝わらない。

```text
悪い依頼：
この人に失礼にならないように、やんわり注意する文面を作って。

よい依頼：
次の状況について、観察、業務影響、要望を分けたうえで、相手を責めずに直接伝える一文を3案作って。
```

### 失敗2: AIで正論を補強する

AIに根拠、反論、過去事例を大量に作らせると、自分の正しさを強化しやすい。相手の合理性を見る依頼を必ず入れる。

```text
追加すべき依頼：
相手側の合理性を最大限に推定してください。
こちらの主張の弱い点も指摘してください。
```

### 失敗3: 本人に言わず、周囲にだけ共有する

相談は必要である。しかし、相談が長期化し、本人に話す予定がないなら、それは準備ではなく回避である。

Backchannel Control Ruleを使い、本人に話す期限か、正式エスカレーション条件を決める。

### 失敗4: 会話後に記録しない

困難な会話をしても、合意が記録されなければ再発する。

Agreement Logを残し、Request Contract、Context Pack、Task Brief、Decision Brief、AI Risk Register、Improvement Backlogへ反映する。

### 失敗5: すべてを1回の会話で解決しようとする

Connection Debtが大きい場合、1回の会話で解決しない。

最初の会話の目的を限定する。

```text
第1回：事実確認と論点整理
第2回：選択肢と責任境界の合意
第3回：運用ルールとフォローアップの合意
```

### 失敗6: 相手の人格を問題にする

「責任感がない」「協力的でない」「分かっていない」と言うと、防衛的な会話になる。

観察可能な行動、業務影響、必要な行動に変換する。

### 失敗7: 安全性が必要な問題を1on1で処理しようとする

ハラスメント、差別、不正、情報漏洩、法令違反、労務問題、報復リスクは、通常の1on1で処理しない。正式なエスカレーションを使う。

## 設計原則

第12章の設計原則は次である。

### 原則1: 対立を成果物に変換する

対立を「感情」や「相性」の問題だけにしない。

次の成果物に落とす。

```text
Connection Debt診断
Conflict Type Map
Difficult Conversation Brief
Perspective Map
Conversation Plan
Agreement Log
Follow-up Contract
Backchannel Control Rule
```

成果物にすることで、会話は属人的な気合ではなく、再現可能な業務プロセスになる。

### 原則2: AIは準備、会話は人間

AIは、整理、言い換え、仮説、リハーサル、記録に使う。本人に話すべきことをAIで代替しない。

### 原則3: 相手の合理性を確認する

相手の合理性を理解せずに会話すると、説得や論破になりやすい。Perspective Mapで相手の責任、制約、恐れ、守りたいものを整理する。

### 原則4: 合意を業務システムへ戻す

会話後の合意は、必ず既存成果物へ反映する。

```text
Request Contract
Context Pack
Task Brief
Review Issue Log
AI Risk Register
Decision Brief
Improvement Backlog
```

反映しなければ、同じ対立は再発する。

### 原則5: 対立を実験に変換する

意見が割れるとき、すぐに勝敗を決める必要はない。第13章で扱うように、最小実験に変換できる場合がある。

```text
全社展開するかどうかで対立している
→ 2週間、低リスク問い合わせだけで試す

AIレビューを続けるかで対立している
→ 指摘をBlocker / Major / Learning / Noiseに分類して2週間測る

外部AI利用を許可するかで対立している
→ 匿名化済み公開情報だけで1チームが試す
```

対立は、実験設計の入口になる。

## 章内ワークフロー

第12章の内容を実務で使う場合、次の順序で進める。

```text
1. Connection Debtを疑う
2. Connection Debt診断を作る
3. 対立の種類をConflict Type Mapで分類する
4. AIに事実、解釈、感情、要求を分けさせる
5. Perspective Mapで相手の合理性を整理する
6. Difficult Conversation Briefを作る
7. Conversation Planを作る
8. 必要なら人事、法務、セキュリティ、上長へエスカレーションする
9. 本人または関係者と会話する
10. Agreement Logを残す
11. Follow-up Contractを設定する
12. 合意をRequest Contract、Context Pack、Task Brief、Decision Briefなどへ反映する
13. 対立が判断ではなく検証可能な仮説であれば、第13章のAI実験設計シートへ渡す
```

この順序は、会話を重くするためではない。感情的に難しい会話ほど、構造化して軽く扱うためである。

## 演習1: Connection Debtを診断する

現在、自分が避けている会話を1つ選ぶ。

次のテンプレートを埋める。

```text
Connection Debt ID：
関係者：
言っていないこと：
言っていない期間：
放置している業務コスト：
放置している関係コスト：
対立の種類：
相手が見ている合理性：
共通目的：
AIに準備させること：
人間が直接話すこと：
会話の期限：
```

### 判定基準

- 「言っていないこと」が人格評価ではなく、業務論点になっている
- 放置コストが書かれている
- 対立の種類が分類されている
- 相手の合理性が書かれている
- AIに準備させることと、人間が直接話すことが分かれている

## 演習2: Difficult Conversation Briefを作る

演習1のConnection Debtをもとに、Difficult Conversation Briefを作る。

```text
会話の目的：
会話しない場合のコスト：
会話で決めたいこと：
会話で決めないこと：
確認したい事実：
自分の観察：
自分の解釈：
業務上の影響：
自分の要望：
相手に聞きたいこと：
直接伝える一文：
合意したい次アクション：
```

### 判定基準

- 会話の目的が1つに絞られている
- 自分の観察と解釈が分かれている
- 業務影響が具体的である
- 要望が行動レベルで書かれている
- 相手に聞く質問がある
- 直接伝える一文が、非難ではなく観察、影響、要望になっている

## 演習3: Agreement Logを作る

架空の会話メモをもとに、Agreement Logを作る。

```text
会話メモ：
営業と開発で、7月提案に新機能を含めるかを話した。
営業は顧客の要望が強いと説明した。
開発は仕様未確定で納期を約束できないと説明した。
最終的に、7月提案では確定機能としては書かず、検討中の拡張として記載することにした。
営業は顧客に最小要件を確認する。
開発は3営業日以内に一次見解を返す。
```

提出物：

```text
Agreement Log
Follow-up Contract
更新すべき成果物一覧
```

### 判定基準

- 合意したことと合意していないことが分かれている
- Ownerと期限が書かれている
- 更新すべき成果物が明示されている
- フォローアップ日がある

## まとめ

AIは困難な会話を準備できる。しかし、代替はできない。

AIは、論点を整理し、相手視点を推定し、伝え方を整え、会話をリハーサルし、合意メモを作れる。だが、本人に直接伝えること、相手の反応を受け止めること、責任境界を合意すること、次の行動を決めることは、人間の仕事である。

AIエージェント協働が進むほど、チームは高レバレッジ化する。高レバレッジなチームでは、未処理の対立、曖昧な責任、言えていない懸念が、組織速度を大きく落とす。

したがって、困難な会話は、感情論としてではなく、業務設計として扱う。

```text
Connection Debtを診断する
対立の種類を分類する
AIで論点と相手視点を整理する
人間が直接話す
合意を記録する
成果物へ反映する
フォローアップする
```

第13章では、対立や判断を、失敗耐性と実験文化へ接続する。すべての対立を会議で決着させる必要はない。検証可能な論点は、小さな実験へ変換できる。

# 第4章 Context Packを設計する

## この章で扱う能力

この章では、AIエージェントに渡す前提情報を設計する。読者は、必要情報と不要情報を分け、業務背景、過去経緯、制約、決定済み事項、未決事項、情報鮮度、機密区分、参照してよい範囲を含む **Context Pack** を作れるようになる。

この章の結論は次のとおりである。

```text
AIの出力品質は、依頼文だけでは決まらない。
AIが何を前提に処理するかで決まる。

Context Pack は、資料を丸ごと渡すことではない。
目的に必要な前提を選び、構造化し、鮮度と不確実性と機密区分を明示し、
AIが誤った前提で処理しないようにする制御文書である。
```

第3章では、AIへの依頼を Request Contract として設計した。この章では、その依頼を成立させるために、AIへ何を渡すか、何を渡さないかを設計する。

## 本書全体における位置づけ

| 項目 | 内容 |
|---|---|
| 設計層 | Context |
| 目的 | AIに渡す情報を構造化し、前提、鮮度、不確実性、機密区分を管理する。 |
| 主な成果物 | Context Pack |
| 前章から受け取るもの | Request Contract、AIエージェント依頼書 |
| 次章へ渡すもの | Task Brief に必要な前提、制約、確認事項 |
| 参照 | [Artifact Index](../../artifact-index/)、[付録A: 成果物テンプレート集](../../appendices/appendix-a/)、[Troubleshooting Flow](../../troubleshooting/) |

本書の6層フレームワークでは、第4章は次の位置にある。

```text
Work: どの仕事にAIを使うか
  ↓
Request: AIに何を依頼するか
  ↓
Context: AIに何を渡すか  ← この章
  ↓
Delegation: どう分解し、どこで止めるか
```

Request Contract が「何をしてほしいか」を定義する文書なら、Context Pack は「何を前提にしてよいか」を定義する文書である。

## なぜ必要か

AIの出力品質は、依頼文だけで決まらない。依頼文が明確でも、渡す情報が古い、不完全、矛盾している、または機密情報を含んでいる場合、出力は実務に使いにくくなる。

たとえば、次のような状況を考える。

```text
依頼：顧客向け障害報告の下書きを作る。
渡した情報：過去の障害報告、監視ログ、社内Slackのやり取り、顧客問い合わせ、旧SLA資料。
```

このとき、資料を丸ごと渡すと、AIは次の問題を起こし得る。

```text
古いSLA資料を最新条件として扱う。
社内Slackの未確認発言を確定情報として扱う。
顧客に見せてはいけない内部ログを本文に含める。
原因未確定なのに断定する。
過去事例を今回の原因として混同する。
```

これは「AIが悪い」というより、AIに渡した文脈が制御されていない問題である。

人間同士の仕事でも、前提が違えば成果物はずれる。AIエージェント協働では、そのずれが高速に増幅される。だから、AIに渡す情報は、資料の量ではなく、前提の品質として設計する必要がある。

## Context Pack とは何か

Context Pack とは、AIエージェントに渡す前提情報、制約、関係者、関連資料、情報鮮度、機密区分、不確実性をまとめた文書である。

| 項目 | Request Contract | Context Pack |
|---|---|---|
| 主な問い | AIに何を依頼するか | AIに何を前提として渡すか |
| 中心 | 目的、成果物、制約、受け入れ条件 | 背景、状況、資料、決定済み事項、未決事項 |
| 変化頻度 | 依頼ごとに変わる | 案件や業務の進行で更新される |
| 役割 | 作業の仕様 | 作業の前提 |
| 失敗時の症状 | 出力形式が違う、範囲がずれる | 古い前提、誤った断定、情報漏洩、論点抜け |

Request Contract と Context Pack は分ける。分けることで、同じ依頼を別案件に再利用できる。また、同じ案件の Context Pack を使って、複数の依頼を出せる。

例:

```text
Context Pack：顧客Aのバックアップ障害対応の現状、経緯、制約、未決事項
Request Contract 1：顧客向け一次返信案を作る
Request Contract 2：社内エスカレーション用の要約を作る
Request Contract 3：次回の障害報告書ドラフトを作る
```

Context Pack は、AIのためだけではない。別担当者が途中参加したとき、管理職がレビューするとき、後から失敗を振り返るときにも使える。

## Context Pack の基本構造

Context Pack には、次の要素を含める。

| 要素 | 内容 | 例 |
|---|---|---|
| 案件名 | 対象業務、顧客、プロジェクト名 | 顧客A バックアップジョブ失敗 一次対応 |
| 業務背景 | なぜこの作業が必要か | 顧客から障害疑いの問い合わせが来ている |
| 関係者 | 担当者、承認者、影響を受ける人 | CS担当、SRE、契約責任者 |
| 現在の状況 | 最新ステータス | 原因未確定。影響範囲調査中 |
| 過去の経緯 | 重要な意思決定、失敗、制約 | 過去に障害報告表現で顧客と認識齟齬があった |
| 決定済み事項 | すでに決まっていること | 原因を断定しない。一次返信はCSが送る |
| 未決事項 | これから確認・判断すべきこと | 原因、影響範囲、復旧見込み、SLA影響 |
| 関連資料 | 参照可能な資料、URL、ファイル | 問い合わせ本文、監視アラート要約、FAQ |
| 重要な制約 | 予算、納期、法務、技術、運用 | 内部ログ詳細は顧客に出さない |
| 優先順位 | 何を優先するか | 断定を避け、確認中であることを明確にする |
| 情報の鮮度 | いつ時点の情報か | 2026-06-27 10:00時点 |
| 機密区分 | 公開、社内限定、機密、個人情報、認証情報 | 社内限定。顧客本文は要管理 |
| AIに参照させてよい範囲 | 利用可能情報 | 問い合わせ本文、アラート要約、公開FAQ |
| AIに参照させてはいけない範囲 | 禁止情報 | 認証情報、内部ログ全文、他顧客情報 |
| 確定情報 | 確認済みの事実 | 顧客から問い合わせが来ている |
| 推定情報 | 可能性はあるが未確定 | 一部ジョブに限定される可能性 |
| 要確認事項 | 次に確認すべきこと | 影響範囲、原因、復旧見込み |

この構造は、あらゆる業務で必ず同じ粒度にする必要はない。軽い作業なら短くてよい。顧客影響、法務、障害、価格、人事、セキュリティに関わる場合は、丁寧に作る。

## コンテキストの種類

Context Pack には、複数種類の情報が含まれる。混ぜて書くと、AIも人間も誤解する。種類ごとに分ける。

| 種類 | 内容 | 例 |
|---|---|---|
| 業務コンテキスト | 業務目的、流れ、成果物 | 問い合わせ一次対応、競合調査 |
| ドメインコンテキスト | 業界、製品、技術、顧客属性 | 法人向けバックアップ、SLA、RPO/RTO |
| 組織コンテキスト | 社内ルール、責任者、承認経路 | 顧客送信はCS承認、契約判断は責任者 |
| 案件コンテキスト | 現在の状況、過去経緯 | 顧客Aでジョブ失敗、原因調査中 |
| 成果物コンテキスト | 成果物の利用先、読者、形式 | 経営会議向け、顧客向けメール |
| 判断コンテキスト | 判断基準、優先順位、許容リスク | 速度より正確性、断定回避 |
| 制約コンテキスト | 禁止事項、法務、セキュリティ | 個人情報禁止、内部ログ非公開 |
| 情報源コンテキスト | 参照元、更新日、信頼度 | 公式FAQ、旧資料、未確認メモ |
| ツールコンテキスト | 利用可能ツール、権限 | FAQ検索read-only、チケット下書き |
| 評価コンテキスト | レビュー観点、受け入れ条件 | 出典必須、要確認分離 |

Context Pack を作るときは、情報をただ集めるのではなく、どの種類の情報なのかを明示する。

## 良いコンテキストの条件

良い Context Pack は、単に情報量が多いものではない。次の条件を満たす。

| 条件 | 内容 | 悪い状態 |
|---|---|---|
| 関連性 | 目的に必要な情報だけが入っている | 資料を丸ごと貼る |
| 正確性 | 確認済み情報と未確認情報が分かれている | 推定が事実扱いされている |
| 鮮度 | いつ時点の情報か分かる | 旧資料が最新扱いされる |
| 完全性 | 作業に必要な前提が揃っている | 重要な制約や過去経緯が抜ける |
| 構造化 | 見出し、表、箇条書きで整理されている | 長文が混在している |
| 出典性 | どこから来た情報か分かる | 根拠不明の断定がある |
| 機密管理 | 渡してよい情報だけが入っている | 個人情報、認証情報、内部ログが混ざる |
| 再開性 | 別担当者が読んでも再開できる | 会話履歴依存で分からない |
| 更新性 | 変更点を反映できる | 古い前提が残り続ける |

AIに大量の情報を渡すより、良い Context Pack を渡す方が価値が高い。

## 資料丸投げと Context Pack の違い

資料丸投げは、AIに「重要なものを見つけて」と依頼する状態である。Context Pack は、人間が重要な前提を選び、AIに処理させる状態である。

| 観点 | 資料丸投げ | Context Pack |
|---|---|---|
| 情報量 | 多い | 必要十分 |
| 構造 | 低い | 高い |
| 鮮度 | 混ざりやすい | 明示される |
| 機密 | 混入しやすい | 分類される |
| 未確認情報 | 事実と混ざる | 要確認として分離 |
| レビュー | 難しい | しやすい |
| 再利用 | 低い | 高い |
| 失敗時の原因特定 | 難しい | どの前提が悪かったか追える |

資料丸投げが常に悪いわけではない。探索段階では、AIに資料群から論点を抽出させることもある。ただし、その場合でも、出力をそのまま成果物にせず、抽出結果を Context Pack に整理し直す。

## 情報鮮度を管理する

AIエージェントに渡す情報には、必ず鮮度を付ける。特に、価格、仕様、規程、法令、組織体制、顧客状況、障害情報は変わる。

Context Pack では、次のように書く。

```text
情報の鮮度：
- 顧客問い合わせ本文：2026-06-27 09:30受信
- 監視アラート要約：2026-06-27 10:00時点
- 旧障害報告テンプレート：2025年版。表現ルールが古い可能性あり
- SLA資料：2026年契約版が正。2024年版は参照禁止
```

古い情報を削除できない場合は、「古い可能性がある」「参照禁止」「参考扱い」と明示する。

### 鮮度ラベル

| ラベル | 意味 | 使い方 |
|---|---|---|
| 最新 | 現時点で有効 | 現行規程、最新仕様、直近ログ |
| 参考 | 背景理解には使えるが判断根拠にはしない | 過去調査、旧議事録 |
| 旧版 | 現在は置き換え済み | 旧価格表、旧SLA |
| 未確認 | 事実確認が必要 | Slackの発言、顧客推測、伝聞 |
| 禁止 | AIに参照させない | 認証情報、他顧客情報、未公開財務 |

情報鮮度を明示しない Context Pack は、事故の元になる。

## 確定、推定、仮定、要確認を分ける

AIに渡す情報は、確定情報、推定情報、仮定、要確認事項に分ける。

| 区分 | 意味 | 例 | AIへの扱わせ方 |
|---|---|---|---|
| 確定情報 | 確認済みの事実 | 顧客から問い合わせが来た | 事実として使ってよい |
| 推定情報 | 可能性はあるが未確定 | 一部ジョブのみ影響の可能性 | 断定禁止。推定と明記 |
| 仮定 | 作業上の仮置き | 返信先は既存担当者と仮定 | 仮定欄に分離 |
| 要確認事項 | 次に確認すべきこと | 影響範囲、原因、復旧見込み | 本文と分けて出力 |

悪い Context Pack:

```text
たぶんストレージ障害で、一部顧客に影響が出ています。
```

改善例:

```text
確定情報：バックアップジョブ失敗アラートが複数発生している。
推定情報：ストレージ関連の可能性があるが、原因は未確定。
要確認事項：影響範囲、原因、復旧見込み、顧客影響。
```

この分離により、AIは断定しにくくなる。人間も、何を確認すべきか分かる。

## 情報源の信頼度を分ける

AIに渡す情報は、出典や信頼度を分ける。

| 情報源 | 信頼度 | 扱い |
|---|---:|---|
| 現行の社内規程、公式FAQ、契約書 | 高 | 判断根拠に使える。ただし権限に注意 |
| 最新の監視ログ要約、チケット記録 | 高〜中 | 技術確認に使う。顧客表現には加工が必要 |
| 担当者メモ、Slack発言 | 中〜低 | 未確認情報として扱う |
| 過去の類似事例 | 中 | 参考情報。今回の原因とは断定しない |
| 旧資料、廃止済みテンプレート | 低 | 参考または禁止として明示 |
| 顧客からの推測 | 低 | 顧客認識として扱い、事実とは分ける |

Context Pack では、「どの情報を根拠にしてよいか」を明示する。根拠にしてはいけない情報を混ぜる場合は、参考扱いと書く。

## 機密区分と情報最小化

AIに渡す情報は、必要最小限にする。情報を多く渡すほど、出力に混入するリスク、ログに残るリスク、権限管理のリスクが増える。

Context Pack では、少なくとも次の分類を使う。

| 区分 | 説明 | AI利用の扱い |
|---|---|---|
| 公開 | 公開Webや外部公開済み資料 | 原則利用可 |
| 社内限定 | 社内向け資料、社内FAQ | 許可された環境で利用 |
| 機密 | 事業計画、未公開情報、契約条件 | 原則制限。必要時のみ管理下で利用 |
| 個人情報 | 氏名、連絡先、識別情報、行動履歴 | 原則投入しない。必要時は社内ルールに従う |
| 認証情報 | APIキー、パスワード、トークン | 投入禁止 |
| 他顧客情報 | 別顧客の情報、契約、障害影響 | 原則投入禁止。匿名化が必要 |

情報最小化の基本は次である。

```text
目的に不要な情報は渡さない。
識別子は必要な場合だけ残す。
本文に不要な個人名は役割名へ置き換える。
内部ログは全文ではなく要約を渡す。
認証情報、APIキー、秘密鍵は絶対に渡さない。
```

## Context Pack と RAG、メモリ、ツール結果の関係

AIエージェント協働では、コンテキストは手入力の文章だけではない。RAG、メモリ、ツール結果、会話履歴もコンテキストになる。

| 要素 | 役割 | 注意点 |
|---|---|---|
| Context Pack | その作業の前提を明示する | 人間が選別・構造化する |
| RAG | 外部・社内文書から必要情報を検索する | 検索範囲、鮮度、権限、出典を管理する |
| メモリ | 継続的に使う設定や履歴を保持する | 未確認情報や機密を長期保存しない |
| ツール結果 | API、DB、検索、ログから得た結果 | 実行時点、権限、加工要否を記録する |
| 会話履歴 | これまでのやり取り | 古い前提や誤情報が残る可能性がある |

Context Pack は、これらをすべて置き換えるものではない。むしろ、AIが参照する情報源の上位整理である。

### RAGを使う場合

RAGを使う場合でも、検索対象を指定する。

```text
RAG参照範囲：
- 現行FAQ
- 2026年版サポート手順書
- 顧客向け公開ドキュメント

RAG参照禁止：
- 旧サポート手順書
- 契約個別条件
- 社内Slackログ
- 認証情報を含む可能性があるログ
```

検索結果をAIが使う場合は、出典と更新日を出力させる。出典不明の検索結果は、確定情報として扱わない。

### メモリを使う場合

メモリに保存してよい情報と、保存してはいけない情報を分ける。

| 保存してよい候補 | 保存しないもの |
|---|---|
| チームの文書スタイル | APIキー、パスワード |
| 社内の標準テンプレート | 個人情報 |
| 利用者の出力形式の好み | 顧客固有の機密情報 |
| 承認フローの一般ルール | 未確認の一時情報 |
| よく使うチェックリスト | 旧情報、誤情報 |

メモリは便利だが、間違った情報を長く保持すると、誤りが再発する。定期的な棚卸しと削除条件が必要である。

## 長い作業におけるコンテキスト管理

AIエージェントが長い作業を行う場合、会話履歴、ツール結果、途中出力が増え、重要な前提が埋もれる。これを防ぐため、Context Pack を更新しながら進める。

### 作業中に残すべきメモ

```text
現在の目的：
ここまでの決定：
未決事項：
新しく分かった確定情報：
破棄した仮説：
次の作業：
停止条件に近づいている点：
```

このようなメモを残すと、別セッションや別担当者でも再開しやすい。

### 再開用 Context Pack

長い作業を再開する場合は、会話履歴をそのまま渡すのではなく、再開用に圧縮する。

```text
再開目的：
前回までの成果物：
確定した前提：
変更された前提：
未決事項：
参照してよい資料：
参照してはいけない資料：
次に実施するタスク：
人間確認が必要な条件：
```

これは、第5章で扱う Task Brief や Restart Packet につながる。

## Context Pack の作成手順

実務では、次の順で作る。

### Step 1: Request Contract から必要情報を逆算する

Request Contract の目的、成果物、制約から、必要なコンテキストを洗い出す。

```text
目的 → なぜ必要か、何の判断材料か
成果物 → 誰が読むか、どの粒度が必要か
対象範囲 → どの資料・対象を含めるか
除外範囲 → どの資料・対象を除外するか
制約 → 渡してはいけない情報、断定してはいけない前提
受け入れ条件 → 出典、鮮度、未確認事項の分離
```

### Step 2: 情報源を棚卸しする

次の観点で、使える情報源を列挙する。

```text
公式資料
社内規程
FAQ
過去チケット
議事録
顧客問い合わせ
ログ要約
過去調査
担当者メモ
```

ここではまだAIに渡さない。まず、人間が棚卸しする。

### Step 3: 参照してよい情報と禁止情報を分ける

情報源をそのまま投入しない。参照してよい範囲と禁止範囲を分ける。

```text
参照可：公開FAQ、匿名化済み顧客要望、監視アラート要約
条件付き：社内限定資料、過去チケット要約
禁止：APIキー、個人情報、内部ログ全文、他顧客情報、旧契約書
```

### Step 4: 確定・推定・要確認を分ける

作業時点で分かっていること、分かっていないことを分ける。不明点を隠さない。

### Step 5: 鮮度と出典を付ける

各情報に、日付、版、参照元、信頼度を付ける。

```text
現行FAQ：2026-05-01版
旧FAQ：2024-11-01版。参照禁止
監視アラート要約：2026-06-27 10:00時点
担当者メモ：未確認。参考扱い
```

### Step 6: AIに渡す形に圧縮する

長文資料を渡す前に、目的に必要な部分だけ要約する。全文が必要な場合も、最初に要点と利用ルールを渡す。

### Step 7: Context Pack をレビューする

Context Pack 自体をレビューする。

```text
必要な前提が抜けていないか。
不要な機密情報が入っていないか。
古い情報が最新扱いされていないか。
推定が事実扱いされていないか。
AIに参照させてはいけない範囲が明確か。
別担当者が読んでも作業を再開できるか。
```

## 実務ケース1: 競合調査の Context Pack

第3章の競合調査 Request Contract に対応する Context Pack を作る。

```text
案件名：
法人向けバックアップサービス 競合調査 2026年6月版

業務背景：
SMB向けプランの解約率が上昇しており、価格と機能の見直しを検討している。営業現場から、競合の価格訴求に関する声が上がっているが、比較軸が統一されていない。

関係者：
- 事業責任者：価格改定の最終判断
- 営業責任者：顧客要望の確認
- プロダクト責任者：機能差分の確認
- 調査担当：比較表作成

現在の状況：
- 価格改定は未決定
- 経営会議前の論点整理が必要
- 競合候補は未確定

過去の経緯：
- 前回調査では比較軸が担当者ごとに異なり、会議で再調査になった
- 営業資料に転用した際、出典不明の情報が混ざったことがある

決定済み事項：
- 価格改定の結論はAIに出させない
- 公開情報と許可された社内メモのみ使う
- 顧客名、個別契約価格、未公開商談情報は使わない

未決事項：
- 比較対象5社
- 価格差が解約率に与える影響
- 機能差分の重要度

関連資料：
- 競合候補リスト：2026-06-20作成
- 過去調査メモ：2025-Q4版。古い可能性あり
- 匿名化済み営業ヒアリングメモ：2026-06-10版

重要な制約：
- 価格情報は公開情報を優先する
- 不明な価格は要確認にする
- 個別顧客の契約価格は使わない
- 営業資料ではなく経営会議前の論点整理として作る

優先順位：
1. 比較軸を揃える
2. 出典を付ける
3. 要確認事項を明確にする
4. 価格改定判断は人間に残す

情報の鮮度：
- 2026-06-27時点での調査とする
- 2025-Q4以前の情報は参考扱い

機密区分：
社内限定。顧客名や個別契約価格は含めない。

AIに参照させてよい範囲：
- 公開Web
- 匿名化済み営業ヒアリングメモ
- 過去調査メモ。ただし古い可能性を明記

AIに参照させてはいけない範囲：
- 個別顧客名
- 個別契約価格
- 未公開商談情報
- 認証情報
- 社外秘の価格戦略メモ

確定情報：
- SMB向けプランの解約率が上昇している
- 価格改定は未決定
- 経営会議前に比較表が必要

推定情報：
- 競合価格訴求が解約理由の一部である可能性がある

要確認事項：
- 比較対象5社の確定
- 競合価格の最新性
- 機能差分の重要度
```

この Context Pack があれば、AIは「競合調査」の背景と制約を理解しやすい。重要なのは、過去調査メモを古い可能性ありとして扱い、顧客名や個別契約価格を除外している点である。

## 実務ケース2: 障害一次対応の Context Pack

次に、顧客問い合わせ対応の Context Pack を作る。

```text
案件名：
顧客A バックアップジョブ失敗 一次対応

業務背景：
顧客Aから「昨日からバックアップジョブが失敗している。至急確認してほしい」と問い合わせが来た。一次返信の下書きと社内確認事項を作る。

関係者：
- CS担当：顧客返信の送信判断
- SRE担当：技術調査
- 契約責任者：SLA・補償判断
- 顧客担当営業：顧客関係の確認

現在の状況：
- 顧客から問い合わせを受信済み
- 監視上、ジョブ失敗アラートを確認済み
- 原因は未確定
- 影響範囲は調査中

過去の経緯：
- 顧客Aは過去に障害報告の表現について詳細説明を求めたことがある
- 原因未確定時に断定表現を避ける必要がある

決定済み事項：
- 顧客へは原因を断定しない
- 復旧予定時刻を断定しない
- 一次返信はCS担当がレビュー後に送信する
- SLA影響や補償には触れない

未決事項：
- 原因
- 影響範囲
- 復旧見込み
- SLA影響
- 顧客側で追加取得してほしい情報

関連資料：
- 顧客問い合わせ本文：2026-06-27 09:30受信
- 監視アラート要約：2026-06-27 10:00時点
- 既知障害情報：2026-06-27 10:05時点では該当未確認
- 類似障害報告テンプレート：2026年版

重要な制約：
- 内部ログ全文は顧客向け本文に含めない
- 内部エラーコードの詳細は含めない
- 他顧客情報は含めない
- 原因、復旧予定、補償、SLA違反を断定しない

優先順位：
1. 顧客に受付と確認中であることを伝える
2. 断定を避ける
3. 追加で必要な情報を明示する
4. 社内確認事項を本文と分ける

情報の鮮度：
2026-06-27 10:10時点。以後の調査結果は未反映。

機密区分：
社内限定。顧客問い合わせ本文は要管理。認証情報は含めない。

AIに参照させてよい範囲：
- 顧客問い合わせ本文
- 監視アラートの要約
- 既知障害情報の要約
- 類似障害報告テンプレート

AIに参照させてはいけない範囲：
- 内部ログ全文
- 認証情報
- 他顧客情報
- 内部エラーコードの詳細
- 契約書全文
- 補償判断に関する内部メモ

確定情報：
- 顧客Aから問い合わせが来ている
- バックアップジョブ失敗アラートがある
- 原因は未確定

推定情報：
- 顧客Aの一部ジョブに影響している可能性がある

要確認事項：
- 影響範囲
- 原因
- 復旧見込み
- 顧客から追加で必要な情報
- SLA影響の有無
```

この Context Pack では、AIが使ってよい情報と、使ってはいけない情報が分かれている。また、確定情報と推定情報も分かれている。顧客向けメール下書きを作る場合、この分離が事故防止につながる。

## Context Pack の更新

Context Pack は一度作って終わりではない。業務が進むと、前提が変わる。

更新時には、次を記録する。

```text
更新日時：
更新者：
追加された確定情報：
変更された前提：
破棄された推定：
新しい未決事項：
参照禁止になった情報：
次回レビュー予定：
```

### 更新例

```text
更新日時：2026-06-27 11:30
更新者：SRE担当
追加された確定情報：特定リージョンのバックアップキュー遅延を確認
変更された前提：顧客Aのみではなく、同一リージョンの一部顧客に影響可能性
破棄された推定：顧客側設定ミスの可能性は低い
新しい未決事項：影響顧客数、復旧予定、外部告知要否
参照禁止になった情報：内部ログ詳細は顧客向け文面に含めない
```

更新履歴がないと、AIがどの前提で出力したのか追えない。第8章の Agent Run Log とも接続する。

## Context Pack のレビュー観点

Context Pack 自体をレビューする。AI出力だけではなく、前提文書を確認する。

| 観点 | 確認内容 |
|---|---|
| 目的適合性 | Request Contract に必要な情報が揃っているか |
| 過不足 | 不要な情報が多すぎないか。重要情報が抜けていないか |
| 鮮度 | いつ時点の情報か明示されているか |
| 確度 | 確定、推定、仮定、要確認が分かれているか |
| 出典 | どこから来た情報か分かるか |
| 機密 | 渡してはいけない情報が混ざっていないか |
| 禁止範囲 | AIに参照させてはいけない範囲が明確か |
| 再開性 | 別担当者が読んでも作業できるか |
| 更新性 | 変更履歴や更新予定があるか |
| 整合性 | Request Contract の制約と矛盾していないか |

特に高リスク業務では、Context Pack を作成者以外が確認する。AIに渡す前に、人間が前提をレビューする。

## 典型的な失敗

### 失敗1: 資料を丸ごと渡す

大量の議事録や仕様書をそのまま渡すと、重要情報が埋もれる。AIは要約できるが、どの情報が意思決定上重要かまでは、業務文脈なしには判断できない。

### 失敗2: 古い情報と最新情報を混ぜる

過去の価格表、古い仕様、廃止済みルール、旧顧客条件が混ざると、AIは古い前提で出力する可能性がある。情報には必ず日付と有効性を付ける。

### 失敗3: 暗黙知を渡さない

社内では当然の前提でも、AIには当然ではない。たとえば、「この顧客には以前障害報告で問題があった」「この製品は新規販売停止予定」「この価格は既存顧客だけに適用」といった情報は明示する。

### 失敗4: 機密情報を混ぜる

AIに渡す必要がない顧客名、個人情報、APIキー、社内限定の財務情報を含めると、情報管理上のリスクが増える。Context Pack には機密区分を入れる。

### 失敗5: 不確実性を隠す

未確認の情報、推定、仮説を確定情報のように書くと、AIの出力も断定的になる。Context Pack では、確定、推定、仮定、要確認を分ける。

### 失敗6: 会話履歴に依存する

長い会話の中に前提が散らばっていると、別セッションや別担当者が再開できない。重要な前提は Context Pack に抽出する。

### 失敗7: RAG任せにする

「社内文書を検索できるから大丈夫」と考えると失敗する。検索範囲、旧資料、権限、出典、更新日を管理しなければ、AIは不適切な文書を根拠にする可能性がある。

### 失敗8: メモリを信頼しすぎる

AIが過去の設定や履歴を覚えている場合でも、それが現在も正しいとは限らない。メモリは便利だが、Context Pack で現在の前提を明示する。

## 設計原則

### 原則1: AIに必要な情報だけを渡す

AIに多く渡せばよいわけではない。目的に対して必要な情報を選び、不要な情報は省く。

### 原則2: 情報を構造化する

箇条書き、表、時系列、ID、優先度を使う。AIだけでなく、人間も読み返しやすくなる。

### 原則3: 情報鮮度を明記する

「2026年6月時点」「2025年契約の旧条件」「未確認」「参照禁止」などを明示する。

### 原則4: 機密区分を入れる

AI利用で最初に事故になりやすいのは、必要以上の情報投入である。公開情報、社内限定、機密、個人情報、認証情報を分ける。

### 原則5: 確定情報と不確実情報を分ける

確定、推定、仮定、要確認を分ける。不確実性を隠さない。

### 原則6: 参照してよい範囲を明示する

AIに参照させてよい資料と、参照させてはいけない資料を分ける。RAGや検索ツールを使う場合も同じである。

### 原則7: 再開可能にする

長い仕事では、一度の会話で終わらない。Context Pack は、別のAIセッションや別担当者が見ても再開できるようにする。

### 原則8: 更新できる文書として扱う

Context Pack は固定文書ではない。前提が変わったら更新し、変更履歴を残す。

## 演習

過去1週間以内の自分の業務から、AIに手伝わせたいタスクを1つ選ぶ。そのタスク用の Context Pack を作る。次に、別の人に読んでもらい、以下を確認する。

- 何をしてほしいか分かるか
- なぜその作業が必要か分かるか
- どの情報が最新か分かるか
- どの情報が古いか分かるか
- どの情報が未確認か分かるか
- AIに渡してはいけない情報が除外されているか
- 別の担当者が作業を再開できるか

### 演習の判定基準

次の条件を満たしていれば、この章の演習としては合格である。

- Request Contract に対応する Context Pack になっている
- 業務背景、現在状況、決定済み事項、未決事項が分かれている
- 関連資料に日付または版が付いている
- 確定情報、推定情報、仮定、要確認事項が分かれている
- AIに参照させてよい情報と、参照させてはいけない情報が明記されている
- 個人情報、認証情報、不要な機密情報が含まれていない
- 別担当者が読んでも次に何をすべきか分かる

## まとめ

AIエージェントの品質は、依頼文だけでなく、渡す文脈で決まる。Context Pack は、AIに考えさせるための資料集ではなく、AIが誤った前提で処理しないための制御文書である。

次章では、この Request Contract と Context Pack を使って、複雑な業務をAIに委任可能な単位へ分解する。依頼と文脈が整っても、大きな仕事を丸投げすれば失敗する。第5章では、サブタスク、チェックポイント、停止条件、エスカレーションを設計する。

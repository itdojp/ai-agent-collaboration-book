# 第11章 AI時代の判断力

## Part Vの位置づけと注意

第11章から第14章までのPart Vでは、AIで作業速度と影響範囲が広がった環境において、人間が担う判断、対話、実験、行動実行を扱う。これは、メンタルヘルス支援、産業保健、専門的コーチング、カウンセリング、人事評価、ハラスメント対応の代替ではない。

本Partで扱うのは、判断回避、対立回避、失敗回避、過剰な自己批判が、AIエージェント協働の業務成果、レビュー品質、展開速度、組織学習に与える影響である。個人の性格を評価するのではなく、Decision Brief、Difficult Conversation Brief、Failure Review Sheet、Action Blocker診断のような成果物に落として、業務設計と組織能力として扱う。

次のいずれかに当てはまる場合は、本Partのテンプレートだけで扱わず、人事、産業保健、法務、労務、ハラスメント相談窓口、医療・心理の専門家支援を優先する。

- 健康、安全、ハラスメント、労務、評価、法令、契約に関係する。
- 本人または周囲に深刻な負荷、継続的な不調、心理的安全性の毀損がある。
- 管理職と部下、発注者と受注者など、権限差によって本人が自由に判断できない。
- 記録、共有、評価利用について本人の同意や社内規程が明確でない。

この注意は、第14章の「精神論ではなく成果物で扱う」「専門領域は専門家へ接続する」という方針と同じである。Part Vは、人間側能力を業務成果へ接続するための設計章であり、専門支援を置き換える章ではない。

## この章で扱う能力

この章では、AIエージェントが生成した情報、分析、比較、提案、予測、下書きを材料にしながら、人間がどのように判断するかを扱う。

読者は、次のことができるようになる。

- AIが出した情報、推論、選択肢、推奨と、人間が負うべき判断責任を分ける
- 判断対象、判断基準、判断権限、判断期限、検証指標を明確にする
- AI提案を、採用、修正後採用、棄却、保留、停止、実験化に分類する
- 「やる判断」だけでなく、「やめる判断」「方向転換する判断」「リスクを受容しない判断」を扱う
- 判断を遅らせている本当の理由を、情報不足、対立回避、失敗回避、責任回避に分解する
- AIを使って反証、プレモーテム、代替案、利害関係者影響、前提崩れを確認する
- 判断後に、結果だけでなく判断プロセスをレビューする

第10章までは、AIエージェント協働を安全に運用するための統制を扱った。第11章からは、人間側能力を扱う。AIが仕事を速くするほど、最後にボトルネックになるのは、AIに何を頼むかだけではない。**人間が何を採用し、何を捨て、何を止め、どのリスクを引き受けるか**である。

## 関連図表

- [F-11: 判断責任分界](../../figures/f-11-decision-responsibility-boundary/) — AIが支援する判断材料と、人間が引き受ける判断責任を分ける。

本書の6層モデルで見ると、第11章は `Human / Org` 層の入口に位置する。

ここまでの章では、次のようにAIエージェント協働を設計してきた。

```text
Work：AIに使う仕事を選ぶ
Request：AIへの依頼を契約にする
Context：AIに渡す前提を設計する
Delegation：タスクを分解して委任する
Control：レビュー、権限、ログ、セキュリティ、ガバナンスで制御する
Human / Org：人間が判断し、対話し、実験し、学習する
```

第11章のテーマは、最後の `Human / Org` のうち、**判断**である。

AIは、情報を集める。比較表を作る。リスクを列挙する。選択肢を増やす。もっともらしい提案を書く。反対意見も生成できる。プレモーテムもできる。市場データや社内データの要約もできる。

しかし、AIは次のことを最終的には決められない。

- 何を優先するか
- どの顧客に約束するか
- どのリスクを引き受けるか
- どの施策をやめるか
- 誰に説明するか
- どのタイミングで動くか
- どの失敗を学習コストとして許容するか
- どこで組織の資源を集中させるか

これらは、事実問題だけではない。価値判断、責任、関係者、タイミング、リスク許容度、組織能力の問題である。

## なぜ判断力が必要になるのか

AIを使うと、情報不足は減る。少なくとも、表面的にはそう見える。

以前なら数日かかった調査が短時間で終わる。比較表はすぐ出る。企画案は10案出る。メール文面も、提案書も、コードも、FAQも、議事録も、すぐに生成される。

この変化によって、人間の仕事は単純に軽くなるわけではない。むしろ、次の問題が増える。

```text
・選択肢が増えすぎる
・案を作ることが簡単になり、捨てることが難しくなる
・もっともらしい提案に流されやすくなる
・レビューと承認が増える
・実験の数が増える一方、学習が追いつかない
・意思決定の根拠がAI出力に埋もれる
・誰が判断したのか分からなくなる
・AIが作った言葉で、責任が曖昧になる
```

AI時代に危険なのは、単に「AIが間違うこと」ではない。AIが間違うことは第6章から第10章で扱った。第11章で扱う危険は、**AIが十分に便利で、もっともらしく、速いために、人間が判断したつもりになること**である。

たとえば、次の状態はすべて判断力の問題である。

| 状態 | 表面的には | 実際には |
|---|---|---|
| AIが推奨した案を採用した | データに基づく判断 | 判断責任をAIに移している |
| 追加調査を続ける | 慎重に見える | 決めることを避けている |
| 施策を増やす | 積極的に見える | やめる判断ができていない |
| 無難な文面にする | 配慮しているように見える | 必要な対話を避けている |
| リスクを列挙する | リスク管理しているように見える | どのリスクを許容しないか決めていない |
| PoCを続ける | 実験しているように見える | 本番化または中止の判断を先送りしている |

AIは判断材料を増やす。判断材料が増えた結果、判断が良くなるとは限らない。判断材料が増えすぎると、判断が遅くなり、責任が曖昧になり、選択肢を捨てることが難しくなる。

したがって、AIエージェント協働では、判断力を個人の勘や経験に任せず、成果物として設計する必要がある。

## 本章における判断力の定義

本書では、AI時代の判断力を次のように定義する。

```text
AI時代の判断力とは、AIが提示する情報、推論、選択肢、推奨、文章を、
人間の目的、責任、価値判断、制約、リスク、関係者、タイミングと照合し、
採用、修正、棄却、保留、停止、実験化、方向転換を決める能力である。
```

ここで重要なのは、判断力を「直感が鋭いこと」や「経験豊富であること」と同一視しないことである。経験や直感は重要だが、それだけでは組織内で再現できない。

本書では判断力を、以下の成果物に分解する。

| 成果物 | 役割 |
|---|---|
| Decision Brief | 何を決めるのかを1枚にまとめる |
| 判断前チェックリスト | AI情報、人間判断、リスク、回避している論点を確認する |
| Decision Criteria Matrix | 判断基準と優先順位を明示する |
| Option Evaluation Table | 選択肢を同じ基準で比較する |
| Kill Decision Sheet | やめる、縮小する、保留する判断を扱う |
| Risk Acceptance Memo | 受容するリスクと受容しないリスクを明示する |
| Decision Review Log | 判断後に、結果だけでなく判断プロセスを振り返る |

判断力は、頭の中で完結させない。**判断を文書化し、レビュー可能にし、後から学習可能にする**。

## AIに任せられること、任せられないこと

AIは判断補助に強い。特に、次の作業はAIに任せやすい。

| 領域 | AIに任せやすいこと | 人間が決めること |
|---|---|---|
| 選択肢生成 | 複数案、代替案、極端な案を出す | どの選択肢を残すか |
| 比較 | メリット、デメリット、コスト、リスクを整理する | どの基準を優先するか |
| 反証 | 失敗理由、反対意見、前提崩れを列挙する | それでも進めるか |
| プレモーテム | 失敗した未来から逆算して原因を挙げる | どのリスクに対策するか |
| 要約 | 情報を圧縮する | 何を重要とみなすか |
| 利害関係者整理 | 影響を受ける相手を洗い出す | 誰に説明し、誰を巻き込むか |
| シナリオ作成 | 複数の未来シナリオを作る | どのシナリオに備えるか |
| 感情的論点の可視化 | 対立、恐れ、先送りの可能性を質問する | 何を直接扱うか |

AIに任せてはいけないのは、次である。

```text
・最終責任を伴う判断
・価値観の優先順位
・顧客、社員、社会への説明責任
・高リスク用途のリスク受容
・不可逆または回復困難な決定
・人間関係や組織摩擦を直接扱う判断
・規制、契約、倫理、雇用、評価に関わる最終判断
```

AIは「この案を推奨します」と書くことができる。しかし、その推奨は、モデルが与えられた情報と指示に基づいて生成したテキストである。組織の責任を引き受けるものではない。

## 判断の種類

AIエージェント協働で発生する判断は、単に「採用するかどうか」ではない。少なくとも、次の種類がある。

| 判断の種類 | 問い | 例 |
|---|---|---|
| 採用判断 | AI出力を使うか | 顧客返信案を採用する |
| 修正判断 | どこを直せば使えるか | 原因断定を削除し、未確定事項を追記する |
| 棄却判断 | 使わない理由は何か | 根拠が弱い競合比較を棄却する |
| 保留判断 | 何が揃えば決められるか | 法務確認まで契約レビューAIの本番化を保留する |
| 実験化判断 | いきなり本番化せず、何を検証するか | CS部門の一部顧客だけで返信案生成を試す |
| 拡大判断 | どこまで広げるか | 社内FAQ検索を全社員へ展開する |
| 縮小判断 | どこを減らすか | RAG対象文書を最新FAQだけに限定する |
| 停止判断 | いつ止めるか | 誤案内率が閾値を超えたため停止する |
| やめる判断 | 続けない理由は何か | 利用率が低いAIワークフローを廃止する |
| 方向転換判断 | 何を変えるか | 自動返信ではなく、返信案レビュー支援に変更する |
| リスク受容判断 | どの残リスクを受け入れるか | 低影響業務で一定の誤分類を許容する |
| リスク拒否判断 | どのリスクは受け入れないか | 顧客送信の自動化は許容しない |

この分類を持っていないと、すべての議論が「やるか、やらないか」に潰れる。実務では、第三の選択肢が多い。

```text
本番化しないが、限定実験する
自動送信しないが、下書き生成は使う
全社展開しないが、1チームで継続する
直接更新しないが、更新案は作る
リスクを受容しないが、代替設計で進める
```

判断力とは、白黒を急ぐことではない。**判断の種類を正しく選ぶこと**である。

## 判断レベル

すべての判断に同じ重さのプロセスを使う必要はない。軽い判断に重い会議を要求すると、AI活用は止まる。一方、重い判断をチャットの流れで済ませると、事故が起きる。

本書では、判断レベルを次のように分ける。

| レベル | 名称 | 例 | 必要な記録 |
|---|---|---|---|
| J0 | 個人メモ判断 | 自分用の要約を採用する | 不要または個人メモ |
| J1 | 低影響判断 | 社内向け文案を修正して使う | 判断前チェックリストの簡易版 |
| J2 | チーム判断 | チーム内ワークフローにAIを入れる | Decision Brief、レビュー記録 |
| J3 | 顧客・業務影響判断 | 顧客返信案、CRM更新案、提案書に使う | Decision Brief、Risk Acceptance Memo、承認ログ |
| J4 | 組織標準判断 | 部門展開、外部ツール採用、RAG共通化 | AI System ADR、AI Risk Register、Approval Workflow |
| J5 | 高リスク・不可逆判断 | 本番変更、自動送信、契約・人事・規制領域 | 経営・法務・セキュリティ承認、監査証跡 |

第10章の Governance Level Matrix と連動させるなら、G0〜G5のうち、G3以上ではJ3以上の判断記録を要求する。G4以上では、Decision Briefだけでなく、AI System ADR、Risk Acceptance、承認ログを残す。

判断プロセスの目的は、遅くすることではない。**判断の重さに応じて、必要な証跡を変えること**である。

## 判断設計の基本構造

AI時代の判断は、次の順序で設計する。

```text
1. 判断対象を定義する
2. 判断者を決める
3. 判断期限を決める
4. 判断基準を決める
5. AIに作らせる材料を決める
6. 人間が決める論点を分ける
7. 選択肢を評価する
8. 反証と前提崩れを確認する
9. 採用、修正、棄却、保留、停止、実験化を決める
10. 判断理由と検証指標を残す
11. 判断後にレビューする
```

この順序で重要なのは、AIに分析させる前に、**何を決めるのか**を明確にすることである。

悪い例は次である。

```text
このAI活用案についてどう思う？
```

この依頼では、判断対象が不明確である。AIは一般論、メリット・デメリット、注意点を返すだろう。しかし、それだけでは意思決定にはならない。

よい例は次である。

```text
判断対象：
CS部門の顧客返信案生成AIを、来月から全チームへ展開するかどうか。

判断者：
CS責任者、AI推進責任者、セキュリティ責任者。

判断期限：
2026年7月15日。

判断基準：
・顧客送信前に必ず人間レビューが入る
・Critical Issue率が2%未満
・返信案作成時間が30%以上短縮
・SLA、補償、原因断定がAI単独で出力されない
・CS担当者が運用負荷を許容できる

AIに作らせる材料：
・過去30件のRun Review要約
・Review Issue Logの傾向
・展開リスク
・展開しない場合の機会損失
・限定展開案
```

この形式なら、AIは判断材料を作れる。人間は判断できる。

## Decision Brief

Decision Brief は、判断対象を1枚に整理する成果物である。

AIエージェント協働では、判断が会話の中に埋もれやすい。チャット上では、質問、回答、追加依頼、別案、修正指示が流れていく。その途中で、いつの間にか判断したことになりやすい。

Decision Brief は、その状態を防ぐ。

```text
Decision Brief

Decision ID：
判断対象：
判断の種類：採用 / 修正 / 棄却 / 保留 / 実験化 / 拡大 / 縮小 / 停止 / やめる / 方向転換 / リスク受容
背景：
なぜ今決めるのか：
判断者：
助言者：
承認者：
判断期限：
対象ユースケース / Inventory ID：
関連するRequest Contract：
関連するContext Pack：
関連するレビュー / 評価 / ログ：
選択肢：
判断基準：
主要な事実：
主要な仮説：
不確実性：
許容できるリスク：
許容できないリスク：
関係者影響：
決定内容：
判断理由：
判断後に見る指標：
見直し日：
```

Decision Brief は、長いレポートである必要はない。むしろ、1枚でよい。重要なのは、判断が次の3つを明確にしていることだ。

```text
何を決めたのか
なぜそう決めたのか
いつ、何を見て見直すのか
```

## 判断前チェックリスト

判断前チェックリストは、AIの提案を採用する前に、人間が確認するための最小チェックである。

```text
判断対象：
AIから得た情報：
AIが作った選択肢：
AIが推奨している案：
人間が決めるべき論点：
判断者：
判断期限：
判断基準：
許容できるリスク：
許容できないリスク：
この判断で避けようとしている不快なこと：
言えていない懸念：
本当はやめるべきだが続けているもの：
怖いが試すべきもの：
判断を延期した場合のコスト：
判断後に検証する指標：
見直し日：
```

このチェックリストの特徴は、技術的・論理的な項目だけでなく、次の項目を含むことである。

```text
この判断で避けようとしている不快なこと
言えていない懸念
本当はやめるべきだが続けているもの
怖いが試すべきもの
```

この4つは、判断の質に強く影響する。

実務では、判断が遅れる理由は、情報不足とは限らない。次のような理由で判断が止まる。

- 反対する役員や部門に説明したくない
- 既存施策をやめると、担当者の努力を否定するように見える
- 失敗したときに責任を問われる
- 顧客に厳しいことを言いたくない
- 自分が推してきた案を捨てられない
- AI導入に反対している人と話したくない
- 技術的には可能だが、現場運用が耐えられないことを認めたくない

AIは、これらを自動的には解決しない。むしろ、AIが資料や案を大量に作れるため、判断回避が見えにくくなる。

## Decision Criteria Matrix

Decision Criteria Matrix は、判断基準を明示するための表である。

AIは、与えられた基準に沿って比較することは得意である。しかし、どの基準を優先するかは、人間が決める必要がある。

```text
Decision Criteria Matrix

判断対象：
判断者：
判断期限：

| 判断基準 | 重み | 必須 / 加点 | 測定方法 | 最低条件 | 備考 |
|---|---:|---|---|---|---|
| 顧客影響 | 5 | 必須 | 誤送信、誤案内、SLA影響 | 重大影響なし | 重大影響がある場合は停止 |
| セキュリティ | 5 | 必須 | 情報分類、外部送信、権限 | S3以下または承認済み | S4以上は事前承認 |
| 品質 | 4 | 必須 | Critical Issue率 | 2%未満 | 第6章のReview Issue Logを参照 |
| 効率 | 3 | 加点 | 作業時間削減 | 20%以上 | 最低条件ではない |
| 運用負荷 | 4 | 必須 | レビュー時間、承認待ち | 現場が対応可能 | 承認疲れに注意 |
| 学習価値 | 2 | 加点 | 次の改善につながるか | あり | 実験の場合に重視 |
```

この表を作ると、議論が整理される。

悪い議論は次である。

```text
便利そうだから進めたい。
リスクがありそうだからやめたい。
```

よい議論は次である。

```text
顧客影響とセキュリティは必須条件であり、満たさなければ進めない。
効率は加点条件であり、効率が高くても顧客影響リスクを上書きしない。
運用負荷が高い場合は、自動化ではなく限定利用にする。
```

判断基準を先に決めることで、AIの分析を使いやすくなる。

## Option Evaluation Table

Option Evaluation Table は、選択肢を同じ基準で比較するための表である。

AIに「どの案がよいか」を聞く前に、選択肢と基準を分ける。

```text
Option Evaluation Table

判断対象：
判断基準：Decision Criteria Matrixを参照

| 選択肢 | 内容 | 顧客影響 | セキュリティ | 品質 | 効率 | 運用負荷 | 学習価値 | 主なリスク | 判断 |
|---|---|---|---|---|---|---|---|---|---|
| A | 全社展開 | 中 | 中 | 要改善 | 高 | 高 | 中 | レビュー負荷増 | 保留 |
| B | 1チーム限定展開 | 低 | 低 | 許容 | 中 | 中 | 高 | 効果限定 | 実験化 |
| C | 下書き生成のみ | 低 | 低 | 許容 | 中 | 低 | 中 | 効率効果は限定 | 採用 |
| D | 中止 | なし | なし | なし | なし | なし | 低 | 機会損失 | 棄却 |
```

この表は、AIに作らせてもよい。ただし、判断欄はAIに書かせても、そのまま採用してはいけない。AIには次のように依頼する。

```text
以下の選択肢を、指定した判断基準に沿って比較してください。
ただし、最終判断は人間が行うため、判断欄には「判断材料」を書き、
採用・棄却の最終判断は書かないでください。
不確実な点は「要確認」と書いてください。
```

AIに最終判断を書かせると、意思決定の責任が曖昧になる。AIには、判断材料を作らせる。

## 判断に使うAI依頼パターン

判断力を高めるために、AIには「結論」を聞くよりも、判断材料を作らせる方がよい。

### パターン1: 選択肢生成

```text
目的：
以下の判断対象について、取り得る選択肢を洗い出す。

制約：
最終判断は書かない。
各選択肢について、メリット、デメリット、必要な前提、失敗条件を書く。

出力：
Option Evaluation Table。
```

### パターン2: 反証

```text
以下の提案を採用しない理由を、できるだけ具体的に挙げてください。
特に、顧客影響、セキュリティ、運用負荷、責任境界、現場の摩擦を重視してください。
最後に、致命的な反証と、対策可能な反証を分けてください。
```

### パターン3: プレモーテム

```text
この施策が3か月後に失敗したと仮定してください。
失敗原因を10個挙げ、発生確率、影響度、早期兆候、予防策を表にしてください。
```

### パターン4: 前提崩れ

```text
この提案が成立するための前提を列挙してください。
各前提について、崩れた場合の影響、確認方法、確認責任者を整理してください。
```

### パターン5: やめる判断

```text
この施策を中止または縮小すべき条件を定義してください。
利用率、品質、コスト、現場負荷、リスク、代替手段の観点で、具体的な閾値を提案してください。
```

### パターン6: 利害関係者影響

```text
この判断により影響を受ける関係者を洗い出してください。
各関係者について、期待、懸念、説明すべきこと、反対が出る可能性、事前に確認すべきことを整理してください。
```

### パターン7: 判断延期コスト

```text
この判断を1か月延期した場合のコストを整理してください。
機会損失、現場負荷、顧客影響、リスク増加、学習機会の損失に分けてください。
```

AIは、判断を代替するのではなく、判断材料を増やす。よい依頼は、「答えを出して」ではなく、「判断に必要な材料を、この基準で整理して」である。

## 判断回避のパターン

AI時代の判断力を損なう最大の要因は、情報不足だけではない。判断回避である。

判断回避には、次のパターンがある。

| パターン | 表面的な行動 | 実際に起きていること |
|---|---|---|
| 追加調査依存 | もっと調べる | 決める不快さを避けている |
| 選択肢増殖 | もっと案を出す | 捨てる判断を避けている |
| PoC延命 | もう少し試す | 本番化または中止を避けている |
| 合意待ち | 関係者の意見を聞く | 誰も責任者になっていない |
| リスク列挙だけ | リスクを整理する | 受容・拒否を決めていない |
| AI推奨依存 | AIが推した案を採用する | 責任をAIに移している |
| 無難な文面化 | 角の立たない表現にする | 必要な対話を避けている |
| 指標待ち | KPIが揃うまで待つ | 不確実性をゼロにしようとしている |

判断回避を見つけるには、次の質問が有効である。

```text
この判断で、最も言いにくいことは何か。
この判断で、誰が不快になる可能性があるか。
この判断を延期すると、誰が得をするか。
追加調査で、本当に判断が変わるのか。
いま足りないのは情報か、決断か。
この施策をやめると言えない理由は何か。
AIの推奨を使って、誰の責任を薄めようとしているのか。
```

これらは、心理分析をするためではない。判断の質を上げるためである。

## 感情と判断を分ける

感情を排除することはできない。むしろ、重要な判断には必ず感情が関わる。

- 失敗したくない
- 批判されたくない
- 顧客を失いたくない
- 部下の努力を否定したくない
- 自分の過去判断を間違いと認めたくない
- AIに仕事を奪われると思われたくない
- 強い反対者と対立したくない

問題は、感情があることではない。**感情を認識しないまま、事実や論理のふりをして判断に混ぜること**である。

たとえば、次の言い方は感情が論理に混ざっている可能性がある。

```text
まだ情報が足りない。
```

本当に情報が足りない場合もある。しかし、次の可能性もある。

```text
反対者に説明したくない。
失敗したときに責任を負いたくない。
やめると言うと担当者の士気が下がる。
自分が推してきた施策を否定したくない。
```

したがって、判断前チェックリストには、あえて次を入れる。

```text
この判断で避けようとしている不快なこと：
言えていない懸念：
本当はやめるべきだが続けているもの：
怖いが試すべきもの：
```

これは感情論ではない。意思決定品質の検査項目である。

## やめる判断

AIによって、施策を作るコストは下がる。資料も、プロトタイプも、分析も、コードも、広告文も、FAQも、ワークフローも作りやすくなる。

その結果、組織には次のものが増える。

```text
・プロンプト
・ワークフロー
・RAGデータセット
・AIエージェント
・自動化スクリプト
・ダッシュボード
・PoC
・会議
・レビュー項目
・承認フロー
・KPI
```

作るコストが下がった環境では、やめる判断の価値が上がる。

やめる判断がない組織では、AI活用が進むほど複雑性が増す。現場は効率化されたはずなのに、実際にはレビュー、承認、例外処理、メンテナンス、問い合わせが増える。

### Kill Decision Sheet

やめる判断を属人的にしないために、Kill Decision Sheet を使う。

```text
Kill Decision Sheet

対象施策 / AI利用：
Owner：
開始日：
当初目的：
当初成功条件：
現在の利用状況：
現在の効果：
現在のコスト：
現在のリスク：
継続する理由：
中止する理由：
縮小する選択肢：
代替手段：
やめた場合の影響：
続けた場合の影響：
中止条件 / 縮小条件：
判断：継続 / 縮小 / 保留 / 中止 / 再設計

関係者通知：
廃止・移行タスク：
見直し日：
```

やめる判断では、次を明確にする。

| 問い | 意味 |
|---|---|
| 当初目的はまだ有効か | 目的が消えた施策を惰性で続けていないか |
| 効果は測れているか | 体感だけで継続していないか |
| 維持コストは増えていないか | AI活用が隠れた運用負債になっていないか |
| リスクは変化したか | モデル、データ、外部ツール、業務変更でリスクが増えていないか |
| やめた場合の影響は何か | 本当に困るのか、単に不安なのか |
| 続けた場合の機会損失は何か | 他の実験や改善に資源を回せなくなっていないか |

やめる判断は失敗の宣言ではない。資源配分の判断である。

## 実験化判断

AI活用では、「本番化するか、やめるか」だけでなく、実験化が重要になる。

不確実性が高い場合、いきなり大きく決めるのではなく、小さく検証する。

実験化すべき条件は次である。

```text
・価値はありそうだが、効果が未確認
・リスクはあるが、限定すれば管理できる
・現場負荷が読めない
・顧客影響を限定できる
・技術的には可能だが、運用が未知
・関係者の反応が読めない
・判断に必要なデータを短期間で集められる
```

実験化すべきでない条件は次である。

```text
・失敗時の顧客影響が大きい
・法務、セキュリティ、規制上の事前確認が必要
・ログや停止手段がない
・結果が出ても判断する責任者がいない
・成功条件と失敗条件が曖昧
・実験と称して本番運用している
```

実験化判断では、第13章のAI実験設計シートへ渡す。

```text
判断：本番化ではなく実験化する
理由：効果と現場負荷が未確認
実験範囲：CSチームA、低リスク問い合わせのみ
成功条件：返信案作成時間30%削減、Critical Issue率2%未満
失敗条件：SLA関連誤分類1件以上、承認待ち時間が現状比20%以上増加
停止条件：顧客影響の可能性が出た時点で停止
見直し日：2週間後
```

実験化は、判断の先送りではない。**不確実性を減らすための判断**である。

## リスク受容判断

第9章と第10章では、リスクを分類し、統制する方法を扱った。しかし、すべてのリスクをゼロにはできない。実務では、残リスクを受け入れるか、受け入れないかを判断する必要がある。

リスク受容判断で重要なのは、次の区別である。

```text
分かっていて受け入れるリスク
分からないまま放置しているリスク
```

前者は判断である。後者は統制不備である。

### Risk Acceptance Memo

```text
Risk Acceptance Memo

対象ユースケース / Inventory ID：
対象リスク：
リスク分類：品質 / セキュリティ / プライバシー / 法務 / 顧客影響 / 運用 / コスト / レピュテーション
想定される影響：
発生可能性：高 / 中 / 低
影響度：高 / 中 / 低
既存の統制：
追加できる統制：
残リスク：
受容する理由：
受容しない場合の代替案：
受容期限：
Risk Owner：
承認者：
監視指標：
再評価条件：
```

リスク受容は、永久許可ではない。期限を持つ。再評価条件を持つ。Ownerを持つ。

悪い例は次である。

```text
リスクはあるが、現場で注意して使う。
```

よい例は次である。

```text
顧客返信案生成において、低リスク問い合わせ分類の誤分類率2%未満を残リスクとして受容する。
ただし、SLA、補償、障害、契約、解約、個人情報に関わる問い合わせは対象外とする。
Critical Issueが1件でも発生した場合、対象ユースケースを停止し、Risk Ownerが再評価する。
受容期限は2026年9月30日とする。
```

この違いは大きい。

## 判断後レビュー

判断の質は、その場では分からない。結果が良ければ判断が良かったとは限らない。結果が悪くても判断プロセスが悪かったとは限らない。

重要なのは、結果とプロセスを分けて振り返ることである。

### Decision Review Log

```text
Decision Review Log

Decision ID：
判断日：
判断内容：
当時の判断基準：
当時の主要事実：
当時の主要仮説：
当時の不確実性：
実際の結果：
想定と合っていたこと：
想定と違ったこと：
見落としていた前提：
AI出力の貢献：

判断プロセスの問題：

```

判断後レビューでは、次を分ける。

| 観点 | 問い |
|---|---|
| 結果 | 期待した成果が出たか |
| プロセス | 当時の情報で合理的に判断したか |
| 前提 | どの前提が外れたか |
| AI | AIは何を助け、何を誤らせたか |

| 仕組み | 次回、Request、Context、Review、Governanceをどう変えるか |

AIエージェント協働では、判断後レビューを第8章の Improvement Backlog へ接続する。

```text
Decision Review Log
  ↓
Improvement Backlog
  ↓
Evaluation Plan
  ↓
Request Contract / Context Pack / Governance Level Matrix の改訂
```

判断は、1回の勝ち負けではない。組織の学習サイクルである。

## 実務ケース1: 顧客返信案生成AIを全社展開するか

### 状況

CS部門で、顧客返信案をAIで生成する取り組みを2週間試した。

結果は次のとおりである。

```text
・返信案作成時間は平均35%短縮
・Critical Issueは30件中1件
・Major Issueは30件中5件
・CS担当者のレビュー負荷は想定より高い
・SLA関連問い合わせでは、AIが原因未確定事項を断定する傾向がある
・顧客送信は人間承認必須
・現場からは「助かるが、レビューが怖い」という声がある
```

経営会議では、全CSチームへ展開するか判断する必要がある。

### 悪い判断

```text
作業時間が35%短縮できたので全社展開する。
```

この判断は、効率だけを見ている。Critical Issue、Major Issue、レビュー負荷、SLA関連リスクを十分に扱っていない。

別の悪い判断は次である。

```text
Critical Issueが出たので中止する。
```

これも短絡的である。中止が正しい可能性はあるが、限定展開、対象範囲縮小、SLA関連問い合わせ除外、レビュー条件強化という選択肢を検討していない。

### Decision Brief

```text
Decision ID：DEC-CS-2026-07-001
判断対象：顧客返信案生成AIを全CSチームへ展開するか
判断の種類：拡大 / 実験化 / 縮小 / 保留
判断者：CS責任者、AI推進責任者
助言者：セキュリティ責任者、法務担当、CSリード
判断期限：2026-07-15
関連Inventory ID：AI-CS-REPLY-001
関連ログ：RUN-2026-07-CS-001〜030
主要事実：返信案作成時間35%短縮、Critical Issue 1件、Major Issue 5件
主要仮説：低リスク問い合わせに限定すれば効果が出る
許容できるリスク：低リスク問い合わせでのMinor Issue
許容できないリスク：SLA、補償、原因断定、契約、個人情報の誤案内
選択肢：全社展開、限定展開、対象縮小、中止
判断後指標：Critical Issue率、Major Issue率、レビュー時間、承認拒否率、CS満足度
```

### Option Evaluation Table

| 選択肢 | 顧客影響 | 品質 | 効率 | 運用負荷 | 主なリスク | 判断材料 |
|---|---|---|---|---|---|---|
| 全社展開 | 中 | 要改善 | 高 | 高 | Critical Issue再発 | 時期尚早 |
| 低リスク問い合わせ限定で展開 | 低 | 許容可能 | 中 | 中 | 分類誤り | 実験価値あり |
| SLA関連を除外して継続 | 低 | 改善余地あり | 中 | 中 | 対象範囲管理 | 現実的 |
| 中止 | なし | なし | なし | なし | 機会損失 | 早すぎる可能性 |

### 判断

```text
全社展開はしない。
低リスク問い合わせに限定し、対象チームを2チームに拡大する。
SLA、補償、原因未確定、契約、個人情報に関わる問い合わせはAI返信案生成の対象外にする。
2週間後にCritical Issue率、Major Issue率、レビュー時間、承認拒否率を見て再判断する。
```

この判断は、「やる」でも「やめる」でもない。限定実験である。AI時代の判断では、このような中間判断を設計できることが重要である。

## 実務ケース2: AIによるCRM更新を許可するか

### 状況

営業部門から、AIが商談メモを読み、CRMの次回アクション、確度、顧客課題を自動更新したいという要望が出た。

AI推進担当は、第7章の権限設計に基づき、次の選択肢を出した。

```text
A. AIがCRMを直接更新する
B. AIが更新案を作り、人間が承認して反映する
C. AIが商談メモを要約するだけにする
D. CRM連携は行わない
```

### 判断基準

| 判断基準 | 重み | 最低条件 |
|---|---:|---|
| 顧客情報保護 | 5 | スコープ外顧客を参照しない |
| 営業運用 | 4 | 営業担当が確認可能 |
| データ品質 | 4 | 誤更新を検出できる |
| 効率 | 3 | 入力時間を削減できる |
| 監査可能性 | 5 | 誰が何を承認したか残る |

### 判断

```text
Bを採用する。
AIはCRM更新案を作るが、直接更新はしない。
営業担当が差分を確認し、承認した場合のみCRMへ反映する。
確度変更、金額変更、契約条件、解約リスク、競合情報は、営業責任者承認を必要とする。
```

この判断の要点は、AI活用を拒否していないこと、しかし直接更新も許可していないことである。AI時代の判断は、権限設計とセットで行う。

## 実務ケース3: 使われていないRAGを廃止するか

### 状況

社内Wikiを対象にしたRAG検索エージェントが導入された。しかし、利用状況は低い。

```text
・月間利用者は対象者300人中12人
・回答のうち30%は古い文書を参照している
・文書更新担当が不在
・API費用は小さいが、問い合わせ対応とメンテナンスに時間がかかっている
・一部の社員は便利だと言っている
```

### 判断回避

このような施策は、やめる判断が難しい。

```text
せっかく作ったから残しておこう。
使っている人もいる。
費用は大きくない。
もう少し周知すれば使われるかもしれない。
```

しかし、これらは継続理由として弱い。

### Kill Decision Sheet

```text
対象施策：社内Wiki RAG検索エージェント
Owner：AI推進担当
当初目的：社内情報検索時間の削減
当初成功条件：対象者の30%以上が月1回以上利用、回答満足度80%以上
現在の利用状況：対象者300人中12人
現在の効果：限定的
現在のコスト：小さいが、古い文書参照による問い合わせが発生
現在のリスク：古い情報を正しい前提として使う
継続する理由：一部利用者には有用
中止する理由：利用率低、文書更新体制なし、誤情報リスク
縮小する選択肢：対象文書を最新FAQと運用Runbookに限定
代替手段：社内Wiki検索改善、FAQ整理
判断：縮小して30日後に再評価。改善しなければ廃止
```

ここでは、即時廃止ではなく縮小を選んでいる。重要なのは、見直し日と廃止条件を入れることである。

## 実務ケース4: AI提案書をそのまま使うか

### 状況

営業担当がAIで提案書を作成した。見た目は整っている。構成もよい。顧客課題、導入効果、価格イメージ、導入スケジュールも書かれている。

しかし、レビューで次の問題が見つかった。

```text
・顧客の業界規制に関する記述が一般論である
・価格は過去案件から推定したもので、正式見積ではない
・導入スケジュールが現場のリソース制約を考慮していない
・競合比較に古い情報が含まれている
・「短期間で確実に効果が出る」と過剰表現している
```

### 判断

この場合、判断は「提案書を使うか、捨てるか」ではない。

```text
・構成は採用する
・価格、スケジュール、規制、競合比較は差し戻す
・顧客向け表現はレビュー必須にする
・正式見積と社内リソース確認後に再提出する
```

AI出力は、全体を採用または棄却するものではない。部分採用、部分棄却、差し戻し、再生成に分ける。

## 典型的な失敗

### 失敗1: AI推奨を意思決定に見せる

AIが「推奨案」と書いたものを、そのまま意思決定の根拠にする失敗である。

```text
AIの分析ではA案が最適です。
```

この言い方では、誰がどの基準で判断したのか分からない。正しくは次である。

```text
AIにはA〜D案を比較させた。
人間側の判断基準は、顧客影響、セキュリティ、品質、運用負荷である。
A案は効率が高いが、顧客影響リスクが高いため採用しない。
B案を限定実験として採用する。
```

### 失敗2: 判断基準を後出しする

AIの出力を見てから、都合のよい基準を選ぶ失敗である。

判断基準は、原則としてAI分析の前に定義する。後から変える場合は、なぜ変えたかを記録する。

### 失敗3: 選択肢を増やしすぎる

AIは選択肢を無限に増やせる。選択肢が多いほど良いとは限らない。

実務では、次のように制限する。

```text
・選択肢は3〜5個に絞る
・各案の違いを明確にする
・明らかに採用しない案は参考に回す
・判断期限を設定する
```

### 失敗4: やめる判断を入れない

AI活用は、導入時には注目されるが、廃止時には誰も関心を持たないことが多い。

やめる判断がないと、古いプロンプト、古いRAG、古いワークフロー、使われないエージェントが残り続ける。

### 失敗5: 実験を判断の先送りに使う

実験は有効だが、成功条件、失敗条件、見直し日がなければ、単なる先送りになる。

```text
もう少し試す
```

ではなく、次のように書く。

```text
2週間、対象チーム2つ、低リスク問い合わせに限定して試す。
Critical Issueが1件でも出たら停止する。
返信案作成時間30%削減、Major Issue率5%未満なら継続判断する。
```

### 失敗6: 感情的論点をデータ不足に見せる

「データが足りない」と言っているが、実際には対立、責任、失敗、政治的摩擦を避けている状態である。

この失敗は見えにくい。判断前チェックリストで、避けている不快なことを明文化する。

### 失敗7: すべてを合意で決めようとする

関係者の意見を聞くことは重要である。しかし、すべてを合意で決めようとすると、責任者が不明確になる。

Decision Briefには、判断者、助言者、承認者を分けて書く。

```text
判断者：最終判断をする人
助言者：判断材料や専門意見を出す人
承認者：リスク、権限、予算、法務などの承認をする人
```

### 失敗8: 判断後に学習しない

判断が当たったか外れたかだけを見て、判断プロセスを振り返らない失敗である。

AI時代には、判断回数が増える。判断後レビューがなければ、同じミスが高速に繰り返される。

## 設計原則

### 原則1: AIには判断材料を作らせ、人間が判断する

AIに「何をすべきか」を聞くこと自体は悪くない。ただし、AIの推奨を意思決定として扱ってはいけない。

AIには、選択肢、反証、前提、影響、リスク、見落としを出させる。判断は人間が行う。

### 原則2: 判断基準を先に決める

判断基準が曖昧なままAIに分析させると、AIは整った一般論を返す。判断基準を先に定義すると、AIの出力を評価できる。

### 原則3: やめる判断を標準化する

AI活用では、始めることより、やめることが難しい。Kill Decision Sheetを用意し、施策、プロンプト、RAG、エージェント、ツール連携を定期的に見直す。

### 原則4: 判断をレベル分けする

すべてを重く扱うと遅くなる。すべてを軽く扱うと危険になる。判断レベルJ0〜J5で、記録と承認の重さを変える。

### 原則5: 不確実性をゼロにしようとしない

実務判断では、不確実性は残る。重要なのは、不確実性を明示し、検証指標と見直し日を設定することである。

### 原則6: 感情的回避を検査項目に入れる

恐れ、承認欲求、対立回避、責任回避は判断を歪める。判断前チェックリストに明示的に入れる。

### 原則7: 判断後レビューを改善へ接続する

判断後レビューは反省会ではない。Request Contract、Context Pack、Evaluation Plan、Governance Level Matrix、教育、テンプレートを改善するために行う。

## 判断ワークフロー

本章の成果物を使うと、判断ワークフローは次のようになる。

```text
判断対象を発見する
  ↓
Decision Briefを作る
  ↓
判断レベル J0〜J5 を決める
  ↓
Decision Criteria Matrixを作る
  ↓
AIに選択肢、反証、前提、利害関係者影響を整理させる
  ↓
Option Evaluation Tableを作る
  ↓
判断前チェックリストを確認する
  ↓
採用 / 修正 / 棄却 / 保留 / 実験化 / 停止 / やめる / リスク受容を決める
  ↓
Decision Briefに判断理由と検証指標を残す
  ↓
第8章のRun Reviewまたは第10章のGovernance Reviewへ接続する
  ↓
Decision Review Logで判断プロセスを振り返る
```

このワークフローは、すべての判断にフル適用するものではない。J0〜J1では簡易版でよい。J3以上では、必ず記録を残す。

## 章内演習

### 演習1: AI提案と人間判断を分ける

次のAI出力を読む。

```text
顧客返信案生成AIは、返信作成時間を大幅に削減できるため、全CSチームへ展開することを推奨します。
リスクはありますが、人間レビューを入れれば問題ありません。
```

次を作成する。

```text
1. AIが述べている事実
2. AIが述べている仮説
3. AIが述べている推奨
4. 人間が決めるべき論点
5. 判断前に必要な追加情報
```

### 演習2: 判断前チェックリストを埋める

自部門で進行中のAI活用案を1つ選ぶ。判断前チェックリストを埋める。

特に、次を空欄にしない。

```text
この判断で避けようとしている不快なこと：
言えていない懸念：
本当はやめるべきだが続けているもの：
怖いが試すべきもの：
```

### 演習3: Kill Decision Sheetを作る

現在使われているAIプロンプト、RAG、ワークフロー、エージェント、ダッシュボードのうち、利用率が低いものを1つ選ぶ。

Kill Decision Sheetを作成し、次のいずれかを判断する。

```text
継続
縮小
保留
中止
再設計
```

### 演習4: 判断後レビューを行う

過去1か月以内に行ったAI活用判断を1つ選び、Decision Review Logを作る。

結果ではなく、判断プロセスをレビューする。

```text
当時の前提は何だったか
AIは何を助けたか
AIは何を見落としたか
人間は何を避けたか
次回、何を変えるか
```

## まとめ

AIが仕事を速くするほど、人間には判断の質が問われる。

判断力とは、AIの出力を信じる能力ではない。AIを疑う能力だけでもない。AIが作った情報、選択肢、提案、反証を材料にしながら、人間が目的、責任、価値判断、リスク、関係者、タイミングを引き受け、採用、修正、棄却、保留、実験化、停止、やめる判断を下す能力である。

AI時代の組織では、作ることは簡単になる。だからこそ、何を作るか、何を作らないか、何をやめるか、いつ方向転換するかを決める力が重要になる。

第12章では、判断を妨げる大きな要因である、困難な会話と対立処理を扱う。AIは文面作成や論点整理を助ける。しかし、人間同士で直接扱うべき問題は残る。

# 第16章 組織展開と教材化

## この章を5分で使う場合

- まず作るもの: 1部門または1ロールに限定したRollout Charterと90-Day Rollout Planを作り、誰に何を学ばせるかを1枚で決める。
- 使うテンプレート: Role-based Learning Path、Curriculum Blueprint、Office Hour Runbook、AI Enablement KPI Dashboard、90-Day Rollout Plan。ワークショップ設計は [付録F](../../appendices/appendix-f/)、ガバナンス初期設定は [付録H](../../appendices/appendix-h/) を使う。
- 読まなくてよい節: 能力レベル、合格基準、評価証拠を新しく作り直さない。それらは [第15章](../chapter-15/) と [付録C: スキル評価ルーブリック](../../appendices/appendix-c/) へ戻す。
- 管理職が確認すべき判断: 展開対象、運用責任者、Office Hourの受け皿、高リスク業務の承認条件、KPIの確認頻度が決まっているか。
- 次章または付録へ渡すもの: 本章は最終章なので、決定事項を[付録A](../../appendices/appendix-a/)のテンプレート、[付録B](../../appendices/appendix-b/)のチェックリスト、付録Fのワークショップ設計、付録Hのガバナンス初期設定へ落とす。

## この章で扱う能力

この章では、本書の内容を、読書で終わらせず、社内の教育、標準成果物、評価、認定、権限付与、KPI、継続改善へ展開する方法を扱う。

第1章から第15章までは、AIエージェント協働を成立させるための設計要素を扱った。

- 第1章: AIエージェント協働の全体像。
- 第2章: AIに任せる仕事、任せない仕事。
- 第3章: Request Contract。
- 第4章: Context Pack。
- 第5章: タスク分解と委任。
- 第6章: AI出力レビューと評価。
- 第7章: ツール、権限、HITL。
- 第8章: ログ、トレース、継続改善。
- 第9章: セキュリティとプライバシー。
- 第10章: ガバナンスと統制。
- 第11章: AI時代の判断力。
- 第12章: 困難な会話と対立処理。
- 第13章: 失敗耐性と実験文化。
- 第14章: セルフトークと高レバレッジ環境。
- 第15章: スキルマップと評価。

本章では、これらを **組織展開計画、教材体系、運用体制、KPI、Office Hour、認定運用、継続改善サイクル** に変換する。

読者は本章を通じて、次のことができるようになる。

- 書籍内容を、ロール別の学習パスへ分解できる。
- 全社員向け、業務担当者向け、管理職向け、エンジニア向け、経営層向けの教材を設計できる。
- MVP教材、テンプレート、演習、チェックリスト、認定運用を定義できる。
- AI CoE、部門責任者、管理職、チャンピオン、情シス、セキュリティ、法務、人事の役割を設計できる。
- 90日展開計画を作成できる。
- 学習KPI、業務KPI、リスクKPI、実験KPIを設計できる。
- Office Hour、コミュニティ、改善バックログ、四半期レビューを運用できる。
- 教材、ポリシー、テンプレート、評価、台帳を継続的に更新できる。

この章の目的は、全社員に同じ研修を受けさせることではない。

目的は、AIエージェント協働を、個人の便利技から、チームの標準作業、さらに組織の実行能力へ変えることである。

## 第15章との責務境界

第15章は、能力を定義し、評価する章である。本章は、その評価結果を使って、組織展開を運用する章である。

| 項目 | 第15章 | 第16章 |
|---|---|---|
| 主な問い | 誰が、何を、どの証拠でできると言えるか | 誰に、何を、どの順序で学ばせ、どう運用するか |
| 主な入力 | 第1章〜第14章の成果物 | 第15章のRole Skill Profile、Evidence Portfolio、Learning Gap Handoff、Certification Decision Record |
| 主な出力 | 評価結果、不足領域、認定判定 | 教材、学習パス、Office Hour、KPI、90日展開計画、継続改善 |
| 認定 | 合格基準と判定記録 | 導入順序、権限付与、台帳反映、再評価通知 |
| 学習 | 不足能力を特定する | 教材、OJT、ワークショップ、支援体制を提供する |
| KPI | 評価に必要な証拠を定義する | 経営・現場・CoEが見る運用指標を管理する |

本章では、能力レベルや合格基準を再定義しない。それらは第15章と付録Cを参照する。本章で設計するのは、評価結果をもとにした展開の仕組みである。

## なぜ組織展開が必要か

AIエージェント協働は、個人のスキルだけで完結しない。

個人がうまく使っていても、組織としては次の問題が起きる。

| 状態 | 起きる問題 |
|---|---|
| 各自が自由に使う | 機密入力、誤送信、出力の無検証、ツール誤用が起きる |
| 一部の人だけが詳しい | 成果が属人化し、再現できない |
| プロンプトだけ共有される | 業務文脈、判断基準、制約が共有されない |
| 研修だけ実施する | 現場にテンプレート、レビュー、相談先がなく定着しない |
| ガバナンスだけ作る | 現場が使いにくくなり、シャドーAI利用が増える |
| 成功事例だけ共有する | 失敗から学べず、同じ事故を繰り返す |
| KPIが時間削減だけ | 品質、リスク、実験、学習、廃棄判断が見えない |
| 管理職が評価できない | AI成果物のレビューが個人任せになる |

AI活用は、便利ツール導入ではなく、業務設計の変更である。

業務設計を変えるには、教育、テンプレート、評価、承認、ログ、KPI、改善の接続が必要である。

## 組織展開のゴール

組織展開のゴールは、AI利用者を増やすことではない。

最終ゴールは、次の状態である。

```text
低リスクなAI活用は速く広がる。
高リスクなAI活用は適切に制御される。
AI出力はレビューされる。
AIの実行はログに残る。
失敗は学習に変換される。
教材、テンプレート、ポリシー、評価が継続的に更新される。
```

この状態を作るために、本章では組織展開を6つの設計対象に分ける。

| 設計対象 | 問い | 主な成果物 |
|---|---|---|
| 方針 | 何を目的にAIエージェント協働を展開するか | Rollout Charter |
| 体制 | 誰が何を担うか | AI Enablement Operating Model |
| 教材 | 誰に何を学ばせるか | Role-based Learning Path、Curriculum Blueprint |
| 実務資産 | 現場で何を使わせるか | Template Library、Checklist Pack、Portal Map |
| 運用 | 相談、認定、台帳、改善をどう回すか | Office Hour Runbook、Use Case Portfolio Board |
| 測定 | 何をもって効果と安全性を判断するか | KPI Dashboard、Quarterly Review Report |

## 組織展開の基本原則

### 原則1: 全員に同じ深さを求めない

全社員に必要なものと、特定ロールに必要なものは違う。

全社員には、入力禁止情報、基本的な依頼方法、レビューの基本、セキュリティ基礎が必要である。

一方で、ツール権限、HITL、評価データセット、AI System Inventory、Risk Register、AI Incident Runbookは、AI推進担当、エンジニア、管理職、情シス、セキュリティ担当が深く理解すればよい。

全員に同じ研修を実施すると、浅すぎるか、重すぎるかのどちらかになる。

### 原則2: 教材より先に利用ルールを決める

研修の前に、最低限の利用ルールを決める。

- 使ってよいAIツール。
- 入力してよい情報。
- 入力してはいけない情報。
- 顧客送信前のレビュー条件。
- ツール実行前の承認条件。
- AI起因インシデントの連絡先。
- 例外申請の方法。

ルールが曖昧な状態で研修だけ行うと、受講者は「結局何をしてよいのか」が分からない。

### 原則3: 研修だけでなく、テンプレートを配る

知識は実務に変換されなければ定着しない。

本書の展開では、各教材に最低1つの実務成果物を接続する。

| 教材 | 対応する成果物 |
|---|---|
| AIに任せる仕事の選び方 | AI適用判断シート |
| 依頼設計 | Request Contract |
| コンテキスト設計 | Context Pack |
| タスク分解 | Task Breakdown、Task Brief |
| レビュー | AI出力レビュー表、Review Issue Log |
| ツール権限 | ツール権限マトリクス、HITL承認フロー |
| ログ | Agent Run Log、Improvement Backlog |
| セキュリティ | AI Security Checklist、Data Classification Matrix |
| ガバナンス | AI System Inventory、AI Risk Register |
| 判断 | Decision Brief、Kill Decision Sheet |
| 対立 | Connection Debt診断、Conversation Plan |
| 実験 | AI実験設計シート、Failure Review Sheet |
| 行動 | Action Blocker診断、Minimum Action Contract |
| 評価 | Role Skill Profile、Evidence Portfolio |

### 原則4: 成功事例だけでなく失敗事例を扱う

AI活用は成功事例だけを共有すると危険である。

成功事例だけが共有されると、現場は次のように誤解する。

- AIはうまく使えば常に品質が上がる。
- 失敗は利用者の能力不足である。
- 事故に近い出力は隠した方がよい。
- レビュー工数は本当は不要なのではないか。

組織展開では、成功事例、失敗事例、廃棄判断、停止判断をすべて扱う。

特に重要なのは、廃棄判断である。

AIで実行速度が上がると、やめる判断が弱い組織では、プロンプト、テンプレート、RAG、エージェント、実験が増え続ける。

### 原則5: 低リスク利用を速くし、高リスク利用を止められるようにする

ガバナンスは、すべてを遅くするためにあるのではない。

むしろ、低リスク利用を速く進めるためにある。

| 利用分類 | 方針 |
|---|---|
| 低リスク | テンプレートと基本研修で速く展開する |
| 中リスク | レビュー、ログ、限定範囲、期間を設定する |
| 高リスク | 承認、台帳、評価、セキュリティレビューを必須にする |
| 禁止 | 明示的に禁止し、例外も原則認めない |

ルールがない組織では、低リスク利用も高リスク利用も同じように迷う。

良い統制は、利用を止めるだけでなく、進めてよいものを進めやすくする。

### 原則6: 権限付与と学習を接続する

AI活用能力の評価は、単なる受講履歴で終わらせない。

第15章で定義した認定結果は、利用可能範囲と接続する。

| 認定 | 許可される範囲の例 |
|---|---|
| AI Safe User | 社内許可ツールで低リスクな下書き、要約、整理を行う |
| AI Work Designer | Request Contract、Context Pack、AI出力レビュー表を使って自業務へ適用する |
| AI Agent Operator | ツール権限、ログ、評価、HITLを含む限定的な運用を設計する |
| AI Team Lead | チームのAI活用方針、レビュー、実験、失敗共有を運用する |
| AI Governance Owner | 台帳、リスク、承認、監査、教育、廃止を統括する |

ただし、認定を人事評価や序列化へ直結させると、失敗共有が減る。

初期段階では、認定は学習パス、権限付与、相談導線、リスク管理に使う。

## 展開対象者を分ける

本書の内容を教材化する場合、対象者を少なくとも6つに分ける。

| 対象 | 主目的 | 必須章 | 深く扱う章 |
|---|---|---|---|
| 全社員 | 安全に使い、基本的な依頼とレビューができる | 1,2,3,4,6,9,10 | 14 |
| 業務担当者 | 自業務にAIを組み込む | 2,3,4,5,6,13 | 11 |
| 管理職 | チーム利用、レビュー、承認、判断、対話、実験を運用する | 5,6,7,8,11,12,13,15 | 10,14,16 |
| エンジニア / AI推進担当 | ツール、権限、ログ、評価、セキュリティを設計する | 4,5,6,7,8,9,10 | 15,16 |
| 情シス / セキュリティ / 法務 | リスク、承認、データ、監査、インシデントを管理する | 7,8,9,10 | 6,15,16 |
| 経営層 | 投資判断、リスク許容度、組織体制、人材方針を決める | 1,2,10,11,13,15,16 | 12 |

全員がすべてを深く学ぶ必要はない。

重要なのは、各ロールが自分の責任範囲で必要な成果物を扱えることである。

## 教材体系を設計する

教材は、次の5階層に分ける。

| 階層 | 目的 | 形式 | 例 |
|---|---|---|---|
| Awareness | 最低限の認識を揃える | 15〜30分動画、1枚紙 | AI利用ポリシー、入力禁止情報 |
| Foundation | 基本技能を身につける | e-learning、確認テスト | Request Contract、Context Pack、AI出力レビュー |
| Practice | 実務ケースで練習する | ワークショップ、演習 | 顧客問い合わせ、提案書、競合調査 |
| Operation | チームで運用する | OJT、Office Hour、レビュー会 | HITL、Run Log、実験、失敗レビュー |
| Governance | 統制・監査・更新を回す | 管理者研修、設計会議 | AI System Inventory、Risk Register、Audit Evidence |

教材の形式は、目的ごとに変える。

| 目的 | 向く形式 | 向かない形式 |
|---|---|---|
| ルール周知 | e-learning、1枚紙、確認テスト | 長時間ワークショップ |
| 指示設計 | 添削演習、Before / After | 講義だけ |
| コンテキスト設計 | 資料整理演習、ケース演習 | プロンプト集配布だけ |
| レビュー能力 | 誤り検出演習、採点演習 | 成功事例紹介だけ |
| ツール権限 | 設計演習、権限マトリクス作成 | 抽象的な注意喚起だけ |
| 判断力 | ケース討議、Decision Brief作成 | 知識テストだけ |
| 対立処理 | ロールプレイ、Conversation Plan | AI文面生成だけ |
| 実験文化 | 実験設計、Failure Review | 成果発表だけ |
| ガバナンス | 台帳、承認、監査証跡の設計 | ポリシー読み合わせだけ |

## MVP教材を設計する

最初から全16章を研修化する必要はない。

初期展開では、最小構成で十分である。

### MVP教材20点

| 優先 | コンテンツ | 対象 | 目的 |
|---:|---|---|---|
| 1 | AI利用ポリシー1枚紙 | 全社員 | 許可、禁止、要承認を明確にする |
| 2 | 入力禁止情報チェックリスト | 全社員 | 機密、個人情報、顧客情報の誤入力を防ぐ |
| 3 | 全社員向け基礎e-learning | 全社員 | AIエージェント協働の最低限の理解を揃える |
| 4 | AIエージェント用語集 | 全社員 | チャットAI、RAG、ツール、HITL、ログなどの誤解を減らす |
| 5 | AI適用判断シート | 全社員、管理職 | 仕事選定を標準化する |
| 6 | Request Contractテンプレート | 全社員 | 依頼品質を上げる |
| 7 | Context Packテンプレート | 業務担当者 | 前提情報の渡し方を標準化する |
| 8 | AI出力レビュー表 | 全社員、管理職 | もっともらしい誤りを減らす |
| 9 | Review Issue Log | 管理職、AI推進担当 | 失敗を改善へ接続する |
| 10 | ツール権限マトリクス | エンジニア、情シス、管理職 | AIの実行範囲を制御する |
| 11 | HITL承認フロー | 管理職、承認者 | 重要操作を人間が止める |
| 12 | AI Security Checklist | 全社員、セキュリティ | AI固有リスクを確認する |
| 13 | AI System Inventory | AI推進担当、情シス | 社内AI利用を台帳化する |
| 14 | 職種別ユースケース集 | 全社員 | 使い方を想像しやすくする |
| 15 | 失敗事例集 | 全社員、管理職 | 誤用パターンを学ぶ |
| 16 | 2時間ワークショップ教材 | 業務担当者、管理職 | 実務適用まで進める |
| 17 | Skill Diagnosis Form | 全社員、管理職 | 不足領域を見える化する |
| 18 | Role Skill Profile | 管理職、AI推進担当 | ロール別到達目標を定義する |
| 19 | Office Hour Runbook | AI推進担当 | 現場相談を運用する |
| 20 | AI Enablement KPI Dashboard | 経営層、AI推進担当 | 効果とリスクを測る |

MVP教材は、完成度より展開速度を優先する。

ただし、次の4つは初期から必須である。

1. 利用ポリシー。
2. 入力禁止情報チェックリスト。
3. Request Contractテンプレート。
4. AI出力レビュー表。

この4つがない状態で全社展開すると、AI活用は属人化し、リスク判断も個人任せになる。

## Role-based Learning Path

ロール別学習パスは、第15章のRole Skill ProfileとLearning Gap Handoffを入力にして作る。

本章で定義するのは、カリキュラム、形式、所要時間、演習、支援、再評価タイミングである。ロール別の目標レベル、合格基準、評価証拠は第15章と付録Cで定義する。

### 全社員向け学習パス

| Module | 内容 | 形式 | 成果物 |
|---:|---|---|---|
| 1 | AIエージェント協働の基礎 | e-learning | 確認テスト |
| 2 | 社内AI利用ルール | e-learning | ポリシー同意 |
| 3 | 入力禁止情報と情報分類 | ケース問題 | 判定結果 |

RACIは、組織規模に応じて簡略化してよい。

## Office Hourを設計する

Office Hourは、AI活用を定着させるうえで重要である。

理由は、AIエージェント協働では、現場ごとに文脈が違うためである。

全員向け教材だけでは、次のような質問に答えられない。

- この顧客情報はAIに入れてよいのか。
- この問い合わせは低リスク扱いでよいのか。
- AI出力のこの表現は顧客に送ってよいのか。
- このプロンプトはRequest Contractとして不足しているか。
- このRAG文書は古いが、参照させてよいのか。
- このツール権限はapproval-requiredにすべきか。
- 実験として扱えるのか、正式なユースケース登録が必要か。

Office Hourは、個別相談窓口であると同時に、改善バックログの入力口である。

### Office Hourの基本運用

| 項目 | 設計 |
|---|---|
| 頻度 | 初期は週1回、定着後は隔週または月1回 |
| 形式 | 予約制 + 公開Q&A |
| 受付内容 | Request、Context、Review、Security、Governance、Experiment、Skill Evaluation |
| 事前提出物 | 相談テンプレート、対象成果物、扱う情報区分、期待する判断 |
| 記録 | 相談ログ、再利用可能なQ&A、改善バックログ |
| 公開範囲 | 機密を除き、抽象化して共有 |
| エスカレーション | セキュリティ、法務、情シス、部門責任者 |

Office Hourは、回答係を疲弊させる運用にしてはいけない。

相談を受けたら、次のいずれかに分類する。

| 分類 | 処理 |
|---|---|
| 個別回答で終わる | 相談ログへ記録する |
| FAQ化できる | ポータルへ追加する |
| テンプレート不足 | Template Libraryへ改善を追加する |
| ポリシー不明確 | Governance Reviewへ上げる |
| 教材不足 | Curriculum Backlogへ追加する |
| 高リスク | 承認フローまたはインシデント対応へ渡す |

## Use Case Portfolioを管理する

AI活用は、ユースケース単位で管理する。

すべてのAI利用を同じ重さで台帳登録する必要はないが、組織として追うべきユースケースは明確にする。

Use Case Portfolio Boardには、次の列を置く。

| 列 | 内容 |
|---|---|
| Idea | まだ仮説段階のAI活用案 |
| Intake | 目的、対象、データ、リスクを整理中 |
| Experiment | 限定範囲で小実験中 |
| Review | 結果、リスク、品質、ROIをレビュー中 |
| Scale Candidate | 展開候補 |
| Production | 標準業務として運用中 |
| Watch | 品質、コスト、リスクを監視中 |
| Retire | 廃止、統合、停止中 |
| Rejected | 採用しないと判断した案 |

各ユースケースには、最低限次を持たせる。

```text
ユースケース名：
Owner：
対象部門：
対象業務：
期待効果：
扱う情報：
リスク分類：
現在ステージ：
必要な成果物：
次の判断日：
続ける条件：
やめる条件：
```

Use Case Portfolioは、AI System Inventoryと同じではない。

| 区分 | 目的 |
|---|---|
| Use Case Portfolio | アイデア、実験、展開、廃棄を管理する |
| AI System Inventory | 運用中または統制対象のAI利用を台帳化する |

実験段階ではUse Case Portfolioで管理し、本番化や高リスク化したらAI System Inventoryへ登録する。

## 90日展開計画

組織展開は、90日で最初のサイクルを回す。

90日で完成させる必要はない。

90日で、次の状態まで進める。

- 利用ルールがある。
- MVP教材がある。
- 1〜2部門でパイロットが走っている。
- テンプレートが使われている。
- Office Hourが開始している。
- スキル診断と実技課題が試されている。
- KPIの初期値が取れている。
- 改善バックログが動いている。

### Phase 0: 事前準備

| 項目 | 内容 |
|---|---|
| 期間 | 開始前1〜2週間 |
| 目的 | 展開の目的、対象、体制を決める |
| 主担当 | Executive Sponsor、AI CoE |
| 成果物 | Rollout Charter、Operating Model、初期リスク方針 |

主な作業は次である。

- 経営上の目的を定義する。
- 対象部門を選ぶ。
- 初期ユースケース候補を集める。
- 利用禁止事項を仮決めする。
- AI CoEまたは暫定推進チームを置く。
- セキュリティ、法務、情シス、人事の連携先を決める。

### Phase 1: 設計

| 項目 | 内容 |
|---|---|
| 期間 | 1〜2週 |
| 目的 | 教材、ルール、テンプレート、診断の骨格を作る |
| 主担当 | AI CoE、HR / Learning、Security |
| 成果物 | AI Use Policy、MVP教材一覧、Skill Diagnosis Form、Use Case Intake |

主な作業は次である。

- AI利用ポリシー1枚紙を作る。
- 入力禁止情報チェックリストを作る。
- Request Contract、Context Pack、AI出力レビュー表を配布可能にする。
- スキル診断を作る。
- パイロット部門のRole Skill Profileを作る。
- Office Hourの受付方法を決める。

### Phase 2: MVP教材作成

| 項目 | 内容 |
|---|---|
| 期間 | 3〜4週 |
| 目的 | 最小限の教材と演習を作る |
| 主担当 | AI CoE、HR / Learning、部門代表 |
| 成果物 | e-learning、ワークショップ、演習、テンプレートライブラリ |

主な作業は次である。

- 全社員向け基礎e-learningを作る。
- 2時間ワークショップを作る。
- ケース演習を作る。
- 職種別ユースケース集を作る。
- 失敗事例集の初版を作る。
- Template Libraryを社内ポータルに置く。

### Phase 3: パイロット

| 項目 | 内容 |
|---|---|
| 期間 | 4〜8週 |
| 目的 | 1〜2部門で実務に適用する |
| 主担当 | Department Owner、Manager、AI CoE |
| 成果物 | パイロット実施ログ、事前・事後診断、Review Issue Log、Learning Log |

主な作業は次である。

- 1〜2部門を選んで実施する。
- 低リスクユースケースから始める。
- 事前診断を行う。
- ワークショップを実施する。
- 現場でRequest Contract、Context Pack、レビュー表を使う。
- Office Hourを開く。
- 実験結果、失敗、質問を改善バックログへ入れる。

### Phase 4: 評価と補正

| 項目 | 内容 |
|---|---|
| 期間 | 8〜10週 |
| 目的 | 教材、ルール、テンプレート、KPIを補正する |
| 主担当 | AI CoE、Manager、Security、HR |
| 成果物 | Pilot Review Report、Improvement Backlog、改訂版教材 |

主な作業は次である。

- 事後診断を行う。
- 実技課題の提出物をレビューする。
- AI出力レビュー表の指摘傾向を集計する。
- Office Hour相談を分類する。
- インシデントまたはヒヤリハットを確認する。
- 教材とテンプレートを改訂する。
- 高リスクユースケースを本番化しない判断も記録する。

### Phase 5: 展開判断

| 項目 | 内容 |
|---|---|
| 期間 | 10〜12週 |
| 目的 | 全社展開、部門追加、継続実験、停止を判断する |
| 主担当 | Executive Sponsor、AI CoE、Department Owner |
| 成果物 | Scale Decision Brief、Quarterly Review Report、次四半期計画 |

主な判断は次である。

- 全社展開する教材は何か。
- 対象部門を増やすか。
- どのユースケースを継続、拡大、停止、廃棄するか。

- 権限付与と認定をどこまで接続するか。
- 次四半期に強化する能力領域はどこか。

## 2時間ワークショップ設計

初期展開では、2時間ワークショップが有効である。

目的は、AIの概念理解ではなく、実際に1つの業務をAIエージェント協働の型へ変換することである。

| 時間 | 内容 | 成果物 |
|---:|---|---|
| 0:00-0:10 | 目的と利用ルール | AI利用ポリシー確認 |
| 0:10-0:25 | AIに任せる仕事、任せない仕事 | AI適用判断シート |
| 0:25-0:45 | Request Contract | AIエージェント依頼書 |
| 0:45-1:05 | Context Pack | Context Pack |
| 1:05-1:25 | AI出力レビュー | AI出力レビュー表 |
| 1:25-1:40 | リスクと承認 | 禁止事項・承認事項チェック |
| 1:40-1:55 | 小実験設計 | AI実験設計シート |
| 1:55-2:00 | 次の行動 | Minimum Action Contract |

- 自分の業務1件のAI適用判断シート。

- 1週間以内に行う最小実験。

## 認定制度への接続

第15章では、認定の合格基準と判定記録を定義した。本章では、その認定を組織運用へ接続する。

認定制度は、初期から重くしすぎない。最初に必要なのは、全員に広く求める最小確認と、高リスク利用に必要な限定認定を分けることである。

| 運用項目 | 本章で決めること |
|---|---|
| 導入順序 | どの認定から開始するか |
| 受講導線 | どの教材・演習・Office Hourへ接続するか |
| 権限付与 | どのツール権限・承認者権限と接続するか |
| 台帳反映 | 認定結果をどの台帳に記録するか |
| 再評価 | いつ、何を根拠に再評価するか |
| 例外処理 | 認定済みでも個別承認が必要な条件をどう扱うか |

推奨する展開順序は次である。

| 段階 | 運用内容 |
|---|---|
| Step 1 | 全社員向けAI Safe User確認を実施する |
| Step 2 | 低リスク業務のAI Work Designer実技課題をパイロットする |
| Step 3 | 管理職向けAI Team Leadワークショップを実施する |
| Step 4 | ツール権限を扱うAI Agent Operator認定を限定導入する |
| Step 5 | AI Governance OwnerをAI CoE、情シス、セキュリティ、法務、管理部門へ広げる |

認定と権限は、直結させすぎても危険である。認定は能力の証拠であり、個別ユースケースのリスク承認ではない。

| 利用範囲 | 認定で確認すること | 追加で必要な運用 |
|---|---|---|
| 低リスクな要約、下書き | AI Safe User | 入力禁止情報チェック |
| 自業務のAI適用設計 | AI Work Designer | 上長レビュー、必要に応じたOffice Hour |
| チーム利用のレビューと実験運用 | AI Team Lead | チーム方針、失敗レビュー、KPI運用 |
| ツール権限、HITL、ログ設計 | AI Agent Operator | 個別Tool Approval、Permission Test |
| 高リスクAI利用の承認、台帳、監査 | AI Governance Owner | Risk Register、Audit Evidence、四半期レビュー |

高リスク操作では、認定済みであっても、個別の承認、ログ、リスクレビューが必要である。

## KPI設計

AIエージェント協働のKPIは、時間削減だけにしてはいけない。

時間削減は重要だが、それだけでは次を見落とす。

- AI出力の品質。
- レビュー負荷。
- 誤出力、誤送信、情報漏洩。
- 実験数と学習数。
- 廃棄判断。
- 認定と権限付与。
- 教材更新。
- チーム内の対立や行動停止。

### KPIカテゴリ

Skillカテゴリの元データは、第15章のSkill Diagnosis Form、Evidence Portfolio、Certification Decision Recordから取得する。本章では、それらを組織展開のダッシュボードへ載せる。

| カテゴリ | KPI例 |
|---|---|
| Adoption | 受講率、テンプレート利用率、Office Hour利用数、ユースケース登録数 |
| Skill | 診断スコア、実技合格率、認定数、再評価合格率 |
| Productivity | 作業時間削減、リードタイム短縮、下書き作成時間、調査時間 |
| Quality | Review Issue件数、Critical Issue率、差し戻し率、再利用率 |
| Risk | AI起因インシデント、ヒヤリハット、禁止情報入力、承認拒否率 |
| Governance | AI System Inventory登録率、Risk Register更新率、Audit Evidence完成度 |
| Experiment | 実験数、失敗共有数、廃棄判断数、Learning Log数、Pace / Spin |
| Cost | ツール費用、API費用、レビュー工数、再実行コスト |
| Human / Org | Connection Debt解消数、Action Blocker改善数、レビュー会実施数 |

### ダッシュボードで見るべき組み合わせ

単一KPIだけで判断してはいけない。

| 見たいこと | 組み合わせるKPI |
|---|---|
| AI利用が安全に広がっているか | 利用数 + Review Issue率 + インシデント数 |
| 生産性が上がっているか | 作業時間削減 + レビュー工数 + 手戻り率 |
| 実験文化があるか | 実験数 + 失敗共有数 + 廃棄判断数 |
| ガバナンスが機能しているか | Inventory登録率 + 承認ログ + Risk Register更新率 |
| 教材が効いているか | 受講率 + 実技合格率 + テンプレート利用率 + Office Hour相談内容 |
| チームが止まっていないか | Action Blocker数 + Pace / Spin + 判断保留期間 |

KPIは経営会議用と現場運用用で分ける。

### 経営会議向けKPI

- 主要ユースケース数。
- 本番化ユースケース数。
- 高リスクユースケース数。
- 作業時間削減見込み。
- AI起因インシデント件数。
- 認定者数。
- 廃棄判断数。
- 次四半期の投資判断。

### 現場運用向けKPI

- Request Contract作成数。
- Context Pack作成数。
- Review Issue Log件数。
- Office Hour相談カテゴリ。
- Critical Issue率。
- 実験のPace / Spin。
- Improvement Backlogの滞留日数。
- 教材更新件数。

## 継続改善サイクル

AIエージェント協働は、一度教材を作って終わりではない。

モデル、ツール、社内業務、法務条件、セキュリティ要件、組織体制は変わる。

そのため、次のサイクルを回す。

```text
利用する
↓
記録する
↓
レビューする
↓
改善する
↓
教材・テンプレート・ポリシーへ反映する
↓
再教育する
↓
再評価する
```

### 月次運用

| 活動 | 内容 |
|---|---|
| Office Hour集計 | 相談内容を分類し、FAQ、教材、ポリシーへ反映する |
| Improvement Backlogレビュー | 未対応の改善項目を優先順位付けする |
| Use Case Portfolio更新 | 実験、本番、停止、廃棄を更新する |
| インシデント・ヒヤリハット確認 | 重大度と再発防止策を確認する |
| 教材小改訂 | FAQ、テンプレート、チェックリストを更新する |

### 四半期運用

| 活動 | 内容 |
|---|---|
| KPIレビュー | 学習、業務、品質、リスク、実験、コストを確認する |
| 教材改訂 | 章、演習、テンプレート、認定課題を更新する |
| ポリシー改訂 | 新ツール、新リスク、例外運用を反映する |
| 認定レビュー | 合格基準、評価者校正、再評価条件を見直す |
| ユースケース棚卸し | 継続、拡大、縮小、停止、廃棄を判断する |
| 経営報告 | 次四半期の投資、リスク、体制を判断する |

### 年次運用

| 活動 | 内容 |
|---|---|
| AI活用戦略レビュー | 事業戦略、組織体制、投資方針との整合を確認する |
| ガバナンス監査 | 台帳、承認、ログ、教育、インシデント、ベンダーを確認する |
| スキルマップ更新 | 新しいロール、ツール、業務リスクを反映する |
| 教材体系更新 | 旧教材、旧テンプレート、旧ポリシーを廃止する |

## 典型的な失敗

### 失敗1: 研修を実施して終わる

受講率だけが上がり、業務は変わらない。

原因は、テンプレート、実技課題、現場相談、管理職レビューがないことである。

対策は、研修ごとに成果物提出を必須にすることである。

### 失敗2: プロンプト集だけを配る

プロンプト集は便利だが、文脈が違うと品質は再現しない。

Request Contract、Context Pack、Review Issue Logとセットで配る。

### 失敗3: ガバナンスが重すぎる

すべてのAI利用に承認を求めると、現場は使わなくなるか、非公式に使う。

低リスク利用は簡単に始められるようにし、高リスク利用だけを明確に止める。

### 失敗4: 成功事例だけを共有する

成功事例だけでは、リスク判断が育たない。

Critical Issue、ヒヤリハット、やめた実験、廃棄したテンプレートも共有する。

### 失敗5: AI推進担当にすべてを背負わせる

AI CoEだけでは現場の文脈を持てない。

部門Owner、Manager、AI Championを置き、部門ごとの責任を明確にする。

### 失敗6: 認定制度が目的化する

認定を取ることが目的になると、実務成果物が弱くなる。

認定は、学習パス、権限付与、OJT、再評価と接続する。

### 失敗7: KPIが時間削減だけになる

時間削減だけを見ると、レビュー省略、リスク見落とし、失敗隠しが起きる。

品質、リスク、実験、廃棄、学習、コストを同時に見る。

### 失敗8: 教材が更新されない

AIツール、社内ルール、利用実態は変わる。

教材は四半期更新を前提にする。

### 失敗9: 管理職が関与しない

現場担当者だけがAIを使い、管理職がレビュー基準を持たない。

チーム単位のAI活用には、管理職向け教材とレビュー演習が必要である。

### 失敗10: 対立や不安を扱わない

AI活用は、既存の責任境界、品質基準、業務範囲、評価不安に触れる。

第12章、第14章の内容を、管理職研修とOffice Hourに入れる。

## 実務ケース1: CS部門への展開

### 状況

CS部門では、問い合わせ回答案の作成に時間がかかっている。

過去FAQ、ナレッジ記事、過去回答例はあるが、回答品質にばらつきがある。

AIを使えば下書き時間を短縮できる可能性があるが、次のリスクがある。

- 原因未確定事項を断定する。
- SLA、補償、契約に触れる。
- 顧客情報を不適切に入力する。
- 社内メモを顧客向け文面に混ぜる。

### 展開設計

| 項目 | 設計 |
|---|---|
| 対象 | 低リスク問い合わせの回答案作成 |
| 除外 | SLA、補償、契約、障害原因未確定、個人情報、セキュリティ関連 |
| 必須教材 | AI利用ポリシー、Request Contract、Context Pack、AI出力レビュー、セキュリティ基礎 |
| 必須成果物 | AI適用判断シート、Request Contract、Context Pack、AI出力レビュー表 |
| 承認 | 顧客送信前に人間レビュー必須 |
| KPI | 回答案作成時間、Review Issue率、Critical Issue率、承認拒否率、Office Hour相談数 |
| 初期実験 | 2週間、2名、低リスク問い合わせ20件 |
| 停止条件 | Critical Issue 1件以上、顧客影響のある誤送信、禁止情報入力 |

### 学習パス

CS担当者には、AI Work Designer相当の到達目標を設定する。

管理職には、AI Team Lead相当のレビュー基準と承認フロー設計を求める。

### 展開判断

2週間後に、次を見て判断する。

- 作業時間は短縮されたか。
- レビュー工数は増えすぎていないか。
- Critical Issueは発生したか。
- 低リスク分類は妥当だったか。
- 現場はRequest Contractとレビュー表を使えているか。
- 顧客送信前レビューは機能しているか。

全社展開ではなく、対象問い合わせを限定して拡大するのが妥当である。

## 実務ケース2: 開発チームへの展開

### 状況

開発チームでは、GitHub IssueからPR説明、テスト観点、変更方針を作る作業にAIを使いたい。

一方で、AIにコード変更やPR作成を任せると、次のリスクがある。

- secretや内部情報を読み込む。
- テスト不足の変更を提案する。
- 直接pushや強制mergeを行う。
- 既存設計意図を誤解する。

### 展開設計

| 項目 | 設計 |
|---|---|
| 対象 | Issue整理、PR説明、テスト観点、変更案作成 |
| 除外 | 直接push、強制merge、本番変更、secret参照 |
| 必須教材 | Task Brief、Review Issue Log、Tool Spec Card、HITL、Agent Run Log、セキュリティ |
| 必須成果物 | Task Breakdown、Tool Permission Matrix、Permission Test、Evaluation Plan |
| 承認 | PRは人間レビュー必須。mergeはAI不可 |
| KPI | PR作成時間、レビュー指摘、CI失敗率、差し戻し率、Permission Test通過率 |
| 初期実験 | 非本番、低リスクrepo、3名、2週間 |
| 停止条件 | secret露出、直接push試行、本番影響、重大なテスト欠落 |

### 学習パス

開発者には、AI Work DesignerからAI Agent Operatorへの段階的到達を設定する。

テックリードには、AI Team Leadとしてレビュー基準、HITL、実験KPIを運用させる。

## 実務ケース3: 管理職研修への展開

### 状況

AI利用は現場で増えているが、管理職がレビュー基準を持っていない。

その結果、次の問題が起きている。

- AI出力をそのまま使っている人がいる。
- 逆に、AI出力を過度に疑い、活用が進まない人がいる。
- 困難な会話をAI文面生成で先送りしている。
- 実験の失敗が共有されない。

### 研修設計

| Module | 内容 | 成果物 |
|---:|---|---|
| 1 | チームAI活用方針 | Team AI Use Charter |
| 2 | AI成果物レビュー | チームレビュー基準、Review Issue Log |
| 3 | 承認フロー | HITL承認フロー |
| 4 | 判断力 | Decision Brief |
| 5 | 対立処理 | Connection Debt診断、Conversation Plan |
| 6 | 実験文化 | Experiment Portfolio、Failure Review Sheet |
| 7 | 行動ブロッカー | Action Blocker診断 |

管理職研修では、プロンプトの細かい書き方よりも、次を重視する。

- どのAI出力をレビューするか。
- どの操作を承認制にするか。
- どの失敗を共有するか。
- どの対立を会話で処理するか。
- どの実験を続け、どれをやめるか。

## 変更管理と社内コミュニケーション

AI展開では、正しい内容を作るだけでなく、どう伝えるかが重要である。

特に、次の誤解を避ける。

| 誤解 | 伝えるべきこと |
|---|---|
| AIで人を置き換える | まずは業務の一部を安全に委任し、人間の判断とレビューを強化する |
| AI利用は任意の個人技 | 組織の標準成果物とレビューに接続する |
| 失敗したら禁止される | 失敗を隠す方が危険。失敗は条件付きで共有し、改善へ接続する |
| ガバナンスはブレーキ | 低リスク利用を速くし、高リスク利用を止めるための仕組み |
| 認定は査定 | 初期は学習パス、権限付与、リスク管理のために使う |
| AI推進担当が全部やる | 部門Owner、Manager、Championが現場運用を担う |

### コミュニケーション計画

| 対象 | メッセージ | チャネル |
|---|---|---|
| 全社員 | 使ってよいこと、禁止事項、最初の学習パス | 全社会議、社内ポータル、メール |
| 管理職 | チームレビューと承認の責任 | 管理職会議、ワークショップ |
| エンジニア | ツール権限、ログ、HITL、評価の設計 | 技術会議、ハンズオン |
| 情シス / セキュリティ | 外部ツール、ログ、インシデント、台帳 | セキュリティ定例 |
| 経営層 | KPI、リスク、投資、組織体制 | 経営会議 |
| AI Champion | 現場相談、テンプレート利用、事例共有 | コミュニティ |

変更管理では、禁止事項だけを強く伝えない。

同時に、許可される低リスク利用、使えるテンプレート、相談先、実験の進め方を明示する。

## 教材更新と廃止

教材は増え続けると、現場が使わなくなる。

教材にもライフサイクルを持たせる。

| ステージ | 内容 |
|---|---|
| Draft | パイロット用の暫定教材 |
| Active | 現在の標準教材 |
| Needs Review | モデル、ツール、社内ルール変更により見直しが必要 |
| Deprecated | 新教材に置き換え済み |
| Retired | 利用停止、参照不可またはアーカイブ |

テンプレートにも同じ管理が必要である。

特に、古いプロンプト、古いContext Pack、旧承認フロー、旧RAG参照先は、誤用の原因になる。

四半期レビューでは、次を確認する。

- 古い教材は残っていないか。
- 現在の利用ツールと教材が一致しているか。
- 新しいリスクが教材に反映されているか。
- 使われていないテンプレートは廃止するか。
- 失敗事例は教材に反映されたか。

## 組織展開ワークフロー

第16章の実務ワークフローは次である。

```text
1. Rollout Charterを作る
2. Operating Modelを決める
3. 対象ロールと対象部門を決める
4. MVP教材を作る
5. Role-based Learning Pathを作る
6. Use Case Portfolioを作る
7. Office Hourを開始する
8. 1〜2部門でパイロットする
9. KPIとReview Issueを確認する
10. 教材、テンプレート、ポリシーを改訂する
11. 認定と権限付与へ接続する
12. 四半期レビューで拡大、停止、廃棄を判断する
```

このワークフローは、直線ではない。

Office Hour、Review Issue Log、Failure Review、Incident Runbook、KPIから、常に前段の教材、テンプレート、ガバナンスへ戻る。

## 設計原則

本章の設計原則を整理する。

### 原則1: 書籍をそのまま研修にしない

書籍は体系であり、研修は行動変容のための設計である。

研修では、ロール別に必要章を選び、成果物作成、演習、レビューへ落とす。

### 原則2: 研修、テンプレート、Office Hourをセットにする

研修だけでは現場で止まる。

テンプレートだけでは理解が浅い。

Office Hourだけでは属人化する。

3つをセットにする。

### 原則3: 低リスク利用から始める

最初から高リスク業務にAIエージェントを入れない。

低リスクな下書き、分類、要約、レビュー補助、小実験から始める。

### 原則4: 管理職を外さない

AI活用は現場担当者だけの問題ではない。

管理職がレビュー、承認、実験、失敗共有、対立処理を扱えないと、組織展開は止まる。

### 原則5: 失敗と廃棄を測る

失敗共有数、廃棄判断数、停止判断数は、実験文化の健全性を示す。

成功事例数だけでは不十分である。

### 原則6: ポリシーと教材を一緒に更新する

ポリシーだけ変えても、現場は行動を変えない。

教材、テンプレート、FAQ、Office Hour、認定課題を同時に更新する。

### 原則7: 経営報告はKPIだけでなく判断を含める

経営層には、数字だけでなく次の判断を提示する。

- 拡大するユースケース。
- 停止するユースケース。
- 追加投資する領域。
- 許容しないリスク。
- 強化する人材能力。

## 演習

### 目的

自組織向けに、AIエージェント協働の90日展開計画を作る。

### シナリオ

あなたは、IT企業のAI推進担当である。

経営層から、AIエージェント協働を社内に展開するよう依頼された。

現在の状況は次である。

- 一部の社員はAIを日常的に使っている。
- 社内ルールはあるが、具体的な入力禁止情報と承認条件が曖昧である。
- CS部門は問い合わせ回答案生成に関心がある。
- 開発部門はIssue整理、PR説明、テスト観点作成に関心がある。
- 営業部門は提案書ドラフトと競合比較に関心がある。
- セキュリティ担当は外部AIツールとMCP連携を懸念している。
- 管理職はAI成果物のレビュー基準を持っていない。
- 経営層は90日以内に効果とリスクを可視化したい。

### 提出物

次を作成する。

1. Rollout Charter。
2. AI Enablement Operating Model。
3. 対象部門と初期ユースケース。
4. Role-based Learning Path。
5. MVP教材20点の優先順位。
6. 2時間ワークショップ設計。
7. Office Hour Runbook。
8. Use Case Portfolio Board。
9. AI Enablement KPI Dashboard。
10. 90-Day Rollout Plan。
11. Change Communication Plan。
12. Quarterly Review Reportの雛形。

### 合格基準

- AI利用拡大そのものではなく、業務成果、品質、安全性、学習を目的にしている。
- 低リスク利用と高リスク利用が分かれている。
- 対象ロールごとに学習パスが異なる。
- 管理職向けのレビュー、承認、判断、対立処理、実験文化が含まれている。
- セキュリティ、法務、情シス、人事の関与ポイントが定義されている。
- CS、開発、営業の初期ユースケースが、リスクと成果物に分解されている。
- Request Contract、Context Pack、AI出力レビュー表がMVPに含まれている。
- Office Hourが単なる相談窓口ではなく、FAQ、教材、ポリシー、改善バックログへ接続している。
- KPIが時間削減だけでなく、品質、リスク、実験、失敗共有、廃棄判断を含んでいる。
- 90日計画に、設計、教材作成、パイロット、評価、展開判断が含まれている。
- 拡大条件だけでなく、停止条件、廃棄条件がある。
- 教材とポリシーの更新責任者が明確である。

## Source Notes

本章は、社内でAIエージェント協働を展開するための運用設計を扱う。研修、評価、認定、ログ、AI利用台帳、KPI、Office Hour、コミュニティ運営は、各社の人事制度、情報セキュリティ規程、個人情報保護方針、労務管理、教育記録の扱いに合わせて調整する必要がある。

実導入時には、AIリテラシー要件、教育履歴の保存期間、評価結果の利用範囲、職務権限との接続、外部AIツールの契約条件、ログの二次利用、部門別リスク分類を確認する。特に、認定結果を権限付与や人事評価へ接続する場合は、透明性、公平性、異議申し立て、更新条件を明確にする。

## まとめ

AIエージェント協働は、個人の使い方で終わらせてはいけない。

組織能力に変えるには、書籍、教材、テンプレート、チェックリスト、評価、認定、Office Hour、KPI、ガバナンス、継続改善を接続する必要がある。

本章の要点は次である。

- 全員に同じ研修を実施するのではなく、ロール別学習パスを作る。
- 研修だけでなく、テンプレート、チェックリスト、演習、Office Hourをセットにする。
- 低リスク利用は速く広げ、高リスク利用は承認、ログ、評価、台帳で制御する。
- 成功事例だけでなく、失敗事例、廃棄判断、停止判断を共有する。
- KPIは時間削減だけでなく、品質、リスク、実験、失敗共有、廃棄、コストを含める。
- 管理職を、レビュー、承認、判断、対立処理、実験文化の運用者として育成する。
- 教材、ポリシー、テンプレート、評価は四半期単位で更新する。

AIエージェント協働の最終形は、AIをよく使う個人が増えることではない。

最終形は、組織が次の問いに答えられることである。

```text
どの仕事にAIを使うのか。
何をAIに依頼するのか。
どの前提を渡すのか。
どこまで委任するのか。
どの出力をどうレビューするのか。
どの操作を止めるのか。
どの失敗から何を学ぶのか。
何を広げ、何をやめるのか。
誰が責任を持つのか。
```

この問いに、成果物、運用、KPIで答えられる状態が、AIエージェント協働が組織能力になった状態である。

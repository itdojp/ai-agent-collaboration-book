# 執筆ガイド

## 本書の文体

- 断定すべき設計原則は明確に断定する。
- モデル性能、料金、UI、API仕様など変化しやすい情報は断定せず、公式情報確認を促す。
- 「AIが考える」「AIが理解する」といった擬人化は避ける。必要な場合は、生成、分類、検索、推定、ツール実行、状態更新と書く。
- 読者の行動に落ちる表現を優先する。「重要である」だけで終えず、「何を作るか」「どう確認するか」を書く。
- 社内研修へ転用しやすいように、章末に演習と成果物を置く。

## 中核フレームワーク

本書のすべての章は、次の6層のどこに位置するかを明確にする。

| 層 | 内容 | 対応章 |
|---|---|---|
| Work | 仕事を選ぶ | 第2章 |
| Request | 依頼を契約にする | 第3章 |
| Context | 文脈を設計する | 第4章 |
| Delegation | 作業を分解して委任する | 第5章 |
| Control | 検証、権限、ログ、統制を設計する | 第6〜10章 |
| Human / Org | 判断、対話、実験、展開を設計する | 第11〜16章 |

各章では、概念説明の後に必ず「この層で読者が作る成果物」を置く。

```text

## チェックリスト

```

## 用語方針

| 用語 | 使い方 |
|---|---|
| AIエージェント | 状態、ツール、複数ステップ実行、承認、評価を伴うAI利用形態 |
| チャットAI | 単発または短い会話で回答するAI利用形態 |
| Request Contract | 目的、入力、制約、出力、受け入れ条件を含む依頼仕様 |
| Context Pack | AIに渡す前提情報を構造化した文書 |
| HITL | Human-in-the-loop。人間確認、承認、差し戻しを含む設計 |
| ガードレール | 禁止事項、入力・出力検査、権限制限、承認ゲートなど |
| Discernment | AIからの出力と人間の責任判断を分離する識別力 |
| Connection Debt | 言うべきことを言わないことで蓄積する関係・組織上の負債 |

## 出典方針

- AIツール、API、標準、法規制、リスクフレームワークは一次情報または公式情報を優先する。
- Lenny's Newsletter の記事は、経験知・実践仮説として扱う。科学的に確立済みの法則としては扱わない。
- 引用は短くし、本文では要約を中心にする。
- 各章の最後に「Source Notes」を置くか、全体の `source-notes.md` にまとめる。

## 章ごとの完成条件

1. 学習目標が明確である。
2. 典型的な失敗が3つ以上ある。
3. 実務ケースが1つ以上ある。
4. 読者が作る成果物が1つ以上ある。
5. チェックリストがある。
6. 演習がある。
7. 関連書籍への接続がある。
8. 変化しやすい情報には確認条件がある。

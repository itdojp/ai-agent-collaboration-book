## 1. 総評

本書は、初稿としては十分に体系化されている。特に強い点は以下である。

- 6層モデル `Work / Request / Context / Delegation / Control / Human / Org` が全体の背骨として機能している。
- 各章が「概念」ではなく「成果物」に接続されている。
- 第1章〜第10章で、AIエージェント協働を業務システムとして扱えている。
- 第11章〜第14章で、人間側の判断、対立、実験、行動停止を業務設計として扱えている。
- 第15章〜第16章で、個人スキルを組織展開、研修、認定、KPIへ接続できている。

現時点の最大の課題は、内容不足ではない。むしろ、初稿執筆を連続的に進めたため、**章間の粒度差、重複、章末セクション名の不統一、付録D〜Hの古さ、図表未整備、Source Notesの整理不足**が残っている。

したがって、次工程は「本文追加」ではなく、**編集・圧縮・統一・図表化・出典確認**である。

## 4. 構成面の評価

### 4.1 章順は妥当

現在の章順は妥当である。

```text
0 読み方
1〜2 基礎・仕事選定
3〜5 依頼・コンテキスト・委任
6〜8 レビュー・ツール・ログ
9〜10 セキュリティ・ガバナンス
11〜14 人間側能力
15〜16 評価・組織展開
```

この順序は、実務導入時の流れとも一致している。

```text
仕事を選ぶ
→ 依頼する
→ 文脈を渡す
→ 委任する
→ レビューする
→ 権限で制御する
→ ログを残す
→ セキュリティ・ガバナンスに載せる
→ 人間が判断・対話・実験する
→ スキル評価・組織展開へ接続する
```

### 4.2 第0章は長すぎる

第0章は実務導線として有用だが、導入章としては情報量が多い。第0章だけで約2万字あり、読者が本編に入る前に疲れる可能性がある。

推奨対応は以下。

- 第0章は10,000〜12,000字程度に圧縮する。
- 詳細な読者別ルートは `quickstart.md` または `artifact-index.md` に移す。
- 「CS部門」「開発チーム」「経営層」の詳細ケースは第16章または付録Fへ移す。
- 第0章には、6層モデル、読む順序、最初の3テンプレート、30日以内の小実験だけを残す。

### 4.3 第15章と第16章は一部重複している

第15章はスキルマップ・評価、 第16章は組織展開・教材化であり、役割は分かれている。ただし、ロール別学習パス、認定、評価、KPI、権限付与が両章で重なっている。

推奨する責務分離は以下。

| 領域 | 第15章 | 第16章 |
|---|---|---|
| スキル定義 | 主担当 | 参照のみ |
| レベル・ルーブリック | 主担当 | 参照のみ |
| 実技課題 | 主担当 | 研修設計へ接続 |
| 認定 | 合格基準を定義 | 運用制度に載せる |
| 学習パス | ロール別到達目標 | 研修カリキュラム化 |
| KPI | 能力向上指標 | 展開・業務効果指標 |
| 権限付与 | 認定条件 | ワークフロー運用 |

### 4.4 第7章と第10章の承認フロー重複を整理する

第7章は「操作単位の権限・HITL」、第10章は「組織統制としての承認」である。現状でも概ね分かれているが、読者には重複して見える可能性がある。

推奨する責務分離は以下。

| 領域 | 第7章 | 第10章 |
|---|---|---|
| 承認対象 | ツール操作 | ユースケース、システム、例外、リスク受容 |
| 成果物 | Tool Spec Card、Permission Test | AI Use Policy、AI System Inventory、AI Risk Register |
| 判断単位 | API呼び出し、送信、更新、削除 | 部門導入、本番化、外部AI利用、継続利用 |
| 承認者 | 操作責任者、業務責任者 | AI CoE、Security、Legal、部門長、経営 |

第7章冒頭か第10章冒頭に、この差分表を置くとよい。

### 5.2 章末Source Notesの名称を統一する

推奨修正:

```text

```

に全章統一する。第0章は外部出典に依存しない旨を短く置く。第14章、第16章は章末にもSource Notes節を追加する。

### 5.3 日本語・英語併記のルールを明示する

本書は実務成果物名に英語を多く使っている。これは有用だが、初出時ルールを統一した方がよい。

推奨ルール:

```text
初出: 日本語名 / English Name
以降: English Nameを主表記。ただし章見出しでは日本語名を併記。
```

例:

```text

判断メモ / Decision Brief
実行ログ / Agent Run Log
```

## 8. 出典・Source Notesレビュー

現時点のSource Notesは、出典候補メモとしては有効である。ただし公開版にはそのままでは不十分である。

主な問題:

- `source-notes.md` の冒頭に第0章メモが置かれ、その後に全体メモが続くため順序が不自然。
- 第1章〜第8章の章別Source Notesが薄い。

- 外部資料のURL確認・用語確認・公開範囲確認がまだ最終化されていない。
- 本文内の出典注記が少ない。

推奨する `source-notes.md` 構成:

```text
# Source Notes

## 全体方針
## 既存シリーズ関連
## 公式・標準・セキュリティ関連
## 人間側能力・チーム設計
## 章別メモ
  第0章
  第1章
  ...
  第16章
## 公開前に再確認するURL
## 本文へ脚注化する候補
```

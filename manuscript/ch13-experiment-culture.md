# 第13章 失敗耐性と実験文化

## この章で扱う能力

この章では、AIエージェントを単なる効率化ツールではなく、**仮説検証と学習速度を上げるための道具**として使う方法を扱う。

読者は、次のことができるようになる。

- AI活用を、効率化、品質向上、実験、展開の4種類に分けて扱う
- 大きな施策を、安全な小実験へ分解する
- AI実験の仮説、成功条件、失敗条件、やめる条件を定義する
- 失敗を責任追及ではなく、学習、改善、廃棄判断へ接続する
- 実験数だけでなく、学習速度、廃棄判断、レビュー負荷、リスクを測る
- Pace / Spinを使って、チームの実験速度と空回りを診断する
- 第11章の判断、第12章の対立を、検証可能な小実験へ変換する
- AIを使って実験案、反証、プロトタイプ、評価ケース、失敗レビューを作る

第11章では、AI時代の判断力を扱った。AIが情報や選択肢を作れるほど、人間は採用、棄却、停止、やめる判断を引き受ける必要がある。

第12章では、困難な会話と対立処理を扱った。判断が止まる理由は、情報不足だけではない。責任境界、優先順位、リスク認識、言えていない懸念が、未処理のまま残ることもある。

第13章では、その次の段階を扱う。すべての判断を会議で決着させる必要はない。すべての対立を説得で解消する必要もない。**検証可能な論点は、小さな実験へ変換できる**。

```text

Context：AIに渡す前提を設計する

```

第13章の対象は、「AIを使えば失敗を気にせず何でも試せる」という話ではない。

逆である。

AIで試行コストが下がるほど、雑な実験、未統制な試行、ログのない利用、やめられない施策、成功事例だけの共有が増えやすい。

そのため、AI時代の実験文化では、次を同時に設計する。

```text
小さく試す
安全に試す
速く学ぶ
失敗を記録する
次の仮説へ変換する
使えないものを捨てる
必要なものだけ展開する
```

## なぜAI時代に実験文化が重要になるのか

AIエージェントは、既存業務を速くする。

資料の下書き、議事録要約、競合比較、コード補助、レビュー補助、問い合わせ分類、FAQ検索、提案書構成、障害報告書ドラフトは短時間で作れる。

しかし、既存業務を速くするだけでは、競争優位は限定的である。

多くの組織が同じようにAIを導入すれば、単純な時間削減は早晩コモディティ化する。重要なのは、削減した時間を何に使うかである。

選択肢は2つある。

```text
1. 既存業務を速く処理し、余った時間を別の既存業務で埋める
2. AIで試行回数を増やし、未知の仮説をより多く検証する
```

1だけでは、組織は忙しいままである。AIでタスク量が増え、レビュー負荷が増え、承認が詰まり、結局ボトルネックが人間側に移る。

2へ進むと、AIは単なる作業短縮ではなく、学習速度を上げる装置になる。

たとえば、次のような使い方である。

| 目的 | AIの使い方 | 人間が判断すること |
|---|---|---|
| 新しい提案仮説を試す | 業種別提案書の初稿を複数案作る | どの仮説を顧客に当てるか |
| FAQ改善を試す | 問い合わせログから不足ナレッジを抽出する | どのFAQを正式化するか |
| 開発体験を改善する | IssueからPR下書き、テスト候補を作る | どこまで自動化を許すか |
| 顧客返信を改善する | 返信案を作り、レビューIssueを分類する | 顧客送信できる条件は何か |
| 対立を検証に変える | 対立論点を仮説と小実験へ分解する | 実験結果を見て何を決めるか |

AI時代の実験文化とは、勢いや楽観ではない。**判断できない論点を、検証できる論点へ変換する能力**である。

## 効率化、改善、実験、展開を分ける

AI活用施策では、「試している」の意味が曖昧になりやすい。

次の4種類を分ける。

| 区分 | 目的 | 代表例 | 主な成果物 |
|---|---|---|---|
| 効率化 | 既存業務を速く安く行う | 議事録要約、定型メール下書き | Request Contract、レビュー表 |
| 改善 | 既存成果物の品質を上げる | 提案書レビュー、FAQ改善、誤り検出 | Review Issue Log、Evaluation Plan |
| 実験 | 未確定の仮説を検証する | 新しい提案導線、AI返信補助、RAG改善 | AI実験設計シート、Learning Log |
| 展開 | 検証済みの仕組みを広げる | 部門展開、全社導入、運用標準化 | AI System Inventory、Approval Workflow、Training Register |

実験を展開と混同してはいけない。

低リスクな実験は速く行う。高リスクな展開は統制する。この切り分けがないと、次のどちらかになる。

```text
・何でも承認待ちになり、実験速度が落ちる
・実験のつもりで始めたものが、統制なしに本番運用になる
```

本章では、低リスクな小実験を増やしつつ、リスクの高い利用を第7章〜第10章の統制へ戻す。

## 実験とは何か

本書では、実験を次のように定義する。

```text
実験とは、未確定の仮説を、限られた対象、期間、データ、権限、コストで試し、
成功、失敗、継続、変更、停止、展開の判断に使う学習を得る活動である。
```

重要なのは、実験の目的が「成功すること」ではない点である。

実験の目的は、**次の判断に必要な学習を得ること**である。

したがって、よい実験は、失敗しても価値がある。

たとえば、次の結果は失敗に見えるが、学習としては有効である。

```text
・提案書初稿作成は速くなったが、レビュー時間が増えた
・返信案は自然だが、原因未確定事項を断定しやすい
・RAGは正しい文書を拾うが、古い文書を優先する傾向がある
・PR下書きは作れるが、テスト観点が浅い
・営業と開発の対立は、説得ではなく顧客要件確認の不足だった
```

これらは、実験が無駄だったことを意味しない。次に直すべき制約、Context Pack、評価ケース、HITL、やめる条件が見えたということである。

## 実験であってはならないもの

次のものは、実験と呼ばない。

| 状態 | 問題 |
|---|---|
| 目的が曖昧な試用 | 学習が残らない |
| 期間がない試用 | いつまでも続く |
| 成功条件がない試用 | 判断できない |
| 失敗条件がない試用 | 危険信号を無視する |
| やめる条件がない試用 | 施策が増殖する |
| ログがない試用 | 再現も共有もできない |
| Ownerがいない試用 | 責任境界が曖昧になる |
| 本番影響があるのに承認がない試用 | インシデントになる |
| 顧客や従業員へ影響するのにレビューがない試用 | 信頼を損なう |

「ちょっと試す」は悪くない。ただし、業務で使うなら、最低限次を決める。

```text
何を検証するのか
どこまでなら安全か
何が起きたら止めるのか
何を学習として残すのか
```

## 実験レベル X0〜X5

AI活用実験は、リスクと影響度に応じてレベルを分ける。

| レベル | 状態 | 例 | 必要な統制 |
|---|---|---|---|
| X0 | 個人メモ実験 | 自分の文章を要約、思考整理 | 個人ルール、機密情報を入れない |
| X1 | 個人業務補助 | 提案書構成案、会議準備、調査観点出し | Request Contract、自己レビュー |
| X2 | チーム内実験 | 1チームでFAQ改善、レビュー補助 | AI実験設計シート、レビュー表、Learning Log |
| X3 | 限定業務実験 | 顧客返信案、CRM下書き、RAG検索 | HITL、ログ、リスク分類、Owner |
| X4 | 本番近接実験 | 顧客影響、本番データ、業務システム連携 | 事前承認、AI Risk Register、評価、監査証跡 |
| X5 | 展開前検証 | 部門展開、全社標準化、外部提供 | Governance Level Matrix、Training Register、Runbook |

X0、X1は軽く扱う。X3以上は、第7章〜第10章の統制へ接続する。

実験文化とは、統制をなくすことではない。**リスクの低い実験を軽くし、リスクの高い実験を重くすること**である。

## 仮説の書き方

よい仮説は、次の形式に近い。

```text
もし [変更すること] を行えば、
[対象者 / 対象業務] において、
[望ましい変化] が起きる。
なぜなら [根拠となる前提] だからである。
```

例：

```text
もし顧客問い合わせを4分類し、低リスク問い合わせだけAI返信案を作成すれば、
CSチームAにおいて、返信案作成時間を30%以上短縮できる。
なぜなら、低リスク問い合わせはFAQ参照と定型説明で対応できる割合が高いからである。
```

悪い仮説は、次のようなものだ。

```text
AIを使えば問い合わせ対応が良くなる。
営業提案をAIで効率化する。
RAGを導入すればナレッジ活用が進む。
AIで開発生産性を上げる。
```

これらは方向性としては分かるが、何を試し、何を測り、何が分かれば次へ進むのかが不明である。

## 最小実験にする

AI活用施策は、最初から大きくしない。

次のように小さくする。

| 大きすぎる施策 | 最小実験への分解 |
|---|---|
| 全社の問い合わせ対応をAI化する | 1チーム、低リスク問い合わせ20件で返信案を作る |
| 営業提案書をAIで自動生成する | 既存顧客3件について提案書構成案だけ作る |
| 社内WikiをRAG化する | 障害対応FAQのうち30文書だけで検索品質を測る |
| PR作成をAIエージェントに任せる | テスト追加PRだけをAIに下書きさせる |
| AIで会議を効率化する | 1種類の定例会議で決定事項とアクション抽出だけ試す |

最小実験では、次を絞る。

```text
対象者
対象業務
対象データ
対象期間
利用ツール
出力形式
承認範囲
成功条件
失敗条件
```

「小さくする」とは、野心を捨てることではない。大きな仮説を、学習可能な単位に分けることである。

## 成功条件、失敗条件、即時停止条件、やめる条件

実験では、条件を4種類に分ける。

| 条件 | 意味 | 例 |
|---|---|---|
| 成功条件 | 次へ進むための条件 | 作業時間30%削減、Major Issue率10%未満 |
| 失敗条件 | 仮説が弱いと判断する条件 | レビュー時間が削減時間を上回る |
| 即時停止条件 | 途中で止める条件 | 顧客機密の誤出力、承認なし送信、契約事項の断定 |
| やめる条件 | 継続・展開しない条件 | 3回改善してもCritical Issueが消えない |

失敗条件と即時停止条件を混同してはいけない。

失敗条件は、学習として扱える。即時停止条件は、安全上または信頼上、止める必要がある。

やめる条件も重要である。AI施策は、始めるよりやめる方が難しい。プロンプト、RAG、ワークフロー、社内Bot、テンプレート、教育資料は放置されやすい。使われない仕組みを残すと、検索結果、ナレッジ、権限、運用コスト、誤利用リスクが増える。

したがって、実験設計時点で次を書く。

```text
どの条件なら続けるか
どの条件なら変更するか
どの条件なら止めるか
どの条件なら展開するか
```

## 20 Experiments Sheet

大きな目標は、20個の小実験へ分解する。

これは数を増やすためではない。最初に思いついた1案へ執着しないためである。

例として、「営業提案をAIで改善する」という目標を分解する。

| No | 小実験 | 学習したいこと |
|---:|---|---|
| 1 | 業種別Context Packで提案書構成案を作る | 業種文脈が品質に効くか |
| 2 | 既存提案書3件をAIにレビューさせる | どのレビュー観点が有効か |
| 3 | 顧客課題から提案仮説を10案作らせる | 仮説生成の幅が出るか |
| 4 | 営業担当の商談メモを構造化する | 入力情報の不足が見えるか |
| 5 | 競合比較表を提案書に反映する | 競合情報が説得力を上げるか |
| 6 | 提案書のリスク表現だけレビューする | 過剰約束を減らせるか |
| 7 | 価格訴求版と運用負荷訴求版を作る | 顧客セグメントに合う軸が分かるか |
| 8 | 技術者向け説明と経営者向け説明を分ける | 読者別の表現差が有効か |
| 9 | 1ページ提案サマリーをAIに作らせる | 要点抽出が商談準備に使えるか |
| 10 | AIに反対意見を出させる | 弱点を事前に見つけられるか |
| 11 | 過去失注理由をContext Packに入れる | 失注パターン回避に効くか |
| 12 | 顧客業界ニュースを使わずに提案する | 外部情報なしでも品質が出るか |
| 13 | 顧客業界ニュースを入れて提案する | 外部情報が改善に効くか |
| 14 | セキュリティ担当向けFAQを作る | 技術懸念に対応できるか |
| 15 | 導入後30日計画を作る | 実行可能性が上がるか |
| 16 | 顧客の反論想定集を作る | 商談準備に効くか |
| 17 | 顧客名を除いた匿名Contextで作る | 情報最小化でも品質が保てるか |
| 18 | 人間作成版とAI支援版を比較する | レビュー負荷を含めた効果があるか |
| 19 | 提案書を5分説明スクリプトに変換する | 営業トークへ転用できるか |
| 20 | 商談後メモから次回提案改善案を作る | 学習サイクルが回るか |

20個すべてを実施する必要はない。重要なのは、選択肢を広げたうえで、最も安全で学習量の多いものを選ぶことである。

## Experiment Portfolio

個別実験だけを見ると、局所最適になる。組織としては、実験ポートフォリオを見る必要がある。

| 区分 | 目的 | 例 | 目安 |
|---|---|---|---|
| 効率化実験 | 時間削減 | 議事録、下書き、要約 | 多めでよい |
| 品質実験 | ミス削減、レビュー強化 | AI出力レビュー、FAQ改善 | 継続的に行う |
| 学習実験 | 新しい仮説検証 | 新提案、導線、業務フロー | 少数でも重要 |
| 統制実験 | 権限、ログ、承認、評価 | HITL、Permission Test | 高リスク領域で必須 |
| 廃棄実験 | やめる判断 | 使われないBot停止 | 定期的に行う |

実験ポートフォリオでは、次を確認する。

```text
効率化だけに偏っていないか
新しい仮説検証があるか
統制を強化する実験があるか
やめる実験があるか
失敗から次の実験へつながっているか
```

AI活用で最も起きやすい偏りは、効率化実験ばかりになることである。効率化は重要だが、学習実験と廃棄実験がないと、組織の方向転換能力は上がらない。

## Pace / Spin

本書では、AI活用実験の状態を見るために、Pace と Spin という簡易指標を使う。

| 指標 | 意味 |
|---|---|
| Pace | 仮説から試行、学習、次の判断までの速度 |
| Spin | 不安、迷い、調整過多、責任回避、対立回避により空回りしている度合い |

Pace は、速ければよいという単純な話ではない。高リスク領域では慎重さが必要である。しかし、低リスク領域でPaceが極端に低いなら、AI活用以前に、組織の実験設計が重すぎる可能性がある。

Spin は、忙しさではない。**進んでいるように見えるが、実際には学習が進んでいない状態**である。

Spinが高いと、次が増える。

```text
事前調整ばかり増える
会議は多いが実験が始まらない
誰も反対していないが誰も決めない
レビュー観点が増え続ける
成功条件が曖昧なまま進む
失敗したときに責任者探しになる
対立を避けて、実験条件が丸くなりすぎる
```

Pace / Spin Reviewでは、次を記録する。

```text
対象期間：
実験数：
実施までの平均日数：
学習ログ作成率：
次の判断までの平均日数：
廃棄判断数：
Pace：0〜150%
Spin：0〜100%
Spinの原因：承認過多 / 対立回避 / 完璧主義 / 情報不足 / Owner不明 / リスク不明 / その他
次に減らす摩擦：
次に増やす安全策：
次の小実験：
```

Paceを上げるには、承認を減らすだけでは不十分である。安全な範囲を明確にし、低リスク実験を軽くし、高リスク実験を正しく重くする必要がある。

## 失敗耐性とは何か

失敗耐性とは、失敗を歓迎することではない。

失敗耐性とは、次の能力である。

```text
失敗しても致命傷にならない範囲で試す
失敗条件を事前に決める
失敗を隠さず記録する
責任追及ではなく設計改善に変換する
次の仮説を作る
不要な施策を捨てる
```

失敗耐性が低い組織では、次が起きる。

```text
成功事例だけが共有される
失敗したプロンプトが残らない
使われなかったAIワークフローが放置される
レビューで見つかった問題が個人の能力問題になる
リスクを恐れて低リスク実験まで止まる
一度始めたAI施策をやめられない
```

失敗耐性が高い組織では、次が起きる。

```text
失敗条件が明確である
小さく失敗する
失敗レビューが短時間で行われる
学習ログが残る
次の実験に変換される
やめる判断が評価される
```

## 責任ある失敗と無責任な失敗

実験文化を誤解すると、「失敗してもよい」という雑なメッセージになる。

本書では、責任ある失敗と無責任な失敗を分ける。

| 区分 | 責任ある失敗 | 無責任な失敗 |
|---|---|---|
| 範囲 | 小さく限定されている | 本番・顧客・全社へ広がる |
| 条件 | 成功・失敗・停止条件がある | 何が失敗か不明 |
| データ | 情報分類されている | 機密情報を雑に使う |
| 権限 | 最小権限、HITLあり | AIが広い権限を持つ |
| 記録 | Run Log、Learning Logあり | ログがない |
| 判断 | 続ける、変える、やめるが決まる | 惰性で続く |
| 共有 | 教訓が共有される | 失敗が隠れる |

AIエージェント協働では、無責任な失敗を許容しない。責任ある失敗を増やす。

## Failure Review Sheet

失敗レビューは、個人の責任追及ではなく、設計改善に使う。

```text
Failure Review ID：
Experiment ID：
実験名：
日付：
参加者：
当初の仮説：
成功条件：
失敗条件：
即時停止条件：
実際の結果：
何が想定と違ったか：
失敗分類：仮説 / Context / Request / Tool / Permission / Review / Governance / Human / Other
発生した影響：
安全策は機能したか：
学習できたこと：
続けること：
変えること：
やめること：
次の実験候補：
更新する成果物：Request Contract / Context Pack / Task Brief / Evaluation Plan / Tool Spec Card / AI Risk Register / その他
共有先：
```

失敗分類は重要である。

たとえば、AI返信案の品質が悪かった場合でも、原因は複数ある。

| 失敗分類 | 例 |
|---|---|
| 仮説 | 低リスク問い合わせと思ったが、実際は例外が多かった |
| Context | FAQが古かった、顧客契約条件が入っていなかった |
| Request | 原因未確定事項を断定しない制約がなかった |
| Tool | CRM検索結果が多すぎた、関連度が低かった |
| Permission | AIが下書きだけでなく更新操作を提案した |
| Review | レビュー観点にSLAや補償表現が入っていなかった |
| Governance | 実験レベルに対して承認が不足していた |
| Human | レビュー担当が不安を言えず、承認した |

失敗を「AIがダメだった」で終わらせると、何も改善しない。

## Learning Log

失敗レビューをしても、学習が残らなければ組織能力にならない。

Learning Logは、実験から得た学習を再利用可能な形で残す。

```text
Learning ID：
Experiment ID：
学習タイトル：
分かったこと：
有効だった条件：
有効でなかった条件：
再利用できる場面：
避けるべき場面：
更新したテンプレート：
追加した評価ケース：
追加した禁止事項：
次の仮説：
共有先：
```

Learning Logは、成功事例にも失敗事例にも使う。

特に重要なのは、**有効でなかった条件**である。

成功事例だけを共有すると、文脈が抜け落ちる。あるチームで成功したプロンプトが、別チームで失敗する理由は、対象業務、データ、レビュー体制、リスク許容度、顧客文脈が違うからである。

## Side Quest Charter

大きなロードマップ外の小実験を許可する場合、Side Questとして扱う。

Side Questは、勝手な本番導入ではない。低リスク・短期間・限定範囲の探索枠である。

```text
Side Quest ID：
提案者：
テーマ：
試したい理由：
対象範囲：
対象外範囲：
使ってよいデータ：
使ってはいけないデータ：
使ってよいツール：
禁止操作：
期間：
必要なレビュー：
共有する学習：
停止条件：
展開禁止条件：
```

Side Questを制度化すると、低リスクな探索が進む。一方で、制度化しないと、非公式なAI利用が増え、台帳にもログにも残らない。

したがって、Side Questには軽い入口と明確な出口が必要である。

```text
入口：短いCharterで始められる
出口：Learning Log、廃棄判断、次の実験、正式Use Case Intakeのいずれかに接続する
```

## 実験を設計する8ステップ

AI活用実験は、次の順で設計する。

```text
1. 判断または対立を特定する
2. 仮説へ変換する
3. 最小実験へ分解する
4. リスクと実験レベルを決める
5. 成功条件、失敗条件、停止条件、やめる条件を決める
6. 必要なRequest Contract、Context Pack、Task Brief、HITLを用意する
7. 実行し、Run LogとLearning Logを残す
8. 続ける、変える、やめる、展開するを判断する
```

### 1. 判断または対立を特定する

第11章のDecision Briefや第12章のConnection Debt診断から、判断できていない論点を選ぶ。

例：

```text
顧客返信案生成AIを全社展開するか判断できない
営業と開発が、新機能を提案書に含めるか対立している
RAGを導入したが、検索品質が十分か判断できない
```

### 2. 仮説へ変換する

判断や対立を、検証可能な仮説へ変換する。

```text
低リスク問い合わせに限定すれば、返信案作成時間を30%削減し、Critical Issueを0件にできる
新機能を「確定機能」ではなく「検討中の拡張」として表現すれば、顧客期待を過剰に上げずに商談機会を維持できる
RAG対象文書を30件に絞り、情報鮮度ラベルを付ければ、検索結果の要確認率を下げられる
```

### 3. 最小実験へ分解する

対象を限定する。

```text
CSチームA、問い合わせ20件、2週間
重要顧客1件、提案書1章だけ、営業と開発レビュー1回
障害FAQ30件、検索クエリ20件、RAG評価のみ
```

### 4. リスクと実験レベルを決める

X0〜X5で分類する。

顧客影響、本番データ、個人情報、外部送信、ツール実行がある場合は、X3以上として扱う。

### 5. 条件を決める

成功、失敗、停止、やめる条件を分ける。

### 6. 必要な成果物を用意する

実験の性質に応じて、必要な前工程成果物を選ぶ。

| 実験内容 | 必要な成果物 |
|---|---|
| AIに出力させる | Request Contract、Acceptance Criteria |
| 社内資料を使う | Context Pack、Data Classification Matrix |
| 複数ステップで進む | Task Brief、Checkpoint Plan |
| ツールを使う | Tool Spec Card、HITL承認フロー |
| 顧客影響がある | AI出力レビュー表、Approval Workflow |
| 高リスク | AI Risk Register、Risk Acceptance Memo |

### 7. 実行して記録する

実験は、Run Log、Review Issue Log、Learning Logへ残す。

### 8. 判断する

実験後は、次のいずれかを決める。

```text
続ける
条件を変えて再実験する
対象を広げる
対象を狭める
停止する
廃棄する
正式運用へ進める
```

実験後に判断しないなら、実験ではなく作業である。

## AIに実験設計を手伝わせる

AIは、実験設計の補助に使える。

### 依頼パターン1: 大きな目標を小実験へ分解する

```text
目的：
営業提案書作成にAIを活用するための小実験を設計したい。

前提：
顧客情報は機密。最初は既存顧客3件、社内レビューのみ。

依頼：
この目標を20個の小実験に分解してください。
各実験について、検証したい仮説、必要データ、成功条件、失敗条件、リスクを表にしてください。
顧客送信やCRM更新は含めないでください。
```

### 依頼パターン2: 失敗条件を出させる

```text
次のAI活用実験について、失敗条件、即時停止条件、やめる条件を分けて提案してください。
過度に楽観的な条件ではなく、実務で止める判断に使える条件にしてください。
```

### 依頼パターン3: 反証を作らせる

```text
次の仮説が間違っている可能性を挙げてください。
特に、Context不足、レビュー負荷、顧客影響、権限、運用負荷、現場抵抗の観点で反証してください。
```

### 依頼パターン4: 失敗レビューを整理させる

```text
以下の実験ログとレビューIssueをもとに、Failure Review Sheetを作成してください。
個人の責任追及ではなく、仮説、Context、Request、Tool、Review、Governance、Humanのどこに改善余地があるかを分類してください。
```

AIは実験設計の壁打ち相手として有効である。ただし、実験を実施してよいか、リスクを受容してよいか、顧客に影響させてよいかは、人間が判断する。

## 実務ケース1: 営業提案の小実験

Northstar Backupでは、営業提案書の作成に時間がかかっている。営業部門は、AIで業種別提案書を自動生成したいと考えている。

しかし、最初から全営業組織へ展開しない。まず、限定実験にする。

### AI実験設計シート

```text
Experiment ID：EXP-SALES-2026-07-001
実験名：業種別Context Packによる提案書初稿作成
Owner：営業企画責任者
関係者：営業担当3名、プリセールス、AI推進担当
実験レベル：X2
対象業務：既存顧客向け提案書初稿作成

仮説：
業種別Context Packと既存提案書テンプレートを使えば、提案書初稿作成時間を50%削減できる。

検証したいこと：
AIが作る初稿が、営業担当のレビューで使える水準に達するか。

最小実験：
既存顧客3件について、顧客名を匿名化したContext Packを作り、提案書の構成案と1ページサマリーだけをAIに作らせる。

対象範囲：
構成案、1ページサマリー、想定質問。

対象外範囲：
価格、契約条件、納期、個別顧客名、社外送信。

使用するAIエージェント / ツール：
社内承認済みチャットAI。外部検索なし。

必要なContext Pack：
業種、課題、既存構成、利用中サービス、制約、提案目的。

使用してよいデータ：
匿名化した商談メモ、公開済み業種情報、標準サービス説明。

使用してはいけないデータ：
顧客名、契約金額、未公開障害情報、個人名。

成功条件：
初稿作成時間が50%以上削減。
営業レビューでCritical Issue 0件、Major Issue 3件以下。

失敗条件：
レビュー時間が人手作成と同等以上。
提案論点が一般論に寄りすぎる。

即時停止条件：
顧客名、契約条件、未公開情報が出力に含まれる。

やめる条件：
3件試して、レビュー負荷が削減効果を上回る。

学習できること：
営業提案でContext Packが効く情報と効かない情報。

測定指標：
初稿作成時間、レビュー時間、Critical Issue数、Major Issue数、再利用可能な構成要素数。

実施期間：
2週間。

次の判断日：
2026-07-31。
```

### 判断

この実験は、顧客送信を含まない。顧客名も匿名化する。したがってX2として扱える。

もし社外送信やCRM更新を含めるなら、X3以上へ上げ、第7章のHITL、第9章のプライバシー、第10章の承認フローが必要になる。

## 実務ケース2: 顧客返信案生成AIの限定実験

CS部門は、顧客問い合わせへの返信案作成にAIを使いたい。

この実験は、顧客影響があるためX3以上で扱う。

### 失敗条件を先に決める

```text
成功条件：
低リスク問い合わせ20件で、返信案作成時間を30%以上削減。
Critical Issue 0件。
Major Issue率10%未満。

失敗条件：
Major Issue率が20%以上。
レビュー時間が削減時間を上回る。
FAQ参照ミスが3件以上。

即時停止条件：
顧客へ未承認送信しようとする。
原因未確定事項を断定する。
SLA補償、契約条件、価格、法的責任を記載する。
他顧客情報を含む。

やめる条件：
2回改善してもCritical Issueが発生する。
レビュー担当者が心理的に送信責任を持てない状態が続く。
```

ここで重要なのは、作成時間だけを成功指標にしないことだ。

返信案生成では、次も見る。

```text
顧客影響
レビュー負荷
承認拒否率
断定表現の発生率
FAQ参照の正確性
担当者の説明可能性
```

## 実務ケース3: RAG改善実験

社内FAQをRAGで検索させたが、古い情報を拾うことがある。

これを「RAGは使えない」と結論づける前に、仮説を分ける。

| 仮説 | 小実験 |
|---|---|
| 古い文書が混ざっている | 30文書だけに絞り、情報鮮度ラベルを付ける |
| クエリが曖昧 | 問い合わせ分類後に検索クエリを生成する |
| 文書構造が悪い | FAQを質問、回答、対象バージョン、更新日で構造化する |
| 評価ケースがない | 既知質問20件で正解文書が上位に出るか測る |
| 権限境界が不十分 | 部門別・顧客別に検索スコープを分ける |

RAG改善は、AIモデルの問題に見えるが、多くの場合、文書、メタデータ、検索設計、権限、評価ケースの問題である。

実験設計では、変更点を一度に増やしすぎない。

```text
悪い例：
文書を増やし、埋め込みモデルを変え、プロンプトも変え、UIも変える。

良い例：
文書30件に限定し、情報鮮度ラベルだけを追加して、既知質問20件で比較する。
```

## 実務ケース4: 開発PR作成エージェント

開発チームは、AIにIssueからPR下書きを作らせたい。

最初から任せる範囲を広げると危険である。次のように段階化する。

| 段階 | 実験 | 権限 |
|---|---|---|
| 1 | Issueを読み、変更方針案を作る | read-only |
| 2 | テスト観点を列挙する | read-only |
| 3 | ローカル差分案を作る | sandbox |
| 4 | draft PRを作る | approval-required |
| 5 | CI結果を読み、修正案を出す | read-only + proposal |
| 6 | 人間レビュー後に限定merge | 原則として人間 |

PR作成エージェントの実験では、次を成功条件にする。

```text
対象：テスト追加、ドキュメント修正、軽微なリファクタリングに限定
成功条件：人間レビューでCritical Issue 0、Major Issue 2件以下
失敗条件：仕様変更を勝手に含める、テストが意味を持たない、レビュー負荷が増える
即時停止条件：本番設定変更、認証情報出力、権限外ファイル変更、未承認push
```

開発系実験では、実験文化とセキュリティが密接に関係する。速く試すためには、サンドボックス、ブランチ制限、CI、レビュー、権限設計が必要である。

## 実務ケース5: 対立を実験へ変換する

第12章の営業と開発の対立を考える。

営業は新オプションを重要顧客提案に含めたい。開発は実装影響が不明で、確定機能として書くべきではないと考えている。

会議で「どちらが正しいか」を決めようとすると、対立が強くなる。

この論点は、実験へ変換できる。

### 対立から仮説へ

```text
対立：
営業は提案に含めたい。開発は含めるべきではない。

仮説：
新オプションを確定機能ではなく「検討中の拡張」として表現し、顧客に最小要件を確認すれば、
実装不能な約束を避けながら、顧客ニーズの有無を検証できる。
```

### 最小実験

```text
対象：重要顧客1社
出力：提案書の1節だけ
表現：確定機能ではなく、検討中の拡張
人間承認：営業責任者、開発責任者、事業責任者
成功条件：顧客が最小要件を具体化する
失敗条件：顧客が確定機能として受け取る、開発影響が過大、提案書レビューでCritical Issue
即時停止条件：納期、価格、実装確約を含む表現が残る
やめる条件：顧客要件が曖昧なまま、次回提案に進もうとする
```

このようにすると、対立は勝敗ではなく、学習の設計になる。

## 実験KPI

AI活用の効果測定では、時間削減だけでは足りない。

実験文化を見るなら、次を測る。

| KPI | 意味 |
|---|---|
| 実験数 | 期間内に実施したAI活用実験数 |
| 実験着手リードタイム | 仮説から最初の試行までの時間 |
| 学習ログ作成率 | 実験後にLearning Logが残った割合 |
| 次仮説化率 | 実験結果が次の仮説へ接続された割合 |
| 廃棄判断数 | やめたプロンプト、Bot、RAG、ワークフローの数 |
| 展開判断数 | 限定実験から正式運用へ進んだ数 |
| Review Issue率 | AI出力の重大指摘率 |
| 承認拒否率 | HITLで承認されなかった割合 |
| インシデント数 | AI起因の事故またはNear Miss |
| Pace | 実験速度 |
| Spin | 空回り度 |

重要なのは、実験数だけを評価しないことだ。

実験数だけを評価すると、軽い試行が増える。学習ログ、廃棄判断、レビュー品質、リスク指標を一緒に見る。

## 典型的な失敗

### 失敗1: 効率化を実験文化と誤解する

議事録要約、メール下書き、資料要約が増えても、それだけでは実験文化ではない。既存業務を速くするだけでなく、未知の仮説を試しているかを見る。

### 失敗2: 企画を重くしすぎる

低リスクな小実験にも、大きな企画書、稟議、ROI試算、全社承認を求めると、誰も試さなくなる。X0〜X2は軽く始められるようにする。

### 失敗3: 成功事例だけ共有する

成功事例だけが共有されると、現場は失敗を隠す。失敗したプロンプト、使われなかったRAG、レビュー負荷が高すぎた施策も重要な学習である。

### 失敗4: やめる条件がない

AI施策は、試すことよりやめることが難しい。使われないBot、古いプロンプト、精度の低いRAG、メンテされないテンプレートはリスクになる。

### 失敗5: 実験数だけを追う

実験数は重要だが、学習に変換されなければ意味がない。実験数、Learning Log、廃棄判断、次の仮説をセットで見る。

### 失敗6: 失敗を個人の能力問題にする

AI出力が悪いとき、担当者のプロンプトが悪いと責めるだけでは改善しない。Request、Context、Tool、Review、Governance、Humanのどこに原因があるかを見る。

### 失敗7: 低リスク実験と高リスク実験を同じ扱いにする

顧客影響のない個人実験と、顧客送信、契約、CRM更新、本番データ利用を同じ承認で扱ってはいけない。

### 失敗8: 実験が本番化しているのに台帳へ載らない

限定実験が便利だからといって、いつの間にか常用されることがある。X3以上、または継続利用になったものはAI System Inventoryへ登録する。

## 設計原則

### 原則1: 小さく試すが、目的は小さくしない

小実験は、野心の小ささではない。大きな目的へ向かうために、検証単位を小さくする。

### 原則2: 失敗条件を先に書く

失敗してから失敗条件を決めると、都合のよい解釈になる。始める前に、何が起きたら仮説を弱めるのかを書く。

### 原則3: やめる条件を設計する

AI活用では、始めることよりやめることが難しい。やめる条件がない施策は、将来の運用負債になる。

### 原則4: 低リスク実験は軽く、高リスク実験は重くする

実験文化は統制の否定ではない。統制の粒度をリスクに合わせることである。

### 原則5: 失敗を成果物に変換する

失敗は、Failure Review Sheet、Learning Log、Evaluation Plan、Improvement Backlog、Kill Decision Sheetへ落とす。

### 原則6: 成功事例だけでなく、廃棄判断を共有する

使えるものを増やすだけでなく、使えないものを捨てる能力が必要である。

### 原則7: AIを実験設計の補助に使い、人間がリスクと判断を引き受ける

AIは小実験案、反証、失敗条件、レビュー観点を作れる。実施可否、リスク受容、展開判断は人間が担う。

## 演習

### 演習13-1: 20個の小実験に分解する

自分の業務改善テーマを1つ選び、20個の小実験に分解する。

条件：

```text
・顧客影響のない実験を5件以上含める
・レビュー品質を高める実験を3件以上含める
・やめる判断をするための実験を2件以上含める
・1日以内に実施できる実験を3件以上含める
```

### 演習13-2: AI実験設計シートを作る

20個の小実験から1つ選び、AI実験設計シートを作る。

必ず含める項目：

```text
仮説
最小実験
実験レベル
対象範囲
対象外範囲
使用データ
成功条件
失敗条件
即時停止条件
やめる条件
測定指標
判断日
```

### 演習13-3: Failure Review Sheetを作る

次の失敗ケースについて、Failure Review Sheetを作る。

```text
CSチームで、低リスク問い合わせ20件を対象にAI返信案生成を試した。
返信案作成時間は40%削減できた。
しかし、20件中4件でMajor Issueが出た。
そのうち2件は、FAQが古いことに起因していた。
1件は、AIが原因未確定事項を断定していた。
1件は、担当者がレビュー観点を見落としていた。
顧客送信はすべて人間承認だったため、顧客影響はなかった。
```

提出物：

```text
Failure Review Sheet
Learning Log
更新すべき成果物一覧
次の実験候補
やめる条件の見直し
```

### 判定基準

- 失敗を「AIが悪い」「担当者が悪い」で終わらせていない
- 仮説、Context、Request、Review、Governance、Humanに分類している
- FAQ鮮度、断定表現、レビュー観点の改善先が明確である
- 次の実験が、前回より小さく安全になっている
- 顧客影響がなかった理由がHITLとして説明されている
- 続ける、変える、やめるの判断が書かれている

## まとめ

AIの価値は、既存業務の効率化だけではない。

AIを使えば、調査、下書き、比較、プロトタイプ、レビュー、反証、評価ケース作成が速くなる。したがって、組織はより多くの仮説を試せる。

ただし、試行回数が増えるほど、失敗、放置、権限過大、ログ不足、レビュー負荷、やめられない施策も増えやすい。

したがって、AI時代の実験文化は、楽観ではなく設計である。

```text
仮説を書く
小さく試す
リスクで実験レベルを分ける
成功条件、失敗条件、停止条件、やめる条件を書く
Run LogとLearning Logを残す
失敗を設計改善へ変換する
不要な施策を捨てる
次の判断へ接続する
```

第14章では、実験を止める個人側の要因を扱う。多くの場合、人が動けない理由は、能力不足ではない。完璧主義、自己批判、失敗への恐れ、評価不安、先送りが、小さな行動を止める。第14章では、それをAction Blocker診断として扱う。

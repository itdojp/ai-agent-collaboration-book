# 第14章 セルフトークと高レバレッジ環境

## この章で扱う能力

この章では、AI時代の高レバレッジ環境で、自己批判、完璧主義、先送り、評価不安、AIへの不安が、判断力と行動量を下げる問題を扱う。

読者は、次のことができるようになる。

- セルフトークを、性格論ではなく業務上の行動ブロッカーとして扱う
- 健全なレビュー、品質基準、リスク管理と、過剰な自己批判を区別する
- 完璧主義、評価不安、AI不安、過剰責任、回避を診断する
- 止まっている行動を、24時間以内に実行できる最小行動へ変換する
- AIを、自己批判の増幅ではなく、行動設計、レビュー準備、実験分解に使う
- 管理職として、部下のAI活用停滞を人格問題ではなく、業務設計問題として扱う
- 第13章の実験文化を、実際の個人行動へ接続する

第11章では、AI時代の判断力を扱った。AIが情報、選択肢、推奨、反証を作れるほど、人間は判断基準、責任、リスク、やめる条件を引き受ける必要がある。

第12章では、困難な会話と対立処理を扱った。AIは会話準備を助けられるが、本人に直接話すこと、合意すること、フォローアップすることは代替しない。

第13章では、失敗耐性と実験文化を扱った。AIの価値は既存業務の効率化だけでなく、試行回数を増やし、未知の仮説を検証することにある。

第14章では、そこからさらに一段下にある問題を扱う。

```text
判断すべきことは分かっている。
対話すべき相手も分かっている。
小さく実験すればよいことも分かっている。
それでも行動が止まる。
```

このとき、問題はプロンプトではない。ツールでもない。ガバナンス不足でもない。人間側の内的な摩擦が、業務成果を止めている。

本章では、その摩擦を「セルフトーク」として扱い、業務上の最小行動へ変換する。


第14章は、心理療法やメンタルヘルス支援を代替するものではない。

扱うのは、業務上の行動停止である。

```text
・初稿を出せない
・AI活用実験を共有できない
・レビュー依頼を送れない
・AIに任せられる作業を自分で抱える
・AIへの不安で学習を避ける
・失敗レビューを人格評価として受け取る
・完璧なプロンプトを探し続け、実験を始めない
```

これらを性格の問題として扱うと、改善しにくい。業務上の行動ブロッカーとして扱えば、設計できる。

## なぜAI時代にセルフトークが業務能力になるのか

AIエージェントは、個人の処理能力を上げる。

調査、要約、ドラフト、レビュー、比較、仮説出し、テストケース作成、議事録、FAQ整理、提案書構成、コード補助は速くなる。

すると、次のような変化が起きる。

| 変化 | 人間側の新しい負荷 |
|---|---|
| 下書きが大量に作れる | どれを採用するか判断する負荷が増える |
| 小実験を速く作れる | 失敗を表に出す回数が増える |
| AIレビューが簡単にできる | 指摘される量が増え、自己批判が刺激される |
| AIに委任できる作業が増える | 自分の価値をどこに置くか再定義が必要になる |
| 少人数で大きな成果を出せる | 1人の停滞、回避、過剰レビューの影響が大きくなる |
| ログと評価が残る | 自分の判断や成果物が見える化される |

高レバレッジ環境では、同じ内的摩擦でも影響が大きくなる。

たとえば、以前なら「提案書ドラフトを1日遅らせた」だけだったものが、AI時代には次のような影響になる。

```text
・AI活用実験のレビューが遅れる
・学習ログが残らない
・他チームへの展開判断が遅れる
・同じ失敗を別チームが繰り返す
・管理職がAI活用の効果を判断できない
・結果として、組織全体の実験速度が落ちる
```

AI時代のボトルネックは、作業速度だけではない。

```text
判断を避けること
対話を避けること
失敗を避けること
レビューを避けること
学習を避けること
```

これらが、組織の速度を落とす。

そのため、セルフトークは個人の気分の問題ではなく、業務能力の一部として扱う必要がある。

## セルフトークとは何か

本書では、セルフトークを次のように定義する。

```text
セルフトークとは、行動の前後に頭の中で繰り返される、自分、他者、成果物、失敗、評価、リスクに関する内的な説明である。
```

セルフトークは、必ずしも悪いものではない。

次のようなセルフトークは有効である。

```text
・まだ根拠が足りない。一次情報を確認しよう。
・顧客影響がある。HITLを入れよう。
・この出力は自然だが、事実確認が必要だ。
・完璧ではないが、レビュー可能な下書きとして出そう。
・失敗条件に触れたので、停止して学習ログを残そう。
```

問題になるのは、セルフトークが行動を止める場合である。

```text
・まだ完成度が低いから出せない
・突っ込まれたら終わりだ
・AIを使うと自分の価値がなくなる
・自分がやった方が早い
・どうせ使われない
・レビューで否定される
・失敗したら評価が下がる
・完璧なプロンプトができるまで始めない方がよい
```

これらは、事実ではなく仮説である。

仮説であれば、検証できる。

## セルフトークを禁止しない

セルフトークを消そうとしてはいけない。

重要なのは、セルフトークを次のように扱うことである。

```text
1. 気づく
2. 書き出す
3. 守ろうとしているものを特定する
4. 実際の業務リスクと分ける
5. 安全に試せる最小行動へ変換する
```

多くの否定的なセルフトークは、何かを守ろうとしている。

| セルフトーク | 守ろうとしているもの |
|---|---|
| 失敗したら終わり | 評価、信頼、立場 |
| まだ完成度が低い | 品質、専門性、顧客影響 |
| AIに負ける | 自分の価値、職能、将来 |
| 自分がやった方が早い | 納期、品質、コントロール |
| レビューされるのが怖い | 自尊心、関係性、評判 |
| どうせ使われない | 労力、期待、失望回避 |
| 相談すると迷惑だ | 他者の時間、関係性 |

守ろうとしているもの自体は、悪くない。

問題は、守るための手段が「行動停止」になっていることである。

本章の目的は、守るべきものを守りながら、行動を止めない設計に変えることである。

## セルフトークの業務影響

セルフトークは、行動として現れる。

| セルフトーク | 業務上の現れ | 発生する損失 |
|---|---|---|
| まだ完成度が低い | 初稿を共有しない | レビュー開始が遅れる |
| 突っ込まれたら終わり | レビュー依頼を避ける | 品質問題が後工程で出る |
| AIに負ける | AI利用の学習を避ける | 職能更新が遅れる |
| 自分がやった方が早い | 委任しない | 属人化、スケールしない |
| 失敗したら評価が下がる | 小実験を避ける | 学習速度が落ちる |
| 正解が分からない | 判断を保留する | 意思決定が滞留する |
| 相手に嫌われる | 困難な会話を避ける | Connection Debtが増える |
| ルールが怖い | 低リスク実験まで止める | 組織のAI活用が進まない |

ここで重要なのは、「不安があるからダメ」ではない。

不安があるのは普通である。

問題は、不安が次のいずれかに変換されることである。

```text
回避
過剰準備
過剰レビュー
過剰統制
過剰な自己否定
他者への責任転嫁
AIへの過信
AIへの過剰防衛
```

## 健全な品質基準と完璧主義の違い

AI活用では、品質基準を下げてはいけない。

ただし、品質基準と完璧主義は違う。

| 区分 | 健全な品質基準 | 完璧主義 |
|---|---|---|
| 対象 | 成果物、プロセス、受け入れ条件 | 自分の能力、評価、失敗回避 |
| 基準 | 明文化されている | 頭の中で膨らみ続ける |
| 期限 | ある | ない、または守れない |
| 共有 | 途中で出す | 完成するまで出さない |
| レビュー | 改善のために受ける | 否定されないために避ける |
| 出力 | 次アクション | 先送り |
| リスク対応 | 具体的なHITL、テスト、制約 | 漠然とした不安 |

健全な品質基準は、次のように書ける。

```text
この顧客返信案は、原因未確定事項を断定していないこと。
SLA、補償、契約変更に触れていないこと。
顧客固有情報が混入していないこと。
人間レビュー後にしか送信しないこと。
```

完璧主義は、次のように曖昧である。

```text
もっと良くしてから出す。
まだ突っ込まれそう。
これでは不十分な気がする。
ちゃんとしたものにしてから見せる。
```

完璧主義を品質管理に戻すには、曖昧な不安を受け入れ条件に変換する。

```text
不安：突っ込まれそう
変換：どの観点でレビューしてほしいかを明記する

不安：完成度が低い
変換：これは初稿であり、構成、事実、リスク表現の3点だけレビューしてもらう

不安：顧客影響が怖い
変換：顧客送信は禁止。社内レビュー用ドラフトとして扱う
```

## 健全なレビューと自己批判の違い

AI時代には、レビューが増える。

AI出力レビュー、コードレビュー、提案書レビュー、Context Packレビュー、Evaluation Planレビュー、Prompt Injection Testレビュー、AI Risk Registerレビューが発生する。

レビュー文化が弱い組織では、AI活用は危険である。

一方で、レビューを人格評価として受け取ると、行動が止まる。

| 区分 | 健全なレビュー | 自己批判 |
|---|---|---|
| 対象 | 成果物、判断、プロセス、リスク | 人格、能力全体、価値 |
| 目的 | 品質向上、事故防止、学習 | 萎縮、回避、自己防衛 |
| 言葉 | 「この根拠が不足している」 | 「自分は向いていない」 |
| 出力 | 修正点、次アクション、評価ケース | 行動停止、過剰防衛 |
| 時間軸 | 次に良くする | 過去を責め続ける |
| 記録 | Review Issue Log、Learning Log | 頭の中だけで反芻 |

レビューを受ける側は、次の分離を行う。

```text
成果物への指摘
判断プロセスへの指摘
前提情報への指摘
リスク制御への指摘
自分の人格への評価
```

通常、業務レビューで必要なのは前4つである。

管理職やレビュアーは、レビューを人格評価にしない。

悪いレビューは次のようになる。

```text
なぜこんな雑なものを出したのか。
AIを使ってこれなら意味がない。
この程度なら自分でやった方がいい。
```

よいレビューは次のようになる。

```text
これは初稿として扱う。
Blockerは2件ある。
1つ目は、原因未確定事項を断定していること。
2つ目は、SLA補償に触れていること。
次はContext Packに禁止表現を追加し、Review Issue Logへ評価ケースとして登録する。
```

厳しいレビューは必要である。

人格攻撃は不要である。

## 行動ブロッカーの分類

行動停止を扱うには、ブロッカーを分類する。

| 分類 | 典型的なセルフトーク | 業務上の現れ | 有効な対処 |
|---|---|---|---|
| 完璧主義 | まだ出せない | 初稿共有が遅い | Draft扱いを明示し、レビュー観点を限定する |
| 評価不安 | 突っ込まれたら終わり | レビュー依頼を避ける | レビュー基準とSeverityを先に決める |
| AI不安 | AIに価値を奪われる | 学習を避ける | 自分が担う判断、対話、責任を定義する |
| 過剰責任 | 自分で全部見ないと危ない | 委任しない | HITL、権限、停止条件で分担する |
| 過剰防衛 | どうせ反対される | 提案しない | 小実験として出し、成功条件と失敗条件を置く |
| 曖昧性不耐性 | 正解がないと進めない | 判断保留 | 仮説として扱い、AI実験設計シートへ変換する |
| 対立回避 | 相手に言いづらい | 裏でだけ相談する | Difficult Conversation Briefへ接続する |
| 学習回避 | どうせ自分には難しい | AIツールを触らない | 15分のX0/X1実験から始める |
| 過剰統制 | ルールがないから全部禁止 | 低リスク利用も止める | X0〜X5、G0〜G5でリスクを分ける |
| 過信 | AIが言うなら大丈夫 | レビューせず採用 | AI出力レビュー表へ戻す |

この分類は診断名ではない。業務上の切り分けである。

重要なのは、分類した後に最小行動を決めることである。

## Action Blocker診断

Action Blocker診断は、行動停止を小さな実行へ変換するための成果物である。

```text
止まっている行動：
本来やるべき小さな次アクション：
頭の中のセルフトーク：
そのセルフトークが守ろうとしているもの：
実際の業務リスク：
想像上のリスク：
リスク制御策：
安全に試せる最小行動：
成果物の扱い：Draft / Review Only / Experiment / Production Candidate
レビューしてほしい観点：
レビューしてほしくない観点：
誰にレビューしてもらうか：
いつまでに実行するか：
実行後に確認すること：
```

この診断で重要なのは、セルフトークを否定しないことである。

たとえば、「突っ込まれたら終わり」というセルフトークがある場合、次のように扱う。

```text
頭の中のセルフトーク：
突っ込まれたら終わり。完成度が低いと言われそう。

守ろうとしているもの：
自分の評価、チームからの信頼、無駄なレビュー依頼を避けること。

実際の業務リスク：
顧客影響はない。社内ドラフトである。
事実誤認がある可能性はある。

リスク制御策：
「事実確認前の構成案」と明記する。
レビュー観点を「構成、抜け漏れ、リスク表現」の3点に限定する。

安全に試せる最小行動：
30分でドラフトを作り、1名にReview Onlyとして共有する。
```

このように、セルフトークを行動設計へ変換する。

## Self-talk Transcript

セルフトークは、頭の中だけで扱うと強くなる。

一度書き出す。

```text
対象行動：
行動前に出ている言葉：
行動中に出ている言葉：
行動後に出ている言葉：
繰り返し出るフレーズ：
その言葉が前提にしていること：
その前提は事実か仮説か：
反証となる事実：
業務上、守るべきリスク：
業務上、過剰に恐れている可能性があるもの：
行動を止めない言い換え：
```

例：

```text
対象行動：
AI返信案生成の実験結果をチームに共有する。

行動前に出ている言葉：
まだサンプルが少ない。突っ込まれそう。ちゃんとした分析になっていない。

その言葉が前提にしていること：
完成度が低いものを共有すると評価が下がる。

その前提は事実か仮説か：
仮説。過去にも初稿レビューは行っている。

反証となる事実：
前回のFAQ改善では、粗い初稿を出したことで早く改善できた。

業務上、守るべきリスク：
顧客影響のある結果を断定しない。Critical Issueを隠さない。

行動を止めない言い換え：
これは完成版ではなく、学習ログの共有である。
Critical Issueと未確定事項を明示して、レビューを受ける。
```

言い換えは、ポジティブな気休めである必要はない。

必要なのは、行動を再開できる業務文である。

## Review vs Self-criticismチェック

AI時代のレビューでは、次のチェックを使う。

```text
受けた指摘：
指摘の対象：成果物 / プロセス / 判断 / リスク / 自分の人格
Severity：Blocker / Critical / Major / Minor / Nit
修正すべき成果物：
次の具体的アクション：
学習として残すこと：
自己批判として混ざっている言葉：
切り離すべき解釈：
レビュアーへ確認すること：
```

例：

```text
受けた指摘：
この返信案は、原因未確定なのに障害原因を断定している。

指摘の対象：
成果物、リスク表現。

Severity：
Critical。

修正すべき成果物：
Context Packの禁止表現、AI出力レビュー表、Evaluation Planの禁止ケース。

次の具体的アクション：
原因未確定時の表現テンプレートを追加する。

自己批判として混ざっている言葉：
自分はレビュー能力が低い。

切り離すべき解釈：
今回のCritical指摘は、レビュー能力全体の評価ではない。
評価ケースが不足していたことを示す。
```

レビューを学習に変えるには、指摘を人格から成果物へ戻す。

## Minimum Action Contract

行動停止が起きているとき、次の目標は「完成」ではない。

次の目標は、**最小行動**である。

Minimum Action Contractは、24時間以内または次の業務日までに実行する小さな行動を定義する。

```text
止まっているテーマ：
24時間以内に行う最小行動：
完了条件：
所要時間上限：
使うAI依頼：
使う既存成果物：
共有先：
共有時の但し書き：Draft / Review Only / Experiment / Learning Share
レビューしてほしい観点：
レビュー不要な観点：
失敗してもよい範囲：
失敗してはいけない範囲：
次の判断：続ける / 変える / やめる / 相談する
```

例：

```text
止まっているテーマ：
AIで作成した営業提案テンプレートを共有できない。

24時間以内に行う最小行動：
テンプレートの1ページ目だけを、営業2名へReview Onlyとして共有する。

完了条件：
Slackに、レビュー観点3つを添えて投稿する。

所要時間上限：
45分。

共有時の但し書き：
これは完成版ではない。構成と顧客への表現だけレビューしてほしい。

失敗してもよい範囲：
表現が粗い、項目が足りない、使いにくいと言われる。

失敗してはいけない範囲：
顧客固有情報、未確定機能、価格、契約条件を含める。
```

最小行動は、心理的に楽である必要はない。

実行可能で、安全で、学習が残ればよい。

## AIをセルフトーク処理にどう使うか

AIは、セルフトークを消す道具ではない。

AIは、次の用途に使える。

| 用途 | AIに依頼できること | 人間がやること |
|---|---|---|
| 書き出し | セルフトークを分類する | 実際に何を恐れているか認める |
| 行動分解 | 最小行動を複数案にする | 1つ選び、実行する |
| リスク分離 | 実リスクと想像上のリスクを分ける | 許容不可リスクを判断する |
| レビュー準備 | レビュー依頼文を作る | レビューを依頼する |
| 対話準備 | 相手視点や質問を整理する | 本人と話す |
| 実験化 | 小実験案を作る | 実験レベルと停止条件を決める |
| 反証 | 思い込みへの反証候補を出す | 事実かどうか確認する |

ただし、AIにやらせてはいけないこともある。

```text
・自分の責任判断をAIに委ねる
・他者の感情や意図を断定させる
・医療、心理、労務、人事判断を代替させる
・相手を説得、論破、操作する文面を作らせる
・不安をなくすために、都合のよい根拠だけを集める
```

AIは、行動を補助する。

行動の代替にはしない。

## AIへの依頼パターン

### パターン1: 行動停止を分解する

```text
以下の状況で行動が止まっています。
人格評価ではなく、業務上の行動ブロッカーとして整理してください。

状況：
止まっている行動：
頭の中のセルフトーク：
守りたいもの：
実際の業務リスク：

1. 想定される行動ブロッカー分類

4. 各案の安全策
5. レビュー依頼文の下書き

制約：
心理診断や医療的助言は不要。
相手の意図を断定しない。
```

```text
完成度：Draft / Review Only / Production Candidate

レビューしてほしくない観点：

背景：

Slack投稿文
レビュー観点の箇条書き
レビュアーが判断しやすい補足情報
```

### パターン3: 完璧主義を受け入れ条件に変換する


### パターン4: AI不安を職能再定義へ変換する

```text
AIによって自分の仕事の一部が置き換わる不安があります。
以下の業務を、AIが担う作業、人間が担う判断、対話、責任、改善に分けてください。

業務：
現在自分が担っていること：
AIに任せられそうなこと：
不安に感じていること：

出力：
1. AIに委任できる作業
2. 人間が担うべき判断
3. 人間が担うべき対話

5. 自分が伸ばすべき能力
6. 最初の学習実験
```

## 実務ケース1: AI活用実験の共有が遅れる

あるリーダーは、AIで作った営業提案書テンプレートを共有できずにいる。

状況は次のとおりである。

```text
・AIで業種別提案書テンプレートを3案作った
・営業チームに共有してレビューを受ける予定だった
・1週間経っても共有していない
・理由は「まだ完成度が低い」「突っ込まれそう」「使われなかったら意味がない」
```

この状態を「やる気がない」と扱うと、改善しにくい。

Action Blocker診断で扱う。

```text
止まっている行動：
営業提案書テンプレート3案を営業チームに共有すること。

本来やるべき小さな次アクション：
1案だけを営業2名へReview Onlyとして共有する。

頭の中のセルフトーク：
まだ完成度が低い。突っ込まれそう。使われなかったら意味がない。

そのセルフトークが守ろうとしているもの：
自分の評価、営業チームの時間、AI活用推進への信頼。

実際の業務リスク：
未確定機能や価格が入ると危険。
顧客固有情報が混ざると危険。
社内レビューだけなら、表現が粗いこと自体は許容できる。

リスク制御策：
顧客送信禁止。Review Onlyと明記。
レビュー観点を「構成」「顧客に刺さる表現」「禁止表現」の3点に限定。

安全に試せる最小行動：
1ページ目だけを45分で整え、営業2名に共有する。

いつまでに実行するか：
本日17時。
```

このケースで必要なのは、自己肯定ではない。

必要なのは、成果物の扱い、レビュー観点、安全策、期限を明確にすることである。

## 実務ケース2: AIへの不安で学習が止まる

あるベテラン担当者は、問い合わせ対応AIの導入に消極的である。

表向きの理由は「AIは信用できない」である。

実際には、次の不安がある。

```text
・自分の経験価値が下がるのではないか
・若手の方がAIを使いこなし、自分が遅れるのではないか
・AIが出した回答をレビューできないと評価されるのではないか
```

この不安に対して、「AIを使え」と命令しても逆効果である。

業務分解で扱う。

| 領域 | AIが担えること | ベテランが担うこと |
|---|---|---|
| 問い合わせ要約 | 長文の要約、論点抽出 | 顧客背景の解釈 |
| FAQ検索 | 関連文書候補の提示 | どの文書が現状に合うか判断 |
| 返信案作成 | 定型部分の下書き | 表現、顧客関係、リスク判断 |
| レビュー | 禁止表現候補の検出 | 最終責任、例外判断 |
| 改善 | よくある不足観点の抽出 | FAQ更新の優先順位判断 |

この人の価値は、返信文をゼロから書くことだけではない。

```text
顧客の文脈を読むこと
例外を判断すること
若手が見落とすリスクに気づくこと
AI出力の危険な断定を止めること
ナレッジを改善すること
```

Minimum Action Contractは次のようにする。

```text
24時間以内に行う最小行動：
過去問い合わせ3件について、AI要約と自分の要約を比較する。

完了条件：
AIが見落とした顧客文脈を3つ記録する。

共有時の但し書き：
これはAI導入賛否の判断ではなく、レビュー観点を増やすための実験である。

次の判断：
AIに任せられる定型部分と、人間がレビューすべき例外条件を分ける。
```

AI不安を否定するのではなく、職能再定義と小実験へ変換する。

## 実務ケース3: 管理職が部下のAI活用をレビューできない

ある管理職は、部下のAI活用成果物に対して、レビューを避けている。

理由は次のとおりである。

```text
・自分より部下の方がAIに詳しい
・間違ったことを言うと立場がない
・細かく指摘するとAI活用の勢いを止めそう
・一方で、そのまま使わせるのも怖い
```

この管理職のセルフトークは、次のようになっている。

```text
AIに詳しくない自分がレビューしてよいのか。
レビューして間違えたら恥ずかしい。
部下の方が詳しいなら任せた方がよい。
```

ここで管理職が担うべきことは、モデルの細部をすべて理解することではない。

管理職が担うべきことは、次である。

```text
目的が明確か
顧客影響があるか
人間承認があるか
禁止情報を扱っていないか
レビュー済みか
ログが残るか
やめる条件があるか
誰が責任を持つか
```

つまり、第2章、第6章、第7章、第10章、第11章の成果物を使えばよい。

このケースのMinimum Action Contractは次のようにする。

```text
24時間以内に行う最小行動：
部下のAI活用案に対して、技術詳細ではなく、以下5点だけ確認する。

1. 目的
2. 顧客影響
3. 使うデータ
4. 人間承認
5. やめる条件

完了条件：
AI Use Case Intakeの未記入項目を3つ指摘する。

レビューしてほしくない観点：
モデル選定、細かいプロンプト技巧。

レビューしてほしい観点：
業務リスク、責任境界、展開判断。
```

管理職は、AI専門家でなくてもAI活用を管理できる。

必要なのは、全部知っているふりではない。見るべき論点を限定することである。

## 実務ケース4: 失敗レビューが人格評価になる

あるチームでは、AI返信案生成実験でCritical Issueが出た。

AIが、原因未確定の障害について断定表現を使った。

レビュー会で、次のような発言が出た。

```text
やっぱりAIは危ない。
こんなものを出す時点でレビューが甘い。
担当者はもっと慎重にやるべきだった。
```

この会話は、学習を止める。

第13章のFailure Review Sheetへ戻す。

```text
失敗分類：
Context、Review、Governance。

想定と違ったこと：
Context Packに「原因未確定時は禁止表現」が入っていなかった。
Review Issue Logに、原因断定の評価ケースが不足していた。
実験前の即時停止条件はあったが、レビュー担当者のチェック観点が曖昧だった。

学習できたこと：
原因未確定、SLA、補償、契約、個人情報はCritical評価ケースとして追加する。

続けること：
低リスク問い合わせでの社内返信案生成実験。

変えること：
Context Pack、Review Issue Log、Evaluation Plan。

やめること：
原因未確定を含む問い合わせのAI返信案生成。
```

ここで、担当者の人格評価は不要である。

責任を曖昧にしてよいという意味ではない。

責任は成果物とプロセスに落とす。

```text
何が足りなかったか
どの成果物を更新するか
次に同じ失敗を防ぐ評価ケースは何か
どの条件では止めるか
```

これが、失敗耐性を業務能力に変換する方法である。

## 管理職の役割

第14章の内容は、個人だけの問題ではない。

管理職は、チーム内の行動ブロッカーを観察し、業務設計で解消する必要がある。

### 観察すべきシグナル

| シグナル | 可能性のあるブロッカー |
|---|---|
| 初稿が出てこない | 完璧主義、評価不安 |
| AI活用実験が共有されない | 失敗不安、レビュー不安 |
| レビュー依頼が遅い | 自己批判、過剰責任 |
| すべて自分でやる | 委任不安、コントロール欲求 |
| AI利用を極端に拒む | AI不安、職能不安 |
| AI出力をそのまま信じる | 過信、レビュー不足 |
| ルールを理由に何も試さない | 過剰統制、リスク分類不足 |
| 会議で合意したのに進まない | Connection Debt、判断回避 |

管理職は、これを「性格」と見なさない。

次のように問いを変える。

```text
この人は何に怯えているか
ではなく、
どの行動が止まっているか。

やる気があるか
ではなく、
最小行動は定義されているか。

自信があるか
ではなく、
レビュー可能なDraftとして出せる条件は何か。
```

### 管理職が使う問い

```text
いま止まっている行動は何か。
完成版ではなく、レビュー用Draftとして出せる最小範囲はどこか。
レビューしてほしい観点は何か。
レビューしてほしくない観点は何か。
失敗してもよい範囲はどこか。
失敗してはいけない範囲はどこか。
どの成果物へ接続するか。
24時間以内にできる最小行動は何か。
```

これらは、慰めではない。

行動再開のための業務設計である。

## チームで決めるレビュー作法

セルフトークを個人に任せるだけでは不十分である。

チームとして、レビュー作法を定義する。

```text
1. Draft、Review Only、Production Candidateを分ける
2. レビュー対象は成果物、プロセス、リスクであり、人格ではない
3. Severityを使う
4. Blocker / Criticalは遠慮なく止める
5. Minor / Nitを理由に小実験を止めない
6. レビュー観点を依頼者が指定する
7. レビュアーは観点外の大規模指摘を別論点に分ける
8. 指摘はReview Issue LogかLearning Logへ接続する
9. 失敗はFailure Review Sheetへ接続する
10. 繰り返す問題は個人批判ではなく、テンプレート、評価ケース、教育の改善対象にする
```

特に重要なのは、次の区別である。

| 状態 | 意味 | 扱い |
|---|---|---|
| Draft | 粗い下書き | 構成、方向性、論点をレビューする |
| Review Only | 社外利用禁止のレビュー用 | リスク、事実、表現をレビューする |
| Experiment | 限定範囲の試行 | 成功条件、失敗条件、停止条件を確認する |
| Production Candidate | 本番候補 | 厳格レビュー、承認、ログ、統制を適用する |

DraftにProduction Candidate並みのレビューをすると、出す人が減る。

Production CandidateをDraftのように扱うと、事故が起きる。

状態を分けることで、レビュー文化と実験速度を両立する。

## Energy / Attention Budget

AI時代には、処理できる情報量が増える。

その結果、注意資源が枯渇しやすい。

次のような状態は、セルフトークを悪化させる。

```text
・AI出力を大量に読んでいる
・レビュー依頼が同時に多い
・Slack、Issue、PR、AIチャット、会議が並列に流れている
・評価ケース、ガバナンス、ログ、実験が増えている
・常に「もっとできるはず」と感じている
```

高レバレッジ環境では、やれることが増える。

やるべきことを増やすだけでは、燃え尽きる。

Energy / Attention Budgetを使う。

```text
対象期間：
今週の重要判断：
今週の重要実験：
AIに任せる作業：
自分が見るべきレビュー：
見ないレビュー：
委任するレビュー：
止める作業：
深く考える時間帯：
AI出力を読む時間上限：
通知を切る時間帯：
```

これは生産性テクニックではない。

判断力、レビュー品質、実験速度を保つための統制である。

## 個人のセルフトークを組織能力へ変える

Action Blocker診断は、個人用で終わらせない。

同じブロッカーが複数人に出るなら、組織側の設計問題である。

| 多く出るブロッカー | 組織側の改善 |
|---|---|
| 初稿を出せない | Draft共有ルール、レビュー観点テンプレートを作る |
| AI利用が怖い | X0/X1の低リスク実験メニューを作る |
| レビューが怖い | Severity、Review Issue Log、レビュー作法を標準化する |
| 失敗共有が怖い | Failure Reviewを責任追及から分離する |
| 管理職がレビューできない | 管理職向けAIレビュー観点を作る |
| ルールが曖昧で止まる | AI Use Policy、G0〜G5分類を整備する |
| 何を学べばよいか分からない | ロール別学習パスへ接続する |
| 実験が続かない | Pace / Spin ReviewとExperiment Portfolioを運用する |

個人の内的摩擦は、組織設計の入力になる。

第15章では、これらをスキルマップと評価へ接続する。

## Action Blocker診断の作成手順

### Step 1: 止まっている行動を書く

抽象的に書かない。

悪い例：

```text
AI活用が進まない。
```

よい例：

```text
AIで作ったFAQ改善案を、CSチームにReview Onlyとして共有できていない。
```

行動は、観察可能でなければならない。

### Step 2: 小さな次アクションを書く

次の行動が大きすぎると、また止まる。

悪い例：

```text
AI活用案を完成させる。
```

よい例：

```text
FAQ改善案のうち、上位3件だけをレビュー依頼する。
```

### Step 3: セルフトークを書く

きれいに書く必要はない。

実際に頭の中で出ている言葉を書く。

```text
こんな粗いものを出したら、準備不足だと思われる。
AIに作らせたと分かると、手抜きに見えるかもしれない。
```

### Step 4: 守ろうとしているものを書く

セルフトークの背後にある保護対象を書く。

```text
品質、信頼、顧客影響、チームの時間、自分の専門性。
```

### Step 5: 実リスクと想像上のリスクを分ける

```text
実リスク：
古いFAQを根拠にしている可能性がある。
顧客向け表現としては未確認。

想像上のリスク：
レビューで人格否定される。
AIを使ったこと自体が悪く見られる。
```

想像上のリスクが絶対に起きないとは限らない。

しかし、業務設計で扱うには、まず分ける必要がある。

### Step 6: 安全策を書く

```text
Review Onlyと明記する。
顧客送信禁止。
レビュー観点を「FAQ鮮度」「顧客表現」「禁止表現」に限定する。
古い可能性のある情報は要確認と書く。
```

### Step 7: 最小行動を契約にする

```text
本日16時までに、FAQ改善案3件をReview OnlyとしてCSリーダー1名に送る。
レビュー観点は3つに限定する。
レビュー結果はReview Issue Logへ記録する。
```

行動は、期限と完了条件を持つ。

## 典型的な失敗

### 失敗1: セルフトークを精神論として扱う

「前向きに考えよう」「自信を持とう」だけでは、行動は変わらない。

必要なのは、次のような業務設計である。

```text
Draftとして出す
レビュー観点を限定する
顧客送信を禁止する
期限を切る
Review Issue Logへ接続する
```

### 失敗2: 不安をすべて無視する

不安の中には、実際のリスクが含まれる。

「怖がらずにやる」ではなく、リスクを制御して試す。

```text
個人情報が入っているならマスキングする
顧客送信前にはHITLを入れる
未確定事項は断定しない
高リスク用途はAI Risk Registerへ入れる
```

### 失敗3: AIで自己批判を増幅する

AIに「この案の問題点を徹底的に洗い出して」と依頼すると、問題点は大量に出る。

それ自体は有用だが、目的を定義しないと、行動停止につながる。

悪い依頼：

```text
この案のダメなところを全部指摘して。
```

よい依頼：

```text
この案はReview Onlyの初稿である。
今は、顧客影響、事実誤認、禁止表現の3点だけをBlocker / Critical / Majorに分類して指摘して。
Minorな表現改善は後回しにする。
```

### 失敗4: レビューを優しくしすぎる

自己批判を避けるために、レビューを弱めてはいけない。

必要なのは、厳密さを保ったまま、対象を成果物とプロセスに限定することである。

```text
原因未確定の断定はCritical。
顧客固有情報の混入はBlocker。
誤字はNit。
```

Severityを分ければ、厳しさと学習を両立できる。

### 失敗5: 個人にだけ改善を求める

同じ行動停止が複数人に出るなら、組織設計の問題である。

```text
レビューが怖い人が多い
→ レビュー作法とSeverityが曖昧。

AI利用が怖い人が多い
→ 低リスク実験の入口がない。

管理職が判断できない
→ 判断基準と承認フローがない。

失敗共有が少ない
→ Failure Reviewが責任追及になっている。
```

個人支援と組織改善を分けない。

## 設計原則

1. **セルフトークは消す対象ではなく、観察して行動設計へ変換する対象である。** 気持ちを否定するのではなく、止まっている行動を特定する。
2. **精神論ではなく成果物で扱う。** Action Blocker診断、Self-talk Transcript、Minimum Action Contractとして書き出す。
3. **レビュー対象は人格ではなく成果物とプロセスに限定する。** Severity、Acceptance Criteria、Review Onlyを明示する。
4. **実リスクと想像上のリスクを分ける。** 顧客影響、法務、セキュリティは制御し、評価不安や完璧主義は最小行動へ変換する。
5. **AIを自己批判の増幅器にしない。** AIには、行動を止める指摘ではなく、レビュー可能な最小行動、リスク制御策、依頼文の作成を依頼する。
6. **管理職は個人の問題として閉じない。** 同じ行動停止が複数人に出るなら、レビュー作法、低リスク実験、権限、教育、心理的安全性の設計問題として扱う。
7. **専門領域は専門家へ接続する。** 医療、心理療法、労務、人事、ハラスメント、深刻なメンタルヘルス問題を本章のテンプレートで代替しない。

## 演習

自分、または架空ケースの担当者が止めているAI活用行動を1つ選ぶ。

次の成果物を作成する。

```text
1. Action Blocker診断
2. Self-talk Transcript
3. Review vs Self-criticismチェック
4. Minimum Action Contract
5. AIへの依頼文
6. レビュー依頼文
7. 組織側で改善すべき項目
```

題材は、次のいずれかを使う。

```text
・AIで作った提案書テンプレートを共有できない
・AI返信案生成実験の失敗を共有できない
・AI活用に不安があり、X0/X1実験を始められない
・管理職が部下のAI活用案をレビューできない
・AI出力へのレビュー指摘を人格評価として受け取ってしまう
```

## 章の受け入れ基準

- Action Blocker診断で、行動停止を業務上の最小行動へ変換できる
- 健全なレビュー、品質基準、リスク管理と、過剰な自己批判を区別できる
- 完璧主義をAcceptance Criteria、レビュー観点、HITL、停止条件へ変換できる
- AI不安を、AIが担う作業と人間が担う判断、対話、責任へ分解できる
- レビュー結果を人格評価ではなく、Review Issue Log、Learning Log、Improvement Backlogへ接続できる
- 管理職として、チームの行動ブロッカーを観察し、Draft共有、レビュー作法、低リスク実験、教育へ接続できる

詳細は [Chapter Acceptance Criteria](../planning/chapter-acceptance-criteria.md) を参照する。

## まとめ

AI時代の高レバレッジ環境では、自己批判、完璧主義、評価不安、AI不安が、個人の気分だけでなく、組織の実験速度と判断品質に影響する。

必要なのは、精神論ではない。

```text
止まっている行動を書く
セルフトークを書き出す
守ろうとしているものを確認する
実リスクと想像上のリスクを分ける
安全策を置く
最小行動を決める
レビュー可能なDraftとして出す
学習ログへ接続する
```

AI時代に必要なのは、自己批判をなくすことではない。

成果物を厳密にレビューしながら、人格評価から切り離し、行動を止めないことである。

第15章では、ここまでの能力を、スキルマップ、レベル定義、評価ルーブリック、認定課題へ整理する。

# 第5章 タスク分解と委任

## この章で扱う能力

この章では、複雑な業務をAIエージェントに委任可能な単位へ分解する。読者は、Request Contract と Context Pack を前提に、サブタスク、依存関係、チェックポイント、停止条件、エスカレーション条件、再開用メモを設計できるようになる。

この章の結論は次のとおりである。

```text
AIへの委任は、丸投げではない。
委任とは、AIに任せる処理、人間が決める判断、
途中で確認する地点、止める条件、引き継ぐ条件を設計することである。

複雑な仕事は、AIに渡す前に小さくする。
小さくしたタスクは、成果物、入力、依存関係、受け入れ条件を持つ。
人間は、AIの作業そのものではなく、作業の境界と確認点を設計する。
```

第3章では、AIへの依頼を Request Contract として定義した。第4章では、AIに渡す前提情報を Context Pack として設計した。この章では、それらを使って複雑な仕事を複数の Task Brief に分解する。仕事選定が Work 層、依頼設計が Request 層、前提情報設計が Context 層なら、この章は Delegation 層である。

## 本書全体における位置づけ

| 項目 | 内容 |
|---|---|
| 設計層 | Delegation |
| 目的 | 複雑な仕事を、AIに委任可能でレビュー可能な作業単位へ分解する。 |
| 主な成果物 | Task Brief、Task Breakdown、Checkpoint Plan、Stop / Escalation Rules、Restart Packet |
| 前章から受け取るもの | Request Contract、Context Pack |
| 次章へ渡すもの | レビュー対象となる中間成果物、AI出力レビュー表に必要な受け入れ条件 |
| 参照 | [Artifact Index](../artifact-index.md)、[付録A: 成果物テンプレート集](../appendices/a-templates.md)、[Troubleshooting Flow](../troubleshooting.md) |

本書の6層フレームワークでは、第5章は次の位置にある。

```text
Work: どの仕事にAIを使うか
  ↓
Request: AIに何を依頼するか
  ↓
Context: AIに何を渡すか
  ↓
Delegation: どう分解し、どこで止めるか  ← この章
  ↓
Control: どう検証し、どう制御するか
  ↓
Human / Org: 人間が何を判断し、どう学習するか
```

第5章の役割は、AIに「自由に進めてよい」と言うことではない。AIが安全に進められる作業単位を作り、どこで人間が確認し、どの条件で停止し、どの条件で別の担当者へ渡すかを明確にすることである。

## なぜ必要か

AIエージェントは、複数ステップの作業を実行できる。資料を読み、論点を整理し、下書きを作り、別案を比較し、ツールを呼び、次の手順を提案することがある。これにより、AIは単なる文章生成ツールではなく、業務遂行の一部を担う存在になる。

しかし、複数ステップを実行できることと、複雑な仕事を丸ごと任せてよいことは別である。実務の仕事には、次のような要素が混ざっている。

```text
情報を集める作業
情報を整理する作業
事実か推定かを分ける作業
判断材料を作る作業
人間が決める判断
社内承認が必要な判断
顧客影響を伴う判断
法務・契約・セキュリティに関わる判断
外部送信や本番操作を伴う実行
```

これらを分けずにAIへ渡すと、AIは「作業」と「判断」と「実行」を同じ流れで処理してしまう。結果として、次のような失敗が起きる。

```text
誤った前提で調査を進める。
根拠が薄いまま結論を出す。
人間が決めるべき判断まで提案ではなく断定する。
顧客に見せてはいけない表現を含む文面を作る。
法務確認が必要な内容を通常業務として処理する。
本来止まるべき条件で作業を続ける。
中断後に、何をどこまで確認したか分からなくなる。
```

この問題は、プロンプトを丁寧に書くだけでは解決しない。Request Contract で目的と受け入れ条件を定義し、Context Pack で前提情報を渡しても、作業が大きすぎれば失敗する。複雑な仕事には、分解と委任設計が必要である。

## 委任とは何か

本書でいう委任とは、AIに作業を渡すことである。ただし、委任は放任ではない。委任には次の4つを含める。

| 要素 | 内容 |
|---|---|
| 作業の分解 | 大きな業務を、AIが処理できる小さなタスクに分ける |
| 責任境界の定義 | AIが処理することと、人間が判断することを分ける |
| 確認点の設計 | 途中で人間がレビューする地点を置く |
| 停止・引き継ぎ条件 | AIが進めてはいけない条件、別担当へ渡す条件を明示する |

委任を正しく設計すると、AIは作業を進めやすくなる。人間もレビューしやすくなる。逆に、委任が曖昧だと、AIは広すぎる範囲を推測で埋め、人間は最後に巨大な成果物を見て修正することになる。

### 丸投げと委任の違い

| 観点 | 丸投げ | 委任 |
|---|---|---|
| 依頼 | 「提案書を作って」 | 「調査、論点整理、構成案、下書き、レビューに分ける」 |
| 成果物 | 期待が曖昧 | 各タスクごとに成果物がある |
| 判断 | AIが暗黙に補う | 人間が判断する箇所を明示する |
| 確認 | 最後に見る | 中間で確認する |
| 停止 | いつ止まるか不明 | 条件で止まる |
| 再開 | 会話履歴依存 | Restart Packet で再開できる |
| リスク | 誤前提・過剰実行・責任不明 | 境界と承認で制御できる |

委任は、AIを信用しないための手続きではない。AIの力を実務で使える形にするための設計である。

## Request Contract、Context Pack、Task Brief の関係

第3章と第4章で作った成果物は、この章で次のように使う。

| 成果物 | 役割 | 第5章での使い方 |
|---|---|---|
| AI適用判断シート | どの業務にAIを使うかを決める | 委任対象の業務を選ぶ |
| Request Contract | 目的、成果物、制約、受け入れ条件を定義する | 全体タスクの上位仕様として使う |
| Context Pack | 前提、資料、制約、情報鮮度を渡す | 各Task Briefの入力として使う |
| Task Brief | 個別タスクの仕様を定義する | AIに渡す最小作業単位として使う |
| AI出力レビュー表 | 出力を検証する | 各Task Briefの成果物をレビューする |

Request Contract は「何を達成したいか」を定義する。Context Pack は「何を前提にするか」を定義する。Task Brief は「そのために、今どの作業をAIに任せるか」を定義する。

たとえば、「法人向けバックアップサービスの競合調査」という仕事があるとする。

```text
Request Contract：
価格改定検討のため、主要競合5社の比較表と論点メモを作る。

Context Pack：
自社サービスの対象顧客、現行価格、顧客要望、過去調査、参照可能な公開情報、機密区分を整理する。

Task Brief 1：
対象競合候補を10社抽出し、対象範囲に合う5社を選ぶための候補表を作る。

Task Brief 2：
承認された5社について、価格体系と主要機能を整理する。

Task Brief 3：
競合比較表から、価格改定の検討論点を抽出する。ただし、価格改定の結論は出さない。
```

このように分けると、人間は「競合候補を選ぶ」「調査範囲を承認する」「価格判断は人間が行う」という確認点を持てる。

## Task Brief とは何か

Task Brief とは、AIに委任する個別作業の仕様書である。Request Contract より小さく、Context Pack より作業寄りである。

Task Brief は、次の問いに答える。

```text
このタスクで、AIは何をするのか。
何をしないのか。
何を入力として使うのか。
何を出力するのか。
先に終わっている必要がある作業は何か。
どこで人間が確認するのか。
どの条件で止めるのか。
どの条件で人間または別担当へ渡すのか。
途中で止まった場合、どう再開するのか。
```

Task Brief の基本形は次のとおりである。

```text
タスク名：
目的：
親Request Contract：
使用するContext Pack：
AIに任せる作業：
AIに任せない判断：
入力：
出力：
受け入れ条件：
依存関係：
チェックポイント：
停止条件：
エスカレーション条件：
再開メモ：
ログに残す情報：
```

この章では、この Task Brief を作るための考え方を順に扱う。

## タスク分解の基本単位

タスク分解では、1つのタスクを1つの成果物に対応させる。これが最も重要である。

### 原則1: 1タスク1成果物

タスクが複数の成果物を持つと、AIも人間も完了判定しにくい。

悪い例：

```text
競合を調べて、価格改定案を作って、営業資料も作って。
```

改善例：

```text
Task 1：競合候補一覧を作る。
Task 2：承認された競合5社の価格・機能比較表を作る。
Task 3：価格改定に関わる論点を抽出する。
Task 4：人間が決めた方針をもとに、営業向け説明資料の構成案を作る。
```

1タスク1成果物にすると、各タスクのレビューが可能になる。どの段階で誤りが入ったかも追跡しやすい。

### 原則2: 作業と判断を分ける

AIに向いているのは、情報整理、分類、下書き、比較、候補生成、レビュー補助である。一方、責任を伴う判断は人間が担う。

| AIに任せやすい作業 | 人間が担う判断 |
|---|---|
| 候補を列挙する | どの候補を採用するか |
| 論点を整理する | どの論点を重視するか |
| 表を作る | 表の結果をもとに意思決定するか |
| メール下書きを作る | 顧客へ送信するか |
| リスク候補を挙げる | どのリスクを許容するか |
| コード修正案を出す | 本番へ反映するか |

Task Brief では、AIに任せる作業と、AIに任せない判断を別項目として書く。

### 原則3: 入力と出力を固定する

タスクには入力と出力が必要である。

```text
入力が曖昧なタスクは、前提を勝手に補いやすい。
出力が曖昧なタスクは、レビューできない。
```

入力には、Context Pack、資料、データ、ログ、議事録、既存ドラフト、URL、チケット、過去判断メモなどがある。出力には、表、箇条書き、判断材料、下書き、比較表、レビュー指摘、Issue案、チェックリストなどがある。

### 原則4: 中間成果物を置く

複雑な作業では、一度で最終成果物を作らせない。次のような中間成果物を置く。

| 中間成果物 | 目的 |
|---|---|
| 論点一覧 | 何を考えるべきかを確認する |
| 対象リスト | 調査対象や処理対象を確定する |
| 情報不足リスト | 追加確認が必要な情報を出す |
| 構成案 | 下書き前に流れを確認する |
| 初稿 | 叩き台としてレビューする |
| 差分表 | 変更内容を確認する |
| リスク一覧 | 実行前に確認する |

中間成果物は、手戻りを減らす。AIが誤った方向へ進んでいても、早い段階で修正できる。

### 原則5: 終了条件を持たせる

タスクは、完了条件と停止条件を持つ。

| 条件 | 意味 | 例 |
|---|---|---|
| 完了条件 | 成果物が受け入れ条件を満たした状態 | 比較表に5社分の情報と要確認事項が揃った |
| 停止条件 | 作業を進めてはいけない状態 | 機密情報が必要になった、根拠が不足した、権限外ツールが必要になった |

完了条件だけでは不十分である。AIエージェントは、情報が足りないときに推測で補う場合がある。停止条件を先に定義しておくことで、推測による継続を防げる。

## タスク分解の7ステップ

複雑な仕事を委任する場合は、次の順で分解する。

```text
Step 1: 最終成果物を確認する
Step 2: 成果物を構成要素へ分ける
Step 3: 各構成要素に必要な入力を確認する
Step 4: 作業と判断を分ける
Step 5: 依存関係を並べる
Step 6: チェックポイント、停止条件、エスカレーション条件を置く
Step 7: Task Brief と Restart Packet に落とす
```

### Step 1: 最終成果物を確認する

最初に、Request Contract の最終成果物を確認する。

```text
最終成果物は何か。
誰が使うのか。
何の意思決定に使うのか。
どの形式で必要か。
どこまでAIが作ってよいのか。
どこから人間が判断するのか。
```

最終成果物が曖昧なまま分解してはいけない。分解は、成果物から逆算する。

### Step 2: 成果物を構成要素へ分ける

次に、成果物を部品に分ける。たとえば「障害報告書」なら、次の部品がある。

```text
事象概要
発生日時
影響範囲
時系列
原因
暫定対応
恒久対応
再発防止策
顧客向け説明
社内向け振り返り
未確認事項
```

これらをすべて一度にAIへ渡すのではなく、作業単位に分ける。

### Step 3: 各構成要素に必要な入力を確認する

各部品には必要な入力がある。

| 構成要素 | 必要な入力 |
|---|---|
| 時系列 | 監視通知、チケット、Slackログ、作業メモ |
| 影響範囲 | 顧客影響一覧、サービスステータス、問い合わせ件数 |
| 原因仮説 | ログ、メトリクス、変更履歴、担当者メモ |
| 再発防止策 | 原因、検知漏れ、運用手順、責任範囲 |
| 顧客向け説明 | 確定情報、表現ルール、法務・CS確認事項 |

入力が足りないタスクは、先に情報不足リストを作るタスクへ分ける。

### Step 4: 作業と判断を分ける

AIが作れるものと、人間が決めるものを分ける。

```text
AIができる：タイムライン候補を作る
人間が決める：正式な発生時刻を確定する

AIができる：原因仮説を列挙する
人間が決める：根本原因を確定する

AIができる：顧客向け文面を下書きする
人間が決める：送信してよいか判断する
```

判断をAIに任せないだけでは足りない。Task Brief の中で「AIに任せない判断」として明記する。

### Step 5: 依存関係を並べる

タスクには順序がある。依存関係を無視すると、AIは未確定の前提で次の作業へ進む。

たとえば障害報告書では、次の順序になる。

```text
1. 入力資料を確認する
2. タイムライン候補を作る
3. 人間がタイムラインを確定する
4. 影響範囲候補を整理する
5. CS責任者が顧客影響を確認する
6. 原因仮説を列挙する
7. 技術責任者が原因を確定する
8. 報告書下書きを作る
9. 人間が顧客送信前レビューを行う
```

依存関係は、表または簡単なフローで書く。

### Step 6: チェックポイント、停止条件、エスカレーション条件を置く

各タスクには、次の3種類の制御点を置く。

| 制御点 | 目的 | 例 |
|---|---|---|
| チェックポイント | 人間が途中で確認する | 調査対象確定前、初稿作成前、送信前 |
| 停止条件 | AIが作業を進めず止まる | 不明情報が多い、機密情報が必要、権限外操作が必要 |
| エスカレーション条件 | 別担当へ渡す | 法務、セキュリティ、顧客影響、本番変更が関わる |

チェックポイントは品質のために置く。停止条件は事故防止のために置く。エスカレーション条件は責任分界のために置く。

### Step 7: Task Brief と Restart Packet に落とす

最後に、個別タスクを Task Brief として書く。長い作業や中断の可能性がある作業では、Restart Packet も作る。

Restart Packet は、別のAIセッション、別担当者、翌日の自分が作業を再開するための引き継ぎメモである。

```text
現在の目的：
完了したタスク：
未完了タスク：
確認済み情報：
未確認情報：
人間が判断したこと：
次にAIへ依頼すること：
使ってよいContext Pack：
使ってはいけない情報：
残リスク：
```

Restart Packet がないと、長い作業は会話履歴に依存する。会話履歴だけに依存すると、前提の変化、確認済み事項、未確認事項が見えにくくなる。

## タスク粒度を決める

タスク分解で難しいのは、粒度である。細かすぎると管理コストが高くなる。大きすぎるとAIが誤る。

### 粒度が大きすぎるサイン

次の条件に当てはまる場合、タスクは大きすぎる。

```text
成果物が複数ある。
AIが複数の判断を暗黙に行う必要がある。
入力資料が多すぎる。
途中で人間確認が必要なのに確認点がない。
出力をレビューするのに時間がかかりすぎる。
失敗したときに、どこで誤ったか分からない。
```

### 粒度が細かすぎるサイン

次の条件に当てはまる場合、タスクは細かすぎる。

```text
各タスクの成果物が独立していない。
人間の確認ばかりで作業が進まない。
AIに任せるより指示管理の方が重い。
同じContext Packを毎回長く渡している。
タスク間の引き継ぎが過剰に増えている。
```

### 適切な粒度の目安

実務では、次を目安にする。

| 観点 | 目安 |
|---|---|
| 成果物 | 1タスク1成果物 |
| レビュー | 5〜15分でレビューできる大きさ |
| 入力 | 必要なContext Packが明確 |
| 判断 | 重要判断が混ざっていない |
| 失敗時 | やり直し範囲が限定できる |
| 再利用 | 次回も似た形で使える |

「5〜15分でレビューできる」は厳密なルールではない。目安である。重要なのは、AIが作ったものを人間が確認できる大きさに保つことである。

## 委任レベル

AIへの委任には段階がある。すべての業務を同じレベルで任せるべきではない。

| レベル | 名称 | AIの役割 | 人間の役割 | 適用例 |
|---:|---|---|---|---|
| 0 | 不使用 | AIを使わない | 人間が実施 | 高リスク判断、規制判断 |
| 1 | 補助 | メモ整理、要約、候補出し | 全判断と成果物作成 | 個人メモ、論点整理 |
| 2 | 下書き | 初稿や表を作る | レビュー、修正、採否判断 | 議事録、メール下書き |
| 3 | 分担 | サブタスクを担当する | 中間確認、承認、統合 | 調査、比較、レビュー補助 |
| 4 | 半自律 | 条件内で複数ステップを進める | チェックポイントで確認 | チケット整理、FAQ候補作成 |
| 5 | 自律実行 | 許可された範囲で実行する | 例外時と定期監査 | 低リスク定型処理 |

多くの組織は、いきなりレベル4や5を目指す必要はない。実務上は、レベル2から3で十分な効果が出ることが多い。高リスク業務は、レベル1または2に抑える。

委任レベルを決めるときは、次を確認する。

```text
情報の機密度
顧客影響
金銭影響
法務・契約影響
本番環境影響
出力の可逆性
人間レビューの容易さ
ログの取得可能性
失敗時の復旧容易性
```

リスクが高いほど、委任レベルを下げる。レビューしやすく、可逆性が高く、ログが残り、失敗時に戻せる業務は、委任レベルを上げやすい。

## チェックポイント設計

チェックポイントとは、AIが次の段階へ進む前に、人間が確認する地点である。

チェックポイントは、AIの自由を制限するだけのものではない。むしろ、AIが安全に大きな仕事を進めるための足場である。

### チェックポイントの種類

| 種類 | 目的 | 例 |
|---|---|---|
| 方針確認 | 作業の方向性を確認する | 調査観点、比較軸、構成案 |
| 対象確認 | 処理対象を確定する | 競合リスト、顧客セグメント、対象チケット |
| 情報確認 | 不足情報や未確認情報を確認する | 要確認リスト、根拠不足リスト |
| 初稿確認 | 成果物の初稿を確認する | メール、報告書、提案書 |
| リスク確認 | 実行前にリスクを見る | 顧客影響、法務、セキュリティ |
| 実行前承認 | 外部送信や変更前に承認する | メール送信、CRM更新、PR作成、本番操作 |
| 完了確認 | 最終成果物を受け入れる | レビュー完了、承認、記録 |

### どこにチェックポイントを置くか

チェックポイントは、次の場所に置く。

```text
前提が確定していない場所
対象範囲を決める場所
判断が必要な場所
外部影響が発生する前
コストや時間が大きく増える前
別部門に影響する前
顧客、契約、本番環境に触れる前
```

たとえば、競合調査では「対象競合の確定前」にチェックポイントを置く。対象を誤ると、その後の調査がすべて無駄になるからである。

障害報告では「原因断定前」と「顧客送信前」にチェックポイントを置く。原因の断定と顧客向け説明は責任を伴うからである。

### チェックポイントの書き方

チェックポイントは、曖昧に書かない。

悪い例：

```text
適宜確認する。
必要なら人間に聞く。
重要そうなら止める。
```

改善例：

```text
Checkpoint 1：調査対象候補を10件出した時点で停止し、人間が対象5件を選ぶ。
Checkpoint 2：比較表の初稿を作成した時点で停止し、出典不足と要確認事項を人間が確認する。
Checkpoint 3：価格改定に関わる提案文を作る前に停止する。AIは価格改定の結論を出さない。
```

「どの成果物が出た時点で」「誰が」「何を確認し」「次に進めてよい条件は何か」を書く。

## 停止条件設計

停止条件とは、AIが作業を進めずに止まる条件である。停止条件は、委任設計の中でも特に重要である。

AIは、不足情報があっても、推測で出力できてしまう。実務では、その推測が事故になる。停止条件は、AIに「ここから先は進むな」と明示するために書く。

### 停止条件の種類

| 種類 | 停止条件の例 |
|---|---|
| 情報不足 | 必要情報がない。出典が見つからない。未確認情報しかない。 |
| 矛盾 | Context Pack内の情報が矛盾している。最新版が不明。 |
| 機密 | 個人情報、認証情報、顧客固有情報が必要になる。 |
| 権限 | 許可されていないツール、API、DB、ファイルが必要になる。 |
| 判断 | 法務、契約、価格、人事、顧客影響の判断が必要になる。 |
| 外部影響 | 顧客送信、公開、CRM更新、本番変更が発生する。 |
| 品質 | 根拠不足、確信度不足、検証不能、レビュー不能。 |
| コスト | 指定した時間、検索件数、API利用、試行回数を超える。 |
| 安全 | プロンプトインジェクション、怪しい指示、想定外の命令を検知する。 |

停止条件は、組織のリスクルールに合わせる。第7章以降で扱うツール権限やHITL設計とも接続する。

### 停止条件の書き方

停止条件は、行動可能な形で書く。

悪い例：

```text
危険な場合は止める。
```

改善例：

```text
次のいずれかに該当する場合は作業を停止し、要確認事項として出力する。
- 価格、契約、SLA、返金、補償に関わる判断が必要になった場合
- 顧客名、個人情報、認証情報、未公開財務情報が必要になった場合
- 公開情報と社内Context Packの内容が矛盾した場合
- 出典が確認できない情報を根拠に結論を出す必要がある場合
- 顧客向け送信文を確定する必要がある場合
```

停止後の出力形式も決めておく。

```text
停止理由：
停止した箇所：
必要な追加情報：
人間に確認してほしい判断：
ここまでに作成した中間成果物：
再開時に使うべき入力：
```

停止は失敗ではない。停止条件に従って止まることは、設計どおりの正常動作である。

## エスカレーション条件設計

エスカレーションとは、AIが処理を続けず、人間または別担当へ渡すことである。停止が「進めない」ことだとすれば、エスカレーションは「誰に渡すか」を決めることである。

### エスカレーション先

| 条件 | エスカレーション先 |
|---|---|
| 法務・契約判断 | 法務担当、契約責任者 |
| 顧客影響 | CS責任者、アカウント担当、事業責任者 |
| セキュリティ | セキュリティ担当、情シス、CSIRT |
| 本番環境 | SRE、運用責任者、変更承認者 |
| 価格・割引 | 事業責任者、営業責任者、経営層 |
| 人事・評価 | 人事責任者、直属管理職 |
| 広報・外部公開 | 広報、経営層、法務 |
| 不明確な責任 | 業務オーナー、AI推進担当 |

### エスカレーション条件の例

顧客問い合わせ対応の場合は、次のように書く。

```text
次の条件では、AIは回答案の作成を停止し、CS責任者へエスカレーションする。
- 返金、補償、契約解除に関する問い合わせ
- セキュリティ事故、情報漏洩の可能性がある問い合わせ
- 障害影響を受けた顧客からの正式な苦情
- 既存FAQと矛盾する回答が必要な問い合わせ
- 個別契約条件を参照しないと回答できない問い合わせ
```

エスカレーションでは、単に「人間へ」と書かない。誰に、何を渡すかを決める。

```text
エスカレーション先：CS責任者
渡す情報：問い合わせ本文、分類結果、参照したFAQ、回答できない理由、必要な確認事項
AIが作ってよいもの：回答案ではなく、論点整理と確認事項一覧
AIが作ってはいけないもの：顧客送信用の確定回答
```

## 依存関係設計

タスクは、独立しているとは限らない。先に終わっていなければならない作業、先に人間が判断しなければならない事項がある。

依存関係を整理しないと、AIは未確定の前提で次のタスクへ進む。

### 依存関係の種類

| 種類 | 内容 | 例 |
|---|---|---|
| 情報依存 | 先に情報が必要 | 対象競合が決まらないと比較できない |
| 判断依存 | 人間判断が必要 | 方針が決まらないと提案書を書けない |
| 承認依存 | 承認が必要 | 顧客送信前に責任者承認が必要 |
| 技術依存 | 技術作業が必要 | ログ抽出が終わらないと原因分析できない |
| 法務依存 | 法務確認が必要 | 契約条件が不明だと回答できない |
| ツール依存 | 権限やツールが必要 | CRM参照権限がないと顧客状況を確認できない |

### 依存関係表

Task Brief とは別に、複数タスクを扱う場合は依存関係表を作る。

| ID | タスク | 成果物 | 依存 | チェックポイント | 担当 |
|---|---|---|---|---|---|
| T1 | 対象情報の確認 | 入力資料一覧 | なし | 資料不足確認 | AI + 人間 |
| T2 | タイムライン候補作成 | 時系列表 | T1 | 技術担当確認 | AI |
| T3 | 影響範囲候補整理 | 影響候補表 | T1 | CS確認 | AI |
| T4 | 原因仮説列挙 | 原因仮説一覧 | T2 | 技術責任者確認 | AI |
| T5 | 報告書構成案 | 構成案 | T2, T3, T4 | 送信前方針確認 | AI |
| T6 | 顧客向け下書き | 下書き | T5 + 人間承認 | 送信前承認 | AI |

この表により、どのタスクが止まると後続に影響するかが分かる。

## 長い作業の委任

長い作業では、1回の会話や1回の実行で完了しない。途中で中断し、別の担当者が引き継ぎ、後日再開することがある。

長い作業では、次の問題が起きる。

```text
どこまで完了したか分からない。
どの情報が確認済みか分からない。
どの判断が人間によって承認済みか分からない。
AIが以前の仮定を確定情報のように扱う。
会話履歴が長くなり、重要な前提が埋もれる。
別のAIセッションで再開できない。
```

これを防ぐために、長い作業では Restart Packet を作る。

### Restart Packet の役割

Restart Packet は、作業の再開に必要な情報だけをまとめた引き継ぎ文書である。Context Pack が業務前提をまとめるのに対して、Restart Packet は進行状況をまとめる。

| 文書 | 目的 |
|---|---|
| Context Pack | 作業の前提情報を渡す |
| Task Brief | 今回の作業単位を定義する |
| Restart Packet | 中断後に再開するための状態を渡す |
| Agent Run Log | 実行履歴と改善点を記録する |

Restart Packet には、次を入れる。

```text
作業名：
親Request Contract：
使用したContext Pack：
完了したタスク：
未完了タスク：
確認済み情報：
未確認情報：
人間が判断したこと：
AIが仮定したこと：
破棄した案：
残リスク：
次に実行すべきTask Brief：
停止中の理由：
再開条件：
```

AIに「続きからやって」と言うのではなく、Restart Packet を渡して再開する。

### 再開時の依頼例

```text
以下のRestart Packetに基づき、未完了タスク T4 のみを実行してください。
T1〜T3の結果は確認済み情報として扱ってよいです。
ただし、T3の影響範囲には未確認事項が残っているため、顧客向け文面では断定しないでください。
出力は、原因仮説一覧と追加確認事項に限定してください。
原因の確定、顧客向け説明、再発防止策の優先順位付けは行わないでください。
```

再開時には、完了済み情報、未確認情報、禁止判断を明示する。

## 実務ケース1: 障害報告書の作成

ここでは、障害報告書の作成を例に、タスク分解を行う。

### 悪い委任

```text
昨日の障害について、顧客向け報告書を作って。
```

この依頼は危険である。理由は次のとおりである。

```text
障害の正式な発生時刻が確定しているか分からない。
顧客影響が確認済みか分からない。
原因が確定しているか分からない。
顧客向けに言えることと言えないことが分かれていない。
法務、CS、事業責任者の確認が必要か不明である。
AIが顧客向け表現を断定する可能性がある。
```

### 改善した分解

| ID | サブタスク | AIの役割 | 成果物 | 人間確認 | 停止条件 |
|---|---|---|---|---|---|
| T1 | 入力資料確認 | 監視通知、Slack、チケット、作業メモを整理する | 入力資料一覧 | 不足資料確認 | 認証情報や個人情報が必要 |
| T2 | タイムライン候補作成 | 時系列候補を作る | タイムライン表 | 技術担当が発生・復旧時刻を確認 | ログが矛盾する |
| T3 | 影響範囲候補整理 | 顧客影響候補を抽出する | 影響候補表 | CS責任者が確認 | 顧客固有契約が必要 |
| T4 | 原因仮説列挙 | 原因候補と追加確認事項を出す | 原因仮説一覧 | 技術責任者が確認 | 原因断定が必要 |
| T5 | 報告書構成案作成 | 顧客向け報告書の構成案を作る | 構成案 | CS・技術責任者が確認 | 未確認情報が多い |
| T6 | 報告書下書き | 承認済み情報だけで下書きする | 顧客向け下書き | 送信前承認 | 補償、返金、契約判断が必要 |

### Task Brief 記入例: T2 タイムライン候補作成

```text
タスク名：障害タイムライン候補作成
目的：顧客向け報告書の前提となる時系列を整理する。
親Request Contract：障害報告書作成 Request Contract
使用するContext Pack：障害対応 Context Pack v2026-06-27

AIに任せる作業：
- 監視通知、チケット、作業メモから時刻付きイベントを抽出する
- イベントを時系列に並べる
- 発生、検知、対応開始、暫定復旧、完全復旧の候補を分ける
- 矛盾する時刻、未確認の時刻を要確認として分離する

AIに任せない判断：
- 正式な障害発生時刻の確定
- 正式な復旧時刻の確定
- 顧客向け説明への採用判断

入力：
- 監視通知一覧
- インシデントチケット
- 作業メモ
- Slack要約

出力：
- Markdown表
- 列は「時刻」「イベント」「情報源」「確度」「要確認事項」

受け入れ条件：
- 情報源が明記されている
- 確定、推定、要確認が分かれている
- 正式時刻を断定していない

チェックポイント：
- タイムライン候補を作成した時点で停止し、技術担当が確認する

停止条件：
- 入力資料間で時刻が矛盾する場合
- 認証情報、顧客個別情報、未許可ログが必要になる場合

エスカレーション条件：
- 監視ログと作業メモが矛盾し、正式な発生時刻判断が必要な場合は技術責任者へ渡す

再開メモ：
- 技術担当が正式時刻を確認した後、T3 影響範囲候補整理へ進む
```

このTask Briefであれば、AIはタイムライン候補作成に集中できる。正式な発生時刻や顧客向け採用判断は人間に残る。

## 実務ケース2: 競合調査から提案書へ

次に、競合調査から社内提案書を作るケースを考える。

### 悪い委任

```text
競合調査して、当社の価格改定提案を作って。
```

この依頼では、調査、比較、判断、提案、承認が混ざっている。価格改定は事業判断であり、AIが結論を出すべきではない。

### 改善したタスク分解

| ID | タスク | AIの役割 | 人間の役割 |
|---|---|---|---|
| T1 | 調査対象候補の抽出 | 候補10社を出し、対象条件に合うか整理する | 調査対象5社を選ぶ |
| T2 | 比較軸の確認 | 価格、機能、対象顧客、サポート、導入事例などの比較軸案を出す | 比較軸を確定する |
| T3 | 競合比較表の作成 | 承認済み5社について情報を整理する | 出典と要確認事項を確認する |
| T4 | 論点整理 | 比較表から価格改定に関わる論点を抽出する | 重要論点を選ぶ |
| T5 | 提案書構成案 | 人間が選んだ論点をもとに構成案を作る | 方針を確認する |
| T6 | 提案書下書き | 構成案に沿って下書きを作る | 経営会議提出可否を判断する |

### チェックポイント

```text
Checkpoint 1：T1完了後、人間が調査対象5社を確定する。
Checkpoint 2：T2完了後、人間が比較軸を確定する。
Checkpoint 3：T3完了後、要確認事項を確認する。
Checkpoint 4：T4完了後、人間が価格改定に関する論点を選ぶ。
Checkpoint 5：T6完了後、経営会議提出前レビューを行う。
```

### 停止条件

```text
- 価格情報が公開情報で確認できない場合、推測せず要確認とする。
- 競合の非公開情報が必要な場合、作業を停止する。
- AIは価格改定の最終結論を出さない。
- 顧客向け説明や外部公開資料への転用は、人間承認前に行わない。
```

この分解により、AIは調査・整理・論点抽出・下書きに集中する。価格判断は人間が担う。

## 実務ケース3: 顧客問い合わせ対応

顧客問い合わせ対応では、AIの活用効果が出やすい。一方で、誤回答、過剰約束、契約違反、情報漏洩のリスクもある。

### 悪い委任

```text
顧客からの問い合わせに自動で回答して。
```

この依頼は危険である。問い合わせの種類によって、AIが回答案を作ってよいものと、人間に渡すべきものがある。

### 改善した委任フロー

```text
1. 問い合わせを分類する
2. 低リスク問い合わせのみFAQ候補を検索する
3. 回答案を作る
4. リスク判定を行う
5. 人間が承認する
6. 送信後にナレッジ改善候補を記録する
```

### タスク分解

| ID | タスク | AIの役割 | 停止・エスカレーション |
|---|---|---|---|
| T1 | 問い合わせ分類 | 問い合わせをカテゴリ分類する | セキュリティ、契約、苦情は停止 |
| T2 | FAQ検索 | 許可されたFAQから関連項目を探す | FAQが矛盾する場合は停止 |
| T3 | 回答案下書き | FAQに基づき回答案を作る | 個別契約条件が必要なら停止 |
| T4 | リスク判定 | 断定表現、過剰約束、顧客影響を確認する | 中・高リスクはCS責任者へ |
| T5 | 送信前確認 | 人間が送信可否を判断する | AI単独送信不可 |
| T6 | ナレッジ改善候補 | FAQ不足や改善点を記録する | FAQ更新は承認制 |

### Task Brief 記入例: T1 問い合わせ分類

```text
タスク名：問い合わせ分類
目的：顧客問い合わせを、回答案作成可能な低リスク問い合わせと、エスカレーションが必要な問い合わせに分ける。

AIに任せる作業：
- 問い合わせ本文を分類する
- 該当しそうなカテゴリを1つまたは複数付ける
- 回答案作成に進めるか、エスカレーションすべきかを判定候補として出す

AIに任せない判断：
- 顧客への最終回答
- 契約条件の解釈
- 返金、補償、障害責任の判断

出力：
- カテゴリ
- リスク分類：低 / 中 / 高
- 推奨次アクション：FAQ検索 / 人間確認 / エスカレーション
- 理由
- 要確認事項

停止条件：
- セキュリティ事故、情報漏洩、返金、契約、障害苦情、法務判断に関わる場合

エスカレーション条件：
- 中リスク以上はCS責任者へ渡す
```

この設計では、AIがいきなり回答するのではなく、分類とリスク判定から始める。

## 実務ケース4: 開発タスクの分解

開発業務でも、AIへの委任には分解が必要である。

### 悪い委任

```text
このIssueを実装して。
```

この依頼では、要件理解、設計、コード変更、テスト、レビュー、PR作成が混ざっている。AIがすべてを一気に進めると、要件誤解や過剰実装が起きやすい。

### 改善した分解

| ID | タスク | AIの役割 | 人間確認 |
|---|---|---|---|
| T1 | Issue理解 | 要件、制約、不明点を整理する | PMまたはTech Leadが確認 |
| T2 | 影響範囲調査 | 関連ファイル、テスト、既存仕様を調べる | 設計方針確認 |
| T3 | 実装方針案 | 複数案とトレードオフを出す | 方針決定は人間 |
| T4 | 最小差分実装 | 承認された方針に沿って変更案を作る | コードレビュー |
| T5 | テスト案作成 | 追加・更新すべきテストを提案する | テスト妥当性確認 |
| T6 | PR説明文作成 | 変更理由、確認方法、リスクを書く | PR提出前確認 |

### 停止条件

```text
- 要件が矛盾している場合
- 仕様変更の判断が必要な場合
- 既存API互換性に影響する場合
- セキュリティ、認証、課金、権限に関わる場合
- テストなしで本番影響がある変更が必要な場合
```

開発業務では、AIにコードを作らせる前に、Issue理解と影響範囲調査を分ける。AIがコードを書く速度は速いが、誤った方針で速く進むと手戻りが大きい。

## 委任で使う出力形式

Task Briefでは、出力形式を明確にする。特に、中間成果物では表形式が有効である。

### タスク分解表

```text
| ID | タスク | 目的 | 入力 | 出力 | AIの役割 | 人間確認 | 停止条件 |
|---|---|---|---|---|---|---|---|
```

### チェックポイント表

```text
| ID | チェックポイント | 何を確認するか | 確認者 | 次へ進む条件 | 差し戻し条件 |
|---|---|---|---|---|---|
```

### 停止条件表

```text
| 条件 | 停止理由 | AIが出力するもの | 人間が判断すること | 再開条件 |
|---|---|---|---|---|
```

### エスカレーション表

```text
| 条件 | エスカレーション先 | 渡す情報 | AIが作ってよいもの | AIが作ってはいけないもの |
|---|---|---|---|---|
```

これらの表は、後続章のレビュー、権限設計、ログ設計にも使える。

## 典型的な失敗

### 失敗1: 大きすぎるタスクを渡す

```text
営業戦略を作って。
障害対応して。
このIssueを解決して。
契約書を見て問題を教えて。
```

これらは、複数の作業と判断が混ざっている。AIには、調査、整理、論点抽出、下書き、レビュー補助などの単位で渡す。

### 失敗2: 作業と判断を分けない

AIが作れる「判断材料」と、人間が行う「判断」を分けないと、AIの出力が意思決定として扱われる。

```text
AIに任せる：契約条項の論点候補を整理する
人間が判断する：契約上のリスク許容可否を決める
```

この区別をTask Briefに書く。

### 失敗3: 中間確認がない

AIが最後まで進んでからレビューすると、誤った前提に基づく成果物全体を修正することになる。対象範囲、比較軸、構成案、初稿の段階で確認する。

### 失敗4: 停止条件がない

停止条件がないと、AIは不足情報を推測で補いやすい。特に、出典不足、情報矛盾、権限外ツール、顧客影響、法務判断が関わる場合は停止条件を明記する。

### 失敗5: エスカレーション先が曖昧

「人間に確認する」とだけ書いても、誰が見るのか分からない。業務オーナー、CS責任者、法務、セキュリティ、SRE、事業責任者など、条件ごとに渡す先を決める。

### 失敗6: 再開できない

長い作業で Restart Packet がないと、会話履歴に依存する。別セッション、別担当、翌日再開に弱くなる。完了したこと、未確認事項、残リスクを残す。

### 失敗7: AIに管理まで任せる

AIに「自分で判断して適宜進めて」と依頼すると、作業管理と責任判断が曖昧になる。進行管理は人間が設計し、AIには定義された範囲で作業させる。

## 設計原則

### 原則1: 1タスク1成果物にする

1つのTask Briefには、1つの主要成果物を持たせる。複数成果物があるなら分ける。

### 原則2: 判断と処理を分ける

AIに処理させることと、人間が判断することを別項目として書く。

### 原則3: 入力と出力を明示する

AIが何を使い、何を作るかを固定する。入力外の情報を使う場合は、明示的に許可する。

### 原則4: 中間成果物をレビュー可能にする

最終成果物だけでなく、途中の成果物を人間が見られる形にする。

### 原則5: 停止条件を先に書く

危険になったら止まるのではなく、何が危険かを事前に定義する。

### 原則6: エスカレーション先を具体化する

「人間に確認」ではなく、「誰に、何を、どの形式で渡すか」を書く。

### 原則7: 再開可能にする

長い作業は、Restart Packet を作る。会話履歴だけに依存しない。

### 原則8: 低リスクから委任レベルを上げる

最初から自律実行を目指さない。低リスクな下書き、整理、分類、レビュー補助から始める。

## 作成する成果物: Task Brief

この章で作る中心成果物は Task Brief である。

```text
タスク名：
目的：
親Request Contract：
使用するContext Pack：
AIに任せる作業：
AIに任せない判断：
入力：
出力：
受け入れ条件：
依存関係：
チェックポイント：
停止条件：
エスカレーション条件：
再開メモ：
ログに残す情報：
```

### Task Brief の最小版

軽い作業では、次の項目だけでもよい。

```text
タスク名：
目的：
AIに任せる作業：
AIに任せない判断：
入力：
出力：
停止条件：
```

ただし、顧客影響、法務、セキュリティ、本番環境、金銭影響がある場合は、最小版では不足する。フル版を使う。

## 作成する成果物: Task Breakdown

複数タスクに分解する場合は、Task Breakdown を作る。

```text
業務名：
親Request Contract：
使用するContext Pack：

| ID | タスク | 目的 | 入力 | 出力 | AIの役割 | 人間確認 | 依存 | 停止条件 |
|---|---|---|---|---|---|---|---|---|
```

Task Breakdown は、AIに一括で渡すための文書ではない。人間が全体構造を確認し、個別の Task Brief を作るための設計図である。

## 作成する成果物: Checkpoint Plan

チェックポイントを明示する場合は、Checkpoint Plan を作る。

```text
業務名：
対象タスク：

| ID | チェックポイント | 何を確認するか | 確認者 | 次へ進む条件 | 差し戻し条件 |
|---|---|---|---|---|---|
```

Checkpoint Plan は、管理職やレビュー担当者にも有用である。AIがどこで止まるか、誰が見ればよいかが分かる。

## 作成する成果物: Stop / Escalation Rules

停止条件とエスカレーション条件は、個別タスクに書いてもよいが、リスクの高い業務では独立した表にする。

```text
業務名：
対象範囲：

| 条件 | AIの動作 | エスカレーション先 | 渡す情報 | 再開条件 |
|---|---|---|---|---|
```

たとえば、問い合わせ対応、契約確認、障害対応、採用、人事、セキュリティ、インフラ変更では、独立した Stop / Escalation Rules を作る。

## 作成する成果物: Restart Packet

長い作業や中断可能性がある作業では、Restart Packet を残す。

```text
作業名：
親Request Contract：
使用したContext Pack：
現在の状態：未着手 / 進行中 / 停止中 / レビュー待ち / 完了
完了したタスク：
未完了タスク：
確認済み情報：
未確認情報：
人間が判断したこと：
AIが仮定したこと：
破棄した案：
残リスク：
停止中の理由：
次に実行すべきTask Brief：
再開条件：
ログ参照先：
```

Restart Packet は、後続の Agent Run Log や改善バックログにも接続する。

## 演習

次の業務を、AIに委任可能な5つ以上のサブタスクへ分解する。

```text
新サービスの社内提案書を作る。
```

条件は次のとおりである。

```text
目的：新しい法人向けバックアップオプションの事業化可否を検討する。
利用者：事業責任者、営業責任者、開発責任者。
AIに任せてよいこと：調査、論点整理、比較表、構成案、下書き、レビュー補助。
AIに任せないこと：事業化判断、価格決定、開発優先順位決定、顧客向け発表。
```

作成するものは次の4つである。

```text
1. Task Breakdown
2. 主要タスク1つのTask Brief
3. Checkpoint Plan
4. Stop / Escalation Rules
```

### 演習の判定基準

次の条件を満たしていれば、この章の演習としては合格である。

- 5つ以上のサブタスクに分解されている
- 各サブタスクが1成果物に対応している
- AIに任せる作業と、人間が判断することが分かれている
- 少なくとも2つ以上のチェックポイントがある
- 価格、顧客影響、法務、セキュリティ、開発優先順位に関する停止条件またはエスカレーション条件がある
- 主要タスク1つについて、入力、出力、受け入れ条件が明記されている
- 中断後に再開できるメモ項目がある

## 章の受け入れ基準

この章を読み終えた時点で、読者は次をできる必要がある。

- 複雑な業務を、AIに委任可能なサブタスクへ分解できる
- Request Contract、Context Pack、Task Brief の役割を分けて説明できる
- 1タスク1成果物の原則に沿ってタスク粒度を調整できる
- AIに任せる作業と、人間が担う判断を分けられる
- 依存関係を表で整理できる
- チェックポイントを設計できる
- 停止条件を具体的に書ける
- エスカレーション先と渡す情報を定義できる
- 長い作業のために Restart Packet を作れる
- 第6章のAI出力レビューにつながる中間成果物を残せる

詳細は [Chapter Acceptance Criteria](../planning/chapter-acceptance-criteria.md) を参照する。

## まとめ

AIへの委任は、丸投げではなく分解である。Request Contract で目的と受け入れ条件を定義し、Context Pack で前提情報を整えても、仕事が大きすぎれば失敗する。

複雑な仕事は、1タスク1成果物に分ける。AIに任せる作業と、人間が担う判断を分ける。依存関係を並べ、チェックポイントを置き、停止条件とエスカレーション条件を明示する。長い作業では、Restart Packet を残す。

次章では、AIが作った成果物をどうレビューし、品質をどう評価するかを扱う。委任によって中間成果物が残れば、レビューは容易になる。レビュー可能な形で委任することが、AIエージェント協働の品質管理の入口である。

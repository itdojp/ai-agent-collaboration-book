# Style Decisions

## 本書の文体

- 結論を先に書く。
- 抽象論の後には、必ず成果物またはチェックリストを置く。
- AIを擬人化しない。
- 「便利」「すごい」「革新的」といった評価語は原則使わない。
- 読者に作業を要求する場合は、入力、手順、出力を明記する。

## 禁止または抑制する表現

| 避ける表現 | 置き換え |
|---|---|
| AIに考えさせる | AIに候補を生成させる / 論点を整理させる |
| AIが理解する | AIが文脈から推定する / 入力に基づいて生成する |
| プロンプトを投げる | Request Contractを渡す |
| AIに丸投げする | タスクを分解して委任する |
| AIが正解を出す | AIが出力候補を生成する |
| 失敗を恐れない | 失敗しても安全な実験範囲を設計する |

## ケース方針

- 実在企業の非公開情報は使わない。
- ケースは原則として `examples/recurring-case.md` の架空会社を使う。
- 法務、規制、価格、仕様などの断定が必要な場合は「要確認」とする。
